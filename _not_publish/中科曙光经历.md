# 中科曙光实习经历（约三个月）

## 一、项目背景

曙光自研的国产化分布式闪存存储系统。我所在的组负责 Java 管理侧（MS），接受 CLI/GUI 指令，向底层 C 或 Python 进程发出管理命令。Java 侧承担任务调度、参数验证、缓存管理、集群管理、性能计算等工作。

项目特点：
- **国产化全自研**：软硬件自研，出于安全可控的需求，框架层面也倾向于自研而非直接使用开源方案（Resource 框架、NAL 通信框架、EventBus 等）。
- **多语言协作**：Java、C、Python 多端交互，开发过程中需要跨模块对接接口、联调和问题定位。部分场景涉及 JNI 调用和 GDB 调试。
- **分布式架构**：集群部署，涉及 scale-out/scale-up、节点间通信、请求路由转发等。
- 实习时项目已进入后期阶段，主要工作集中在性能指标、监控告警、灾备等方向。

## 二、核心自研框架

这些框架贯穿后续各个业务模块，先整体梳理。

### 2.1 Resource 框架（分布式内存缓存）

JVM 内存级别的资源管理框架，本质是分布式一致性 KV 缓存。

- 使用 Map 存储 BO（业务对象），通过注解标记字段，自动建立等值索引和模糊索引。
- 不仅存储本机数据，还存储集群其他节点的数据。通过**魔改的 Raft 协议**实现节点间元数据的一致性同步。
- 自己实现了事务操作，保证 KVDB（底层自研数据库，基于 LSM Tree）与 JVM 内存的一致性。注意：事务一致性是本机维度的，不保证跨节点的强一致。
- 提供加锁/不加锁、互斥/非互斥的多种访问方式，向上层屏蔽实现细节。

**与 Redis/Caffeine 的区别**：Redis 和 Caffeine 是通用缓存，不感知业务元数据结构；Resource 框架是面向本系统设计的，内置了 Raft 同步、注解索引、与 KVDB 的事务保证，属于业务定制的分布式缓存。

**请求路由机制**：当某节点收到针对其他节点硬件的操作指令（如调整风扇转速），会根据本机 Resource 中的元数据，将请求路由转发到目标节点执行。写入操作都在目标主 MS 上完成，避免多源写入问题。

### 2.2 Job 框架（任务调度）

自研的定时任务调度框架，局限于集群内部，由 leader 节点指派任务，不是无状态的通用分布式调度。性能指标采集、告警扫描等周期性工作都基于此框架。

### 2.3 EventBus 框架

自研的发布-订阅消息队列，用于模块间解耦。底层实现是 Map + 队列的结构。

## 三、主要业务工作

### 3.1 性能指标计算

**背景**：为用户提供性能指标的可视化和历史记录，需要周期性地采集硬件数据并计算各项指标。

**实现细节**：

1. **XML 驱动配置**：解析 XML 文件，将指标计算任务配置为内部数据结构，包含采集源、计算方法等。
2. **策略模式**：不同指标有不同的计算策略（加减乘除及更复杂的复合计算），通过策略模式实现指定的计算类方法。
3. **多线程并发采集与计算**：
   - 多个 Task 并发执行，使用 CompletableFuture 分段采集和计算，统一返回结果。
   - 自定义线程池，数据密集型场景配置为 CPU*2 的线程数。在 64C/128G 的环境下处理百万级对象。
   - 拒绝策略使用 CallerRunsPolicy。
4. **读多写少场景优化的数据结构**：
   - `CopyOnWriteArrayList`：线程安全，适合读多写少。
   - `StampedLock`（邮戳锁）：乐观读锁优化读性能。
   - `HashBasedTable`（Guava）：二维 Hash 表，加快多维查找。
   - `ImmutableList`（Guava）：不可变列表，无锁读。
5. **性能数据**：优化后单轮计算耗时从 100ms 降至 40ms。

**GC 事故排查（亮点）**：

数据量较大时（定时任务 5s 一轮，约 3 万对象）出现 SpringBoot 假死，不响应任何请求。

- **定位过程**：通过 `jstack` 发现 GC 线程占满 CPU；通过 JProfiler 分析堆内存分布，定位到关键代码。
- **根因**：信息采集时看板 key 对象在每轮任务中被频繁 new 出来再 clear 掉，导致 Young GC 压力极大。
- **解决方案**：将 List 改为 Map（以 id 为 key），对象不存在时才创建，已存在时复用——本质上是享元模式的思路，用缓存层减少对象的频繁创建和销毁。

### 3.2 硬件信息采集

**背景**：MS 作为管理侧，需要定期读取底层硬件状态数据，包括盘框状态、风扇状态/转速、M.2 盘状态、电源状态等。

**实现细节**：

1. 定时任务通过 SSH 执行底层 Python 脚本采集硬件信息，返回 JSON 字符串。
2. 编写多个 Handler（策略模式），将 JSON 解析为对应的 BO。部分采集涉及 JNI 调用。
3. 解析后的 BO 存入 Resource 框架，供性能指标计算和告警模块使用。

### 3.3 监控告警

**背景**：存储系统需要对硬件和软件异常进行及时告警和自动消警。

**实现细节**：

1. **XML 配置驱动**：解析 XML 配置告警指标、阈值、计算方法，支持动态的预警/告警阈值配置和多级告警策略。
2. **周期性扫描**：定时任务约 10s 一个周期，从 Resource 中读取数据，计算各告警指标，判断是否触发告警或消警。
3. **责任链模式**：告警处理流程使用责任链，提供 5 个默认阶段的 Processor，可扩展。
4. **告警联动与抑制**：不同告警之间存在关联关系，使用告警树（组合模式）管理。例如：盘框告警会抑制其下所有盘的拔盘告警，因为用户此时只需要知道盘框连接异常，不需要处理一堆独立的拔盘告警。这要求开发时不仅考虑技术实现，也要考虑用户体验。
5. **误告警处理**：人工介入（如运维操作）导致 CPU 突增等场景，通过周期性扫描 + 消警条件自动处理，达到消警条件即触发消警。

**框架学习**：除了基于告警框架实现业务告警，也主动学习了框架源码，了解设计者如何做抽象、如何让框架可扩展可维护。

## 四、灾备与双活

了解层面，非主要开发工作。

- Java 管理侧采用类似 MySQL MHA 的灾备方案，物理设备使用虚拟 IP，支持 IP 漂移实现故障切换。
- **双活架构**：两个副本 + 一个仲裁节点（不参与读写，只监控其余二者状态，类似 Redis 哨兵）。当主节点故障时，仲裁节点主动触发切换。
- LIF（逻辑接口）无缝漂移：切换过程中对上层透明。

**待明确的点**（自己还没完全搞清楚）：

- 切换时的数据一致性如何保障？是否有双写机制？

## 五、工程实践收获

1. **企业级开发流程**：设计评审 → 代码实现 → 代码 Review → 转测验证 → 上库，完整的质量保证流程。
2. **Git 分支管理**：公司级别的 Git 管理规范，分支策略，常用 Git 技巧。
3. **跨平台开发与调试**：Java/C/Python 多语言联调，JNI、GDB 调试经验。
4. **存储领域知识**：盘框、存储池、快照、克隆、创删池等存储概念。
5. **设计模式的实际应用**：策略模式、责任链模式、组合模式、享元模式，都是在真实业务场景中使用的，不是教科书式的理解。
6. **自研框架的设计思考**：为什么在有开源方案的情况下选择自研？答案是业务定制性和安全可控性。自研框架与业务深度绑定，比通用方案更贴合场景。
