# 蚂蚁集团 OceanBase 实习经历

## 一、背景

在 OceanBase 存储团队实习，岗位偏测试开发（测开）。团队负责 OB 存储层的质量保障，工作内容覆盖功能测试、性能压测、自动化工具建设。

实习期间主要交付了四块工作：共享存储冷热缓存测试、可观测性 Agent（dooba）、造数工具（slap）、日志服务测试框架与 OBGuard 支持。

---

## 二、主要工作

### 2.1 共享存储冷热缓存测试

**背景**：OB 4.x 引入共享存储架构后，支持冷热数据分层缓存。需要验证缓存分层策略的正确性和性能表现。

**工作内容**：
- **功能测试**：通过内部状态表判定数据所在的缓存层级（热/冷），验证手动冷热迁移的正确性。
- **性能测试**：对缓存分层场景进行压测，关注 QPS、RT、IO 吞吐、缓存命中率等指标。
- **成果**：推动修复了缓存策略缺陷，缓存命中率提升，IO 压力下降。

### 2.2 dooba —— 可观测性 Agent

**背景**：自动化测试流程中缺少统一、可复用的性能指标观测工具。已有的监控手段零散，没有结构化的采集、存储和分析链路。

**最终形态**：面向 QA 和 RD 的边缘可观测 Agent，从 0 到 1 构建。

**核心能力**：
- 监控多项性能指标：QPS、TPS、缓存命中率、CPU 使用率、MinIO QPS 等。
- 30+ 采集任务，10+ 插件（MinIO、AI、钉钉告警等）。
- 基于时间轮的任务调度系统，支持插件化扩展。
- 存储层经历了三次演进：自实现 mini LSM-Tree → SQLite → DuckDB。
- 计算层委托给 DuckDB，支持时间窗口聚合、降采样、下钻分析。
- 联动告警插件 + 钉钉插件，支持自动 TopSQL 下钻。

**演进过程**（这是这个项目最有价值的部分——在模糊需求下迭代）：

1. **V1 流式内存处理**：最初只是一个单文件 UI 监控脚本，纯内存处理，无持久化。
2. **V2 引入存储**：数据量增大后需要持久化，手写了一个 mini LSM-Tree（MemTable 用 SkipList + HashMap，SSTable 按时间组织文件，支持简单的 compaction）。选择自己写是因为当时环境只有 Python 3.6，DuckDB 要求 3.10+，SQLite 的时序聚合写起来别扭，而 LSM-Tree 天然适合追加写入场景。
3. **V3 引入 Web 白屏展示**：加入浏览器端可视化。
4. **V4 重构为插件系统**：将 Web、分析模块抽取为插件，核心层只保留任务调度和存储检索。引入 SQLite 存储和时间轮调度。
5. **V5 引入 DuckDB**：大数据量的抽取和计算成为瓶颈，Python 性能不够。引入 DuckDB 后将存储和计算全部委托，Python 版本升至 3.10，抛弃 import 导入方式，改为 19M 的打包二进制分发。同时支持从 Git 网络导入任务和插件。
6. **V6 告警与联动**：引入告警插件和钉钉插件，联动 TopSQL 做下钻分析。

**最大的难点不是编码，是需求**：

- 需求边界完全模糊。最开始得到的指示不太明确，作为刚入职的实习生，对公司自动化测试流程和目标不够了解，导致需求理解有偏差。
- 主管需要的可能只是：在自动化测试流程中转储性能指标，同时给出是否存在性能抖动的结果。我期望它能承载更多功能，中间有些开发不太符合预期。
- 演示过两次，似乎都与预期不完全相符。
- 后来回到学校后才意识到，自己要做的实际上是一个简化版的可观测性平台中 Metric 的部分，依托 DuckDB 做简化版产品。当时没有这个认知，导致产品形态模糊，途中反复重构。
- "性能抖动"本身是一个很模糊的需求，检测方式有很多种，不能简单把检测逻辑封死在内部，需要将控制反转给用户（配置驱动），这带来了设计上的复杂度。

**成果**：已稳定支撑日常巡检与性能 Job，发现数个性能抖动问题。

### 2.3 slap —— 造数工具

**背景**：OB 已有的造数工具版本落后、性能低：一种通过构造 INSERT 流量插入（需要走 compaction）；另一种用 Python 生成 CSV（可以走 OB 旁路导入绕过合并，但输出只有 40MB/s）。大规模存储测试和性能压测时等待时间过长。

**设计**：Python 驱动 + Rust 造数，配置驱动。

- **Python 端**：自动分区、数据范围分配、任务编排，输出无状态的 task 文件。
- **Rust 端**：深度性能调优，800MB/s 吞吐量输出 CSV。支持压缩算法，上传至 S3。
- **水平扩展**：任务文件是无状态的，可以在多机上分布执行，充分利用硬件资源。

**Rust 端优化细节**：
- ryu 高性能浮点数转字符串。
- SIMD 模板字符串填充（在 OceanBase 数据库大赛做向量索引优化时积累的 SIMD 经验在这里用上了）。
- 高性能随机数 crate。
- 循环无锁队列。
- IO 层 sendfile 优化。

**多种写入策略**：
- 行定长时：`O_DIRECT` + 多线程顺序写，不需要缓冲区，每个线程直接写磁盘绕过 page cache（要求 512B 对齐）。
- 行不定长时：环形缓冲区 + 多生产者单消费者模型。

**性能数据**：相比 OB 内部之前的 Python 造数方案，速度提升 10+ 倍（40MB/s → 800MB/s）。

**成果**：支持多个大数据量业务场景的数据构造和 bug 复现，显著减少等待时间。

### 2.4 日志服务测试框架 & OBGuard 支持

**背景**：OB 4.4.0 之后独立出了 Rust 编写的日志服务集群（LogService），需要端到端的测试能力。同时 OBGuard（OB 的边车进程，负责监控 OB 日志并告警，提供 trace、一键提 bug 等功能）需要支持新的日志服务。

**工作内容**：
- 为 Rust 日志服务集群编写基于 JavaCPP 的端到端测试框架（面试时 KPI 面试官专门追问了 JavaCPP 与 JNI 的区别和实现原理）。
- 为 OBGuard 支持 LogService 的日志告警和部署。

## 三、OceanBase 数据库大赛（赛前经历，非实习期间）

参加 OceanBase 数据库大赛，基于 MiniOB（教学用单机数据库）和 OceanBase 做优化，最终取得满意成绩。这段经历是后来拿到 OB 实习 offer 的重要背景。

### MiniOB 赛道

从零实现数据库核心特性：
- SQL 执行全链路：网络层 → Parser（Yacc）→ Statement → 执行计划 → 存储引擎。
- 实现 NULL 值处理、UNIQUE 约束、事务控制、B+ 树索引、向量索引等。
- Benchmark 优化：修复内存泄漏、砍掉冗余对象拷贝、优化类型转换。

### OceanBase 赛道（向量索引优化）

在成熟的分布式系统上做极致性能调优：
- **火焰图 Profiling**：第一次将火焰图应用于 C++ 系统性能分析，精准定位瓶颈函数。
- **SIMD 指令集优化**：针对固定维度的向量距离计算，用 SIMD 指令彻底提速。
- **分区堆优化**：针对超大 LIMIT 查询场景，改良 Partitioned Heap 结构。
- **量化技术**：应用 SQ4、SQ8、PQ4、PQ8 量化技术，用极低的计算代价换取接近的精度，配合多线程优化和剪枝策略。

### 架构认知收获

- **分布式调度执行**：理解租户与集群隔离、Master 统筹下发算子给 Worker 节点的机制。
- **计算存储分离**：计算下推至数据节点，管理节点只保存元数据并下达指令，避免数据大规模网络拷贝。
- **LSM-Tree 体系**：数据先写入 MemTable，通过 LSM-Tree 分层模型流转到多级 SSTable，实现高并发读写和冷热分离。

## 四、技术沉淀

### 性能优化方法论

通过大赛和实习积累的性能优化六原则：
1. **缓存预取**：让数据在被用到前到达缓存。
2. **异步化**：将阻塞操作剥离出主干流水线。
3. **并发并行**：细粒度分片压榨算力上限。
4. **利用局部性**：高频数据放在离计算核心最近的存储介质上。
5. **顺序优于随机**：顺序读写速度数量级领先。
6. **增量优于全量**：处理 Delta 而不是重新全盘计算。

### 工具链

- **性能分析**：火焰图、perf、pidstat、syscount-bpfcc、jstack、JProfiler。
- **存储引擎**：手写 LSM-Tree、DuckDB、SQLite，理解各自适用场景。
- **系统编程**：SIMD、O_DIRECT、sendfile、mmap、环形缓冲区、无锁队列。

### 测开视角的独特价值

面试时被问到"测开经历去面开发岗怎么转变"，核心回答：测试和开发是两个不同的视角，测开经历让我对软件质量有更深的认识。具体体现在：
- 对压测流程、性能基线、回归验证有实操经验。
- 习惯从"怎么验证它是对的"而不只是"怎么实现它"的角度思考。
- 构建工具的能力：dooba 和 slap 都是从 0 到 1 的工具，按开源项目标准维护。

## 五、面试准备要点

### 一句话介绍

> 在 OceanBase 存储团队做测试开发实习，主要交付了冷热缓存测试、可观测性 Agent、高性能造数工具、日志服务测试框架四块工作。之前还参加过 OceanBase 数据库大赛，做向量索引优化。

### 可能被深挖的方向

| 方向 | 关键点 | 高频追问 |
|------|--------|----------|
| dooba（可观测 Agent） | 需求模糊下的迭代、LSM-Tree 手写细节、DuckDB 替换决策、插件系统设计 | "存储层为什么换了三次？""LSM-Tree 的 compaction 策略？""MemTable 用什么数据结构？" |
| slap（造数工具） | Rust 优化细节、800MB/s 怎么来的、多机水平扩展、O_DIRECT vs page cache | "Rust 比 Python 快的本质原因？""定长和不定长行的写入策略为什么不同？" |
| 冷热缓存 | 怎么判定缓存层级、压测指标选择 | "缓存命中率怎么测的？" |

### 应对技巧

- **dooba 的迭代故事**是最好的亮点素材，能体现"在模糊需求下独立交付"的能力，但要控制讲述节奏，不要把五个版本都展开，挑 2-3 个关键转折点。
- **slap 的性能数据**很硬（40MB/s → 800MB/s，10+ 倍提升），要能说清瓶颈在哪、优化了什么、为什么有效。
- **LSM-Tree 手写细节**面试被追问过且答得不好（MemTable 数据结构、布隆过滤器、compaction 策略），需要重新过一遍。
- **测开转开发**的叙事：不要回避测开身份，强调工具建设能力和质量视角的独特价值。DD 三面面试官评价积极。
- **OB 大赛经历**可以作为"为什么对系统性能感兴趣"的起源故事来讲，串联整个技术成长线。
- **代码量**会被问到（DD 二面问过），提前准备一个合理的数字。
