## 开场：为什么明明能用，还是不舒服？

**听众通常已经有这个感受：**

- Claude Code / Agent 确实能帮忙，很多时候甚至很强
- 但用久了会发现，它不是“稳定同事”，更像一个每次都要重新带入状态的高手
- 不舒服的地方往往说不清楚：不是完全不能用，而是很难稳定复用

**这些痛点很具体：**

| 日常现象 | 背后的问题 |
|----------|------------|
| 上次已经解释过的项目背景，下次又要重讲 | 没有可靠的长期状态 |
| 需求聊着聊着变了，最后和最初目标不一致 | 需求只存在聊天历史里 |
| Agent 做完说完成，但你不确定对不对 | 缺少自动验证和质量门禁 |
| 同样的坑反复踩，经验没有沉淀 | 记忆不等于可复用方法论 |
| 跨天任务接不上，计划和进度断掉 | 会话内规划没有项目化 |
| 记忆、RAG、Hooks、Skills、Specs 名词很多 | 缺少统一心智模型 |

**这场分享不是先介绍工具，而是先回答：**

> 这些不舒服分别来自大模型的哪个局限？
> 后面的机制和工具，分别用什么规则去补这些短板？

## 核心观点

**一句话总结整场分享：**

> 后面所有机制和工具，都不是为了追新名词；
> 它们都是围绕大模型的局限，以及日常使用中的真实痛点，去补状态、上下文、流程、验证和经验沉淀。

**一句话总结上下文工程：**

> 记忆、RAG、网络搜索、Skills、Specs... 这些名词的本质都是同一件事：
> **在合适的时机，将合适的上下文放入 LLM 的上下文窗口。**

**一句话总结 Harness：**

> Harness 是补齐大模型缺失能力的外围系统：
> 状态保存、工具调用、记忆管理、行为引导。
> 不同实现只是选择补齐哪些、怎么补齐。

**一句话防止过度简化：**

> 上下文工程解决的是“模型知道什么”；
> 脚本、工具、测试、Guard 解决的是“系统能稳定做什么”。
> 一个好用的工作区需要两者同时存在。

## Harness vs Agent Framework：关键区别

| 维度 | Agent Framework | Harness |
|------|-----------------|---------|
| **决策权** | 程序硬编码 | 模型自主判断 |
| **工作流** | 固定流程图 | 动态适应场景 |
| **工具调用** | 规则触发 | 信任模型决定何时 |
| **记忆** | 系统管理 | 文件驱动，模型可读 |
| **控制方式** | 代码逻辑 | 上下文引导 |
| **灵活性** | 需改代码 | 改配置/文档即可 |

**核心哲学：**
> Harness 不替模型做决定，它给模型能力，然后信任模型。

## Claude Code 增强机制：Hooks —— 无感增强的入口

### 什么是 Hooks？

```
┌─────────────────────────────────────────────────────────┐
│              Hooks：生命周期的拦截点                        │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Claude Code 提供了 Agent 生命周期的事件拦截点            │
│  → 这些拦截点就是 Hooks                                   │
│                                                         │
│  类比：                                                  │
│  ┌──────────────────────────────────────┐              │
│  │ Git Hooks                              │              │
│  │ • pre-commit  → 提交前检查             │              │
│  │ • post-commit → 提交后通知             │              │
│  │                                        │              │
│  │ Claude Code Hooks                      │              │
│  │ • SessionStart → 会话开始时注入        │              │
│  │ • PreToolUse  → 工具调用前注入         │              │
│  │ • PostToolUse → 工具调用后捕获         │              │
│  │ • SubagentStop → 子Agent停止时验证     │              │
│  └──────────────────────────────────────┘              │
│                                                         │
│  核心思想：                                              │
│  ┌──────────────────────────────────────┐              │
│  │ 在特定时机，自动执行你的代码            │              │
│  │ → 可以注入上下文                       │              │
│  │ → 可以捕获变更                         │              │
│  │ → 可以阻止/放行                        │              │
│  └──────────────────────────────────────┘              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Hooks 的四种用途

```
┌─────────────────────────────────────────────────────────┐
│              Hooks 的四种用途                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 注入上下文 (Inject)                                  │
│  ┌──────────────────────────────────────┐              │
│  │ SessionStart:                         │              │
│  │ • 加载 CLAUDE.md                       │              │
│  │ • 加载 workflow 规则                   │              │
│  │ • 注入 git status                      │              │
│  │ → 模型一启动就知道项目规则              │              │
│  └──────────────────────────────────────┘              │
│                                                         │
│  2. 捕获变更 (Capture)                                   │
│  ┌──────────────────────────────────────┐              │
│  │ PostToolUse:                          │              │
│  │ • Bash 执行 → 捕获命令输出             │              │
│  │ • Write 文件 → 捕获新文件              │              │

│  └──────────────────────────────────────┘              │
│                                                         │
│  3. 质量门禁 (Gate)                                      │
│  ┌──────────────────────────────────────┐              │
│  │ SubagentStop:                         │              │
│  │ • Agent 说"我做完了"                   │              │
│  │ • Hook 先跑 lint + test               │              │
│  │ • 失败 → 阻止停止，Agent 继续修        │              │

│  └──────────────────────────────────────┘              │
│                                                         │
│  4. 状态管理 (Persist)                                   │
│  ┌──────────────────────────────────────┐              │
│  │ SessionEnd / PostToolUse:             │              │
│  │ • 压缩会话内容                         │              │
│  │ • 写入 memory 文件                     │              │
│  │ • 下次自动加载                         │              │
│  └──────────────────────────────────────┘              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Hooks 配置示例

```json
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",  // 匹配 Bash 工具
        "hooks": [
          {
            "type": "command",
            "command": "capture-command-output.py"
          }
        ]
      },
      {
        "matcher": "Write",  // 匹配 Write 工具
        "hooks": [
          {
            "type": "command", 
            "command": "capture-new-file.py"
          }
        ]
      }
    ]
  }
}
```

**关键：matcher 决定拦截哪个工具，hooks 决定执行什么**

# 4. OpenSpec 规范驱动开发

OpenSpec 的价值是把需求、设计、任务和归档变成项目资产，让 Agent 不只跟着当前聊天走，而是围绕明确的变更契约执行。

# 5. 上下文工程：那些名词没那么复杂

本章对应的痛点：**Memory、RAG、搜索、Hooks、Skills、Specs 像不同体系，难以形成统一理解。**

把它们拆开后，核心都是“把什么内容，在什么时机，放进上下文窗口”。

## 一句话总结

> **一切上下文工程的核心目的只有一个：**
> **在合适的时机，将合适的上下文放入 LLM 的上下文窗口。**

但这句话只覆盖一半：

> 上下文负责“让模型知道”；
> 工具、脚本、测试和 Guard 负责“让系统稳定执行”。

## 三者的区别

| 维度 | 记忆 | RAG | 网络搜索 |
|------|------|-----|----------|
| **来源** | 之前的对话 | 预构建的知识库 | 实时互联网 |
| **时效性** | 过去的经验 | 相对静态 | 最新的信息 |
| **控制权** | Agent 自动积累 | 你预先准备 | Agent 实时查询 |
| **适用场景** | "上次怎么解决的" | "公司文档规定" | "最新 API 文档" |
| **实现方式** | Hooks + 文件存储 | 向量检索 + Embedding | API 调用 |

# 6. Skills 组织最佳实践

本章对应的痛点：**经验留不住，或者写了也不容易被 Agent 想起来。**

Skills 组织的关键不是写长文档，而是让触发条件、使用边界、操作流程和常见错误足够清晰。

## SKILL.md 结构

```markdown
---
name: skill-name-with-hyphens
description: Use when [触发条件 + 症状]
---

# Skill Name

## Overview (核心理念 1-2句)

## When to Use
症状列表
何时不适用

## Core Pattern
Before/After 对比

## Quick Reference
操作表格

## Common Mistakes
错误 + 修复
```

## CSO (Claude Search Optimization)

```
┌─────────────────────────────────────────────────────────┐
│                  CSO 关键原则                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Description ≠ 流程摘要                              │
│     └── Claude 会跳过全文，只执行描述                     │
│     └── 只写触发条件，不写工作流                          │
│                                                         │
│  2. 关键词覆盖                                           │
│     └── 错误信息、症状、工具名                           │
│                                                         │
│  3. 动态命名                                             │
│     └── creating-skills > skill-creation                │
│                                                         │
│  4. Token 效率                                           │
│     └── 频繁加载 < 200 words                             │
│     └── 把大文档放到引用文件                              │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## Skills 的 TDD 方法论

| TDD 阶段 | Skills 对应 |
|----------|-------------|
| RED | 无技能时跑压力场景，记录失败行为 |
| GREEN | 写技能针对性解决失败点 |
| REFACTOR | 测试发现新漏洞 → 补漏洞 → 再测 |

**Iron Law：No skill without a failing test first.**

# 7. 长期运行的个人助手：OpenClaw vs Hermes

本章对应的痛点：**跨天、跨会话、跨渠道的长期使用很难接上。**

OpenClaw 和 Hermes 火，是因为它们都在尝试把记忆、技能、会话搜索和入口渠道组合成长期助手；体验难调，也是因为长期记忆天然需要积累和校准。

## 为什么这两个项目火了？

```
┌─────────────────────────────────────────────────────────┐
│              它们的卖点是什么？                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  OpenClaw (高 star 的个人助手项目)                       │
│  ┌──────────────────────────────────────┐              │
│  │ 卖点：                                 │              │
│  │ • "长期记忆" — 跨会话记住你            │              │
│  │ • 多渠道入口 — 聊天软件里直接调度       │              │
│  │                                        │              │
│  │ 但实际体验：                           │              │
│  │ • 配置复杂                             │              │
│  │ • 记忆质量不稳定                       │              │
│  │ • 需要"养"很久才有效果                 │              │
│  └──────────────────────────────────────┘              │
│                                                         │
│  Hermes Agent                                           │
│  ┌──────────────────────────────────────┐              │

│  │ • "自动从经验创建技能"                 │              │

│                                                         │

│  ┌──────────────────────────────────────┐              │
```

# 8. 综合架构对比与展望

本章回到开头的痛点：不同项目不是在发明完全不同的东西，而是在选择补齐大模型的不同短板。

## Harness 是什么？统一视角

```
┌─────────────────────────────────────────────────────────┐
│                Harness 统一视角                           │
├─────────────────────────────────────────────────────────┤
│                                                         │

│                                                         │
```

## 核心设计原则总结

```
┌─────────────────────────────────────────────────────────┐
│                Harness 设计原则                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. Trust the Model                                     │
│     └── Harness 不替模型做决定                           │
│     └── 给能力，让模型判断何时用                          │
│                                                         │
│  2. File as State                                       │
│     └── 一切状态都是文件                                  │
│     └── 模型可读 = 模型可控                              │
│                                                         │
│  3. Context as Control                                  │

│     └── CLAUDE.md/Skills/Hooks 共同塑造                  │
│                                                         │
│  4. TDD for Skills                                      │

│     └── 从错误中提炼规则                                  │
│                                                         │
│  5. Self-Evolution                                      │
│     └── 从经验自动生成技能                                │
│     └── 使用中持续改进                                    │
│                                                         │
│  6. Tools and Evals                                     │
│     └── 共享脚本收束解空间                                │
```

## 个人工作区：推荐目录结构

```text

├── CLAUDE.md                    # 总规则：边界、偏好、常用命令
├── docs/
│   ├── architecture.md          # 项目/工作区怎么组织
│   ├── workflows.md             # 关键流程
│   └── decisions/               # 重要决策记录
├── knowledge/
│   ├── patterns/                # 可复用模式
│   ├── writeups/                # 完成案例
│   └── backlog.md               # 待沉淀经验
├── .claude/
│   ├── rules/                   # 路径/主题级常驻规则
│   ├── skills/                  # 工作区专属 Skills
│   ├── commands/                # 高频入口命令
│   └── hooks/                   # 自动检查/自动注入上下文
```

**最实用的顺序：**

3. 再把高频命令固化到 `scripts/`
4. 再把重复流程沉淀成 `Skills`
5. 再给 Skills 配 smoke / regression
6. 最后才考虑 Hook、索引、会话数据库

```
│  └──────────────────────────────────────┘              │
│                                                         │
```

## CTFAgent 需要三层代码边界

| 位置 | 放什么 | 示例 |
|------|--------|------|
| `tools/ctfagent/` | 项目级共享库和 CLI | memory index、eval runner、spec validator、skill inventory |
| `.claude/skills/__lib__/` | Skill 运行时轻量公共库 | markdown parser、workspace model、通用 query |
| `.claude/skills/<skill>/scripts/` | Skill 专属脚本 | `ssti_probe.py`、`jwt_tool.py`、`exploit_scaffold.py` |

**判断标准：**

| 问题 | 放哪里 |
|------|--------|
| 会被 3 个以上 Skills 调用 | `tools/ctfagent/` |
| 只服务 Skill 脚本内部 import | `.claude/skills/__lib__/` |
| 只服务一个具体 Skill | `skills/<skill>/scripts/` |
| 影响 self-evolution 流程 | `tools/ctfagent/` + `specs/workflows/` |
| 只是一次题目的临时 exploit | `workspaces/<challenge-id>/` |

**语言选择：**

- 默认用 Python：CTF 生态、脚本、测试、二进制和 Web 探测都更顺
- TypeScript 只在浏览器自动化、前端分析、Node 生态工具明显占优时使用
- 不要 Python/TS 各写一套同类工具，统一 CLI 入口更重要

## CTFAgent：什么该自动，什么不该自动？

| 动作 | 建议 | 原因 |
|------|------|------|
| 记录 writeup / pattern | 自动或半自动 | 低风险，高价值 |
| 更新共享索引 / 评估报告 | 自动 | 属于可复算外围状态 |
| 生成 skill 候选草稿 | 自动 | 可以提高沉淀效率 |
| 新增共享脚本 API | 人审后执行 | 会影响多个 skills |
| 修改已有 skill 的小规则 | 半自动 | 需要测试验证 |
| 新增 skill | 人审后执行 | 容易造成技能树膨胀 |
| 修改能力 spec | 必须人审 | 这是系统行为契约 |
| 删除旧规则 | 必须人审 | 容易破坏历史能力 |

**核心原则：**

> 自我进化可以自动发现问题、自动提出补丁，但不应该自动改变系统契约。

## CTFAgent 的下一步优先级

| 优先级 | 改动 | 为什么 |
|--------|------|--------|
| P0 | 建 `tools/ctfagent/` 共享工具层和统一 CLI | 收束记忆、评估、spec、skill 管理的解空间 |
| P0 | 给 `skill-maintainer` 增加 spec-change 判断 | 先区分“补经验”和“改能力边界” |
| P0 | 建 `specs/workflows/skill-evolution.md` | 固化自我进化流程 |
| P1 | 把跨 skill 的脚本从单个 skill 中抽到共享层 | 避免重复脚本和隐式分叉 |
| P1 | 为现有核心 skill 写 capability spec | 明确每个 skill 的验收标准 |
| P1 | 建统一 lab / smoke 命名约定 | 让升级可回放 |
| P1 | 把 `skill-backlog` 拆成候选变更队列 | 避免 backlog 变成杂物堆 |
| P2 | 加 session / writeup 检索索引 | 到这一步再做长期记忆更稳 |
| P2 | 自动生成 skill 草稿 | 在 guard 完善后再开放 |
