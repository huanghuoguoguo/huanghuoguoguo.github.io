# 经历与思考素材库

这个文件不是简历，也不是成稿草稿。

它只做一件事：把以后写博客真正能用上的材料留下来，尤其是那些“只有我能写”的部分。

维护原则：

- 一条素材只写一个认知点
- 先写场景，再写我当时怎么想，最后写后来怎么改
- 不追求完整，宁可短一点，也不要写成说明文
- 能写具体动作就不写抽象判断
- 有情绪就记情绪，但情绪要连着场景

## 已有素材

### 曙光：GC 风暴不是“JVM 不行”，是对象在无脑地反复创建

- 时间：2024 年，曙光实习
- 地点 / 项目：国产化分布式闪存存储系统，Java 管理侧性能指标计算
- 我当时在做什么：5s 一轮跑定时任务，采集和计算大约 3 万对象的指标
- 我先入为主怎么想：一开始只觉得“系统怎么突然假死了”，下意识怀疑线程池、锁竞争或者某段计算太慢
- 真正出问题的地方：看板 key 对象每轮都在 `new`，用完再 `clear`，Young GC 压力很大，服务直接假死
- 我怎么发现的：先 `jstack` 看 GC 线程，再用 JProfiler 看堆分布，最后把热点收敛到那段 key 对象逻辑
- 我后来怎么改：把 `List` 改成按 id 复用对象的 `Map`，不存在才创建，存在就复用
- 这件事让我现在更相信什么：性能问题先别猜，先抓现象和对象流向；很多所谓“大问题”，最后都落在一个很蠢的小地方
- 能落到文章里的句子 / 画面：`jstack` 一打开，CPU 几乎全让 GC 线程吃掉了；我那时第一反应不是“终于找到了”，而是“怎么会这么蠢”
- 暂时还没想透的地方：如果数据规模继续再涨，这套对象复用能撑多久，后面会不会又撞上别的瓶颈

### 曙光：告警抑制不是技术技巧，是用户视角

- 时间：2024 年，曙光实习
- 地点 / 项目：监控告警模块
- 我当时在做什么：实现盘框、磁盘等硬件告警
- 我先入为主怎么想：最开始会觉得“多报一点总比漏报好”
- 真正出问题的地方：盘框一旦异常，下面一串磁盘拔盘告警都会冒出来，技术上没错，运维看起来全是噪音
- 我怎么发现的：顺着告警树往下看，发现用户真正需要知道的是“盘框连接异常”，不是 20 条子告警一起炸脸
- 我后来怎么改：做了告警树和抑制关系，把父级异常优先级提上来
- 这件事让我现在更相信什么：工程里很多“做对了”的东西，到了用户眼里可能还是错的
- 能落到文章里的句子 / 画面：技术上全对，用户还是会骂；因为他想要的是一个能处理的信号，不是一堵红墙
- 暂时还没想透的地方：复杂抑制关系以后会不会把真正有价值的细粒度问题也一起压掉

### OceanBase：dooba 最大的问题不是编码，是我把需求想大了

- 时间：2025 年，OceanBase 实习
- 地点 / 项目：dooba，可观测性 Agent
- 我当时在做什么：从 0 到 1 做一个性能指标采集、存储、分析、展示工具
- 我先入为主怎么想：我当时总想把它往“简化版可观测平台”方向做，觉得功能越完整越好
- 真正出问题的地方：主管真正需要的，也许只是自动化流程里能转储指标，并判断有没有性能抖动；我却把很多精力花在产品形态上
- 我怎么发现的：演示做了两次，都和预期不完全对上；后来回学校再看，才意识到自己在替需求补脑
- 我后来怎么改：开始承认“性能抖动”本身就是模糊需求，尽量把控制权交给配置，而不是一开始就把检测逻辑写死
- 这件事让我现在更相信什么：需求不清的时候，先别急着把系统做大；先追问要交付的结果到底是什么
- 能落到文章里的句子 / 画面：最大的坑不是 Python，也不是 DuckDB，更不是插件系统；坑在于我一开始就把东西想大了
- 暂时还没想透的地方：需求模糊时，产品边界到底该由谁来先收窄

### OceanBase：手写 LSM-Tree 不一定是“有追求”，也可能是我当时太乐观

- 时间：2025 年，OceanBase 实习
- 地点 / 项目：dooba 存储层演进
- 我当时在做什么：V2 需要持久化，又受 Python 版本和依赖限制
- 我先入为主怎么想：LSM-Tree 天然适合追加写，自己写一个应该挺顺
- 真正出问题的地方：后面需求很快变，分析和查询复杂度上来之后，自己写的存储层开始拖后腿
- 我怎么发现的：越往后越发现瓶颈不只是写入，真正麻烦的是抽取、聚合、下钻这些动作
- 我后来怎么改：最后把存储和计算都委托给 DuckDB
- 这件事让我现在更相信什么：自己写一层系统有时候能帮你看清原理，但不等于它适合留下来
- 能落到文章里的句子 / 画面：手写 LSM-Tree 那一阵子我很兴奋，后来越改越知道，这东西不该继续让我扛
- 暂时还没想透的地方：什么场景下“自己先写一版”是值得的，什么场景下只是过度工程

### OceanBase：slap 的性能优化是很硬的，但真正难的是把瓶颈讲清楚

- 时间：2025 年，OceanBase 实习
- 地点 / 项目：slap，造数工具
- 我当时在做什么：把 Python 造数链路重做成 Python 编排 + Rust 生成
- 我先入为主怎么想：只要 Rust 写得够快，问题就解决了
- 真正出问题的地方：性能提升不只是语言切换，背后是写入路径、对齐、缓冲区、多线程模型一起改
- 我怎么发现的：如果只说“Rust 更快”，面试里根本撑不住；必须回到 `O_DIRECT`、环形缓冲区、sendfile、SIMD 这些具体点
- 我后来怎么改：开始用更细的方式拆瓶颈，哪些是 CPU，哪些是 IO，哪些是数据路径设计
- 这件事让我现在更相信什么：性能文章里最怕一句话总结，必须把瓶颈是怎么迁移的写出来
- 能落到文章里的句子 / 画面：40MB/s 到 800MB/s 不是一个“优化结果”，中间其实是一串很具体的妥协
- 暂时还没想透的地方：多机扩展时新的瓶颈会优先出现在网络、对象存储，还是调度编排

### 改 Node.js 文章：不熟的东西直接删，不留半懂不懂的话头

- 时间：2026 年 3 月，改 Node.js 文章
- 地点 / 项目：博客《TypeScript + Node.js：从另一个后端开发者的视角》
- 我当时在做什么：逐段把 AI 帮我补出来的 Node.js / AWS 解释往回收
- 我先入为主怎么想：最开始默认“不熟也可以先留一句边界说明，显得完整一点”
- 真正出问题的地方：这样一写，文章会变成“我仿佛已经知道这些词在系统里怎么工作，只是暂时不展开”，视角一下就站高了
- 我怎么发现的：我会立刻指出“这种问题你就直接删掉，不要留一个话头”“不熟悉就不要讲，不要讲得面面俱到”
- 我后来怎么改：把 SNS、SQS、冷启动这些没真用过的内容直接删掉，只保留自己当时能站住的观察和类比
- 这件事让我现在更相信什么：写陌生技术时，少一点完整感，多一点老实；不熟就别硬挂在那
- 能落到文章里的句子 / 画面：不熟悉就不要讲。不要讲得面面俱到。
- 暂时还没想透的地方：什么时候一个概念算“可以先提一下”，什么时候就应该直接删掉

### OceanBase 测开：AI 不一定先替掉 QA，先替掉的往往是重复劳动

- 时间：2026 年 3 月，改 QA 译文时
- 地点 / 项目：回看 OceanBase 测开经历
- 我当时在做什么：一边翻译 QA 角色存废的文章，一边回想自己在数据库产品里做测开的感受
- 我先入为主怎么想：最开始容易把 QA 理解成“执行回归、收集指标、最后再兜一层的人”
- 真正出问题的地方：很多 QA 的时间会被重复劳动吃掉，执行自动化、搬运结果、反复对环境和数据，真正值钱的领域知识反而没有被释放出来
- 我怎么发现的：一方面是主管在周会上反复强调“不要成为 RD 低级错误的兜底者”，另一方面是结合 OceanBase 这类数据库产品里内核 QA 的权重，再看 AI agent 最擅长的大量重复工作和无尽精力，这个对比会越来越明显
- 我后来怎么改：更愿意把 QA 想成维护领域知识、介入需求分析、设计高价值测试场景、构造自动化任务的人；很多工具以后也应该开始优先为 agent 设计
- 这件事让我现在更相信什么：AI 更可能先重写 QA 的分工，而不是简单取消 QA；最后还是要把人的精力压到高 ROI 的判断上
- 能落到文章里的句子 / 画面：不要把 QA 做成替 RD 兜底的角色；代码和脚本都能丢给 agent 跑，但“这套系统最容易死在哪”这件事，不能没人知道
- 暂时还没想透的地方：agent 时代里 QA 和 RD 的职责边界，最后会怎么重新划

## 以后怎么补

每次实习、项目、比赛、面试复盘后，不要问：

- “这段经历值不值得写成博客？”

先问这三个问题：

1. 我当时最先想错的地方是什么？
2. 是哪一个瞬间把我拧过来的？
3. 这件事让我现在更不相信什么，或者更相信什么？

只要能回答这三个问题，这段经历以后大概率就能长成一篇文章。
