# 新建博客文章

根据用户给出的主题创建新文章。

## 步骤

1. 确认主题、目标读者、文章类型（技术深潜/系统设计/读书笔记/实践复盘）
2. 生成文件：`_posts/YYYY-M-D-标题.md`，日期用当天
3. 填写 frontmatter（title, date, categories, tags）
4. 根据文章类型生成骨架结构，不填充内容
5. 骨架中用注释标注每节应该写什么、大概多少字

## 写作风格：汪曾祺式技术散文

我们写技术说明文，但态度上借用汪曾祺：散淡、不急、允许闲笔、句子短、朴素。

详细的风格规范和 AI 痕迹检测清单见 `.claude/skills/review-post/SKILL.md`，写文章时同样需要遵守。这里只列写作时最常踩的几个坑：

### 写的时候记住这几条

1. **别每段都划重点。** 加粗全文不超过 3-5 处。重点靠内容本身传达。
2. **别用"不是X而是Y"。** 直接说你想说的。
3. **别每节都总结。** 说完就停。结尾可以收一下，中间不用。
4. **留闲笔。** 讲完一个技术点，可以插句题外话再继续。调试的插曲、某个工具名字的来历、当时的场景——这些别删。
5. **句子往短了写。** 一个逗号能断成句号的，断成句号。
6. **少用引号。** 技术术语不加引号，概念不加引号。只有直接引语和确实在特殊含义下使用的词才加。
7. **允许不确定。** "大概是这个原因""我后来想了想也不太对"——这种语气可以出现，比假装什么都确定要好。

### 禁用词

综上所述、总而言之、值得注意的是、不可否认的是、深入探讨、赋能、抓手、闭环、打法、让我们一起来看看、在这个数字时代、不可小觑、至关重要、真正重要的是、归根结底。

## 文章类型模板

### 技术深潜

适用于深入某个技术点的文章（如 eBPF、HNSW、MVCC）。

- 开头：一个具体的问题或场景，不要背景铺垫
- 中间：原理剖析 + 代码/数据验证，允许中途跑题聊相关的有趣细节
- 结尾：适用场景，以及你自己还没想清楚的地方

### 系统设计

适用于架构设计类文章（如 RAG 系统、日志系统）。

- 开头：面试题/业务场景，直接抛问题
- 中间：角色划分 → 选型推导 → 关键决策点。不需要每层都完美递进，真实的取舍和犹豫可以写进去
- 结尾：与其他方案的差异对比，或者自己觉得没解决好的地方

### 读书笔记

适用于技术书籍的读后整理。

- 开头：为什么读这本书，带着什么问题来的
- 中间：核心观点 + 自己的理解和批注。可以跳着写，不用按书的章节顺序
- 结尾：对自己工作/项目的启发，或者不同意作者的地方

### 实践复盘

适用于实际项目的踩坑记录（如 db_gen、OB 大赛）。

- 开头：要解决什么问题
- 中间：迭代过程（V1→V2→...），每版贴数据。不用整理成完美阶梯——V2 没解决 V1 的所有问题、V3 退回了 V1 的某个决策，这些写出来比完美叙事更好
- 结尾：最终方案 + 哪些弯路值得走、哪些不值得
