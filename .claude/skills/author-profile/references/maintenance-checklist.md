# 维护 checklist

## 固定检查清单

每次触发本 skill，都按下面这份 checklist 过一遍，不要凭感觉跳步。

### A. 这轮对话里有没有“值得沉淀”的信息

逐条问自己：

- 用户有没有补新的项目、实习、比赛或开源经历？
- 用户有没有补新的事实、时间线、性能指标、职责边界？
- 用户有没有明确说“我以前怎么想，后来怎么改”？
- 用户有没有明确表达新的技术判断？
- 用户有没有通过删改文章暴露新的风格偏好？
- 用户有没有暴露新的知识盲区，或修复了旧盲区？

如果上面全是“没有”，本轮不更新。

### B. 这条信息属于哪一类

只在下面四类里选，不要发散：

- `硬事实`
- `一手判断`
- `思考方式`
- `风格偏好`

如果一条信息同时属于两类，可以双落点，但要分别写。

### C. 应该写到哪里

按这个顺序判断：

1. 这是不是稳定认识？
   - 是：写 `PROFILE.md`
2. 这是不是一个能长成博客的认知转折？
   - 是：写 `经历与思考素材库.md`
3. 这会不会影响项目事实、简历表述、时间线或指标？
   - 是：同步修对应经历文件 / `jianli.md`

### D. 这条信息要不要覆盖旧内容

更新前问：

- 这是新事实，还是旧事实的修正？
- 这是稳定偏好，还是一次性情绪？
- 这是新的补充角度，还是和旧画像冲突？

处理规则：

- 事实修正：覆盖旧的
- 风格偏好变化：替换旧的
- 新补充但不冲突：追加
- 一次性情绪：不写入画像，只在必要时写素材库

### E. 更新后再做一次自检

更新完，检查下面几条：

- `PROFILE.md` 有没有变得更肿而不是更准？
- 素材库这条卡片是不是只讲了一个认知点？
- 经历文件是不是被改成了散文化总结？
- 有没有把没有证据的猜测写成事实？
- 有没有把一次吐槽上升成稳定人格特征？

如果任何一条是“有”，回退到最小改动重写。

## 更新决策树

1. 本轮有没有新信息？
   - 没有：不改
   - 有：继续
2. 是不是事实修正？
   - 是：先改经历文件 / 简历，再同步画像
   - 不是：继续
3. 是不是认知转折？
   - 是：先补素材库，再决定要不要顺手补画像
   - 不是：继续
4. 是不是稳定判断或稳定偏好？
   - 是：写 `PROFILE.md`
   - 不是：不强行沉淀

## 冲突处理

遇到冲突时按这个优先级：

1. 用户本轮明确表述
2. 具体经历文件中的事实
3. `PROFILE.md` 里的旧归纳

处理方式：

- 如果是事实冲突，修经历文件和简历，再同步画像
- 如果是风格偏好变化，直接替换旧偏好，不要保留两版互相打架的话
- 如果只是新的补充角度，不冲突，可以追加

## 写入风格

### `PROFILE.md`

- 每条 1 到 2 句话
- 用概括句，不展开讲故事
- 重点是“以后 agent 记住什么”

### 素材库

- 一条卡片只写一个认知点
- 先场景，再误解，再转折，再判断
- 允许带一点情绪，但情绪必须连着场景

### 经历文件

- 修事实、修范围、修结果
- 不要把经历文件改成散文化随笔

## 什么时候要同步经历文件

出现以下信息时，不要只改画像：

- 用户补了新的项目结果或性能指标
- 用户修正了职责边界
- 用户补了新的模块细节
- 用户指出之前文档里有误
- 用户对简历表述提出了新的压缩方式或纠偏

## 什么时候要优先补素材库

出现以下信息时，优先补卡片：

- “我当时其实是这么想的”
- “后来我才意识到”
- “那次真正的问题不是 X，是 Y”
- “这件事之后我更相信……”
- “这句话 / 这类话我现在见一次删一次”

补卡片时，优先把以下字段补全：

- 我先入为主怎么想
- 真正出问题的地方
- 我怎么发现的
- 我后来怎么改
- 这件事让我现在更相信什么

如果这五项里有三项以上答不出来，先别急着写成卡片。

## 禁止事项

- 不要把一次随口吐槽直接上升成稳定画像
- 不要把作者包装成全知视角
- 不要往 `PROFILE.md` 里塞长故事
- 不要因为用户讲了一个项目，就自动假设他完全掌握了所有底层细节
- 不要在没有依据时替用户发明风格偏好

## 维护成功的标准

这个 skill 做得对不对，看三个结果：

1. 后续写博客时，agent 能主动调用作者自己的经历和认知转折
2. 画像会越来越清楚，但不会越来越肿
3. 用户聊得越多，agent 越能抓住“这个人是怎么形成判断的”，而不是只记住履历关键词
