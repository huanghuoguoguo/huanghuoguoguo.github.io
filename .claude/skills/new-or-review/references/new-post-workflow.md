# 新建文章工作流

## Stage 0：Interview

在真正写之前，先判断要不要做访谈。

以下任一情况成立，就必须先问作者：

- 主题里有明显超出作者熟悉范围的技术点
- 作者给了一个题目，但没有给足场景、经历、例子或证据
- 文章很依赖作者自己的判断，但现在只看到一个结论，没有来路
- 有多个可能的写法，不知道作者想从哪条线进入
- 某些地方 AI 只能给常规解释，但不知道这是不是作者真正相信的说法
- 怀疑继续写会写成“先有结论，再补材料”的稿子
- 不确定该写到多深，或者可能超出作者当前认知边界

访谈不是形式动作。目的有两个：

1. 把作者已经有但还没说出来的经历、画面、判断、犹豫挖出来。
2. 把作者暂时没想到的问题补出来，帮他把认知链路说完整。

也就是说，AI 在这一步不只是“问素材”，还要像访谈者一样追问：

- 你当时最先是怎么想的？
- 哪一步开始觉得不对？
- 真正拧过来的瞬间是什么？
- 你现在更相信哪个判断？为什么？
- 这地方你自己有没有还拿不准的？
- 这段如果按你的口气写，你更想写得硬一点、烦一点，还是收着一点？

### 什么时候必须停止写作，先访谈

如果在 `plan` 或 `draft` 中出现下面这些信号，必须暂停，先问作者，不要硬往下补：

- 你发现自己只能写出标准答案讲解
- 你发现自己在用常见科普结构填空
- 你不知道某个判断是不是作者本人真的相信
- 你不知道某段该往“经历”写还是往“知识点”写
- 你开始用大量“也许”“通常”“一般来说”来糊过去

### 访谈问题怎么问

要求：

- 一次 3 到 7 个问题，宁可少，不要铺满
- 问题要短，尽量具体
- 优先问能改变文章方向的问题
- 问“你怎么想的”，也问“发生了什么”
- 允许 AI 提出作者自己还没想到的追问方向

优先问题类型：

1. 场景问题
   - 这篇你最想从哪个具体场景切进去？
   - 当时是什么事让你开始在意这个问题？

2. 误解问题
   - 你最早是怎么理解这件事的？
   - 后来发现那个理解哪里不对？

3. 证据问题
   - 这篇里你手上有哪些能落地的材料？命令、日志、数据、图、旧文、代码都算
   - 哪一段如果让我现在写，你最怕它写空？

4. 风格问题
   - 这篇你想写得更硬一点，还是更散一点？
   - 哪些句子你一看就会觉得“这不是我会写的”？

5. 边界问题
   - 哪些地方你其实还没完全想透？
   - 哪些内容你不想让 AI 擅自展开？

### Stage 0 输出

访谈后必须沉淀：

- 可直接写入文章的作者原话或判断
- 可直接调用的具体场景和证据
- 明确的认知边界
- 本文不该硬写的部分
- 作者更偏好的写法方向

如果访谈后材料仍然不足，就先停在 `plan`，不进入 `draft`。

## Stage 1：Plan

先确认：

1. 主题是什么
2. 目标读者是谁
3. 文章类型是什么
4. 当前手里有哪些素材
5. 哪些观点有证据，哪些还没有
6. 哪些地方是作者已经想清楚的，哪些地方还只是模糊理解
7. 这篇如果现在直接写，最容易写假的地方在哪

支持的文章类型：

- 技术深潜
- 系统设计
- 读书笔记
- 实践复盘

如果用户明确要求“写得像某个样本作者，但内容仍然是我自己的经历和思考”，在 `plan` 阶段必须额外确认三件事：

1. 要借的是哪一层：
   - 开头方式
   - 推进骨架
   - 证据组织
   - 语气强度
2. 哪些东西不能借：
   - 样本作者的人设
   - 样本作者的职业路径
   - 样本作者固定口头禅
   - 和作者本人经历无关的情绪或判断
3. 本文自己的锚点是什么：
   - 你的真实项目
   - 你的认知转折
   - 你的疑问
   - 你的证据

### Stage 1 输出

必须先产出：

- 文章标题候选
- 文章类型
- 核心问题
- 章节骨架
- 每节要回答的问题
- 每节依赖的证据
- 证据缺口列表

如果这篇是“借某种文风骨架写”，还必须多产出两项：

- `borrowed skeleton`
  - 明确这篇借的是哪几个结构动作，例如“场景起手 + 主问题推进 + 从入门到深入”
- `self anchors`
  - 明确这篇只能来自作者本人的材料，例如“某次实习排查”、“某次读书误解”、“某个 benchmark 或日志”

如果素材不足，先指出缺口，不要硬写。
如果缺口影响文章是否像作者本人，先回到 `Stage 0` 继续访谈。

## Stage 2：Draft

基于骨架写首稿。

要求：

- 生成文件：`_posts/YYYY-M-D-标题.md`
- `frontmatter` 包含 `title`, `date`, `categories`, `tags`
- 每节围绕一个明确问题展开
- 尽量落到实验、命令、图、表格、日志、代码或具体场景
- 不主动把所有判断都写成确定结论
- 不整理成“完美阶梯递进”叙事
- 优先写“我为什么现在能理解这件事”，不要直接摆出一段抽象定义
- 不要把首稿写成“结构完整、表达稳妥、像已经审过一遍的成稿”
- 允许局部有火气、有偏向，但要能指出火气是从哪来的
- 如果写到某一节时发现自己只能补标准解释，先停下来回访作者，不强写
- 如果在模仿样本时开始出现“样本作者在说话，而不是作者本人在说话”的感觉，立刻回退，保留骨架，换回作者自己的语气和材料

首稿不是终稿。允许粗糙，但必须有真实材料支撑点。

## Stage 3：Review

首稿完成后，必须做 `review`。详细检查项和检测器信号解释见 `references/review-and-revise.md`。

`Stage 3` 必须输出：

- 总体判断：可继续 / 需修订 / 建议人工接管
- 高优先级问题列表
- revision plan
- 文章里最像作者、最值得保留的段落
- 如果需要回访作者，要明确列出要回访的 3 到 5 个问题

## Stage 4：Revise

修订只针对命中问题做局部修改。具体修订规则见 `references/review-and-revise.md` 和 `references/anti-patterns.md`。

禁止：

- 整篇重写
- 把真实细节删掉换成平滑空话
- 为了过检测把语气改得过于油滑
- 明明应该回访作者，却擅自用常识补完整段

## Stage 5：Final Check

修订后再审一次，决定：

- `accept`
- `retry`
- `handoff_to_human`

停止条件满足任一条就不要继续自动回环：

- 高严重度问题已经清空
- 连续两轮改动后收益很小
- 修订开始损伤真实细节
- 文章已经“没 AI 味，但也没你味”
