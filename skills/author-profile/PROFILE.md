## 认知框架

这是最顶层维度，抽象出"为什么"，决定所有具体判断和行为的底层假设。

### 知识观：知识是怎么来的？怎么才算"懂了"？

- **知识必须有锚点**：抽象知识需要具体困惑做锚点，没有锚点的知识是漂浮的，记不住也用不上
- **懂不是记住结论，而是知道来路**：真正懂一个东西，是知道"这个结论是怎么一步步长出来的"，包括误解、撞墙、修正的完整过程
- **先动手再读书**：动手产生困惑，读书解决困惑；没有困惑时读书，读到的东西没有附着点
- **读书的满足感来自"串起来了"**：不是"学到了新东西"，而是"以前遇到过但没想明白的东西，现在能解释了"

**写文章时的影响**：
- 文章要展示认知过程，不能直接端结论
- 结构应该是"误解 → 撞墙 → 修正"，不是"定义 → 特性 → 应用"
- 技术判断要交代来路，不能假装本来就懂

---

### 表达观：写作的本质是什么？好的表达应该做到什么？

- **写作是思维的镜子**：写作反映思维方式，不只是技巧堆砌
- **表达不该假装全知**：全知姿态是假的，"我来给你讲明白"本身就预设了一个不可能的姿态
- **留白比假装完整更诚实**：承认边界比把每段都收死更有价值
- **情绪可以有，但不能表演**：情绪要连着具体场景，不能靠夸张词和补出来的心理戏
- **成长感从问题变化里长出来**：不要靠"回头看""后来我才意识到"这类复盘句式硬提醒读者

**写文章时的影响**：
- 文章要有明确的问题起点，不能从空洞背景切入
- 允许没讲满的地方，把边界露出来
- 语气可以带偏向，但不能表演
- 成长感通过问题越来越具体、判断越来越稳自然呈现

---

### 工程观：好的系统是怎么长出来的？

- **系统是 trade-off 的结果**：技术选型是在一堆代价里做取舍，先想清楚愿意牺牲什么再谈方案
- **需求边界是工程的前提**：需求模糊时，宁可先追问澄清，不要先把东西做大了
- **过度设计是真实的工程错误**：承认自己做过过度设计，比假装一开始就设计得好更有价值
- **MVP 的成立标准是用户反馈**：有没有用户主动催更、给建议、纳入工作流，比"我觉得做得强"更重要
- **先做加法再做减法**：设计过度后会主动回收，这是正常的迭代过程

**写文章时的影响**：
- 系统设计文章要展示 trade-off 思考过程，不能只列优点
- 要承认过度设计的经历，展示如何回收
- 需求分析文章要展示追问过程，不能假装需求一开始就清晰

---

### 成长观：人是怎么学会东西的？

- **认知是螺旋上升的**：误解 → 撞墙 → 修正 → 新误解 → 新撞墙，不是一次性跃迁
- **翻车经历比成功案例更值得写**：翻车暴露了认知边界，写出来比假装顺利更有价值
- **判断从经历里长出来**：不是先有判断再找案例撑，而是先有经历再慢慢形成判断
- **复盘是为了校准，不是为了表演**：复盘要追问"当时为什么这样想"，不是为了总结"我学到了什么"

**写文章时的影响**：
- 文章结构要展示认知螺旋，不能只展示终点判断
- 翻车经历要保留，不能删成完美阶梯
- 判断要交代来路，不能假装一开始就稳

## 画像使用规则

### 写文章时（new-or-review 联动）

1. **先读认知框架**：理解底层假设，决定文章推进方式
2. **结构选择**：
   - 知识观 → 文章要展示认知过程，不能直接端结论
   - 表达观 → 文章要有锚点，允许留白
   - 工程观 → 系统设计要展示 trade-off
   - 成长观 → 文章要展示认知螺旋
3. **风格校准**：风格偏好关联表达观，不是孤立规则
4. **素材调用**：经历是认知框架的证据，调用时要关联认知转折

### 改写时（humanizer-zh 联动）

1. **先读表达观**：理解为什么不喜欢某些表达方式
2. **判断改写方向**：
   - 删掉"回头看" → 因为成长感不该硬提醒
   - 删掉"本质上" → 因为不该假装全知
   - 保留作者语气 → 因为情绪可以有，但不能表演
3. **保留锚点**：改写不能删掉文章的认知来路

### 沉淀时（extraction-and-writeback 联动）

1. **优先抽象认知维度**：新信息先问"这反映了什么认知假设？"
2. **关联认知框架**：每条新判断都要标注来路
3. **分层沉淀**：
   - 认知框架更新 → PROFILE.md 顶层
   - 判断更新 → PROFILE.md 技术判断表
   - 经历更新 → PROFILE.md 经历部分 + 经历文件
   - 表现更新 → PROFILE.md 思考方式/风格偏好表
