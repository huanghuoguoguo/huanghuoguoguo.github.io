# 审稿与修订

## Review 的三部分

首稿完成后，`review` 由三部分组成：

1. 技术事实检查
2. 作者风格检查
3. 运行内置脚本做轻量信号扫描

## 技术事实检查

至少检查：

- 术语、原理、数据是否正确
- 前提是否支撑结论
- 没有数据支撑的地方是否写得过满
- 有没有超出作者知识边界的展开
- `frontmatter` 是否完整

## 作者风格检查

至少检查：

- 开头是不是具体问题或场景
- 有没有 benchmark、命令输出、syscall 计数、日志、ASCII 图、对比表
- 是否写出了 `V1 -> V2 -> V3` 的真实演化
- 句子是否过长、过于整齐
- 有没有允许闲笔和真实犹豫
- 结尾有没有强行升华
- 判断是不是从经历和观察里长出来的，而不是像讲师一样直接宣布“本质”
- 有没有“先有结论，再回头组织文章”的痕迹
- 有没有把原本带情绪的判断磨成过于中性、过于稳妥的审校腔
- 文中有没有至少几处只能由作者本人写出来的材料或语气
- 有没有哪一段一眼看去就像“这里其实应该先访谈，再写”
- 如果这篇借了样本文风，检查借来的是骨架还是人格：
  - 骨架可以借
  - 样本作者的职业身份、强行教学口吻、固定口头禅不能硬套

## 降 AIGC 改写

当内置脚本或人工 review 发现 AI 痕迹、模板味、过度平滑等问题时，调用 `humanizer-zh` skill 执行具体改写。

`humanizer-zh` 提供两轮策略：
- **第1轮**：打破模板化、消除欧式中文（弱动词+动名词组合、定语堆砌）、恢复动词核心
- **第2轮**：清除 AI 写作模式（夸大象征、宣传腔、三段式、否定式排比、模糊归因等）

调用方式：在 review 输出中标注"需要降 AIGC 处理"的段落，然后执行 `/humanizer-zh`。

注意：
- 每次调用只执行一轮，需新开对话继续下一轮
- 改写时不得新增事实、数据、案例或结论
- 必须保留原文术语、逻辑关系、编号结构
- 两轮之间禁止跳轮或逆序
- 自动回环最多2到3轮，之后交给人工裁决

## 运行内置脚本

内置脚本路径：`scripts/review_signals.py`（相对当前 skill 目录）

它只做可程序化的粗筛：模板词、过度总结、句长过匀、认知转折节拍、硬证据缺失等。它不判断作者人格、不替代人工 review，也不决定最终修改。

预期使用方式：

- 对已有文章或首稿运行脚本：

  ```bash
  py .claude/skills/new-or-review/scripts/review_signals.py path/to/article.md
  ```

  如果环境里的 `python` 命令可用，也可以把 `py` 换成 `python`。

- 读取 JSON 输出，重点关注段落级 `signals`
- 把命中项转成 revision plan
- 没有命中不代表没有问题；命中也不代表必须修改，必须结合作者画像和上下文判断

如果脚本返回以下类型信号，按下述方向理解：

- `cliche_phrase`
  - 套话或空洞过渡过多

- `forced_summary`
  - 段落或小节结尾过度收束

- `uniform_sentence_length`
  - 节奏太平，句长太匀

- `dash_usage`
  - 使用了破折号，容易把判断包装成解释腔或金句腔

- `perfect_ladder`
  - 叙事过于工整，不像真实迭代

- `missing_evidence`
  - 该有证据的地方没落地

- `author_fit_low`
  - 整体语气、选材或结构不像作者

- `need_interview`
  - 当前问题不是润色能解决的，应该回访作者补材料

- `narrative_tick`
  - 「后来才」认知转折在一篇文章中出现 3 次以上，形成了可预测的节拍器

- `hedge_habit`
  - 「按我现在的理解」「我更愿意把……理解成」等缓释判断出现超过 1 次，每次下判断都套缓冲层

- `cognitive_formula`
  - 「原来觉得 X → 后来发现不是那么回事 → 现在我理解 Z → 开放尾巴」这套公式重复出现，认知弧变成了流水线

- `no_hard_evidence`
  - 全文无命令行输出、无日志片段、无 benchmark 数据、无性能数字。对于技术深潜和系统设计类文章，这属于结构性证据缺失

## Review 输出

按严重程度排序输出：

- ❌ 事实错误 / 严重 AI 模板 / 严重风格偏差
- ⚠️ 逻辑问题 / 明显 AI 痕迹 / 证据不足
- 💡 风格优化建议 / 节奏优化建议

每条意见至少包含：

1. 具体位置
2. 问题类型
3. 为什么是问题
4. 修改方向
5. 必要时给出局部改写示例

最后补三项总结：

- 当前 AI 痕迹程度：轻 / 中 / 重
- 最优先处理的 2 到 3 个问题
- 文中最有人味、最该保留的段落和原因

## Revise 原则

修订只针对命中问题做局部修改。

禁止：

- 整篇重写
- 把真实细节删掉换成平滑空话
- 为了消掉脚本命中项把语气改得过于油滑
- 明明应该回访作者，却擅自用常识补完整段

## 按信号类型修订

### `cliche_phrase`

- 删除套话
- 直接进入具体观察、事实或动作

### `forced_summary`

- 删除节尾抽象总结
- 让段落停在具体细节、现象或疑问处

### `uniform_sentence_length`

- 拆长句
- 引入短句和停顿
- 不要求每段都匀称好看

### `dash_usage`

- 默认删除破折号
- 不要机械改成逗号
- 先判断破折号两边的语义关系：解释、转折、补充、列举，还是单纯造节奏
- 按上下文拆成两句、改成冒号、括号说明，或直接重写
- 如果破折号两边是在做漂亮解释，优先保留具体判断，删掉包装层

### `perfect_ladder`

- 恢复真实迭代过程
- 写出回退、犹豫、瓶颈迁移和未解决的问题

### `missing_evidence`

- 补命令、日志、数据、图、对比表
- 没证据就收窄结论，不强写

### `author_fit_low`

- 回到作者画像
- 收窄话题范围
- 换成更口语、更朴素、更像项目复盘的叙述方式
- 把“定义式判断”改成“认知来路 + 我的理解”
- 如果问题来自“像样本作者，不像作者本人”，保留结构推进，替换成作者自己的材料、迟疑和判断

### `need_interview`

- 停止继续润色
- 回到作者那里补场景、补认知来路、补边界
- 只在拿到作者回答后继续写

### `narrative_tick`

- 统计「后来才」「后来慢慢」「后来……才」的出现次数
- 超过 2 次：把其中至少一半改成具体触发事件（「翻 perf 输出才发现」「被主管问了一句才觉得不对」）
- 有些认知转折直接删掉「后来」，让新认知自己站住

### `hedge_habit`

- 统计「按我现在」「更愿意把」「更倾向于」的出现次数
- 超过 1 次：保留一处真正需要缓冲的，其余直接下判断
- 把空洞缓冲改成具体限定：「按我现在的理解」→「读了 OSTEP 之后我更相信」

### `cognitive_formula`

- 把文章的认知转折点列出来，检查是否共享同一套公式
- 至少有一处转折不打「原来」也不打「后来」——让两段认知并列
- 至少有一处转折用外部锚（数据、对话、报错）而非内部反思

### `no_hard_evidence`

- 技术深潜和系统设计类文章，至少包含一项：命令输出、日志片段、benchmark 表格、性能数字
- 没有就收窄结论，不强写
- 如果确实没有任何硬证据可用，在文中明确声明「手上没数据，以下是推断」

## Final Check

修订后再审一次，决定：

- `accept`
- `retry`
- `handoff_to_human`

停止条件满足任一条就不要继续自动回环：

- 高严重度问题已经清空
- 连续两轮改动后收益很小
- 修订开始损伤真实细节
- 文章已经“没 AI 味，但也没你味”
