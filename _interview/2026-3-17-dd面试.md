**Q: 线上问题怎么排查？**

明确关注的指标——CPU 打满、CPU 没满但 IO 满，等等。本质上是一个缩小范围的过程，先看现象再排除原因。

**Q: ThreadLocal 怎么解决内存泄漏？**

这个勉勉强答出来了。

> ThreadLocalMap 的 key 是弱引用，ThreadLocal 对象被 GC 后 key 变成 null，但 value 还在。如果线程一直活着（比如线程池），这些 null key 的 entry 就泄漏了。解决办法：用完主动调 `remove()`。ThreadLocalMap 在 `get()`/`set()` 时也会顺手清理 null key 的 entry，但不能完全依赖这个。

**Q: ThreadLocal 父线程放进去的缓存，子线程能拿到吗？**

我记得阿里有一个开源项目 TransmittableThreadLocal，就是为了让子线程拿到父线程的 ThreadLocal。那原生应该是拿不到的。

> `InheritableThreadLocal` 可以在创建子线程时拷贝父线程的值，但线程池场景下线程是复用的，创建时机不对，所以需要阿里的 TTL 来解决。

**Q: 线程池参数？**

忘了时间参数了。

> 七个参数：`corePoolSize`（核心线程数）、`maximumPoolSize`（最大线程数）、`keepAliveTime`（空闲线程存活时间）、`unit`（时间单位）、`workQueue`（任务队列）、`threadFactory`（线程工厂）、`handler`（拒绝策略）。提交任务的流程：先填核心线程，满了进队列，队列满了开到最大线程数，再满就走拒绝策略。

**Q: 线程池的队列选择有哪些？**

没怎么答上来。

> 常见的几种：`LinkedBlockingQueue` 无界（默认 `Integer.MAX_VALUE`），Executors.newFixedThreadPool 用的这个，容易 OOM；`ArrayBlockingQueue` 有界，需要指定容量；`SynchronousQueue` 不存元素，提交任务必须有线程立刻接走，Executors.newCachedThreadPool 用的这个；`PriorityBlockingQueue` 支持优先级排序。实际用的话一般选有界队列，避免任务堆积。

**Q: Redis 数据结构？**

> 五种基础类型：String、List、Hash、Set、Sorted Set（ZSet）。底层编码会根据数据量切换，比如 List 在元素少时用 listpack（以前是 ziplist），多了换 quicklist；ZSet 小的时候 listpack，大了用 skiplist + hashtable。另外还有几个不那么常提的：Bitmap、HyperLogLog、Stream、Geo。这也应证了，数据量小的时候，算法不如简单粗暴的便利。

**Q: Redis 存 Java 对象是怎么存的？**

只说了 JSON 序列化。

> Redis 存的就是字节流，选什么序列化方式都行。JSON 可读性好但体积大；Java 原生序列化方便但跨语言不行、体积也大；Protobuf/Kryo 体积小速度快，适合高性能场景。Spring 里通过 `RedisSerializer` 切换，常见选 Jackson 或 GenericJackson2JsonRedisSerializer。

**Q: Redis 大 Key？**

> 大 Key 指单个 key 的 value 特别大，比如 String 超过 10KB、集合类元素超过 5000 个。危害：读写阻塞（Redis 单线程）、网络带宽打满、内存不均（集群场景某个分片偏大）、删除时阻塞（`DEL` 大 key 是同步的）。排查用 `redis-cli --bigkeys` 或 `MEMORY USAGE key`。解决：拆分（hash 分桶、list 分段）、异步删除用 `UNLINK` 代替 `DEL`。

**Q: Redis SETNX 和 SET 的区别？**

> `SETNX` 只在 key 不存在时设置，存在则不做任何操作，返回 0。早期用来做分布式锁。`SET` 从 Redis 2.6.12 开始支持 `NX`、`EX`、`PX` 参数，一条命令搞定 set-if-not-exist + 过期时间，原子性有保证。现在做分布式锁推荐 `SET key value NX EX seconds`，不再单独用 `SETNX`。

**Q: MySQL 死锁？产生原因，排查思路？**

面试的时候还聊到了意向锁。一个事务要改某些行，先在表级别加意向排他锁（IX），再在行级别加排他锁（X）。意向锁之间是兼容的，多个事务改不同行不冲突。它的作用是让想加表级锁的事务能快速判断表里有没有行锁，不用逐行扫。

> 产生原因：两个事务互相持有对方需要的锁。典型场景是两个事务以不同顺序更新同一批行，或者间隙锁（Gap Lock）交叉。比如事务 A 锁了行 1 等行 2，事务 B 锁了行 2 等行 1。
>
> 排查：`SHOW ENGINE INNODB STATUS` 里的 LATEST DETECTED DEADLOCK 段会打印死锁的两个事务分别持有和等待的锁。也可以开 `innodb_print_all_deadlocks` 把每次死锁都写进 error log。拿到信息后看是哪两条 SQL、锁的哪些索引，调整加锁顺序或缩小事务范围。

**Q: MySQL 索引失效的情况？**

> 常见的几种：对索引列用函数或运算（`WHERE YEAR(create_time) = 2026`）；隐式类型转换（varchar 列用数字查）；`LIKE` 左模糊（`LIKE '%abc'`）；`OR` 连接的条件中有非索引列；联合索引不满足最左前缀；`NOT IN`、`!=` 在某些情况下优化器放弃索引；数据量太小优化器觉得全表扫更快。

**Q: 在 OB 的代码量是多少？**

**Q: Rust 比 Python 快 5 倍的原因是什么？**

> Rust 编译成机器码，零成本抽象，没有 GC 停顿，内存布局紧凑（无对象头、无装箱）。Python 是解释执行，有 GIL 限制多线程并行，动态类型每次操作都要类型检查和间接寻址。5 倍其实算少的，计算密集场景差距能到几十倍。

聊到了线程和协程的区别。

> 线程是 OS 调度的，切换要陷入内核、保存寄存器和栈，开销在微秒级。协程是用户态调度的，切换只需要保存少量寄存器和栈指针，开销在纳秒级。线程栈默认 MB 级别，协程栈可以很小（几 KB）。Go 的 goroutine、Python 的 asyncio、Rust 的 async/await 都是协程模型。

**Q: AI Bot 开源框架的工作？**

**Q: 向量搜索如何提高回答准确率？**

> 几个方向：query 改写（扩展同义词、拆分子问题）；embedding 模型选好的或针对领域微调；chunk 策略调优（大小、重叠、按语义切分）；混合检索（向量 + BM25 关键词检索再 rerank）；加 reranker 对召回结果二次排序；元数据过滤缩小检索范围。

**Q: 向量检索为什么比关系型数据库检索快？**

> 不是所有场景都快，是在语义相似度搜索场景下快。关系型数据库做语义搜索要么全表扫算相似度，要么根本做不了。向量检索用 ANN 索引（HNSW、IVF 等），把高维空间划分成区域，查询时只搜附近的区域，时间复杂度从 O(n) 降到 O(log n) 级别，用精确度换速度。

**Q: 向量能不能精确查找？**

> 能，暴力搜（Flat 索引）就是精确的，算每个向量的距离取 top-k。但 O(n) 复杂度，数据量大了扛不住。ANN 索引牺牲一点精度换速度，recall 通常能做到 95%+ 但不是 100%。如果要精确匹配某个向量，可以把向量哈希后当普通索引用，但这就不是相似度搜索了。

面试官评价：还不错。有一个建议——做压测或性能优化的时候，要建立结构化的关系网格（类似知识图谱），建立关系系数，这样才有直观的工作体现。
