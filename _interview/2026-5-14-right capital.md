## 一面

### 大模型与 Agent 架构

**Q: 现在大模型能力越来越强，外部那层东西是不是会越来越被吸收掉，你怎么看待这个？**

> 面试官指的是 agent harness——把 LLM 变成 agent 的那个执行环境。学术上定义为六元组 H = (E, T, C, S, L, V)：执行循环、工具注册、上下文管理、状态存储、生命周期钩子、评估接口。
>
> 核心观点是：**harness 承担了大部分工作，LLM 的"真实责任"是稀疏的、有边界的**。有研究表明，仅改变工具格式（模型不变），SWE-bench 通过率可以从 6.7% 跳到 68.3%——这说明 agent 的能力很大程度上由 harness 决定。
>
> 回到问题：随着模型能力增强，harness 会被"吸收掉"吗？我觉得边界会移动，但 harness 不会消失。模型的进步会让某些组件变简单（比如规划能力增强后，执行循环可以更轻量），但工程化的问题——状态持久化、容错恢复、安全沙箱、协议互操作——仍然需要外部架构来承载。**"The harness is the chassis; the model is the engine"**——引擎再强，也得有底盘。

**Q: Java 里是不是也有 Copy-on-Write 的概念？**

**Q: Java 控制反转（IoC）、依赖注入。具体举例什么是 IoC，什么时候用？**

> IoC 的核心是"控制权反转"——对象的创建和依赖管理从代码内部移到了外部容器。比如传统的 `new Service(new Dao())` 是自己管理依赖；用 Spring 后只需要 `@Autowired`，容器负责创建和注入。
>
> 使用场景：对象之间有复杂依赖关系时；需要解耦方便单元测试时；需要统一管理对象生命周期时（单例、原型等）。

**Q: AOP 是怎么实现的，有没有具体的业务举例？什么时候用？**

> AOP（面向切面编程）通过代理实现，JDK 动态代理（接口）或 CGLIB（类）。典型场景：事务管理（`@Transactional`）、日志记录、权限校验、性能监控。比如在 Service 层方法上加了 `@Transactional`，Spring 会生成代理对象，在方法执行前后自动开启和提交/回滚事务，业务代码不需要感知。

### 数据库

**Q: 用过什么数据库？**

**Q: Redis 用过什么数据结构？Set 是什么，ZSet 是什么，有什么区别？**

> 五种基础类型都用到过。Set 是无序集合，元素唯一，支持交集、并集、差集操作，适合标签系统、共同好友等场景。ZSet（Sorted Set）是有序集合，每个元素关联一个 score，按 score 排序，适合排行榜、延时队列等场景。底层实现上，Set 小数据量用 listpack，大的用 hashtable；ZSet 小的用 listpack，大的用 skiplist + hashtable。

**Q: 缓存穿透、击穿、雪崩？**

> **穿透**：查询不存在的 key，请求直接打到数据库。解决：布隆过滤器、空值缓存。
>
> **击穿**：热点 key 过期瞬间，大量请求同时打到数据库。解决：互斥锁、热点数据永不过期。
>
> **雪崩**：大量 key 同时过期或 Redis 宕机。解决：过期时间加随机值、高可用架构、限流降级。

**Q: MySQL 存储结构是什么样的，主键索引和唯一索引的区别？**

> InnoDB 存储结构是 B+ 树，数据文件本身就是主键索引（聚簇索引），非主键索引的叶子节点存的是主键值。
>
> 主键索引：聚簇索引，叶子节点存完整行数据，一个表只能有一个。
>
> 唯一索引：辅助索引，叶子节点存索引列值 + 主键值，可以有多个。查询时先走唯一索引拿到主键，再回表查数据（覆盖索引除外）。

**Q: B+ 树是怎么存储数据的？**

> B+ 树是多路平衡搜索树。特点：非叶子节点只存键值和指针，不存数据；叶子节点存数据，且通过双向链表连接（范围查询友好）；树高度低（通常 3 层就能存千万级数据），磁盘 IO 次数少。InnoDB 中，页大小默认 16KB，每个节点对应一个页。

**Q: HTTP GET 和 POST 的区别，携带的参数是怎么携带的？**

> GET 参数拼在 URL 后面（Query String），有长度限制，会被浏览器缓存、记录在日志中，适合幂等的查询操作。POST 参数放在请求体里，没有长度限制，不会被缓存，适合非幂等的修改操作。从语义上讲，GET 是获取资源，POST 是提交数据。

**Q: Cookie 和 Session？**

> Cookie 存在客户端，每次请求自动带上。Session 存在服务端，用 Session ID 关联用户数据。Cookie 可设置过期时间，Session 默认关闭浏览器失效（取决于服务端配置）。Session ID 通常通过 Cookie 传递，禁用 Cookie 时可以用 URL 重写。

**Q: Cookie 是怎么携带信息的，鉴权是怎么鉴权的？**

> Cookie 在 HTTP 响应头 `Set-Cookie` 中设置，后续请求自动在 `Cookie` 头中携带。鉴权流程：用户登录后服务端生成 Session ID 写入 Cookie，后续请求服务端根据 Cookie 中的 Session ID 查找 Session，验证用户身份和权限。

**Q: 如果是 Session 呢？**

**Q: JWT 怎么实现的？**

> JWT 由三部分组成：Header（算法类型）、Payload（用户信息）、Signature（签名）。服务端用密钥签名后返回给客户端，客户端后续请求在 Authorization 头携带，服务端验签即可，不需要存储 Session。优点是无状态、跨服务通用；缺点是无法主动失效（需要黑名单机制）、Payload 不加密（敏感信息不能放）。

**Q: HTTP 状态码了解吗？2xx，1xx？**

> 1xx：信息性状态码，如 100 Continue（继续发送请求体）。
>
> 2xx：成功，如 200 OK、201 Created、204 No Content。
>
> 3xx：重定向，如 301 永久重定向、302 临时重定向、304 Not Modified。
>
> 4xx：客户端错误，如 400 Bad Request、401 Unauthorized、403 Forbidden、404 Not Found。
>
> 5xx：服务端错误，如 500 Internal Server Error、502 Bad Gateway、503 Service Unavailable。

### Git 与 Linux 基础

**Q: git reset 有几种模式？**

> `git reset` 有三种模式：
>
> - `--soft`：只移动 HEAD 指针，暂存区和工作目录不变。相当于撤销 commit，但改动还在暂存区，可以重新 commit。
>
> - `--mixed`（默认）：移动 HEAD 指针，重置暂存区，工作目录不变。相当于撤销 commit + unstage，改动变成未暂存状态。
>
> - `--hard`：移动 HEAD 指针，重置暂存区和工作目录。彻底回退，所有改动丢失（除非有 reflog 可以救）。
>
> 忘掉 `--mixed` 了，不加参数时就是这个。

**Q: 从别的分支 copy commit 过来是什么命令？**

> `git cherry-pick <commit-hash>`。把指定 commit 的改动应用到当前分支，会生成一个新的 commit。
>
> 批量 cherry-pick：`git cherry-pick <start>^..<end>`（闭区间）。

**Q: 现在用的 Mac 还是 Win？Linux 发行版是什么？**

> 开发环境是 Windows 和 Ubuntu，但对不同发行版之间的区别不太了解。

**Q: 文件 read、write、exec 权限是什么？**

> Linux 文件权限用三组 rwx 表示：所有者、所属组、其他人。
>
> - `r`（read）：读取权限，数值 4。对文件是可以读取内容，对目录是可以 ls 列出内容。
> - `w`（write）：写入权限，数值 2。对文件是可以修改内容，对目录是可以创建/删除文件。
> - `x`（exec）：执行权限，数值 1。对文件是可以运行，对目录是可以 cd 进入。
>
> 常见的 755 = rwxr-xr-x（所有者全权限，其他人可读可执行），644 = rw-r--r--（文件默认权限）。
