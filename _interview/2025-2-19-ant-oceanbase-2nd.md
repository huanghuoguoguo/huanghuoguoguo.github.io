## 中科曙光告警指标开发的技术难点

**Q: 中科曙光中关于告警指标的开发，有什么技术难点？**

这个问题没有回答到点子上——告警开发的技术难点在哪里，没有说清楚。

### 项目背景

作为集群的存储器服务器项目，用户需要能够在机器出现异常或者存储快要用光的情况下得到主动通知。

### 技术架构

- 依赖公司内部开发集成的一致性 KV 缓存层
- 调用 Runtime 的 Python 脚本做信息采集，放入缓存层以及持久化的 KVDB 进程
- 告警框架中通过分析数据流的形式，把注册的、通过 XML 注册的告警指标一一进行分析和计算
- 框架定义了计算流程、级联告警、告警优先级、消警过程等
- 使用者只需要编写 XML 文件即可
- 通过**责任链模式**封装了几种情况下的告警消警模式代码，如果有自定义需求只需要继承指定基类

### 技术难点

对我当时感知最大的就是**框架怎么写，怎么抽象各种行为，用设计模式规定执行流程，组织和编码成正常工作的形式**。

### 反思与不足

目前来看，我认为它可能还有一些不足：

- 它是一个定时任务，在数据采集行为之后进行计算和通知，可能有些行为不能及时通知
- 如果 Java 进程挂了，没有其他手段

## TWorkJob 框架

**Q: TWorkJob 框架，也是实习的时候做的吗？**

答：国产服务器产品，所以尽可能减少外部依赖，写了很多业务友好的框架。包括一致性 KV 缓存层、KVDB、告警框架、Job 框架、EventBus 框架。

在我实习过程中，有很大一部分时间是在学习项目内部的框架实现。我认为这个 Job 框架是其中比较有意义的一部分。

### 框架功能

- 接受即时任务的提交
- 规定任务粒度，DAG 的组合任务
- 提供任务重试、告警、降级
- 提供定时任务的提交，XML 方式配置，自动提交、手动提交、执行回调等

在我所能理解的地方我都进行了复现。我认为可能因为项目的迭代，它在某些部分有些臃肿，但是不可否认它的正确性以及高效。我对部分代码做了我认为更简单的形式。

### 状态管理问题

它其实不是一个无状态的任务调度框架，或者说这个 Java 进程的集群也不是无状态的。很多模块依靠的是一致性 KV 缓存层 Resource，但这个 Resource 只有部分元数据是通过 Raft 同步的，其他数据只和本集群内部的几个控制器有关。

比如说，假如用户要操控风扇的转速，并不是对一个集中式的持久化数据做操控，而是做请求转发，转发到指定的机器上再操作。

### 我的改进思路

对于我们认为的后端项目，为了贴合"云开发"这个概念，肯定都是希望数据和任务都是无状态的，存储和计算分离，也就是说在任何一个节点上执行都应该是对等的。所以我也做了这方面的调整。

其中很多实现概念我都是参考它的设计，只是在某些方面，比如锁，我都会尝试替换为一致性的分布式锁等。然后怎么做重试、怎么做隔离、怎么做回调、怎么做负载均衡，都是我需要考虑的。

### 高可用参考

在高可用这个内容上，参考了非常多的开源项目：

- **XXL-Job**：功能最接近，架构是调度器集群，通过 MySQL `FOR UPDATE` 获取锁，然后执行一次调度，分派任务给执行器。几乎是服务于定时任务的。
- 但我在调度框架中还希望有即时的单次任务，用这样的架构可以吗？
  - 思考一下也不是不可以，将任务的提交都放到另一个可靠的持久层，做 Queue，然后谁获取到锁谁就调度一次任务。
  - XXL-Job 是一次调度一个时间周期内的任务，把任务放到时间轮上进行调度。
  - 即时任务的下一次执行时间就是现在，获取任务的形式不是从 MySQL 数据库中扫描一遍然后计算下一次执行时间，而是直接递交到队列中，从队列中获取任务。
- 思考来思考去不是很好，以我的水平还是先考虑别的，暂时用 Redis 代替了。

### 元数据管理方案

- **HDFS、RocketMQ、Redis 哨兵、OceanBase**：将元数据放到一个额外的集群中管理，这可以，但可能比较复杂。
- **Redis Cluster、Kafka**：通过 Raft、KRaft 选举出一个节点存储元数据。
- **我的方案**：直接在 Redis 中存储，弱化这方面的实现。

### 理论学习

参加完比赛之后看了一本书叫《设计数据密集型应用》（DDIA），也非常感谢 OB 比赛以及实习，让我在看的过程当中对大部分名词不再感到非常陌生。然后我对之前的项目和比赛又有新的想法等。总之这也是践行了我当时对分布式理论、数据密集型等的一些设计。

## 测试框架设计

**Q: 如果让你现在开发一套测试代码、测试框架，来验证这个任务调度框架的能力，会有哪些想法，或者大概怎么设计？**

### 测试流程定义

既然是测试框架的话，首先需要定义框架执行的流程：

1. **环境准备**
2. **提交执行**
3. **验证断言**

### 分布式任务调度的特殊考虑

针对我的分布式任务调度框架，需要做一些其他的调整：

- **环境准备**：需要先保证能够正确拉取，元数据维护正常，主节点和从节点能够相互感知
- **提交执行**：需要验证任务是否被正确提交
- **验证断言**：任务是否按照预期在预期节点被执行

### 可观测性

Job 框架还需要向外提供查询状态的信息，提供一些查询能力供外部查询不可变的信息。

### 分层验证

这需要验证多个阶段的能力：

1. **单元测试**：每个部分其实是解耦的——分布式锁、任务调度、任务元数据存储、重试、任务分发、通知等。在代码层面可以先做细粒度的代码覆盖测试，验证单元测试是否符合预期。

2. **集成测试**：验证服务起来之后任务调度的能力。需要一个统一的输入输出、日志追踪、监控检查。
   - **输入**：一个一个任务
   - **输出**：选择操作数据库的任务，去操作一个一致性保证的存储，这样才能体现多任务的形式

3. **验证点**：
   - 是否能够正确提交
   - 是否能够被主节点接受
   - 是否能够被调度
   - 是否按预期进行负载均衡
   - 是否按照预期落盘
   - 是否正常结束
   - 是否正常告警
   - 是否正常隔离

4. **实现方式**：依靠 trace 追踪、验证，然后由编写测试用例的人指定在多少时间之后进行验证操作，然后给出断言。

也就是：**提交 → 执行 → 验证 → 断言**。
