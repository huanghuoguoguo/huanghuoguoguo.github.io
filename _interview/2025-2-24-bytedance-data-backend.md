## 后端应用的性能、可靠性、可扩展性

**Q: 在实习中如何确保后端应用的性能、可靠性和可扩展性？**

这个问题答得比较散，现在复盘可以整理得更系统：

### 性能

- **缓存一致性设计**：项目中有一致性 KV 缓存层，底层是 `ConcurrentHashMap`，通过 Raft 协议同步元数据。缓存层的设计要考虑业务场景——读多写少，所以用性能更友好的数据结构，比如不可变 `List`、邮戳锁（Ticket Lock）、多维 Hashtable。

- **GC 事故复盘**：实习中遇到过一次 GC 问题。没有用池化思想，创建了大量临时对象又频繁回收，导致应用假死，GC 线程占满 CPU。排查手段是用内存转储工具（MAT/JProfiler）分析堆内存，发现某一类对象占比异常。解决方式是引入对象池，复用对象。

- **日志优化**：项目中有大量定时任务，日志输出频繁。优化点是减少不必要的日志打印、压缩日志级别、减少日志中的字符串拼接（避免不必要的内存分配）。

### 可靠性

- **HA 机制**：第三方 C++ 进程负责 HA 拉起，主进程是 Java。C++ 进程监控 Java 进程状态，异常时自动重启。

- **幂等性**：坦白说，这块没做。

### 可扩展性

- **模块化设计**：模块之间的交互主要用内存中的 EventBus 框架实现。将上下游的显式调用改为事件驱动通知，被调用方通过注册事件监听，用发布订阅模式减少耦合。

> 现在回头看，这个回答的问题在于：说得太散，没有主线。应该先说结论——"我们主要通过事件驱动架构降低耦合，通过缓存和对象池优化性能"，再展开细节。

## 蚂蚁金服数据库比赛

**Q: 简单介绍一下蚂蚁金服数据库比赛？**

**Q: 初赛阶段你做了什么？**

答：存储引擎的设计和实现、词法语法分析、SQL 层逻辑算子的实现和优化。

**Q: 国赛阶段的索引优化？**

答了简历上的内容，基于 IVF-HNSW 的向量索引优化。

**Q: SQ4 是什么意思？为什么说它有损？能带来多少提升？**

答：SQ4 是将 0-128 的值域映射至 0-16，损失了一部分精度，但减少了数据量，提高了带宽和运算速度。

**Q: QPS 从 1400 提升到 1600，召回率下降多少？**

答：大概 70%。（现在想想，这个数据其实没准备好，应该查一下当时的实验记录）

**Q: 怎么优化召回率？**

答：在 SQ8 的基础上重排（re-ranking）。

**Q: SIMD 指令解释，实现时用什么语言？**

答：单指令多数据流，用 C++ 实现。能减少指令数量，提高带宽利用率。

> 其实还可以补充：缓存预取（Cache Prefetching）减少 IO 冲突和延迟、多核绑定、流水线优化。当时课程作业还做过数据结构优化——小 limit 时用固定大小数组，类似 Redis QuickList，降低结构复杂度。但这些都没展开说。

## 项目一：高并发秒杀系统

**Q: 介绍一下在高并发场景下如何提升系统性能？**

答：

- **限流方案**：本地限流 + Redis 限流
- **调用链路优化**：同步改异步，非必要的操作交给线程池或消息队列
- **IO 优化**：IO 多路复用、Netty、NIO，或换性能更高的 Web 容器（如 Jetty）
- **CDN 和前端缓存**：静态资源走 CDN，减少服务器压力
- **数据库优化**：索引优化、查询优化
- **锁粒度细化**：通过 Hash、Range 分片，将操作负载均衡，降低锁粒度
- **任务分发**：耗时任务转发到性能更强的机器

> 其实还可以补充：业务场景细分（单库存 vs 多库存的秒杀方案可以不一致）。

## 进程间通信

**Q: 多线程和多进程的区别？不同进程之间的内存能直接访问吗？如何进行进程间通信？**

答：

- 进程间内存不能直接访问，需要借助 IPC 机制。
- 进程间通信方式：管道、Socket、消息队列、共享内存、信号、文件映射、事件、条件变量。
- 实际接触最多的是第三方进程（如 MQ、Redis），通过 Socket 通信，用无状态介质（如 JSON）进行进程间通信。

> LCS 是经典 DP 问题，`dp[i][j]` 表示 text1[0:i] 和 text2[0:j] 的最长公共子序列。状态转移方程：
> ```
> if text1[i-1] == text2[j-1]:
>     dp[i][j] = dp[i-1][j-1] + 1
> else:
>     dp[i][j] = max(dp[i-1][j], dp[i][j-1])
> ```
