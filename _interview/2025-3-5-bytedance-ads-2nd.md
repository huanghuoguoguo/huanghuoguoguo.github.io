## 告警框架细节

**Q: 解释告警框架的细节、结构。**

### 架构

- 依赖公司内部开发集成的一致性 KV 缓存层
- 调用 Runtime 的 Python 脚本做信息采集，放入缓存层以及持久化的 KVDB 进程
- 告警框架中通过分析数据流的形式，把注册的、通过 XML 注册的告警指标一一进行分析和计算
- 框架定义了计算流程、级联告警、告警优先级、消警过程等
- 使用者只需要编写 XML 文件即可
- 通过**责任链模式**封装了几种情况下的告警消警模式代码，如果有自定义需求只需要继承指定基类

### 执行流程

1. Python 脚本采集指标数据
2. 写入一致性 KV 缓存层
3. 告警框架读取数据流
4. 按 XML 注册的指标进行分析和计算
5. 触发告警/消警

## 一致性 KV 缓存层 Resource

**Q: 解释一致性 KV 缓存层 Resource 细节。**

答：它是一个分布式的 KV 缓存层，用于存储元数据和控制器状态。通过 Raft 协议保证多副本之间的一致性。

**Q: 面试官好像不太理解我阐述的是一个多领导者的架构，有并发写的问题。**

这是我的表述问题——没有把架构背景说清楚。

实际情况是：

- 系统中有多台机器，每台机器上有一个控制器
- 用户可以操控风扇转速等操作
- 这些操作不是对集中式持久化数据操作，而是请求转发到指定机器上执行

## 核心拷问

**Q: "我想就是归本溯源的，其实我想知道的是，这个一致性 KV 缓存，到底它的背景以及它的意义是什么？为什么要有这个一致性 KV 缓存？能干什么事情，一致性的以及跟谁的一致性？"**

这个问题直击要害，我当时没有回答清楚。

### 应该这样回答

**背景**：

- 分布式集群环境，多节点部署
- 需要存储元数据（如租户信息、资源配置）和控制器状态
- 多个节点可能并发读写

**意义**：

- 保证元数据在多节点之间的一致性
- 避免并发写冲突
- 提供高可用（节点故障时其他节点可接管）

**一致性对象**：

- 元数据（通过 Raft 强一致同步）
- 控制器状态（部分本地状态，不通过 Raft）

## 缓存更新策略

**Q: 如果只是为了不落 KVDB，为什么不更新之后删除缓存，再重新加载，而是通过 Raft 同步？**

这个问题问到了点子上——Cache-Aside、Write-Through、Write-Behind 等缓存策略的选择。

### 当时没答上来

现在想想：

- **删除缓存 + 重新加载**：适用于读多写少，缓存命中率高的场景
- **Raft 同步**：适用于多节点并发写，需要强一致性的场景

我们选择 Raft 同步是因为：

1. 有并发写需求，不是纯读场景
2. 元数据需要强一致性，不能接受短暂不一致
3. 多节点部署，需要高可用

**Q: 解释控制器是什么？**

## 写冲突解决

**Q: 怎么解决的写冲突？怎么用 Raft 解决写冲突的？**

Raft 通过以下机制解决写冲突：

1. **Leader 选举**：同一时刻只有一个 Leader 能接受写请求
2. **日志复制**：Leader 将写操作复制到 Follower
3. **多数派确认**：超过半数的节点确认后提交
4. **线性化语义**：所有写操作有全局顺序

### 我的表述问题

我没有说清楚 Raft 的具体机制，只是笼统地说"通过 Raft 同步"。

## 多主 vs 单主

**Q: 为什么要多主？单主不行吗？**

这个问题问的是架构选择的权衡。

### 多主的优势

- **高可用**：单主故障时，其他主可以继续服务
- **负载均衡**：写请求可以分散到多个主
- **低延迟**：就近写入，减少跨机房延迟

### 多主的代价

- **冲突解决**：需要处理并发写冲突
- **复杂性**：需要一致性协议（如 Raft）

### 单主的优势

- **简单**：没有冲突问题
- **强一致**：天然线性一致

### 单主的代价

- **单点故障**：需要额外的 HA 机制
- **性能瓶颈**：所有写请求集中到一个主

### 应该这样回答

我们选择多主是因为：

1. 分布式集群环境，需要高可用
2. 有并发写需求，单主性能不足
3. 通过 Raft 保证一致性，牺牲一定性能换取可用性

## LIF 接口、漂移、自动路由

**Q: LIF 接口、漂移是什么？自动路由等。**

这个问题涉及到存储系统的具体实现，我当时答得不好。

### LIF（Logical Interface）

逻辑接口，用于客户端访问存储集群。它可以漂移（Failover），当节点故障时，LIF 漂移到其他节点。

### 漂移

当节点故障时，将 LIF 和控制器漂移到健康节点，实现高可用。

### 自动路由

客户端请求自动路由到正确的节点，无需感知底层拓扑。

## 手写代码

### 快排

```cpp
void quickSort(vector<int>& nums, int left, int right) {
    if (left >= right) return;

    int pivot = nums[right];
    int i = left - 1;

    for (int j = left; j < right; j++) {
        if (nums[j] < pivot) {
            i++;
            swap(nums[i], nums[j]);
        }
    }
    swap(nums[i + 1], nums[right]);

    quickSort(nums, left, i);
    quickSort(nums, i + 2, right);
}
```
