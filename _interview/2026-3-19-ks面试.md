## 一面

**Q: 在 OB 的压测工具的背景，怎么做的？是功能测试还是性能测试？**

**Q: 常用的数据结构有哪些？**

讲了一些，然后提到 OB 国赛用到堆。

**Q: 什么时候用堆？**

**Q: 设计一个业务场景——压测时记录延时指标，要支持多线程写入和时间窗口读取（某个时间段的最小值、最大值、平均值、P99 等），怎么设计？**

从业务需求分析，要支持高并发顺序写入和顺序读取，数组和链表是合适的基础结构。但数组会面临扩容，链表因为物理不连续会 cache miss，所以考虑二者结合。

和列存数据库类似，每个页面（其实是桶）是多个数组的集合，每个数组内支持顺序写入和顺序读取。同时页面内可以记录预计算的指标（最大值、最小值等），读取时直接拿到聚合结果。

页面用跳表的形式组织，提供按时间顺序的快速访问。

虽然没讨论持久化，但如果需要的话可以考虑实现一个简单的 LSM tree。总的来说，这是一个顺序并发写入加时间窗口读取的问题，参考列存数据库（时序数据库）即可。

> opus复盘：整体思路是对的，chunked array + 预聚合的方向没问题，和时序数据库（InfluxDB、DuckDB）的设计思路一致。但有几个点值得推敲：
>
> 1. **P99 算不出来**。页面内预计算 min/max/avg 没问题，但 P99 不能从各页面的局部统计值合并得到。要算 P99 要么保留原始数据排序，要么用近似算法（t-digest、DDSketch）。这是回答里最大的漏洞——面试官问的指标里明确包含 P99，但这个设计没法支持它。（那就摁算）
> 2. **跳表不是最优选择**。页面本身是按时间顺序追加写入的，天然有序，用数组/vector 存页面指针然后二分查找时间窗口就够了，O(log n) 且缓存友好。跳表的优势是并发插入时不用全局锁，但页面粒度已经足够粗了，页面级别的并发用简单的 append + atomic index 就能搞定。
> 3. **多线程写入的并发控制没展开**。这是面试官问的核心之一。比较常见的做法是 per-thread buffer（每个线程写自己的 buffer，读的时候 merge），避免写路径上的锁竞争。"多个数组的集合"如果就是这个意思，应该说得更明确。（但是我觉得，写每个页面里的，用atomic index就够了，毕竟是顺序写入）
> 4. **LSM tree 的类比有点跳**。LSM 解决的是"写多读少 + 持久化"，核心是 WAL + memtable + compaction。这个场景是纯内存的指标收集，即使要持久化，直接 mmap 或 append-only file 就够了，不需要 LSM 的 compaction 机制。面试场景下提这个容易被追问细节。
>
> 总结：大方向（chunked + 预聚合 + 类时序数据库）是好的，但 P99 这个点是硬伤，补上 t-digest/DDSketch 的思路。并发控制可以更具体地说 per-thread buffer。

**Q: C++ 有没有遇到过内存问题？**

**Q: 智能指针，shared_ptr 底层有了解过吗？**

猜测是计数，归零时由最后一个线程释放。

> `shared_ptr` 内部维护一个引用计数块（control block），包含 strong count 和 weak count。每次拷贝 `shared_ptr` strong count +1，析构时 -1，降到 0 就释放管理的对象。计数的增减在单线程下是普通操作，多线程下用原子操作（`std::atomic`）保证线程安全。

**Q: 如果是多线程呢？**

这里没回答上来，其实应该说猜测是 CAS。

> 引用计数本身的增减是原子的（用 CAS 或 `fetch_add`），所以多个线程同时拷贝/销毁同一个 `shared_ptr` 的不同副本是安全的。但注意：同一个 `shared_ptr` 对象被多个线程同时读写（比如一个线程赋值、另一个线程读）不是线程安全的，需要加锁或用 `std::atomic<shared_ptr>`（C++20）。

**Q: 一个进程，启动运行加载进内存，内存布局有了解吗？**

> 从低地址到高地址：text 段（代码）、data 段（已初始化全局变量）、bss 段（未初始化全局变量，加载时清零）、heap（向上增长，malloc 分配）、内存映射区（mmap、共享库）、stack（向下增长，函数调用帧）。最顶上是内核空间，用户态不可访问。

**Q: 比如说 `const char *` 一个字符串？**

> 字符串字面量存在 text 段（或 rodata 段），`const char *` 指针本身根据声明位置决定——局部变量在栈上，全局变量在 data/bss 段。指针指向的内容是只读的，试图修改会 segfault。

**Q: malloc 这些申请内存的操作成本比较高，面对这样的情况有没有用过什么解决方法？**

这里其实之前探索过 malloc（还有一些开源的 allocator，比如 jemalloc、tcmalloc，内部是有一些缓存机制的），就以 Java Integer 缓存为例，简单讲了一下享元设计模式。

> 几个方向：内存池（预分配一大块，自己管理分配/回收，避免频繁 syscall）；对象池/享元模式（复用对象，减少分配次数）；换 allocator（jemalloc、tcmalloc 用线程本地缓存减少锁竞争和系统调用）；arena 分配（一批对象一起分配、一起释放，适合生命周期相同的场景）。本质上都是减少向 OS 要内存的次数。
