## 一、"任务完成"正在撒谎

2026 年 3 月，一篇论文对当前主流 Agent 评测范式做了一次直接拆穿：

研究者在 58 条 Agent 运行 trace 中，选出了所有被打分为"完全成功（reward=1.0）"的样本。然后对这些"成功"的运行做了过程审查——结果发现，其中 83% 含有隐性的程序性违规：Agent 取得了正确的最终结果，但走的路是错的，用的工具序列是有问题的，或者中途的推理与行动之间存在裂缝。

这不是个案。同年更早发表于 NeurIPS 2025 的论文对多 Agent 系统做了系统性的失败分类分析，统计出：步骤重复（Step Repetition）是占比最高的单一失败模式，达 17.14%；推理 - 行动不匹配（Reasoning-Action Mismatch）紧随其后，占 13.98%。

这两类失败加起来，覆盖了所有 Agent 失败的将近三分之一。而它们有一个共同特征：几乎不可能被最终输出检查捕获。

Agent 走错了路，但最后到了终点——你的测试通过了。Agent 说要做 A，但实际执行了 B——你的测试通过了。Agent 在第三步陷入死循环，消耗了三倍的 token，然后勉强得出了正确答案——你的测试也通过了。

这就是 Gartner 在 2025 年 6 月预测"超过 40% 的 Agent 项目将在 2027 年前被叫停"时，所指向的核心问题之一：不是模型不够好，而是团队根本没有基础设施来知道模型在哪里不够好。

"任务完成率"是一个危险的指标——它告诉你 Agent 到达了终点，但隐藏了它是怎么到达的。

## 二、为什么 Agent 的评估本质上比 LLM 难一个维度

评估一个 LLM 的逻辑是清晰的：给输入，拿输出，和参考答案比对，打分。输入输出是确定的，评估是一次性的，错误是即时可见的。

Agent 打破了这个逻辑的每一个假设。

1. 行为是跨步骤的，不是单次的：Agent 的一次"回答"可能包含十几次 LLM 调用、五次工具调用、三次状态读写。错误可以发生在任何一步，并向后传播。只看最终输出，相当于只看飞机在哪里坠毁，而不问飞行过程中哪个仪表出了问题。
2. 同一个任务有多条合法路径：两次运行可以采用完全不同的工具序列，最终产出等价的结果——你的 eval 不能因为路径不同就判定失败。但也不能因为结果相同就忽视路径的质量差异——一条路消耗了三倍 token，走了两次弯路，这是需要被捕获的信号。
3. 错误会复利：在一个普通 LLM 应用里，一次坏的输出就是一次坏的输出，边界清晰。在 Agent 系统里，第 2 步的一个错误参数，到第 8 步已经污染了整条推理链——而你的最终输出检查对此毫无感知。

这三点加在一起，意味着 Agent 评估需要一套完全不同的思维框架。

## 三、三层 Eval，每一层看见不同的失败

当前工程实践里逐渐收敛出的一个共识是：生产级 Agent 需要三个层次的 eval 同时运行，任何单层都无法独立覆盖全部风险。

### 第一层：结果评估（Black-box Eval）

最直接的一层：Agent 完成任务了吗？最终状态是否正确？

这是大多数团队唯一在做的评估，也是上文提到的"成功率谎言"的根源。它不是没用——结果正确终究是第一目标——但单独依赖它，你只是在检查飞机有没有坠毁，而不是在维护飞行安全。

结果评估适合作为快速过滤层：明显失败的 run 先通过它筛掉。但它无法告诉你"成功的 run 里有多少是不安全的成功"。

### 第二层：轨迹评估（Trajectory Eval）

轨迹评估评的不是"最终答案对不对"，而是"Agent 走的路对不对"：

1. 工具选择是否正确？有没有用了不该用的工具，或者漏用了必要的工具？
2. 工具调用参数是否正确？有没有传入错误的格式或错误的值？
3. 步骤顺序是否合理？有没有在依赖项完成之前就执行了下游操作？
4. 有没有步骤重复？有没有在没有新信息的情况下重新调用了相同的工具？
5. 推理链和执行动作是否一致？Agent 说"我要查询用户历史记录"，然后实际调用的是什么？

一个实用的起点：为每个核心任务定义一份"期望轨迹"——包括期望的工具调用序列、关键参数的期望格式、以及必要的步骤顺序约束。

轨迹 eval 的任务就是检查实际运行与期望轨迹的偏差。

### 第三层：元评估（Meta-Eval）

元评估的对象不是 Agent，而是你的 Evaluator 本身。

当你用 LLM-as-Judge 对 Agent 输出打分时，这个 Judge 本身也会犯错：它对响应顺序有偏见（先看到哪个答案倾向于选哪个）、对回答长度有偏见（更长的答案往往得更高分）、还会因为模型版本静默更新而产生评分漂移——你的 Agent 没变，评分体系变了，你完全不知道。

元评估要做的事情：定期用同一对回答交换顺序做两次评测，检查 Judge 是否给出了矛盾的结论；锁定 Judge 所用的模型版本；用人工标注校准自动评分，追踪两者相关性。

在航空事故调查里，结果评估说的是"飞机坠毁了"，轨迹评估是飞行数据记录仪和驾驶舱语音记录仪，元评估是检查你的记录仪和重建程序本身是否可信。

## 四、Eval 数据集：工程里最被低估的资产

很多团队在 eval 框架上花了大量时间，但 eval 数据集却是凑出来的——几十个手写用例，主要覆盖的是工程师自己能想到的边界情况，而不是真实用户会触发的那些。

这是一个根本性的结构问题。你的 eval 数据集代表的，就是你的质量体系能"看见"的东西。数据集里没有的失败模式，永远不会被 CI 拦下来，只能等用户告诉你。

当前实践里收敛出的一个具体数字：
在你信任 eval 指标的聚合结论之前，数据集规模至少需要达到 500 个用例。

低于这个量级，评分数字的统计噪声会掩盖真实的质量变化。

数据集从哪里来？分两个来源，性质互补：

**手工构建的核心集（50-100 条）**。工程师和产品专家共同定义的黄金用例，覆盖最关键的用户场景和已知的边界情况。这部分质量最高、可信度最强，是整个 eval 体系的锚点。即使自动化评分系统失效，这部分可以用来人工校准。

**从生产 trace 中挖掘的扩展集（持续增长）**。这是整套体系里复利效应最显著的部分，也是 Eval 工程区别于简单"写测试"的核心所在——每一次生产失败，都是一份免费的、高价值的 eval 数据。

当一个用户触发了一个你没预料到的路径，导致 Agent 失败——这不是坏消息，这是新数据。当线上 Eval 把一条 trace 标记为低分——这不是需要修复后遗忘的事故，这是一个需要被永久保留的测试用例。

今天的生产事故，是明天 CI 的回归测试，是后天 eval 套件更高覆盖度的来源。随着时间推移，你的 eval 数据集会越来越接近真实用户的行为分布，而不是工程师的想象。

Notion 工程团队从这里获得了最直接的收益：把评估工作流从手动构建 JSONL 切换到以生产 trace 为核心数据来源之后，他们每天能处理的问题从 3 个增长到了 30 个。

## 五、把 Eval 接进 CI：让质量成为发布条件

eval 数据集和 eval 框架都有了之后，最后一步是把它们接进发布流程。

这一步听起来简单，但它代表了一个根本性的工程文化转变：把"Agent 质量"从一个主观判断，变成一个有数字的、可版本化的工程指标。

具体来说，这意味着：

每一个 PR——无论是提示词改动、模型替换、工具逻辑调整，还是系统 prompt 的细节修改——在合并之前，都自动触发 eval 套件的运行。评分低于设定阈值，PR 被阻止合并，就像单元测试失败会阻止构建一样。

这个机制解决了 Agent 工程里一个长期被忽视的问题：改动的隐性回归。在传统软件里，改一个函数影响的范围是可以静态分析的；在 Agent 系统里，修改系统 prompt 的一句话，可能让某类用户查询的成功率下降 15%，但你在 code review 阶段完全看不出来，只能等上线后从用户投诉里发现。CI 门控让这类回归在合并前就暴露出来。

有一个细节值得注意：同一套 Scorer 应该同时跑在 CI 和生产监控上。很多团队在 CI 里跑一套 eval，在生产里看另一套指标，两者不对齐——结果就是 CI 通过了，但生产质量还是在悄悄下滑。标准统一，才能让"CI 通过"真正等于"质量有保障"。

## 六、一点思考

- **Harness 工程**解决的是：让 Agent 在跨 session 的长程任务中不崩。
- **可观测性工程**解决的是：让你看清 Agent 每一步在做什么。
- **Eval 工程**解决的是：让你知道 Agent 做得够不够好，以及每次改动之后有没有变差。

三者的关系是线性的：没有 Harness，Agent 跑不起来；没有可观测性，Agent 失败了你找不到根因；没有 Eval，你改了系统、不知道改好了还是改坏了，只能靠用户投诉来发现。
