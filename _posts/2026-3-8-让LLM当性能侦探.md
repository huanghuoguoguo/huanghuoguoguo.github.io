上一篇文章里，我手动优化了一个 Rust 数据生成器，从 1.4M rows/s 折腾到 9.2M rows/s。过程中反复在做同一套动作：跑 `top` 看 CPU → 跑 `syscount` 看系统调用 → 跑 `perf` 看热点 → 猜一个方向 → 改代码 → 重新跑。

做了五轮之后我就在想：这套流程这么固定，能不能让程序自己跑？

<!-- more -->

于是我用 LangGraph 搭了一个 Agent 系统，让 LLM 来当"性能侦探"。这篇文章记一下这个尝试。我现在的感觉很直接：**能跑，但离我真敢放心交给它，还有一段距离。**

---

## 思路

我当时脑子里想的是，性能分析这件事其实很像一个循环：

1. 看一眼系统状态（top、free）
2. 心里有个大概判断（CPU 高？I/O 高？内存紧张？）
3. 决定用什么工具深入看（syscount？perf？/proc/meminfo？）
4. 看到数据，修正判断
5. 重复，直到大概知道问题卡在哪

后来我才意识到，这和 OODA 循环（Observe-Orient-Decide-Act）有点像。LLM 擅长的是顺着上下文往下判断，不太擅长的是老老实实把动作做对。那是不是可以反过来想：动作我先给它封成工具，判断和选择留给它？

---

## 第一版：单个 Agent

最简单的方案：一个 Agent，给它四个工具，让它自己跑。

```python
tools = [
    get_system_metrics,   # 跑 top -b -n 1
    get_syscall_counts,   # 跑 syscount-bpfcc
    read_proc_file,       # 读 /proc/meminfo 等
    get_process_stats,    # 跑 pidstat
]
llm = ChatOpenAI(model="kimi-k2-turbo-preview").bind_tools(tools)
```

LangGraph 的状态机很简单：

```
用户提问 → Agent 推理 → 调工具？─→ 是 → 执行工具 → 结果回传 → Agent 继续推理
                                  └→ 否 → 输出报告 → 结束
```

跑起来之后，Agent 一般会先调 `get_system_metrics()`，看到 db_gen 进程 CPU 很高；再去调 `get_syscall_counts(1)`，看到大量 write 调用；最后给一个报告，说"瓶颈在 I/O 密集的 write 系统调用"。

大方向没跑偏，但也就到这里了。往深里走，它接不上代码层面的分析。另一个问题更实际：诊断、读代码、改代码全塞给一个 Agent 之后，提示词会长得离谱。LLM 很容易"走神"，跑着跑着就忘了自己现在到底该查系统，还是该看代码。

## 第二版：三个 Agent 分工

把一个全能选手拆成三个专家：

```
Detective（侦探）──→ Architect（架构师）──→ Coder（程序员）
   查系统指标           看代码找原因           改代码修复
```

每个 Agent 有自己的工具集和职责，共享同一条消息链。就像一个群聊——侦探说完自己的发现，架构师接着看，不用重复解释上下文。

**侦探**只管 OS 层面的事：

```python
detective_tools = [get_system_metrics, get_syscall_counts,
                   read_proc_file, get_process_stats, execute_shell_command]
```

它的任务就是先把系统级瓶颈摸出来。比如"CPU 95%，主要消耗在 sys 态，write 系统调用占比最高"。摸到这一步就输出一个 `DIAGNOSIS_COMPLETE: ...`，状态机再把控制权交给架构师。

**架构师**只管代码层面：

```python
architect_tools = [list_directory, read_code_file]
```

它读侦探的分析结果，再去看 `db_gen/src/main.rs`，尽量把问题落到具体代码上。最后吐一个 `FINAL_REPORT: ...`，交给程序员。

**程序员**负责改代码：

```python
coder_tools = [read_code_file, overwrite_file, execute_shell_command]
```

读架构师的建议，改代码，跑 `cargo build --release` 验证编译通过。

### 实际效果

一次典型运行：

1. 侦探跑 `top`，看到 db_gen CPU 95%
2. 侦探跑 `syscount`，发现 write 和 futex 很多
3. 侦探输出诊断："高 sys CPU，write 系统调用频率过高"
4. 架构师读 `src/main.rs`，指出"没有 BufWriter，每行一次 write 系统调用"
5. （程序员阶段目前禁用了——原因见后面）

侦探到架构师这段还算靠谱。问题出在程序员阶段：LLM 改 Rust 代码的准确率不够高，经常编译不过，或者改的地方不对。`overwrite_file` 工具要求提供**完整文件内容**（没有 patch 功能），几百行的 Rust 文件让 LLM 整个重写一遍，出错概率很大。

现在这个版本里，我把程序员阶段先关掉了，就停在架构师给建议这里。

### 一些工程细节

**怎么防止 Agent 死循环？**

LLM Agent 最怕的事是陷入循环——反复调同一个工具，或者反复输出差不多的分析。加了两个保险：

```python
if len(messages) > 15:
    return "architect"  # 强制跳下一阶段
```

每个 Agent 最多 15 条消息，超了就硬切。

**PID 怎么找？**

侦探需要 db_gen 的 PID 才能用 `pidstat` 和 `syscount` 针对性分析。提示词里写了一个 fallback 链：

```
1. pidof db_gen
2. 如果失败，pgrep -f db_gen
3. 如果多个 PID，用 get_process_stats 找 CPU 最高的那个
```

实际跑的时候这段逻辑大部分时候能用，偶尔 LLM 会忘记执行这个步骤。

## 第三版：Operator 模式

到第三版时，我把侦探改成了 Operator（运维工程师），又补了两个工具：

```python
start_background_process(command)  # 后台启动进程
kill_process(pid)                  # 杀进程
```

这样 Operator 就能自己启动 db_gen，自己观测，自己停掉。整条链路算是接上了，不用我先手动把 db_gen 跑起来。

```
Operator（运维）──→ Architect（架构师）──→ END
  启动程序、观测、诊断      看代码、给建议
```

## 目前的局限

说实话，这个系统现在是能跑通的，但离"好用"还差得挺远：

**LLM 的分析深度有限。** 侦探能发现"CPU 高、write 多"这种表面现象，但不太能做出"这是 ext4 inode 锁导致的"这种深层推断。这种判断需要对内核实现有了解，不是光看 `top` 和 `syscount` 输出就能得到的。

**改代码不可靠。** 让 LLM 重写几百行 Rust 文件，准确率不够。如果有类似 `sed` 的 patch 工具会好很多，但目前的 `overwrite_file` 只能全量覆盖。

**工具输出太大。** `top` 和 `syscount` 的输出经常超过 LLM 的注意力范围，关键信息被淹没在一堆噪音里。如果能对工具输出先做一轮预处理（比如只保留 top 5 进程、只保留前 10 的系统调用），效果会好很多。

**提示词工程很脆弱。** 换一个 LLM 模型，或者系统负载稍有不同，Agent 的行为就可能完全不一样。"DIAGNOSIS_COMPLETE:" 这种约定信号，有些模型会忘记输出。

---

## 收获

虽然现在毛病还很多，但这个尝试至少让我确认了一件事：**把性能工程师那套排查顺序拆成工具和状态机，这条路是能走通的。**

我现在更愿意把它理解成一个分工问题。工具负责把动作做实，提示词负责把边界讲清楚，LLM 负责在中间做一点选择。说到底，它做的还是"什么时候该看什么"这件事。

如果后面还继续折腾，我大概会先试这几件事：

- 工具输出预处理：先用小模型或规则过滤噪音，只把关键指标给到 Agent
- Patch 而非 overwrite：代码修改用 diff/patch 模式，减少 LLM 的出错面
- 经验库：把之前的诊断案例存起来，RAG 检索，给 Agent 提供参考
- 人机协作：Agent 跑完诊断后暂停，人确认后再执行修改

## 再想远一点：Agent 的权限边界

上面运行 Agent 需要 `sudo`，因为 `syscount-bpfcc` 和 `pidstat` 需要 root 权限。这其实暴露了一个更本质的问题：**Agent 的能力越强，权限越高，出事的后果也越大。**

像 OpenClaw（一个全权限的 Claude Agent 项目）这类方案，Agent 可以读写文件、执行任意命令、SSH 到远程机器。能力确实强，但这也意味着一个错误的判断就可能导致误删文件、误杀进程、甚至搞崩生产环境。LLM 的输出本身就有概率性，把高权限操作交给一个"不确定的"决策者，这个组合让人不太放心。

我在 OB 实习的时候其实就想过，Agent 到底更适合碰哪一类运维问题。直接去动生产环境，我现在还是很犹豫。风险太高了。可 QA 自动化监控这种场景，反而让我觉得有点意思。

想象一下这样的场景：CI/CD 流水线上跑着几百个自动化任务，每天晚上批量执行。现在某个 Job 失败了，正常流程是什么？

1. 值班的人看到告警
2. SSH 到对应机器
3. 看日志、看环境变量、看资源状态
4. 判断是环境问题还是代码问题
5. 记录现场、提 Bug 单

这套流程中，1~4 都是信息收集和初步判断，5 是格式化输出。**没有任何一步需要修改环境。** 全是只读操作。

对我来说，这才像一个更合适的 Agent 场景：给它 SSH 到目标机器的权限，但**只给只读工具**：

```python
tools = [
    read_log(path),           # 读日志文件
    check_env_vars(),         # 查看环境变量
    check_disk_and_memory(),  # df -h, free -m
    check_process_status(),   # ps aux, systemctl status
    get_recent_deployments(), # 最近的部署记录
    get_test_report(job_id),  # 测试报告详情
]
# 注意：没有 write、没有 kill、没有 restart
```

你可以把它想成一个 24 小时盯任务的实习生。某个 Job 半夜三点挂了，它就去做这些事：

- SSH 上去查当时的环境状态（内存、磁盘、进程）
- 读相关日志，定位报错堆栈
- 对比最近的部署变更记录
- 把所有现场信息汇总，做一个初步判断
- 自动提一个 Bug 单，附上所有现场信息

等第二天早上工程师上班，看到的就不只是那条干巴巴的"Job #3847 Failed"了。至少能先看到一份像样的现场报告：哪个测试挂了，报错栈是什么，当时机器资源怎么样，最近相关变更是谁提的。是不是代码回归，也能先给一个初步猜测。

这里最关键的一点还是：**它只收集信息，只做判断，不改环境。** 最后拍板的人还是工程师。这样一来，Agent 的价值和它的风险就能先分开。至少我会更敢用一点。

### 顺着这个思路再往前想

如果 QA 监控这种只读场景真跑顺了，慢慢建立起信任，后面也许可以一点点把权限放开：

**第一层：只读诊断（现在就能做）**
- 自动化任务失败时，先把现场收起来，再给一个初步归因
- 盯慢查询日志，定期吐性能报告
- 扫配置文件，看看有没有明显的错误模式

**第二层：测试环境里的轻量自愈**
- 自动重启失败服务。先只在测试环境做
- 自动清理磁盘空间，比如 /tmp 和过期日志
- 在预设范围里扩一点测试集群资源

**第三层：受控的生产操作**
- Agent 先出方案，人审完再执行
- 灰度发布可以让它建议节奏，但每一步都得确认
- 故障自愈也只能限定在写死的 runbook 里

我现在会先用一个很笨的标准区分这三层：这个动作出了错，能不能收回来，影响面有多大。只读天然最安全。清理 /tmp 这种事，最多重跑。可生产服务一重启，事情就完全不是一个级别了。

所以我现在对人机协作这件事反而更有感觉了。Agent 去做那些重复的、机械的、半夜也得有人盯着的活儿；人去接最后的决策和高风险操作。这个分工至少目前看还算顺。

## 代码结构

```
perf_agent/
├── agent.py          # V1: 单 Agent OODA 循环
├── multi_agent.py    # V2: Detective → Architect → Coder
├── main.py           # V3: Operator → Architect
├── tools.py          # 所有工具定义
├── config.py         # LLM 配置
├── state.py          # 状态定义
└── agents/           # V3 的 Agent 模块
    ├── operator.py
    ├── architect.py
    └── coder.py
```

运行：

```bash
cd perf_agent

# 先在另一个终端启动 db_gen（用压力测试版）
cd ../db_gen && cargo run --release --bin db_gen

# 然后运行 Agent
sudo python3 main.py "诊断 db_gen 的性能瓶颈"
```

这里要加 `sudo`，主要是 `syscount-bpfcc` 和 `pidstat` 这两个工具本身就要更高权限。
