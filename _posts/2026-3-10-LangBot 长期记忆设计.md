前几天有个用户在 LangBot 的群里说了一句话，让我印象很深："我跟机器人聊了一个月，每次还要重新自我介绍，感觉它得了阿尔茨海默。"

这不是 LangBot 的 bug，这是所有 LLM chatbot 的通病。在前面的 RAG 文章里我讲过，大模型是无状态的——每次对话都是一次独立的函数调用，参数不会变，上下文窗口也只管当前这一轮会话。**会话结束，一切归零。**

我们熟悉的 RAG 解决了"怎么让模型知道你的文档里写了什么"，但似乎没解决"怎么让模型记住你是谁"。至少表面上看，一个是知识注入，一个是记忆——但这两件事的边界可能没那么清晰，后面会展开。

这篇文章记录我在 LangBot 上设计长期记忆插件的过程和思路。

## 先理清概念：短期记忆和长期记忆

类比人的记忆系统，chatbot 的"记忆"可以分成两层：

**短期记忆 = 上下文窗口。** 就是当前会话里的对话历史。你说了什么、模型回了什么，全部拼成一个消息列表传给 LLM。这就是 RAG 那篇文章里讲的 `[system, user, assistant, user, assistant, ...]` 结构。它的特点是：

- 生命周期：仅限当前会话
- 容量：受上下文窗口大小限制（4K~1M tokens）
- 读写：自动的，每轮对话自动追加

**长期记忆 = 跨会话持久化的信息。** 用户的名字、偏好、上周聊过的话题、约定好的事情——这些信息需要在会话结束后还能保留，下次对话时还能想起来。它的特点是：

- 生命周期：跨会话持久化
- 容量：理论上无限，但需要检索机制
- 读写：需要显式的存储和检索

LLM 天生就有短期记忆（上下文窗口），但完全没有长期记忆。**所谓"给 chatbot 造记忆"，就是在造这个长期记忆。**

```
┌─────────────────────────────────────────────────┐
│ 一次对话                                          │
│                                                   │
│  System Prompt ← 长期记忆注入到这里                  │
│  User: "帮我推荐咖啡"                               │
│  Assistant: "你之前提过喜欢美式，要不要试试..."       │  ← 靠长期记忆才能这样回答
│  User: "好的"                                      │
│  ...                                              │
│                                                   │
│  ↑ 这些是短期记忆（上下文窗口）                       │
└─────────────────────────────────────────────────┘
         ↕ 会话结束后，短期记忆消失
┌─────────────────────────────────────────────────┐
│ 长期记忆存储                                       │
│                                                   │
│  核心画像：小明，喜欢美式咖啡，软件工程师              │
│  情景记忆：[03-08] 小明说下周二要去上海出差             │
│           [03-05] 小红提到在学 Rust                  │
└─────────────────────────────────────────────────┘
```

## 其实都是 RAG

说到这里先停一下。长期记忆、短期记忆、跨 session 记忆、工作记忆、情景记忆——这些名词看起来花里胡哨，但如果你往后退一步看，它们在做的事情其实是同一件：**把数据存起来，在合适的时候检索出来，作为上下文给到 LLM。**

这不就是 RAG 吗？

我们熟知的 RAG 流程是：文档切块 → Embedding → 存进向量数据库 → 用户提问时做语义检索 → 把命中的片段塞进 prompt。但这只是 RAG 最早、最常见的一种实现。RAG 的本质不是向量数据库，而是"Retrieval-Augmented Generation"这个模式本身——**先检索，再生成**。至于检索的数据源是什么、存储介质是什么、检索方式是什么，都是具体实现的选择。

```
广义 RAG 的本质：

  存储                   检索                    增强
  把数据放在某个地方 →  在需要时找到相关的数据 →  塞进 LLM 的上下文

具体实现可以是：

  向量数据库 + Embedding 检索        ← 经典 RAG
  Markdown 文件 + LLM 全文阅读      ← OpenClaw 的记忆方案
  KV 存储 + 每次全量注入             ← 用户画像方案
  对话历史 + 滑动窗口截断            ← 最朴素的"短期记忆"
```

你看，连"上下文窗口里保留最近 N 轮对话"这种最基本的操作，本质上也是一种 RAG——只不过"存储"是内存里的消息列表，"检索"策略是"取最近 N 条"，"增强"方式是直接拼进 messages 数组。

理解了这一点，后面的设计决策就会清晰很多：我们要做的不是发明什么新机制，而是针对"用户记忆"这个特定数据源，选择合适的存储方式和检索策略。不同的选择，造出来的"记忆"就有不同的特性。

## 先看一个参照：OpenClaw 的记忆方案

在动手设计之前，我研究了一圈现有方案。2026 年初爆火的 OpenClaw（社区叫它"小龙虾"）在记忆设计上走了一条跟主流 Agent 框架完全不同的路，给了我不少启发——尤其是"让 Agent 自己管理记忆"这个核心思路，后面会看到 LangBot 直接借鉴了这一点。但它的存储方案在 LangBot 的场景下行不通，这也是后面做取舍的起点。

OpenClaw 是一个本地优先的个人 Agent 框架——不做独立前端，寄生在 Telegram、Slack、飞书等现有 IM 里，用户提供自己的 API Key，所有数据留在本地。它的核心设计原则是"你的基础设施，你的密钥，你的数据"。

### 三层记忆模型

OpenClaw 把记忆分成三层：

```
┌──────────────────────────────────────────────────┐
│  Working Memory（工作记忆）                         │
│  当前对话的最近 N 轮消息，就是上下文窗口               │
│  易失性，进程重启即丢失                              │
└──────────────────────────────────────────────────┘
                  ↓  周期性总结 / 提取
┌──────────────────────────────────────────────────┐
│  Short-term Memory（短期记忆）                      │
│  当前会话的摘要和关键事实，存在内存中                   │
│  会话结束时持久化到磁盘                              │
└──────────────────────────────────────────────────┘
                  ↓  每日归档
┌──────────────────────────────────────────────────┐
│  Long-term Memory（长期记忆）                       │
│  持久化的 Markdown 文件：MEMORY.md                   │
│  按时间线组织的"日记"形式，用户可直接阅读和编辑         │
└──────────────────────────────────────────────────┘
```

前两层跟大多数 Agent 框架的做法差不多。真正有意思的是第三层——长期记忆不用向量数据库，而是存成纯文本 Markdown 文件。

### 用文本文件当记忆库

OpenClaw 的长期记忆就是一个 `MEMORY.md`：

```markdown
<!-- ~/.openclaw/memory/MEMORY.md -->

# User Memory

## 2026-03-10
- 用户在研究 OpenClaw 的架构设计
- 重要：用户偏好深入的技术原理分析，而非表面介绍

## 2026-03-09
- 用户正在做 DGA 恶意域名检测的论文
- 用户已入门 Rust，寻找高性能项目练手
- 技术栈偏好：系统级编程、网络安全、AI 基础设施
```

按日期倒序写"日记"。Agent 定期从对话中提取关键信息追加写入，用户可以随时用任何文本编辑器打开查看、修改、删除。

另外还有一个 `SOUL.md`，定义 Agent 的行为准则和人格。Agent 会基于交互经验自动更新这个文件——比如发现用户偏好技术原理而非表面介绍，就调整自己的回复策略。这是一种"元认知"：Agent 通过读自己的记忆来反思和进化行为方式。

### 为什么不用向量数据库？

OpenClaw 的设计者认为：个人助手的记忆规模根本用不到向量索引。

算一下：假设每天交互 50 轮，每轮 500 tokens，一年的原始对话量大概 9MB。但 `MEMORY.md` 存的不是原始对话，而是压缩后的摘要和关键事实——再加上定期归档清理过时条目，实际文件大小远小于原始对话量。对于大多数个人用户，用几个月积累的记忆文件在 200K token 的上下文窗口内基本放得下。LLM 直接全文阅读就能理解，不需要 embedding + cosine similarity 那套检索机制。

文本方案带来几个实际好处：

- **透明性**：Agent 行为异常时，用户打开 `MEMORY.md` 就能检查它到底记了什么、是不是记错了。不像向量数据库是个黑盒，人类无法直接阅读 embedding 向量。
- **可编辑**：发现过时或错误的记忆，直接改文件。不需要写代码操作向量索引。
- **版本控制**：纯文本天然适合 git 管理，记忆变更有完整历史。也能用 Dropbox 之类的云盘跨设备同步。
- **引用完整性**：按时间线组织的叙事保留了上下文。向量检索的一个隐蔽问题是，召回的片段脱离了原始语境——"deadline 是周五"被检索出来后，"哪一周的周五"这个信息可能丢失了。时间线叙事不存在这个问题。

向量数据库在 OpenClaw 中是可选的加速层，只在记忆文件膨胀到很大时才有必要启用。真相来源（source of truth）始终是那个可读的 Markdown 文件。

### 这个方案的边界

文本方案对个人助手很合适，但有明显的适用范围：

| 维度 | 文本方案（OpenClaw） | 向量方案 |
|------|---------------------|---------|
| 适用规模 | 个人助手，记忆文件 < 数 MB | 不限 |
| 检索方式 | LLM 全文阅读 | 语义相似度精确召回 |
| 多用户 | 天然单用户 | 支持多用户隔离 |
| 群聊场景 | 未设计 | 可做 per-session / per-user 隔离 |

OpenClaw 选择文本记忆是因为它定位为"个人 Agent"——一个人、一份记忆、一个文件。但 LangBot 面对的是不同的场景：多用户、群聊、共享 bot。一个 500 人的群不可能用一个 Markdown 文件装所有人的记忆，也不可能每轮对话让 LLM 全文扫描所有历史记录。

所以 LangBot 的记忆设计虽然借鉴了 OpenClaw "Agent 自主管理记忆"的核心思路，但在存储和检索方案上走了不同的路。下面的几个决策会逐步展开。

## 第一个决策：谁来决定"记什么"？

这是整个设计里最关键的一个问题，方案选错了后面全是坑。

### 方案 A：自动提取

最直觉的做法：每轮对话结束后，单独发一次 LLM 调用，让模型从对话中提取值得记住的信息。

```
对话结束
  → 额外调一次 LLM："从以下对话中提取值得长期记住的信息"
  → LLM 返回提取结果
  → 存入记忆库
```

这个方案的问题很明显：

- **额外的 LLM 调用开销**：不管是每条消息都提取还是每个 session 结束时批量提取，都绕不开一次独立的 LLM 调用。即使用便宜的小模型，积少成多也是一笔不小的成本。
- **提取质量不可控**：LLM 不知道哪些信息对这个用户真正重要。"今天天气不错"要不要记？"我喜欢 Java"要不要记？自动提取要么太激进（记一堆垃圾），要么太保守（漏掉关键信息）。

### 方案 B：Agent 驱动（Tool 调用）

换一个思路：不在对话外面额外做提取，而是**给 LLM 一个 `remember` 工具，让它在对话过程中自主决定什么时候记、记什么**。

```yaml
- name: remember
  description: |
    记住一条关于用户的信息。当用户分享了值得长期记住的个人偏好、
    事实、计划或重要信息时调用此工具。不要记录闲聊或临时性内容。
  parameters:
    content: string   # 要记住的内容
    tags: string[]    # 分类标签
    importance: int   # 1-5，5 最重要
```

这就是上一篇文章讲的 Agent 模式——给模型工具，让它自己决策。

关键优势：**不需要额外的 LLM 调用**。`remember` 是当前对话的 function calling 的一部分，LLM 在回复用户的同时顺手调一下 Tool，不需要单独再发一次请求。唯一的额外开销是 Tool 定义本身会占一些 system prompt 的 token，但跟单独发一次 LLM 请求比，几乎可以忽略。

```
方案 A（自动提取）：每轮 = 1 次正常 LLM 调用 + 1 次提取 LLM 调用
方案 B（Tool 驱动）：每轮 = 1 次正常 LLM 调用（Tool 调用在其中完成）
```

而且 LLM 在对话语境中做"该不该记"的判断，比事后回看对话记录做提取准确得多——它知道用户说"我喜欢美式咖啡"是在认真表达偏好，还是在随口举个例子。

这个思路就是前面介绍的 OpenClaw 模式的核心：Agent 自己管理自己的记忆，而不是由外部系统替它管。OpenClaw 的简化版 [NanoClaw](https://github.com/anthropics/NanoClaw) 更直接——让 Claude 通过编辑一个 `CLAUDE.md` 文件来管理对用户的记忆，下次对话时这个文件注入 System Prompt。

最终选了方案 B。

## 第二个决策：记忆存成什么结构？

确定了"LLM 自己决定记什么"之后，下一个问题是：记下来的东西怎么组织？

想清楚两种不同性质的信息：

- **"小明是软件工程师，喜欢美式咖啡"** — 这是画像类信息，低频变化，每次对话都可能用到
- **"小明说下周二要去上海出差"** — 这是事件类信息，跟特定时间关联，只在相关话题时才需要

这两种信息的使用模式完全不同。画像信息量小（几百字），应该每次都注入；事件信息会持续积累，不能全塞进去，需要按需检索。

所以设计成了**双层架构**：

```
┌──────────────────────┬──────────────────────────────┐
│  L1 核心画像          │  L2 情景记忆                   │
│  (Core Profile)      │  (Episodic Memory)            │
├──────────────────────┼──────────────────────────────┤
│  大小：< 2KB          │  按条目增长                    │
│  注入：每次对话必注入   │  按相关性检索 Top-K            │
│  类比：System Prompt   │  类比：RAG                    │
│  更新：update_profile  │  更新：remember               │
└──────────────────────┴──────────────────────────────┘
```

**L1 核心画像**始终注入 System Prompt，就像一个始终生效的"用户说明书"：

```json
{
  "name": "小明",
  "traits": ["软件工程师", "养了一只猫叫橘子"],
  "preferences": ["回复用中文", "不喜欢表情包"],
  "notes": "周三通常不在线"
}
```

**L2 情景记忆**是一条一条的事件记录，存在向量数据库里，按语义相关性检索，只取跟当前对话相关的 Top-K 条注入：

```json
{"content": "小明说下周二要去上海出差", "tags": ["出差", "上海"], "importance": 3}
{"content": "小红提到正在学 Rust",     "tags": ["编程", "Rust"],  "importance": 2}
```

L1 和 L2 通过不同通道注入，LLM 最终看到的消息结构大致是这样：

```
[System Prompt]
  你是一个智能助手...

  ## Memory (Profile)               ← L1 画像（EventListener 注入 system prompt）
  - Name: 小明
  - Traits: 软件工程师, 养了一只猫叫橘子
  - Preferences: 回复用中文
  - Notes: 周三通常不在线

  ## Current Speaker                ← 说话人身份（EventListener 注入）
  - Name: 小明
  - ID: 12345

[User Message]                     ← L2 情景记忆（KnowledgeEngine 注入 RAG context）
  <context>
  - [2026-03-08T10:30:00Z] (importance:3) 小明说下周二要去上海出差
  - [2026-03-05T14:22:00Z] (importance:2) 小红提到正在学 Rust
  </context>
  <user_message>
  帮我推荐个咖啡
  </user_message>
```

L1 画像始终在 system prompt 里，每轮对话都能看到。L2 情景记忆走 RAG 通道——根据当前消息做语义检索，命中的条目作为上下文拼在用户消息前面。两层注入互不干扰。

如果你读过 RAG 迁移那篇文章，会发现 L2 情景记忆跟 RAG 的检索逻辑几乎一模一样——只不过 RAG 检索的是用户上传的文档，这里检索的是历史对话中记下的信息。**本质上，长期记忆的 L2 就是对会话历史做 RAG。**

---

## 第三个决策：记忆怎么接入 LangBot？

既然 L2 本质是 RAG，或者说，长期记忆本质上就是检索增强，一个自然的想法就是：直接复用 LangBot 现有的知识库架构。这让我们不用思考怎么存，只需要知道怎么用。

LangBot 有一套 KnowledgeEngine 抽象层——RAG 迁移那篇文章搭的基础设施。一个 KnowledgeEngine 负责一种知识源的检索逻辑，LangBot 在每轮对话前自动调用绑定的 KnowledgeEngine 做检索，把结果作为 RAG context 注入。RAG 文档走这条路，记忆也可以走这条路。

所以我把记忆做成了一种特殊的知识库。插件实现了一个 `LongTermMemoryEngine`，继承 LangBot 的 `KnowledgeEngine` 接口。用户的操作流程是这样的：

```
1. 安装 langbot-plugin-memory 插件
2. 在 LangBot UI 创建知识库
   → 引擎选择 "LongTermMemory"
   → 配置嵌入模型（用于情景记忆的语义检索）
   → 配置隔离模式（session / bot）
3. 在目标 Pipeline 中绑定该知识库
   → 该 Pipeline 获得完整长期记忆能力
```

绑定之后，LangBot 在每轮对话前自动调用 `KnowledgeEngine.retrieve()`，用当前用户消息做语义检索，把命中的记忆条目注入 RAG context。这就是 L2 的读取路径——完全复用 LangBot 已有的 RAG 注入机制，零新增逻辑。

写入路径同样直接：LLM 调用 `remember` Tool 时，插件调 `invoke_embedding` 做向量化，再 `vector_upsert` 写入向量数据库：

```python
# remember Tool 的核心逻辑
embedding = await self.ctx.invoke_embedding(content)
await self.ctx.vector_upsert(
    collection_id=kb_collection_id,
    doc_id=str(uuid.uuid4()),
    embedding=embedding,
    metadata={"content": content, "tags": tags, "importance": importance, ...}
)
```

这里复用了 RAG 迁移时设计的那套 RPC 接口（`invoke_embedding`、`vector_upsert`、`vector_search`）。当时设计那套接口的时候就考虑过记忆场景，所以现在几乎零适配成本就能用上。这也算是 RAG 插件化架构的一个额外收益——基础设施是共享的。

这个设计还带来一个额外好处：**per-pipeline 的精细控制**。不同 Pipeline 可以选择绑定或不绑定记忆 KB，形成三个档位：

| 档位 | 配置 | 效果 |
|------|------|------|
| 完整记忆 | 启用插件 + 绑定记忆 KB | L1 画像 + L2 情景检索 + Tools |
| 仅画像 | 启用插件，不绑 KB | L1 画像 + Tools（情景记忆不被检索） |
| 完全关闭 | 不启用插件 | 无注入、无 Tools，完全透明 |

一个设计约束：**插件只允许创建一个记忆 KB 实例**。多个 Pipeline 共用同一个 KB，通过 metadata 中的 `user_key` 字段实现隔离。这样 `remember` Tool 写入时不需要判断"写到哪个 KB"——只有一个，不会写错。

这里其实有个妥协：LangBot 框架目前不支持插件按 pipeline 创建独立的 KB，所以所有 session 的记忆混在同一个向量集合里，靠 metadata 过滤做逻辑隔离。代价是检索时需要扫描整个集合再按 `user_key` 过滤，数据量大了之后检索速度会受影响。但对于当前阶段"先跑起来"的目标，这个 trade-off 是值得的。

---

## 第四个决策：群聊里的记忆归谁？

LangBot 很大一部分场景是群聊。群聊里的记忆隔离是个有意思的问题。

第一反应是按用户隔离——A 的记忆 B 看不到。但仔细想想，群聊本来就是公开的。A 在群里说"项目用 React"，B 当时就能看到。如果 bot 的记忆反而把这条信息藏起来，只有 A 问的时候才知道项目用 React，这就不自然了。

而且 per-user 隔离会让画像维度爆炸——一个 500 人的群要维护 500 份独立画像。大多数群聊场景下（团队群、兴趣群），共享记忆才是合理的默认行为。

所以最终的方案是：**记忆按 session 隔离，同一个 session 内所有人共享**。每个群有一份群画像，每个私聊有一份用户画像。不做 sender 维度的隔离。

```
QQ群A → session_key = "group_123"  → 群内所有人共享画像和记忆
QQ群B → session_key = "group_456"  → 独立的另一份
私聊C → session_key = "person_789" → 用户C独享
```

但这带来一个新问题：如果群里所有人共享记忆，A 说"我喜欢咖啡"存进去了，B 来问"推荐个饮品"，bot 会不会把 A 的偏好推荐给 B？

解法是**说话人感知**（Speaker Awareness），分三层：

1. **注入说话人身份**：EventListener 在每轮对话的 System Prompt 里注入当前说话人的名字和 ID，LLM 知道"现在跟我说话的是谁"
2. **Tool prompt 引导归因**：`remember` 的 prompt 要求 LLM 在记忆内容中标注说话人——存"小明说他喜欢咖啡"而不是"喜欢咖啡"
3. **语义检索自然匹配**：B 问"推荐个饮品"时，检索结果是"小明说他喜欢咖啡"，LLM 看到这是小明的偏好，不会张冠李戴

```
System Prompt:
  ## Current Speaker
  - Name: 小红
  - ID: 67890

L2 检索结果:
  - 小明说他喜欢喝咖啡
  - 小红说她在减肥

→ LLM 知道当前说话人是小红，会基于"小红在减肥"来推荐，不会推荐咖啡
```

这个方案的好处是**不需要在存储层做复杂的隔离，归因在语义层解决**。向量搜索本身就能通过语义匹配找到相关的记忆，LLM 有足够的能力理解"这条记忆是关于谁的"。

需要承认的是，这不是硬隔离——三层机制全部依赖 LLM 对 prompt 的遵从。LLM 可能忘记在记忆里标注说话人，也可能在推理时把别人的偏好张冠李戴。这是个 best-effort 的方案，不是绝对可靠的。但在群聊这种本身就是公开场景下，偶尔的归因偏差是可接受的。

在 session 隔离之上，还有一个更高层的维度：

| 模式 | user_key | 效果 | 场景 |
|------|---------|------|------|
| `session`（默认） | `session_key` | 每个群/私聊独立记忆 | 各个群/私聊互不干扰 |
| `bot` | `"global"` | 所有群/私聊共享记忆 | bot 跨场景共享上下文 |

`session` 模式下，QQ 群 A 的记忆和 QQ 群 B 的记忆互不干扰，私聊的记忆也是独立的。`bot` 模式下，bot 在所有场景中共享一份全局记忆和画像——适合需要跨群共享上下文的场景。

存储键的推导非常简单：

```python
def get_user_key(session_key: str, isolation: str) -> str:
    if isolation == "session":
        return session_key       # e.g. "group_123456"
    return "global"              # bot 模式，全局共享
```

## 记忆的管理

LLM 有两个写入工具来管理记忆：

| 工具 | 操作 | 目标 | 用途 |
|------|------|------|------|
| `remember` | 写入 | L2 情景记忆 (向量数据库) | 存事件、计划、经历 |
| `update_profile` | 写入 | L1 核心画像 (Binary Storage) | 存姓名、特征、偏好 |

`remember` 只管情景信息——有时间性的事件、计划、项目进展。`update_profile` 只管身份信息——稳定的个人特征、偏好。两个工具的边界很清楚，不会打架。

读取侧完全自动化：L1 画像由 EventListener 在每轮对话时自动注入 System Prompt，L2 情景记忆由 KnowledgeEngine 在每轮对话前自动检索注入 RAG context。LLM 不需要主动"搜记忆"——需要的信息已经在上下文里了。

用户也可以通过命令手动查看记忆：

```
!memory                  - 查看记忆概况（隔离模式、session key、画像摘要、KB 状态）
!memory profile          - 查看核心画像详情
!memory search <query>   - 搜索情景记忆（向量语义检索）
```

## 成本对比

这是我在做方案选型时最关心的一个维度。

| 环节 | 自动提取方案 | Tool 驱动方案（本方案） |
|------|------------|---------------------|
| 记忆写入 | 每轮：1 次额外 LLM + 1 次 Embedding | 仅 LLM 认为有价值时：Tool 调用（零额外 LLM） |
| 记忆检索 | 每轮：1 次 Embedding + 1 次向量查询 | 每轮：1 次 Embedding + 1 次向量查询 |
| 画像注入 | 无（没有画像概念） | 每轮：1 次 KV 读取（几乎为零） |
| 每轮额外 LLM 调用 | 1 次 | 0 次 |

两个方案的检索开销是一样的——都是一次 Embedding + 一次向量查询。差别在写入侧：自动提取方案需要额外的 LLM 调用做记忆提取，而 Tool 驱动方案的写入发生在当前 LLM 调用中，不需要独立请求。L1 画像的注入成本几乎可以忽略——从 Binary Storage 读一个 JSON 对象，毫秒级别。记忆数据通过 LangBot 的向量数据库抽象层存储，插件本身不绑定具体实现。

## 还没完：记忆会腐烂

当前版本的记忆是只增不减的。LLM 通过 Tool 写入后没有自动整理机制。用久了会出现重复、矛盾、过时的记忆——"小明说他喜欢喝咖啡"和"小明说他现在改喝茶了"同时存在，到底信哪个？

这是后续迭代要解决的问题，按优先级排了三个方向：

**写入时去重（P1）**：`remember` 写入前先做一次向量搜索，发现高相似度的已有记忆时调 LLM 判断——是矛盾（替换旧的）、补充（保留两条）、还是重复（跳过写入）。在源头拦截垃圾记忆，成本是每次写入多一次向量搜索加偶尔一次 LLM 调用。

```
remember("小明说他喜欢喝茶")
  ↓
向量搜索：发现已有 "小明说他喜欢喝咖啡"（相似度 0.92）
  ↓
调用 LLM 判断：
  - 矛盾 → 替换旧记忆（偏好变了）
  - 补充 → 保留两条（咖啡和茶都喜欢）
  - 重复 → 跳过写入
```

**定期整理（P2）**：每 50 轮对话或距上次整理超过 7 天时，触发一次批量整理——合并相似记忆、标记过时信息、把反复出现的稳定事实提升到 L1 画像。异步执行，不阻塞当前对话。

**检索时筛选（P3）**：检索出 Top-K 后让 LLM 做一轮相关性筛选再返回主对话。优先级最低——向量相似度本身已经是不错的筛选了，额外加一次 LLM 调用的收益不一定值得延迟的代价。

这三个都依赖插件 SDK 提供 `invoke_llm()` API（让插件主动调用 LLM），目前还在规划中。设计原则是所有维护功能可选、失败时静默降级、不影响正常对话。

## 回到那个根本问题

前面两篇文章一直在追一个线索：**显式的外部架构会不会被模型的原生能力吸收？**

记忆也一样。目前我们在做的事情——用 Tool 调用写入记忆、用检索注入上下文——本质上是在模型外面手动搭了一套记忆系统。但如果有一天模型原生就支持跨会话记忆呢？

其实这已经开始发生了。ChatGPT 的 Memory 功能就是平台层面对长期记忆的原生支持——自动从对话中提取用户偏好并持久化。它做的事情跟我们这个插件没有本质区别——都是在 System Prompt 里注入持久化的用户信息。

区别在于控制权。平台原生的记忆功能是黑箱的——你不知道它记了什么、为什么记、什么时候会忘。而插件方案是完全透明的：用户可以 `!memory search` 检索记忆条目，可以 `!memory profile` 查看画像，LLM 的每次 `remember` 调用都是可审计的。

对于 LangBot 这样的开源平台来说，这种透明性和可控性是核心价值。用户部署的是自己的机器人，记忆数据存在自己的数据库里，不依赖任何外部平台。

至于这些外部架构最终会不会被模型完全内化——大概率会，但那是模型能力发展的事。在那一天到来之前，我们能做的就是把脚手架搭得尽量干净，等到不需要的时候能轻松拆掉。
