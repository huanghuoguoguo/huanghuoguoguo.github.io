前几天有个用户在 LangBot 的群里说了一句话，让我印象很深："我跟机器人聊了一个月，每次还要重新自我介绍，感觉它得了阿尔茨海默。"

这不是 LangBot 的 bug，这是所有 LLM chatbot 的通病。在前面的 RAG 文章里我讲过，大模型是无状态的——每次对话都是一次独立的函数调用，参数不会变，上下文窗口也只管当前这一轮会话。**会话结束，一切归零。**

RAG 解决了"怎么让模型知道你的文档里写了什么"，但没解决"怎么让模型记住你是谁"。这是两个不同的问题：一个是知识注入，一个是记忆。

这篇文章记录我在 LangBot 上设计长期记忆插件的过程和思路。

## 先理清概念：短期记忆和长期记忆

类比人的记忆系统，chatbot 的"记忆"可以分成两层：

**短期记忆 = 上下文窗口。** 就是当前会话里的对话历史。你说了什么、模型回了什么，全部拼成一个消息列表传给 LLM。这就是 RAG 那篇文章里讲的 `[system, user, assistant, user, assistant, ...]` 结构。它的特点是：

- 生命周期：仅限当前会话
- 容量：受上下文窗口大小限制（4K~1M tokens）
- 读写：自动的，每轮对话自动追加

**长期记忆 = 跨会话持久化的信息。** 用户的名字、偏好、上周聊过的话题、约定好的事情——这些信息需要在会话结束后还能保留，下次对话时还能想起来。它的特点是：

- 生命周期：跨会话持久化
- 容量：理论上无限，但需要检索机制
- 读写：需要显式的存储和检索

LLM 天生就有短期记忆（上下文窗口），但完全没有长期记忆。**所谓"给 chatbot 造记忆"，就是在造这个长期记忆。**

```
┌─────────────────────────────────────────────────┐
│ 一次对话                                          │
│                                                   │
│  System Prompt ← 长期记忆注入到这里                  │
│  User: "帮我推荐咖啡"                               │
│  Assistant: "你之前提过喜欢美式，要不要试试..."       │  ← 靠长期记忆才能这样回答
│  User: "好的"                                      │
│  ...                                              │
│                                                   │
│  ↑ 这些是短期记忆（上下文窗口）                       │
└─────────────────────────────────────────────────┘
         ↕ 会话结束后，短期记忆消失
┌─────────────────────────────────────────────────┐
│ 长期记忆存储                                       │
│                                                   │
│  核心画像：小明，喜欢美式咖啡，软件工程师              │
│  情景记忆：[03-08] 说下周二要去上海出差               │
│           [03-05] 提到在学 Rust                    │
└─────────────────────────────────────────────────┘
```

## 第一个决策：谁来决定"记什么"？

这是整个设计里最关键的一个问题，方案选错了后面全是坑。

### 方案 A：自动提取

最直觉的做法：每轮对话结束后，单独发一次 LLM 调用，让模型从对话中提取值得记住的信息。

```
对话结束
  → 额外调一次 LLM："从以下对话中提取值得长期记住的信息"
  → LLM 返回提取结果
  → 存入记忆库
```

这个方案的问题很明显：

- **每轮对话多一次 LLM 调用**：用户聊十条消息，你就额外调十次 LLM 做记忆提取。成本直接翻倍。
- **提取质量不可控**：LLM 不知道哪些信息对这个用户真正重要。"今天天气不错"要不要记？"我喜欢 Java"要不要记？自动提取要么太激进（记一堆垃圾），要么太保守（漏掉关键信息）。

### 方案 B：Agent 驱动（Tool 调用）

换一个思路：不在对话外面额外做提取，而是**给 LLM 一个 `remember` 工具，让它在对话过程中自主决定什么时候记、记什么**。

```yaml
- name: remember
  description: |
    记住一条关于用户的信息。当用户分享了值得长期记住的个人偏好、
    事实、计划或重要信息时调用此工具。不要记录闲聊或临时性内容。
  parameters:
    content: string   # 要记住的内容
    tags: string[]    # 分类标签
    importance: int   # 1-5，5 最重要
```

这就是上一篇文章讲的 Agent 模式——给模型工具，让它自己决策。

关键优势：**零额外 LLM 调用**。`remember` 是当前对话的 function calling 的一部分，LLM 在回复用户的同时顺手调一下 Tool，不需要单独再发一次请求。成本上跟没有记忆功能的对话完全一样。

```
方案 A（自动提取）：每轮 = 1 次正常 LLM 调用 + 1 次提取 LLM 调用
方案 B（Tool 驱动）：每轮 = 1 次正常 LLM 调用（Tool 调用在其中完成）
```

而且 LLM 在对话语境中做"该不该记"的判断，比事后回看对话记录做提取准确得多——它知道用户说"我喜欢美式咖啡"是在认真表达偏好，还是在随口举个例子。

这个思路借鉴了一个叫 [NanoClaw](https://github.com/anthropics/NanoClaw) 的开源项目（OpenClaw 的简化版）。它的做法是让 Claude 通过直接编辑一个 `CLAUDE.md` 文件来管理对用户的记忆——Agent 自己往文件里写"这个用户喜欢什么、做过什么"，下次对话时这个文件会被注入 System Prompt。本质上就是：**Agent 自己管理自己的记忆，而不是由外部系统替它管。**

最终选了方案 B。

## 第二个决策：记忆存成什么结构？

确定了"LLM 自己决定记什么"之后，下一个问题是：记下来的东西怎么组织？

想清楚两种不同性质的信息：

- **"小明是软件工程师，喜欢美式咖啡"** — 这是画像类信息，低频变化，每次对话都可能用到
- **"小明说下周二要去上海出差"** — 这是事件类信息，跟特定时间关联，只在相关话题时才需要

这两种信息的使用模式完全不同。画像信息量小（几百字），应该每次都注入；事件信息会持续积累，不能全塞进去，需要按需检索。

所以设计成了**双层架构**：

```
┌──────────────────────┬──────────────────────────────┐
│  L1 核心画像          │  L2 情景记忆                   │
│  (Core Profile)      │  (Episodic Memory)            │
├──────────────────────┼──────────────────────────────┤
│  大小：< 2KB          │  按条目增长                    │
│  注入：每次对话必注入   │  按相关性检索 Top-K            │
│  类比：System Prompt   │  类比：RAG                    │
│  更新：update_profile  │  更新：remember               │
└──────────────────────┴──────────────────────────────┘
```

**L1 核心画像**始终注入 System Prompt，就像一个始终生效的"用户说明书"：

```json
{
  "name": "小明",
  "traits": ["软件工程师", "养了一只猫叫橘子"],
  "preferences": ["回复用中文", "不喜欢表情包"],
  "notes": "周三通常不在线"
}
```

**L2 情景记忆**是一条一条的事件记录，存在 ChromaDB 向量库里，按语义相关性检索，只取跟当前对话相关的 Top-K 条注入：

```json
{"content": "用户说下周二要去上海出差", "tags": ["出差", "上海"], "importance": 3}
{"content": "用户提到正在学 Rust",     "tags": ["编程", "Rust"],  "importance": 2}
```

L1 和 L2 通过不同通道注入，LLM 最终看到的消息结构大致是这样：

```
[System Prompt]
  你是一个智能助手...

  ## 关于该用户的信息              ← L1 画像（EventListener 注入 system prompt）
  - 称呼：小明
  - 特征：软件工程师, 养了一只猫叫橘子
  - 偏好：回复用中文

[User Message]                     ← L2 情景记忆（KnowledgeEngine 注入 RAG context）
  <context>
  - [2026-03-08] 用户说下周二要去上海出差
  - [2026-03-05] 用户提到正在学 Rust
  </context>
  <user_message>
  帮我推荐个咖啡
  </user_message>
```

L1 画像始终在 system prompt 里，每轮对话都能看到。L2 情景记忆走 RAG 通道——根据当前消息做语义检索，命中的条目作为上下文拼在用户消息前面。两层注入互不干扰。

如果你读过 RAG 迁移那篇文章，会发现 L2 情景记忆跟 RAG 的检索逻辑几乎一模一样——只不过 RAG 检索的是用户上传的文档，这里检索的是历史对话中记下的信息。**本质上，长期记忆的 L2 就是对会话历史做 RAG。**

## 第三个决策：记忆怎么接入 LangBot？

既然 L2 本质是 RAG，一个自然的想法就是：直接复用 LangBot 现有的知识库架构。

LangBot 有一套 KnowledgeEngine 抽象层——RAG 迁移那篇文章搭的基础设施。一个 KnowledgeEngine 负责一种知识源的检索逻辑，LangBot 在每轮对话前自动调用绑定的 KnowledgeEngine 做检索，把结果作为 RAG context 注入。RAG 文档走这条路，记忆也可以走这条路。

所以我把记忆做成了一种特殊的知识库。插件实现了一个 `LongTermMemoryEngine`，继承 LangBot 的 `KnowledgeEngine` 接口。用户的操作流程是这样的：

```
1. 安装 langbot-plugin-memory 插件
2. 在 LangBot UI 创建知识库
   → 引擎选择 "LongTermMemory"
   → 配置嵌入模型（用于情景记忆的语义检索）
   → 配置隔离模式（shared / per_user）
3. 在目标 Pipeline 中绑定该知识库
   → 该 Pipeline 获得完整长期记忆能力
```

绑定之后，LangBot 在每轮对话前自动调用 `KnowledgeEngine.retrieve()`，用当前用户消息做语义检索，把命中的记忆条目注入 RAG context。这就是 L2 的读取路径——完全复用 LangBot 已有的 RAG 注入机制，零新增逻辑。

写入路径同样直接：LLM 调用 `remember` Tool 时，插件调 `invoke_embedding` 做向量化，再 `vector_upsert` 写入 LangBot 自带的 ChromaDB：

```python
# remember Tool 的核心逻辑
embedding = await self.ctx.invoke_embedding(content)
await self.ctx.vector_upsert(
    collection_id=kb_collection_id,
    doc_id=str(uuid.uuid4()),
    embedding=embedding,
    metadata={"content": content, "tags": tags, "importance": importance, ...}
)
```

这里复用了 RAG 迁移时设计的那套 RPC 接口（`invoke_embedding`、`vector_upsert`、`vector_search`）。当时设计那套接口的时候就考虑过记忆场景，所以现在几乎零适配成本就能用上。这也算是 RAG 插件化架构的一个额外收益——基础设施是共享的。

这个设计还带来一个额外好处：**per-pipeline 的精细控制**。不同 Pipeline 可以选择绑定或不绑定记忆 KB，形成三个档位：

| 档位 | 配置 | 效果 |
|------|------|------|
| 完整记忆 | 启用插件 + 绑定记忆 KB | L1 画像 + L2 情景检索 + Tools |
| 仅画像 | 启用插件，不绑 KB | L1 画像 + Tools（情景记忆不被检索） |
| 完全关闭 | 不启用插件 | 无注入、无 Tools，完全透明 |

## 第四个决策：群聊里的记忆归谁？

LangBot 很大一部分场景是群聊。群聊里的记忆隔离是个有意思的问题。

想象一个 QQ 群里有个 LangBot 机器人。用户 A 说"我喜欢咖啡"，用户 B 说"项目用的是 React"。下次 A 问"推荐个饮品"的时候，bot 应该知道 A 喜欢咖啡；但 bot 该不该知道"项目用 React"这个信息？

这取决于群的性质：

- **团队工作群**："项目用 React"这种信息应该是共享的，所有人都能受益
- **私聊 / 隐私群**：每个人的记忆应该互相隔离，A 的偏好 B 不应该知道

所以做了两种隔离模式，在创建知识库时配置：

**`shared` 模式（默认）**：所有用户共享一个记忆池。

```
用户A："项目用 React" → 存入 {session_key}:shared
用户B："部署在 k8s"  → 存入 {session_key}:shared
任何人提问 → 检索共享池（知道项目用 React、部署在 k8s）
```

shared 模式下，L1 核心画像从"用户画像"变成"群画像"——描述的不是某个人，而是这个群的集体特征和共识。所有群成员共同"调教"同一个 bot。

**`per_user` 模式**：每个用户独立的记忆池。

```
用户A："我喜欢咖啡" → 存入 {session_key}:{userA}
用户B："部署在 k8s" → 存入 {session_key}:{userB}
A 提问 → 只检索 A 的记忆（知道他喜欢咖啡）
B 提问 → 只检索 B 的记忆（不知道有人喜欢咖啡）
```

隔离模式在创建知识库时配置，不同 Pipeline 可以绑定不同隔离模式的知识库。

存储键的推导逻辑：

```python
# session_key = "{launcher_type}_{launcher_id}"，标识一个聊天会话
def get_user_key(session_key, sender_id, isolation):
    if isolation == "per_user" and sender_id:
        return f"{session_key}:{sender_id}"
    return f"{session_key}:shared"
```

## 记忆的管理

LLM 有三个工具来管理记忆，职责明确分离：

| 工具 | 操作 | 目标 | 用途 |
|------|------|------|------|
| `remember` | 写入 | L2 情景记忆 (ChromaDB) | 存事件、计划、经历 |
| `update_profile` | 写入 | L1 核心画像 (Binary Storage) | 存姓名、特征、偏好 |
| `recall` | 只读 | L1 核心画像 | 查看当前画像 |

`remember` 只管情景信息——有时间性的事件、计划、项目进展。`update_profile` 只管身份信息——稳定的个人特征、偏好。两个写入工具的边界很清楚，不会打架。

`recall` 只读取 L1 画像，不检索 L2 情景记忆。L2 的检索在每轮对话开始时由 KnowledgeEngine 自动完成——相关的情景记忆已经在 RAG context 里了，LLM 不需要主动去"搜记忆"。

用户也可以通过命令手动管理记忆：

```
!memory           - 查看记忆概况（隔离模式、画像摘要、KB 状态）
!memory profile   - 查看核心画像详情
!memory search <query> - 搜索情景记忆（向量语义检索）
!memory clear     - 清除所有记忆（画像 + 情景）
```

## 成本对比

这是我在做方案选型时最关心的一个维度。

| 环节 | 自动提取方案 | Tool 驱动方案（本方案） |
|------|------------|---------------------|
| 记忆写入 | 每轮：1 次额外 LLM + 1 次 Embedding | 仅 LLM 认为有价值时：Tool 调用（零额外 LLM） |
| 记忆检索 | 每轮：1 次 Embedding + 1 次向量查询 | 每轮：1 次 Embedding + 1 次 ChromaDB 查询 |
| 画像注入 | 无（没有画像概念） | 每轮：1 次 KV 读取（几乎为零） |
| 每轮额外 LLM 调用 | 1 次 | 0 次 |

两个方案的检索开销是一样的——都是一次 Embedding + 一次向量查询。差别在写入侧：自动提取方案每轮都要额外调一次 LLM 做记忆提取，而 Tool 驱动方案的写入发生在当前 LLM 调用中，零额外开销。L1 画像的注入成本几乎可以忽略——从 Binary Storage 读一个 JSON 对象，毫秒级别。记忆数据存在 LangBot 自带的 ChromaDB 里，不需要额外的基础设施。

## 跟 RAG 的关系

写到这里你可能已经发现了：**长期记忆的 L2 层本质上就是 RAG，只不过索引的对象从"用户上传的文档"变成了"LLM 自己记下的对话片段"。**

```
RAG:     用户上传文档 → 分块 → Embedding → 向量库 → 按相关性检索 → 注入 RAG context
记忆:    LLM 调 remember → 打标签 → Embedding → 向量库 → 按相关性检索 → 注入 RAG context
```

流程几乎一样，区别在于数据来源和写入方式。RAG 的数据由用户主动上传，记忆的数据由 LLM 在对话中自主提取。

这也是为什么这个记忆插件能复用 RAG 迁移时搭建的基础设施——同样的 `invoke_embedding`、`vector_upsert`、`vector_search` RPC 接口，同样的向量数据库抽象层。当时做 RAG 插件化架构的时候，把"宿主提供基础设施、插件定义业务策略"这个分层做干净了，现在记忆插件直接受益。

而 L1 核心画像则更接近 System Prompt 注入——体积小、始终生效，不需要检索。这跟 RAG 没什么关系，更像是一种持久化的 System Prompt 片段。

## 回到那个根本问题

前面两篇文章一直在追一个线索：**显式的外部架构会不会被模型的原生能力吸收？**

记忆也一样。目前我们在做的事情——用 Tool 调用写入记忆、用检索注入上下文——本质上是在模型外面手动搭了一套记忆系统。但如果有一天模型原生就支持跨会话记忆呢？

其实这已经开始发生了。ChatGPT 的 Memory 功能、Claude 的 Project Knowledge，都是平台层面对长期记忆的原生支持。它们做的事情跟我们这个插件没有本质区别——都是在 System Prompt 里注入持久化的用户信息。

区别在于控制权。平台原生的记忆功能是黑箱的——你不知道它记了什么、为什么记、什么时候会忘。而插件方案是完全透明的：用户可以 `!memory search` 检索记忆条目，可以 `!memory clear` 清除记忆，LLM 的每次 `remember` 调用都是可审计的。

对于 LangBot 这样的开源平台来说，这种透明性和可控性是核心价值。用户部署的是自己的机器人，记忆数据存在自己的数据库里，不依赖任何外部平台。

至于这些外部架构最终会不会被模型完全内化——大概率会，但那是模型能力发展的事。在那一天到来之前，我们能做的就是把脚手架搭得尽量干净，等到不需要的时候能轻松拆掉。
