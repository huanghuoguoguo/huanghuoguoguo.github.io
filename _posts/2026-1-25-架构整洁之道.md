## 引言

看书这件事，最好带着问题去翻页。

我一边读《架构整洁之道》，一边在心里反复念叨：项目一路迭代，业务方总催"快上快上"，研发方却想"别崩别崩"。快和稳就像拔河，绳子却只有一根。变化来之前，我能不能提前瞄到哪儿会松动？时间那么少，怎样算出 ROI，把有限的人力一把押在刀刃上？到底是处处先铺扩展，还是留些坑等真痛了再填？一旦不得不重构，又怎能让火车不熄火，一边换轨道一边持续交付？

这些疑惑不是凭空来的。上一份实习，我一个人扛了架构、需求、迭代、维护、用户故事，像守着小型开源项目（虽然的确是公司内部使用的小型项目），build system 的熟练度倒是涨了不少，可烂尾和遗憾也留下一堆。现在我想让书里的经验替我答疑，把曾经的弯路捋直。

在翻开《架构整洁之道》之前，我先补了三本"前传"，每本都给我一点提示，也留下一堆新问号：

- **《人月神话》**：我原本指望它告诉我"架构怎么跟着版本一起长大"，结果它把重点放在"人和月"对不上——人力往上加，进度并不会乖乖等比例缩水，沟通曲线反而翘得比房价还陡。读完我才意识到，项目不是搬砖，信息传递也有成本，必须先给团队尺寸划一条"弹性边界"，否则人越多越乱。可具体怎么划、划在哪，书里只给了警钟，没给尺子。
- **《测试驱动开发》**：我想借它给模块间搭些"护栏"，免得功能越长越歪。现实却是——边界一开始没想清，测试用例写得再密，也挡不住引用关系乱成蜘蛛网；后来把 AI 拉进来 vibe coding，上下文长度一满，提示词和 rules 说破嘴，AI 还是把代码带到沟里。
- **《持续交付》**：这本离我的困惑最近——它把"怎样让软件一直能跑、一直能发"拆成流水线。可惜我学生气未脱，看见新工具就想尝鲜，依赖 AI 飙车，重构比 commit 还勤快，结果步子太大，交付日期只能往后挪。

带着这三本留下的"半成品答案"，我打开《架构整洁之道》，想找补剩下的拼图：**怎样让架构在业务狂奔时依旧稳得住，又不必天天"炸楼重建"。** 如果再重来一次，在开始之初，我应该怎么做？

## 设计与架构到底是什么

> 工程师们经常相信的另外一个错误观点是："在工程中容忍糟糕的代码存在可以在短期内加快该工程上线的速度，未来这些代码会造成一些额外的工作量，但是并没有什么大不了。"相信这些鬼话的工程师对自己清理乱麻代码的能力过于自信了。无论是从短期还是长期来看，胡乱编写代码的工作速度其实比循规蹈矩更慢。

我理解好的架构能够带来更好的工作效率，但**如何在项目开始之初就意识到并加以设计？** 因为很多架构上的问题是在需求变更之后才发现的。冻结需求再评估最好的架构？显然不现实。

> The only way to go fast, is to go well.
> 要想跑得快，先要跑得稳。

持续交付解决了开发过程中因分支混乱导致的延期问题，使版本发布节奏可控；架构方法则提供了可遵循的设计原则，让我能在项目初期就建立清晰、可演进的技术框架，减少后期返工。

### 两个价值维度

> 对于每个软件系统，我们都可以通过行为和架构两个维度来体现它的实际价值。不幸的是，他们往往只关注一个维度，而忽视了另外一个维度。更不幸的是，他们常常关注的还是错误的维度，这导致了系统的价值最终趋降为零。

这就是我在迭代 Dooba 时的问题——我忽略了用户故事，迟迟没有将其交付于真实场景中。尽管架构在往我认为更好的方向走，但始终没有体现出**行为价值**。系统价值迟迟没有体现。

> 需求变更的范畴与形状，是决定对应软件变更实施成本高低的关键。从系统相关方的角度来看，变更需求的范畴都是类似的，成本也应该是固定的。但从研发者角度来看，持续不断的变更需求就像是要求他们不停地用一堆不同形状的拼图块拼成新的形状。

这里有一个体会很深的问题：对外部使用者来说，不知道修改需求对内部的变化有多大。在他们看来，加一个参数、引起一些额外行为的成本似乎很低，但有时实现起来很困难。

> 如果系统的架构设计偏向某种特定的"形状"，那么新的变更就会越来越难以实施。好的系统架构设计应该尽可能做到与"形状"无关。

如果是**可插拔的插件设计**，是否能让系统对"形状"更加中立？后面读到 OCP 的时候，我找到了答案。

> 如果事后业务部门提出了一项需求，而你的预估工作量大大超出他们的预期，这帮家伙通常会对你放任系统混乱到无法变更的状态而勃然大怒。

章末，作者强调软件行为与架构是所有人的事，架构师尤其避不开。我的看法是：不管项目多小，都得按可持续的标准来，一次性"能跑"不是终点，**可演进才是首要目标**。

## 编程范式总览

### 结构化编程

消除 goto 让不确定的行为变成确定的结果。用"可证伪"的顺序/分支/循环把 goto 的随意跳转关进笼子，第一次让程序流程变得**可推理、可调试**。

### 面向对象编程

封装、多态、继承通常用来描述面向对象，但 OO 不止于此——三大特性是"具象化的工具"，而面向对象本身是**更抽象的设计哲学**，用对象（职责+协作）去对业务建模。把 OO 等同于"有 class、能继承"就本末倒置了。

换一种说法：用封装+多态+继承把"变化"关进笼子，让修改被局限在有限的对象边界里，降低因需求变动带来的连锁反应。

### 函数式编程

用"不可变+纯函数"把"副作用"关进笼子，让并发、回滚、测试更容易被推理。

> 所有的竞争问题、死锁问题、并发更新问题都是由可变变量导致的。如果变量永远不会被更改，那就不可能产生竞争或者并发更新问题。

不可变和 Rust 的所有权看起来很像——都在把不确定性往零压。在架构层面，不可变真正的价值是把"时间"这个最难追踪的维度从程序里抽掉：数据一旦不会原地变脸，任何线程、任何时刻看到的都是同一份快照。Rust 的所有权模型做的是同一件事——通过编译期规则把"可变别名"直接封死，让"数据竞争"在编译器面前变成语法错误，而不是上线后的随机 Bug。

问题不是"不可变是否可行"，而是：

- 业务边界里哪些状态**必须可变**？把它们集中到明确的"可变核心"（Rust 的 `Mutex<Arc<T>>`，函数式的 STM），外围保持不可变。
- 用所有权/借用检查或纯函数式接口，把"可变别名"变成编译期非法，而不是靠人眼 Code Review。
- 对实在无法消除的可变点，给它们加上代数抽象（`Send + Sync`、`IO Monad`），让副作用显式化、可组合、可测试。

> 一个架构设计良好的应用程序应该将状态修改的部分和不需要修改状态的部分隔离成单独的组件，然后用合适的机制来保护可变量。软件架构师应该着力于将大部分处理逻辑都归于不可变组件中，可变状态组件的逻辑应该越少越好。

### 为什么要先讲范式？

因为前两章抛出的痛点就是"系统随时间腐化、需求变动成本指数级上升"。如果开发者只把"架构"当成包图、层图，而忽略代码里最基础的约束原则，再漂亮的分层也会被随意的副作用、全局可变、滥用继承的代码击穿。

- 没有结构化思维，流程任意跳转，就不存在可测试的单元
- 没有对象范式对"变化"的封装，再多端口-适配器也挡不住业务改动穿透所有层
- 没有函数式的不变性，高并发、高可回滚的持续交付就是空谈

> 这三个编程范式都对程序员提出了新的限制。每个范式都约束了某种编写代码的方式，没有一个编程范式是在增加新能力。也就是说，我们过去 50 年学到的东西主要是——什么不应该做。

> 软件构建并不是一个迅速前进的技术。今天构建软件的规则和 1946 年阿兰·图灵写下电子计算机的第一行代码时是一样的。尽管工具变化了，硬件变化了，但是软件编程的核心没有变。

回头看，学习路上很多收获确实是"什么不应该做"：不应该让分支和架构野蛮生长，不应该不加 review 就接受 AI 生成的代码。

## SOLID 原则

SOLID 的目标：

- 使软件可容忍被改动
- 使软件更容易被理解
- 构建可在多个软件系统中复用的组件

### SRP：单一职责原则

> 任何一个软件模块都应该只对某一类行为者负责。

该设计原则基于**康威定律**（Conway's Law）的推论——一个软件系统的最佳结构高度依赖于开发这个系统的组织的内部结构。

**这能提前看到变化点吗？** 需求访谈时问三句："这份数据/规则现在有谁用？各自目的是什么？过去半年改过多频繁？"只要出现两个部门+两种目的+任一方曾改，就直接记为"潜在劈叉点"。

即便是 MVP 也能用这招——第一周只有 CFO 需求？先只写 `PayCalculator` 接口+实现，其他端口留空接口。`Employee` 里绝不放"公用算法"，保持"无共享代码"习惯。等 COO 需求来了再把 `HoursReporter` 实现填上，依旧零冲突。这叫**"预留接缝（seam）"**——不是预留功能，不会多做功能，却避免了以后拆楼的成本。

最后的实现方法是将一个类中三个不同方向的方法拆成三个子类。虽然内部持有共同类型的私有数据，但对外暴露的行为不一致，且尽量互不影响。

但如果我就是认为要把共有方法抽到父类呢？override？—— **"公共基类+数据上移"在面向复用时很香；在面向变化时，它把共享代码变成共享伤口。** SRP 的核心是把会随不同演员而改变的代码切成互相看不见的碎片——哪怕碎片里暂时长得一样。

### OCP：开闭原则

> 设计良好的计算机软件应该易于扩展，同时抗拒修改。

如果对原始需求的小小延伸就需要对原有系统进行大幅修改，那架构设计显然是失败的。怎么做到？

> By properly separating the things that change for different reasons (the Single Responsibility Principle), and then organizing the dependencies between those things properly (the Dependency Inversion Principle).

先将满足不同需求的代码分组（SRP），再调整分组之间的依赖关系（DIP）。

在 Dooba 的架构演进里，任务和插件被彻底剥离：前者通过本地文件系统扫描自动注册，后者用 git clone 远程拉取再动态装载——要加新能力，只需"丢进去"或"拉下来"，核心代码一行不动。Spring Boot 也是同理：它对扩展敞开大门，只要按约定声明 Bean、实现接口，框架就把你织入生命周期；而容器如何启动、调度、释放，依旧牢牢掌握在框架手里。

**好的框架只暴露行为扩展点，不暴露执行控制权。**

### LSP：里氏替换原则

> 如果想用可替换的组件来构建软件系统，那么这些组件就必须遵守同一个约定，以便让这些组件可以相互替换。

语法上能替换还不够，**语义上必须让客户端毫无感知**。只要替换后需要修改客户端代码、测试用例或业务预期，这个继承关系就是设计失误。

### ISP：接口隔离原则

别让客户端被迫依赖它用不到的方法，更别让它们因为别人的方法被改而重新编译/部署。**接口要按"使用者"而不是"实现者"来划分**，小即是美。

### DIP：依赖反转原则

> 如果想要设计一个灵活的系统，在源代码层次的依赖关系中就应该多引用抽象类型，而非具体实现。

**DIP = "面向接口编程 + 接口归高层所有"**，于是编译箭头掉头，高层再也不被低层的改动牵着走。

在 Dooba 架构中，core 内部的组件只持有"行为契约"——数据库、告警、消息等操作均依赖同层定义的抽象接口。真正的实现被放逐到 core 之外，通过依赖注入在运行时挂载。核心代码永远面向自己的接口编程，对具体实现既无编译依赖，也无运行时耦合。

### SOLID 小结

| 原则 | 一句话 |
|------|--------|
| SRP | 一个类只对一种变化原因负责 |
| OCP | 新增行为靠加代码，不靠改代码 |
| LSP | 子类型替换后，客户端行为不变 |
| ISP | 客户端只依赖它用到的方法 |
| DIP | 高层定义接口，低层实现接口，依赖方向与控制流相反 |

## 组件设计

> 在经历了 50 年的演进之后，组件化的插件式架构已经成为我们习以为常的软件构建形式了。

### 组件聚合

究竟哪些类应该被组合成一个组件？三个原则彼此竞争：

- **REP（复用/发布等同）** + **CCP（共同闭包）**：黏合性原则，让组件变大
- **CRP（共同复用）**：排除性原则，让组件变小

> 一个软件项目的重心会从该三角区域的右侧开始，先期主要牺牲的是复用性。然后，随着项目逐渐成熟，其他项目会逐渐开始对其产生依赖，项目重心就会逐渐向左侧滑动。

**项目的迭代是逐渐调整的，不要妄想在一开始就建设好架构。** 当下适用的分割方式可能明年就不再适用，组件的构成安排应随着项目重心的不同而不断演化。平衡和变化是永恒不变的真理。

### 组件耦合

> 组件结构图是不可能自上而下被设计出来的。它必须随着软件系统的变化而变化和扩张，而不可能在系统构建的最初就被完美设计出来。

组件依赖关系图的目标是**隔离频繁的变更**——不希望无关紧要的 GUI 变更影响到业务逻辑组件，也不希望报表的增删操作影响到高阶策略。如果在设计具体类之前就来设计组件依赖关系，几乎必然失败，因为此时对共同闭包一无所知。**组件依赖关系必须随项目的逻辑设计一起演进。**

在 Dooba 中，告警插件、消息插件、eventbus 等是有依赖关系的。自身调度采集是稳定的，任务是不稳定的。

> 如果一个系统中的所有组件都处于最高稳定性状态，那么系统就一定无法再进行变更了。

**一个组件的抽象化程度应该与其稳定性保持一致**——如果一个组件想要稳定，它就应该由接口和抽象类组成，以便将来做扩展。

## 什么是软件架构

早在大一的时候，一个亲戚跟我说"要做架构师，而不是死写代码的"。他不是做互联网的，不清楚架构师和码农的区别，当时的我也不知道。没人一出校门就是架构师，只有经历产品迭代、踩坑、重构，把关注点从函数级正确性扩展到系统全生命周期，才迈出第一步。

> 软件架构师自身需要是程序员，并且必须一直坚持做一线程序员。如果不亲身承受因系统设计而带来的麻烦，就体会不到设计不佳所带来的痛苦，接着就会逐渐迷失正确的设计方向。

> 软件架构设计的主要目标是支撑软件系统的全生命周期，设计良好的架构可以让系统便于理解、易于修改、方便维护，并且能轻松部署。软件架构的终极目标就是最大化程序员的生产力，同时最小化系统的总运营成本。

这里有个有趣的论点：**软件架构对系统运行的影响远不及它对开发、部署和维护的影响。** 几乎任何运行问题都可以通过增加硬件来解决，而硬件远比人力便宜。在软件系统的所有方面中，**维护所需的成本是最高的**。

> 软件架构师的目标是创建一种系统形态，该形态会以策略为最基本的元素，并让细节与策略脱离关系，以允许在具体决策过程中推迟或延迟与细节相关的内容。

书中列了一堆"不应该过早决定"的东西：数据库、Web 服务、REST 模式、微服务框架、依赖注入框架……高层策略不应该和这些细节有关。**一个优秀的软件架构师应该致力于最大化可选项数量。**

在 Dooba 的设计过程中，一开始就耦合了存储层，这是不对的。后续不管是 DuckDB、LSM 树、还是 CSV 文本文件，都不应该影响上层的决策。

### 团队结构与架构

> 任何一个组织在设计系统时，往往都会复制出一个与该组织内沟通结构相同的系统。

不同的团队结构应该采用不同的架构设计。对一个只有五个开发人员的小团队来说，完全可以高效地共同开发单体系统，软件架构在早期开发中反而可能是障碍。这可能就是许多系统没有设计良好架构的原因——它们的开发团队起初都很小。

前段时间有单体 Spring Boot 取代微服务的公司——**"部署成本越高，可用性越低"**，微服务把"开发期便利"换成了"部署期负担"，当团队规模、日活、预算还没到临界点时，单体反而更香。

### 关于重复

> 架构师们经常会钻进一个牛角尖——害怕重复。

这是 TDD 过程中我经常碰到的问题——我经常会因为想要消除重复而不得不停下来。但书里提醒：有些重复只是表面重复，背后的变化原因不同，强行合并反而违反 SRP。

## 划分边界

> 怎样的决策会被认为是过早且不成熟的呢？答案是那些决策与系统的业务需求（也就是用例）无关。这部分决策包括我们要采用的框架、数据库、Web 服务器、工具库、依赖注入等。

Dooba 最初只是一个单文件脚本。由于需求确认不足、功能边界不清，代码被提前扩展，导致多次重构，持续交付中断。这个故事的本质是**把业务风险和技术风险搞混了**——用技术方案去解决尚未出现的业务问题，把"可能"当成"必然"，把"以后"当成"现在"。**先花成本再等需求，而非先等需求再花成本。**

测试过程中完全可以先把整个系统写成**纯内存实现**，同时用接口把"存储层"与"计算层"彻底切开：

- 计算层只依赖 `Storage` 接口，看不到数据存储层的任何细节
- 内存实现 `InMemoryStorage` 先跑通所有业务路径，单元测试秒级反馈
- 然后逐层替换：引入 `DuckDBStorage` 实现同一接口，只补集成测试
- 最后才写 `DistributedStorage`（Raft、gRPC 等），只测网络分区与恢复逻辑

之前面试被问到怎么测试我写的分布式调度框架，同理：先把调度策略、任务状态机写成内存版，本地单线程测试覆盖 80% 逻辑；再写 `InProcessTransport` 模拟多节点，多线程测并发与锁；最终才用 `NettyTransport` 跑真集群，只测网络超时、重试、分区。**框架可向下缩放：单机单测秒级运行，集成用例通过 Testcontainers 一键起容器。**

### 策略与层次

当高层组件需要调用某个底层实现时，先把这部分能力抽象成接口，并把接口提升到与高层同层（或更高层）。此后高层只依赖该接口，不再引用任何具体实现，编译期箭头便自然"向上"指向更稳定的抽象，从而完成依赖反转。

### 业务逻辑

> 用例并不描述系统与用户之间的接口，它只描述该应用在某些特定情境下的业务逻辑，与数据流入/流出系统的方式无关。

> 请一定不要在数据结构中直接引用业务实体对象。这两个对象存在的意义是非常不一样的，随着时间的推移会以不同的原因、不同的速率发生变更。

实践中的做法：Controller 只见到 Request/Response，Repository 只见到 Entity，两者通过转换器互联。数据库加索引、改列名 → 只改 Entity + Mapper；接口加字段 → 只改 Request/Response + Mapper。任何一层变化都不会强制另一层重新编译发布，SRP 与 CCP 同时得到满足。

## 整洁架构

> 软件的用例层中通常包含的是特定应用场景下的业务逻辑，这些用例引导了数据在业务实体之间的流入/流出，并指挥着业务实体利用其中的关键业务逻辑来实现用例的设计目标。

跨越边界的数据传递需要理解**腐化层**的作用——确保不同层之间的数据模型互不污染。

### 尖叫的软件架构

> 良好的架构设计应该尽可能地允许用户推迟和延后决定采用什么框架、数据库、Web 服务以及其他与环境相关的工具。良好的架构设计应该只关注用例，并能将它们与其他的周边因素隔离。

这里的"用例"不是泛指功能需求，也不是单个输入输出，而是**一条完整的、具备业务价值的处理流程**：谁发起、通过什么通道、系统必须做什么/不允许做什么、返回什么结果。良好架构把技术决策推迟到项目后期，甚至允许随时替换，而不用改动用例代码本身。

> Web 只是一种交付手段——一种 IO 设备。应用程序采用 Web 方式来交付只是一个实现细节，这不应该主导整个项目的结构设计。

选择 Spring Boot、Flask 还是 Django，理由是什么？框架再强大也只是工具，它的文档和教程却常常带着"传教士"口吻。选框架前应该先想清楚：用它能推迟哪些决策？以后换掉要改多少代码？它是否侵入核心实体？**让框架留在细节层，用例层保持框架无感；能随时拔插，才算真正"采用"而非"被采用"。**

### 层次与边界

> 作为软件架构师，我们必须有一点未卜先知的能力。有时候要依靠猜测——当然还要用点脑子。这不是一次性的决定。架构师必须持续观察系统的演进，时刻注意哪里可能需要设计边界。

YAGNI 原则说得对——过度工程设计往往比设计不足还糟糕。但如果发现确实需要设置架构边界却没有事先准备，再添加边界的成本和风险往往很高。

> It takes a watchful eye.

依然是拥抱变化，在变化中持续做出应对。持之以恒，一刻也不能放松。

### Main 组件

> Main 组件是系统中最细节化的部分——也就是底层的策略，它是整个系统的初始点。

想想 Spring Boot 的 main 函数——当我们将 Main 组件视为一种插件时，用架构边界将它与系统其他部分隔离开，系统的配置是不是就变得更容易了？

### 服务：宏观与微观

"只要使用了服务，就等于有了一套架构"——这显然是错误的。

> 服务这种形式说到底不过是一种跨进程/平台边界的函数调用而已。有些服务会具有架构上的意义，有些则没有。

服务之间在变量层面做到了彼此隔离，但它们还是可能通过共享资源而耦合。给服务之间传递的数据记录中增加一个新字段，每个需要操作这个字段的服务都必须做出相应的变更——**服务之间通过共享数据结构而间接耦合。**

### 测试边界

> 当开发者意识到一些简单的修改就会导致大量的测试出错时，他们自然就会抵制修改。

但如果对系统重构，确实会导致一些测试失败，怎么办？答案是**不要依赖于多变的东西**。GUI 往往是多变的，因此通过 GUI 来验证系统的测试一定是脆弱的。业务逻辑不通过 GUI 也应该可以被测试。

在 Dooba 的测试过程中，不应该测试具体的业务逻辑流转，而是针对 core 下的核心组件进行测试。**当把数据层、Web 层全部视作可插拔的插件时，跨越模块的数据将其包装为腐化层，确保数据的流转通路是可确定的。** 组件内的测试仅考虑范围内的流转，假设外部传来的数据是对的。

### 整洁的嵌入式架构

这个顺序很重要：

1. **先让代码工作起来**——如果代码不能工作，就不能产生价值
2. **然后再试图将它变好**——通过重构，让自己和他人更好地理解代码
3. **最后再试着让它运行得更快**——按照性能提升的"需求"来重构代码

## 实现细节

### 数据库只是实现细节

> 从系统架构的角度来看，数据库并不重要——它只是一个实现细节。如果就数据库与整个系统架构的关系打个比方，它们之间就好比是门把手和整个房屋架构的关系。

作者说"这个比喻肯定会招来非议，这种架我吵过很多次了"——他澄清的是：这里讨论的不是数据模型，数据模型当然重要，但数据库只是存取数据的工具。

持久化层中选用什么数据库只是一个实现细节，不管是 DuckDB、MySQL、SQLite 还是大文件文本，在项目设计之初不应该考虑用何种持久化实现——除非系统实现需要依赖这个数据库的特性。**业务逻辑不应该考虑数据存储的方式，只是一味地存取。至于性能，替换数据库对业务逻辑来说不应该有任何感知。**

作者还讲了一个故事：他坚持系统核心架构中不应该引入关系型数据库，但客户的购买合同里"关系型数据库"是必选项——客户自己也不知道为什么需要，但它是真实存在的需求。作者承认自己"错了"，不是在工程层面，而是忽视了客户的非理性但真实的需求。我觉得作者其实可以提供另一种数据安全方案来证明 SQL 不是不可替代的——因为客户想要的本质上是数据安全吧？

### 框架是实现细节

> 框架作者想让我们与框架订终身——这相当于我们要对他们的框架做一个巨大而长期的承诺，而在任何情况下框架作者都不会对我们做出同样的承诺。这种婚姻是单向的。

框架可能违反依赖关系原则，要求我们把代码引入到业务实体中；框架可能朝着我们不需要的方向演进；框架帮助我们实现早期功能，但随着产品成熟，与框架斗争的时间可能比框架帮助我们的时间还长。

像 xxl-job 那样，可以使用 Spring Boot 注入 context，也可以自己实现 context，同时为 Spring Boot 编写额外的 starter，而不是完全依赖 Spring Boot 完成工作。**测试不应该跨组件进行（集成测试除外），框架应该被隔离在组件边界之外。**

## Dooba 复盘：如果重来一次

整本书读下来，Dooba 的影子反复出现。把散落的反思串起来：

1. **过早耦合存储层**：一开始就绑定了具体数据库，导致上层决策被下层变动反复拖累。应该先用 `InMemoryStorage` 跑通所有业务路径，存储层作为插件后挂。
2. **忽略行为价值**：架构一直在往我认为更好的方向走，但迟迟没有交付于真实场景。用户故事被搁置，系统价值没有体现。应该先交付最小可用功能，小步快跑。
3. **重构太勤快**：重构比 commit 还频繁，步子太大，持续交付中断。学到新方法就想推倒重来，忽视了"先跑通再变好"的节奏。
4. **测试不分层**：没有把"可测"与"不可测"彻底分层，测试火力没有集中到核心契约和极端场景。应该确保组件内测试仅考虑范围内的流转，跨模块数据包装为腐化层。
5. **边界没想清就扩展代码**：最初只是单文件脚本，需求确认不足就提前扩展，导致多次重构。应该先问清楚：是持续监控告警，还是收集指标提供接口？

**如果重来**：先交付最小可用功能；用接口切开存储层和计算层；每次改动先跑测试再提交；不为尚未出现的需求预先设计；边界不确定时预留接缝而非预留功能。

## 如何测试分布式调度框架

读完整洁架构后，我重新回答一下面试时被问到的问题——怎么测试我写的分布式调度框架：

**前提**：框架是业务无关的，所以用例必须跟具体业务无关。先圈定功能边界，找到谦卑对象（跟外部交互的部分：网络连接、数据库存储等），把"可测"与"不可测"彻底分层。

然后按层级展开：

- **契约测试**：序列化/反序列化、心跳、重试、超时、幂等 token。JUnit + Testcontainers 起两个节点，本地 loopback 网络，只测消息语义，不测业务。
- **状态机一致性测试**：任务状态（Pending→Running→Success）在任何并发路径下都不可倒退。用 Model-Based Testing 随机生成事件序列，断言状态符合预期模型。
- **容错与恢复测试**：leader 宕机、网络分区、进程 OOM、磁盘只读。Toxiproxy 模拟网络延迟/丢包，Kill -9 节点后验证任务重调度且无重复执行。
- **性能基准测试**：吞吐量（任务/秒）、调度延迟（提交→开始）、Fail-over 时间。固定内存 DB + 本地网络，排除外部波动，得到框架自身上限。
- **时间相关测试**：注入 `Clock` 接口，测试时给出固定时间流，验证超时、重试退避、Cron 表达式精确触发。
- **"业务无关"验证**：让框架分别调度空任务、CPU 密集型计算、IO 密集下载，三者混合后仍满足吞吐/延迟 SLA，即证明与业务逻辑解耦。

**怎么提高系统的可测试性？** 两条做完就够了：

1. **先画边界再写代码**：用例层、领域层、适配器层先切出清晰接口；框架、数据库、网络、文件都挡在适配器里，测试时换成内存桩即可秒级运行。
2. **让决策点都落在多态对象里**：调度策略、计费规则、鉴权逻辑全部抽象成 Strategy/Policy 接口；测试用例直接 new 不同实现类注入，无需启动整个上下文。

边界清晰 → mock 点可枚举；策略多态 → 分支可单独实例化。可测试性从"事后补桩"变成"架构自带"。

## 结语

> 让我们现在编写的代码不对未来的代码产生阻碍是一项非常重要的技能，通常需要花费多年的时间才能掌握。

看完这本书，我最大的收获是**终于能把那些痛点说清楚了**——耦合、过早决策、边界失守、影响范围不可控。

用我自己的话说：

1. 不要操之过急，值得花时间在项目之初考虑系统架构
2. 不要放弃敏捷，拥抱变化，开发过程中策略有调整很正常
3. 不要盲目追新，合适的才是最好的——不仅考虑业务场景，也考虑团队学习成本
4. 看到项目中易变和不易变的部分，做好抽象和边界划分，对测试和部署有莫大好处
5. 用量化方法评测系统是否整洁

把前面几本书串起来看：项目构建远非"加人就能提速"的简单算术，真正决定迭代速度的是边界切得是否干净、模块是否各自可测可部署。让新人第一天就能拉出最小可运行单元，而不必先理解整套"祖传上下文"，才是架构要解决的问题。

实习时中科曙光的调度平台——infra 层只放 Job 框架、资源持久化、事件总线、告警，不引用任何 biz 对象；biz 层通过接口反向依赖 infra。当时只觉得"干净"，如今能说清：这是 DIP+CCP 的活例子。我曾向面试官描述这套设计，却满口"用了内存分布式一致性、用了时间轮"，一句抽象边界都没讲，功能点也随细节一起淹没。

**书没给我万能蓝图，但给了度量尺**：以后看别人的代码，先找"用例是否依赖细节"；讲自己的方案，先说"边界在哪，什么能推迟决定"，再谈实现。

我不想争"架构师"这个头衔，只是一个人迭代工具时，想让它好维护、好扩展、也更好用。这本书给了我一把尺子：每次加功能前，先量一量边界清不清、替换成本高不高、用户故事能不能独立演进。码农还是架构师无所谓，能把变化算清楚，让代码今天跑得动、明年不翻车，就足够了。

我非常推荐大家在看一本书之前，已经积攒了不少疑问——不管是编码中、架构设计中、迭代中、扯皮中的问题。如果没有问题，不建议看书。只有亲自踩过坑、被变化抽过耳光，才真正懂得什么是可演进的设计。
