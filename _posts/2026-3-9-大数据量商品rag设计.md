## 背景

前两天面试被问到一个系统设计题：**设计一个基于京东商品的 RAG 系统，要求能处理海量商品数据、覆盖多种类目，面向京东用户提供智能问答与推荐。**

这个问题的核心挑战在于两点：一是**数据量大**（京东 SKU 数量在十亿级别），二是**商品种类繁多**（从 3C 数码到生鲜食品，属性差异巨大）。

拿到题目后我的第一反应是：这不是一个标准 RAG 能解决的问题。通用 RAG 的流程是"Query → Embedding → 向量检索 → 拼 Prompt → LLM 生成"，但电商场景下这套流程至少有三个致命缺陷：

- **纯语义检索不够精确**：用户说"iPhone 15 Pro 256G 黑色"，这是一个精确的属性筛选需求，走向量近似搜索反而不如结构化过滤。
- **商品数据高度结构化**：价格、库存、SKU 属性这些数据本身就适合传统数据库或搜索引擎来查，没必要全部灌进向量库。
- **时效性要求高**：向量库的更新成本远高于 ES 或 Redis，价格和库存这种分钟级变化的数据不适合放在向量索引里。

所以混合检索不是"锦上添花"，而是**必须的**。在这个前提下来做架构设计。

---

## 1. 架构设计：先定角色，再选实现

和日志系统设计一样，我的习惯是**先把架构中每个位置需要的角色定下来，再去选具体实现**。不要一上来就想"用 Milvus 还是 Pinecone"，而是先确认"这里需要一个语义检索引擎"。具体用什么填充，取决于量级、团队技术栈和运维能力。

整个系统可以拆成四个角色层：

```
┌──────────────────────────────────────────────────────────────┐
│                   查询理解层                                   │
│   角色：理解用户意图，抽取实体，扩展查询                        │
│   子任务：意图识别 / 实体抽取 / 查询扩展 / 用户画像融合          │
└──────────────────────────────────────────────────────────────┘
                               ↓
┌──────────────────────────────────────────────────────────────┐
│                   混合检索层                                   │
│   角色：从海量商品中快速召回候选集                               │
│   ┌────────────┐  ┌────────────┐  ┌────────────────────┐     │
│   │ 语义检索引擎 │  │ 结构化检索  │  │ 关系推理引擎        │     │
│   │ (模糊语义)  │  │ (精确过滤)  │  │ (属性关联/兼容性)   │     │
│   └────────────┘  └────────────┘  └────────────────────┘     │
│                  ↓ 融合排序 (RRF / 加权融合)                    │
└──────────────────────────────────────────────────────────────┘
                               ↓
┌──────────────────────────────────────────────────────────────┐
│                   排序层                                       │
│   角色：对候选集做精细化排序，融入业务规则                       │
│   粗排 (双塔模型) → 精排 (交叉编码器) → 业务过滤 (库存/地域)    │
└──────────────────────────────────────────────────────────────┘
                               ↓
┌──────────────────────────────────────────────────────────────┐
│                   生成层                                       │
│   角色：基于检索结果组装上下文，生成自然语言回答                  │
│   上下文组装 + LLM 生成 + 事实校验                              │
└──────────────────────────────────────────────────────────────┘
```

每一层定义的是**职责**，而不是技术选型。比如"语义检索引擎"这个角色，可以用 Milvus、Faiss、甚至 PostgreSQL 的 pgvector 来实现，取决于数据量级。十亿级向量需要分布式向量库，百万级向量一个单机 Faiss 就够了。

## 2. 数据索引设计：解决数据量大的问题

### 2.1 多级索引策略

```
索引层级：
├── 一级：类目索引（3C数码 / 家电 / 服饰 / 食品等 20+ 大类）
├── 二级：品牌索引（每个类目下的热门品牌）
├── 三级：商品向量索引（按类目分片，每片 < 100 万向量）
└── 四级：实时索引（今日上新 / 促销商品，小时级更新）
```

面试时被追问了一个问题：**为什么按类目分片而不是按 SKU 哈希分片？**

按 SKU 哈希分片负载均衡更好，但有一个很大的缺点：**跨品类查询时需要广播到所有分片**。而实际场景中，用户的查询通常有明确的品类指向（"推荐一个手机"、"买个冰箱"），按类目分片可以在查询理解层确定类目后**只查对应分片**，大幅减少检索范围。

当然也存在跨类目的查询（"送女朋友什么礼物好"），这时候需要先通过意图识别映射到若干可能的类目，再做多路并行召回。这是类目分片的代价，但在大多数查询都有品类指向的前提下，这个代价是值得的。

### 2.2 向量维度的取舍

向量维度的选择本质上是**精度和性能的权衡**：

| 向量类型 | 维度 | 用途 | 说明 |
|---------|------|------|------|
| 商品标题向量 | 384 维 | 粗排召回 | 轻量级，速度优先 |
| 商品详情向量 | 1024 维 | 精排 | 详细语义，精度优先 |
| 图片向量 (CLIP) | 512 维 | 以图搜图 | 多模态检索 |

粗排阶段用低维向量快速筛掉大量不相关商品，精排阶段再用高维向量做细致的语义匹配。这和搜索引擎的"粗排 → 精排"两阶段是同一个思路：**每一阶段用和其候选集规模匹配的计算量。**

### 2.3 存储分层：每个组件只做自己擅长的事

一个关键的设计决策是**不要把所有数据都塞进向量库**。向量库擅长的是 ANN（近似最近邻）检索，不是结构化查询。合理的做法是按数据特征分配到不同的存储角色：

| 存储角色 | 数据内容 | 职责 |
|---------|---------|------|
| 语义检索引擎（向量库） | 向量 + SKU ID | 语义召回 |
| 结构化检索引擎（搜索引擎） | 品牌、价格、品类、属性 | 精确过滤 + 全文检索 |
| 实时数据层（缓存） | 价格、库存、促销 | 高频变动数据的实时读取 |
| 全量数据层（关系型/宽表存储） | 商品详情、评价、FAQ | 生成阶段按需拉取 |

这样每个组件只做自己擅长的事，也方便独立扩缩容。至于具体用 Milvus 还是 Faiss、用 ES 还是 OpenSearch，那是实现层面的选择，架构层面先不锁定。

商品数据的结构大致如下：

```json
{
  "sku_id": "100012043978",
  "structured_data": {
    "title": "Apple iPhone 15 Pro Max 256GB 原色钛金属",
    "category_path": "手机数码 > 手机 > 苹果手机",
    "brand": "Apple",
    "attributes": {
      "屏幕尺寸": "6.7英寸",
      "存储容量": "256GB",
      "颜色": "原色钛金属",
      "5G网络": "支持"
    },
    "price": {
      "current": 9999.00,
      "original": 10999.00,
      "history_low": 9499.00
    },
    "stock": {
      "status": "现货",
      "warehouse": ["北京", "上海", "广州"]
    }
  },
  "textual_content": {
    "detail_text": "商品详细介绍...",
    "user_reviews_summary": "用户好评率98%，主要夸赞拍照效果和续航...",
    "faq": ["是否支持双卡？", "保修期多久？"]
  },
  "temporal_data": {

    "update_time": "2024-06-15T10:00:00Z"
  }
}
```

其中 `structured_data` 走结构化检索引擎，`textual_content` 生成向量后走语义检索引擎，`temporal_data` 走实时数据层。**一份数据，按特征拆到不同的存储角色里。**

## 3. 检索策略：解决商品种类繁多的问题

### 3.1 多路召回

商品种类繁多意味着单一召回路径无法覆盖所有查询场景，需要多路并行召回再融合：

| 召回路径 | 适用场景 | 说明 |
|---------|---------|------|
| 语义向量召回 | 模糊描述（"拍照好的手机"） | 向量相似度检索 |
| 属性过滤召回 | 精确筛选（"iPhone 15 Pro 256G"） | 结构化检索引擎 + 属性倒排索引 |
| 协同过滤召回 | 个性化推荐（"猜你喜欢"） | 用户行为序列建模 |
| 关系推理召回 | 关联查询（"兼容 iPhone 15 的耳机"） | 知识图谱（兼容/配件/替代关系） |
| 实时热召回 | 时效性需求（"今日特价"） | 实时计算 + 缓存 |

多路召回的结果通过 **RRF（Reciprocal Rank Fusion）** 或加权融合来合并排序。RRF 的好处是不需要对各路召回的分数做归一化，实现简单且效果稳定。

### 3.2 查询意图分类

```
意图类型：
├── 商品发现型："推荐一款适合打游戏的笔记本"
│   → 策略：向量检索 + 属性过滤 + 评价摘要生成
│
├── 商品对比型："iPhone 15 和华为 Mate 60 哪个好"
│   → 策略：精准召回两款商品 → 属性对齐 → 生成对比表格
│
├── 售后咨询型："这个冰箱保修多久？"
│   → 策略：精准召回商品 → 抽取售后政策 → 直接回答
│
├── 价格敏感型："历史最低价是多少？"
│   → 策略：召回商品 → 查询价格曲线 → 生成趋势分析
│
└── 凑单查询型："还差50元免运费，推荐个商品"
    → 策略：购物车协同过滤 + 价格区间过滤
```

这里有一个容易踩的坑：**意图分类的粒度不是越细越好**。粒度太细会导致分类准确率下降，而一旦意图识别错了，后面整条链路的策略都会偏。实际工程中更务实的做法是：

1. **粗粒度意图分类**（3-5 类），保证高准确率
2. **在召回阶段用多路融合来兜底**——即使意图判断不够精确，多路召回的冗余也能保证不漏掉相关结果
3. **在重排序阶段再做细粒度调整**，通过 Cross-Encoder 对候选结果做精细打分

这比依赖一个完美的意图分类器要鲁棒得多。不要在链路的最前端放一个精度不够的分类器，然后指望后面的环节能弥补它的错误——错误在链路中是会放大的。

## 4. 生成层：LLM 能做什么、不能做什么

### 4.1 上下文组装策略

LLM 的上下文窗口是有限的，需要合理分配"预算"：

```
上下文预算分配（假设 8K 上下文）：
├── 系统提示词：500 tokens
├── 用户查询 + 对话历史：200 tokens
├── 商品基础信息（Top-3）：1500 tokens
│   ├── 商品1：标题 + 价格 + 核心属性 + 评价摘要 (500)
│   ├── 商品2：同上 (500)
│   └── 商品3：同上 (500)
├── 详细对比数据（如果是比较查询）：2000 tokens
├── 用户评价精选：2000 tokens
├── 促销/售后信息：1000 tokens
└── 生成预留：800 tokens
```

### 4.2 电商场景下的幻觉问题

通用 RAG 出现幻觉可能只是答案不准，但电商场景下后果严重得多：

- **编造价格**：说"当前售价 2999 元"但实际是 3999 元 → 用户投诉，平台赔钱
- **虚构促销**：说"支持满减"但活动已结束 → 信任崩塌
- **属性错误**：说"支持 5G"但实际不支持 → 退货纠纷

所以生成层必须对 LLM 的能力边界做清晰的划分：

| LLM 负责 | LLM 不负责 |
|----------|-----------|
| 组织语言、生成推荐理由 | 价格、库存等事实数据 |
| 对比分析、购买建议 | 促销活动信息 |
| 评价摘要、优缺点总结 | 商品规格参数 |

**核心事实数据必须从实时数据源直接拉取，走模板化填充**，LLM 只负责"说人话"的部分。生成结果中的数字和属性值还要与数据源做交叉比对，作为事后校验的兜底。

这个原则说白了就是：**让 LLM 做它擅长的事（语言组织），不让它做它不擅长的事（记忆事实）。** 架构设计的一个基本思想就是不要让任何组件承担超出其能力边界的职责。

### 4.3 生成回答示例

> **用户**："3000 元左右，拍照好的手机推荐"
>
> **系统回答**：
>
> **推荐 Top-3：**
> 1. **小米 14** — 徕卡光学镜头，主摄 5000 万像素，售价 3299 元
> 2. ...
>
> **对比维度：**
>
> | 型号 | 主摄 | 夜景表现 | 价格 | 适合人群 |
> |------|------|---------|------|---------|
> | 小米 14 | 5000 万 | 优秀 | 3299 | 全能型 |
> | ... | ... | ... | ... | ... |
>
> **购买建议：**
> - 注重人像：选 XX
> - 注重夜景：选 XX
>
> **当前促销：** 小米 14 支持 6 期免息，以旧换新最高抵 800 元

其中"售价 3299 元"和"6 期免息"是从实时数据源拉取的模板数据，不是 LLM 生成的。

## 5. 实时性保障与增量更新

### 5.1 不同数据类型的更新策略

| 数据类型 | 更新频率 | 技术方案 |
|---------|---------|---------|
| 价格/库存 | 分钟级 | 监听数据库变更日志 → 消息队列 → 增量更新缓存和检索引擎 |
| 新品上架 | 小时级 | 批量构建向量 → 灰度发布 → 全量推送 |
| 用户评价 | 天级 | 离线任务生成评价摘要 → 更新索引 |
| 促销信息 | 实时 | 缓存 + 规则引擎，查询时动态注入 |

### 5.2 向量索引的增量更新

很多 RAG 方案讲得好像"数据变了就重新 Embedding 然后更新向量库"很简单，但实际上这是个被低估的工程难题：

- **重新 Embedding 的计算成本很高**：十亿级商品全量重建一次向量，即使用 GPU 集群也要数小时
- **HNSW 这类图索引不支持高效的单点更新**：每次插入新向量都可能触发图结构调整，性能不可预期
- **一致性问题**：结构化检索引擎里的价格已经更新了，但向量库里的向量还是旧的，两边对不上

务实的做法是**区分什么数据需要进向量库、什么数据不需要**：

1. **结构化数据（价格/库存/促销）不走向量库**，直接从缓存或搜索引擎实时拉取
2. **向量只编码语义信息**（标题、描述、评价），这些内容变化频率低，可以接受天级更新
3. **新品通过增量索引处理**，定期做全量索引重建保证一致性

这又回到了前面"存储分层"的设计思路——向量库只负责语义检索这一件事，不要让它承担数据实时性的压力。

## 6. 性能与扩展性

### 6.1 数据分片

```
分片规则：
├── 3C数码（手机/电脑/相机）：单独集群，高频查询
├── 家电（大家电/小家电）：单独集群
├── 快消品（食品/母婴/美妆）：按品牌分片
└── 长尾商品：冷数据压缩存储，查询时按需加载
```

### 6.2 向量索引优化

- **索引类型**：HNSW（`ef_construction=200`，`M=16`），在召回率和查询速度之间取得较好平衡
- **量化压缩**：PQ（Product Quantization）可以把十亿级向量的内存占用从 TB 级降到百 GB 级
- **冷热分离**：近期被查询过的商品向量驻留内存，长尾商品走 SSD 缓存

这些优化本质上都是在做同一件事：**用可接受的精度损失换取可承受的资源成本。** 没有完美的索引参数，只有在具体数据集上 benchmark 后的最优权衡。

## 7. 评估体系

### 离线评估

- **检索准确率**：Top-10 命中率（对比搜索日志做 ground truth）
- **生成相关性**：人工标注回答相关性（1-5 分）
- **幻觉检测**：事实一致性检查（价格/属性是否与数据源一致）

### 在线评估

- **CTR（点击率）**：用户是否点击推荐商品
- **CVR（转化率）**：是否完成购买
- **对话满意度**：显式反馈 + 会话时长分析
- **重复查询率**：用户是否因回答不满意而重新提问

这里要注意一点：**电商 RAG 的评估不能只看检索指标。** 传统 RAG 评估聚焦 Recall@K、MRR 这些检索指标，但电商场景下最终目标是转化。一个 Recall 很高但推荐的都是超出用户预算的高价商品的系统，转化率不会好。评估体系必须把**业务指标（CTR/CVR）作为北极星指标**，检索指标只是过程指标。

## 8. 关键技术选型

| 架构角色 | 京东体量下的选择 | 说明 |
|---------|----------------|------|
| 语义检索引擎 | Milvus / Faiss 集群 | 十亿级向量需要分布式方案 |
| 结构化检索引擎 | Elasticsearch | 属性过滤 + 倒排索引 + 全文检索 |
| 实时数据层 | Redis | 价格/库存/促销的实时读取 |
| 全量数据层 | MySQL / HBase | 商品详情，生成阶段按需拉取 |
| 向量化模型 | BGE-large-zh-v1.5 | 中文语义理解好，开源可微调 |
| 重排序模型 | bge-reranker-large | Cross-Encoder，精排精度高 |
| 生成模型 | Qwen / GLM 系列 | 中文电商场景，支持私有化部署 |
| 关系推理 | 图数据库 | 商品兼容/配件/替代关系 |
| 实时数据同步 | 变更日志 + 消息队列 | 价格/库存变更的分钟级同步 |

左列是**架构角色**，右两列是**实现选择**。架构角色在需求分析阶段就能确定，实现选择可以推迟到技术评估阶段。如果团队已有 OpenSearch 的运维经验，结构化检索引擎用 OpenSearch 替代 ES 也完全没问题——架构不变，只是换了一个实现。

## 9. 与通用 RAG 的关键差异

| 维度 | 通用 RAG | 电商商品 RAG |
|------|---------|-------------|
| 数据特点 | 非结构化文本为主 | 结构化属性 + 非结构化描述混合 |
| 时效性 | 通常容忍天级延迟 | 价格/库存需要分钟级更新 |
| 多模态 | 可选 | 必须（商品图片是核心信息） |
| 业务规则 | 弱 | 强（促销逻辑、地域库存、用户等级） |
| 个性化 | 通常无 | 必须融合用户画像（浏览/购买/收藏历史） |
| 幻觉容忍度 | 中等 | 极低（价格错误直接导致资损） |

如果只能记住一个差异，那就是：**通用 RAG 的核心问题是"找到相关信息"，电商 RAG 的核心问题是"在找到相关信息的基础上，还要保证事实准确、数据实时、结果个性化"。** 后者的复杂度不在检索本身，而在检索之外的一切。

## 10. 个人经历的关联

本科期间做过一个**个性化推荐系统的 Android 社区应用**，规模和复杂度远不及京东这样的系统，但核心思路有相通之处：都需要根据用户画像（浏览/点赞/评论行为）来做内容的个性化排序。那个项目让我对协同过滤和用户行为建模有了实践层面的理解，也是面试中能比较自然地聊到"个性化画像融合"这个环节的基础。

从那个小项目到今天讨论的十亿级商品 RAG 系统，核心的思维方式是一致的：**理解用户意图 → 从海量候选中高效召回 → 排序择优 → 呈现结果。** 变化的是规模和工程复杂度，不变的是这个漏斗式的信息筛选逻辑。而架构设计的价值，就在于让这个漏斗的每一层都能独立演进，互不干扰。
