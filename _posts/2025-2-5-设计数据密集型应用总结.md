## 关于存储引擎的思考

读完第一部分最大的收获是理解了两类存储引擎各自解决的问题。

LSM-Tree 的核心思想是利用局部性原理：查询大概率命中最近写入的 key，所以把最近的数据放在内存（memtable）中，大幅提升写性能。通过异步合并不可变的段文件，控制存储空间。曙光的底层 kvdb 就是一个 LSM-Tree 数据库。

Kafka 是典型的日志结构存储，OceanBase 也是日志结构，它们根据 AP 和 TP 不同的侧重，充分利用存储介质的特性。InnoDB 是典型的面向页面的存储引擎。

经常看到的面试题"用较小的内存处理大数据"，本质上就两种思路：数据压缩（用 bit 替代 int），以及分段处理（MapReduce 思想，归并处理）。LSM-Tree 的合并过程本身就可以看作一种归并。

## 关于分布式的思考

在决定做从曙光偷来的调度 job 引擎的时候，我就考虑过节点间通信的问题。看到 xxl-job 使用 HTTP 进行调度器和执行器之间的交互，我知道必须加上通信机制。所以考虑了几个轻量级的 Raft 实现框架，但没能参透那些状态机和日志的使用方式，最终采用了依靠 Redis Stream 的方式，用轻量级的 MQ 实现节点间的相互通信，用 Redis 分布式锁实现选主。后续计划用 etcd 替代。

说到共识，之前了解过的算法主要是 ZAB、Raft、KRaft、Paxos 等。它们的共同点是：需要解决谁是当前集群中最大最新的（term），然后再决定谁是下一代领导者，决定下一个共识操作怎么被接受。

Redis 为什么引入哨兵机制而不是内置 Raft？因为 Redis 广泛应用于缓存场景，对一致性的要求相对较低，更注重性能和可用性。哨兵机制在这些场景下已经能提供足够的高可用支持。要从应用场景出发，理解设计选择。Redis 用 Gossip 协议传播元数据，分片情况下并不追求强一致。

## 各技术栈的复制与分片对比

| 技术栈 | 复制方式 | 分片方式 |
|-------|---------|---------|
| MySQL | 主从复制（binlog 同步），半同步复制，Galera 多节点同步 | 分库分表（中间件实现，如 ShardingSphere） |
| Redis | 主从复制（RDB + 增量同步），哨兵自动故障转移 | 哈希槽（16384 槽，CRC16） |
| Kafka | 分区副本，ISR 机制保证一致性 | 按 Topic 分区，支持多种分区策略 |
| RocketMQ | DLedger 复制（基于 Raft） | 按 Topic 分配消息队列 |
| OceanBase | Paxos 多副本强一致 | tablet 分片，多节点管理 |

### MySQL vs Redis

| 特性 | MySQL 主从 + 分库分表 | Redis 哨兵 + 分片 |
|------|---------------------|-----------------|
| 数据一致性 | 主从一致，分片独立 | 哨兵主从一致，分片哈希槽分配 |
| 高可用 | 需工具辅助故障切换 | 哨兵/分片内置自动切换 |
| 扩展性 | 分库分表水平扩展 | 增加主从节点 |
| 复杂性 | 需要中间件或应用层支持 | Redis 内置支持 |

MySQL 适用于需要高可用性和水平扩展的场景，但需要额外的中间件。Redis 的哨兵和分片架构由 Redis 内置支持，更轻量。
