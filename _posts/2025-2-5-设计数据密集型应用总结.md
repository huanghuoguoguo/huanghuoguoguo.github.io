从 1 月 16 号开始，到 2 月 4 号完，历时两个多礼拜粗读了这本书。之所以说"粗读"，是因为很多概念没经过实践，光靠目前接触的项目只能吃个大概。但这不影响它成为我读过的信息密度最高的一本技术书——几乎每一章都能和我之前做过的事情对上号，只是以前不知道那些零散的经验背后有一套完整的理论框架。

<!-- more -->

之前看 Redis 设计与实现的时候，根本不清楚 Redis 选用那些策略是为了解决什么问题。直到自己动手开发调度引擎、在曙光实习接触分布式存储，思考过诸多解决方法之后，才发现"原来这些问题前人都想过，而且想得比我深得多"。换做一年前的我，是绝对不会碰这本书的——超前学习的错误我犯过太多次了，这本书的内容正好匹配我当前的认知水平。

---

## 全书主线：三个目标和无处不在的 trade-off

DDIA 有一条非常清晰的主线：**可靠性、可扩展性、可维护性**。这三个词在第一章就提出来了，贯穿全书。

但读完之后我觉得，比这三个目标更重要的是书中反复出现的一个思维方式——**trade-off**。没有完美的方案，只有特定场景下更合适的方案：

- LSM-Tree 写快读慢，B-Tree 读快写慢——存储引擎的读写权衡
- 同步复制一致性高但受限于最慢副本，异步复制性能高但可能丢数据——复制策略的一致性-可用性权衡
- 强一致性实现成本高，最终一致性用户体验差——一致性级别的权衡
- 范围分区支持范围查询但容易热点，哈希分区分布均匀但不支持范围查询——分区策略的权衡

以前做项目的时候，总想找一个"最好的方案"。读完这本书之后我才意识到，技术选型就是在一堆代价里做取舍。你要先想清楚自己愿意牺牲什么，再谈方案。这是整本书给我最大的认知转变。

## 存储引擎：从"知道有这个东西"到"理解它为什么这样"

读完第一部分最大的收获是理解了两类存储引擎各自解决的问题，以及它们背后的设计动机。

### 写密集 vs 读密集

LSM-Tree 的核心思想是利用局部性原理：最近写入的数据大概率会被查询到，所以把最近的数据放在内存（memtable）中，写入只涉及 memtable 和顺序追加的 WAL，写性能极高。通过后台异步合并不可变的 SSTable 文件，控制存储空间。

B-Tree 则是另一个方向：数据存在固定大小的页中，分层组织，4 层 B-Tree 可以存储 256TB 数据。查询时沿着树结构直接定位，读性能极高。

曙光的底层 kvdb 就是一个 LSM-Tree 数据库，当时实习的时候只知道"写快读慢"，但不清楚为什么。读完书才理解：LSM-Tree 读取可能需要查多个 SSTable（从最新到最旧依次查找），这是它用写性能换来的代价。布隆过滤器能快速排除不存在的 key，稀疏索引能缩小搜索范围，但本质上读取的开销就是比 B-Tree 高。

做 OceanBase 大赛的时候，我在 miniob 上实现了 B+ 树索引和 Disk Buffer Pool。当时只是照着接口写，现在回头看，B+ 树的页分裂、WAL 写入、Buffer Pool 的 LRU 淘汰，每一个设计点都能在 DDIA 第三章找到理论依据。**先动手再读书，和先读书再动手，理解深度完全不同。**

### 日志结构思想的普遍性

书里让我特别有感触的一点是：日志结构不只是 LSM-Tree 的专利，它是一种通用的设计思想。

- Kafka 的分区本质上就是一组追加写的日志文件
- OceanBase 把写入先落到内存的 MemTable，再通过 LSM-Tree 分层流转到多级 SSTable
- InnoDB 的 redo log 也是顺序追加的日志结构
- 甚至 CDC（变更数据捕获）的核心思想也是把状态变化记录为日志流

**"日志是数据库真正的主角"**——这是我读完第一部分和第三部分之后总结出来的一句话。数据库的各种花哨功能（事务、复制、恢复），底层都依赖某种形式的日志。

## 分布式：从"拿来主义"到"理解设计选择"

第二部分是和我个人经历关联最密切的部分。

### 调度引擎的教训

在做从曙光偷来的调度 job 引擎的时候，我就遇到了节点间通信的问题。看到 xxl-job 用 HTTP 做调度器和执行器之间的交互，我知道必须加上通信机制。当时考虑了几个轻量级的 Raft 实现框架，但没能参透状态机和日志复制的使用方式，最终采用了 Redis Stream 做节点间通信，Redis 分布式锁做选主。

读完 DDIA 的第八章和第九章之后回头看这个设计，问题很清晰：

1. **Redis 分布式锁做选主不可靠。** 分布式环境下时钟不可靠（第八章的核心论点），Redis 锁依赖过期时间，在 GC 停顿或网络分区的情况下可能出现两个节点同时认为自己持有锁的情况。Martin Kleppmann 在书里专门讨论了这个问题（fencing token 方案）。
2. **没有考虑脑裂。** 网络分区时调度器和执行器可能各自为政。我当时压根没想过这个场景，因为不知道"部分失效是分布式系统的常态"这个基本认知。
3. **用 MQ 代替共识协议，本质上是在回避问题。** Redis Stream 能传递消息，但不能保证所有节点对"谁是 Leader"达成一致。这和真正的共识协议（Raft/Paxos）有本质区别。

如果重新做这个调度引擎，我会直接用 etcd（内置 Raft）做服务发现和选主，把共识问题交给专门的组件解决，而不是自己拿 Redis 锁拼凑一个半吊子方案。

### 共识算法：不只是"选主"

之前对共识算法的理解非常狭隘——以为共识就是选主。读完第九章才知道，**共识算法的核心问题是让多个节点就某个值达成一致，选主只是其中一种应用场景。**

几个纠正：
- Paxos 原始论文里没有 Leader 的概念，它是纯粹的多提案者共识。Multi-Paxos 才引入了稳定 Leader 做优化
- Raft 用 term，ZAB 用 epoch，Paxos 用 proposal number，它们的语义不完全等价，不能一概而论
- 共识算法除了选主，还用于分布式事务的提交决策、日志复制的一致性保证等场景

### Redis 为什么用 Sentinel 而不是 Raft

原文里我写"Redis 对一致性要求低所以不用 Raft"，这个因果关系过于简化。更准确的原因：Redis Sentinel 诞生于 2012 年左右，Raft 论文 2013/2014 年才发表。而且 Redis 的设计哲学是单线程极简架构，Sentinel 作为独立进程做监控和故障转移，符合 Redis"每个组件只做一件事"的理念。Redis Cluster 的 gossip + 投票机制本身也是一种弱共识方案，是在 Redis 已有的架构约束下做出的合理选择。

**要从历史背景和架构约束出发理解设计选择，而不是用结果倒推原因。**

## 第三部分：批处理和流处理的位置

第三部分讲的是衍生数据——当数据不再只存在于一个系统中，如何在多个系统之间移动和转换。批处理（MapReduce/Spark）和流处理（Kafka/Flink）是两种核心范式。

说实话，读这部分的时候体感最弱。批处理和流处理的场景我当时没接触过，很多东西停留在"知道有这么个概念"的层面。但有两个点留下了印象：

**记录系统 vs 衍生数据系统的区分。** 记录系统是数据的权威来源（每个事实只表示一次），衍生数据系统从记录系统转换而来（缓存、索引、物化视图都属于此类）。这个区分帮我理解了一个困扰过我的问题：为什么很多系统要同时用 MySQL 和 Redis 和 ES？因为 MySQL 是记录系统，Redis 缓存和 ES 索引都是衍生数据——它们的数据来源是 MySQL，丢了可以重建。

**流处理和变更数据捕获（CDC）。** 把数据库的每一次变更看作事件流，下游系统订阅这个流来维护自己的衍生数据。这个思路和 Kafka 的分区日志模型完美契合——Kafka 本质上就是一个持久化的、可重放的事件日志。

## 回头看：这本书改变了什么

### 1. 从"记住方案"到"理解 trade-off"

以前学技术的习惯是记结论："MySQL 用 B+ 树""Redis 用跳表""Kafka 用分区日志"。读完这本书之后，每看到一个技术选型，第一反应变成了"它牺牲了什么来换取这个优势？"

比如下面这个表格，以前我可能就是当速查卡来背：

| 技术栈 | 复制方式 | 分片方式 |
|-------|---------|---------|
| MySQL | 主从复制（binlog 同步），半同步复制 | 分库分表（中间件实现） |
| Redis | 主从复制（RDB + 增量同步），哨兵故障转移 | 哈希槽（16384 槽） |
| Kafka | 分区副本，ISR 机制 | 按 Topic 分区 |
| OceanBase | Paxos 多副本强一致 | tablet 分片 |

但现在我会问：为什么 MySQL 的复制默认是异步的？因为主从同步延迟会直接影响写入性能，而大多数业务场景能容忍短暂的读不一致。为什么 Kafka 用 ISR 而不是 Paxos？因为 Kafka 追求的是高吞吐，ISR 只要求一部分副本（而非多数派）确认就算写入成功，在吞吐和一致性之间取了更偏向吞吐的平衡点。为什么 OceanBase 用 Paxos？因为它是金融级数据库，一致性要求最高，宁可牺牲一些延迟也不能丢数据。

**同一个问题（数据复制）的不同解法，反映的是不同的业务取舍。** 不理解取舍，表格背了也没用。

### 2. "瓶颈会迁移"的直觉

第二部分讲的分布式环境三类不可靠（网络、时钟、节点故障），给我一个很强的直觉：**解决一个问题往往会暴露下一个问题。**

后来做 db_gen 项目（用 Rust 写 CSV 数据生成器）的时候，这个直觉被完美验证：V1 解决了单线程的性能问题，暴露了 Channel 锁竞争；V2 消灭了锁竞争，暴露了合并阶段的双倍 I/O；V3 去掉了合并，撞上了 ext4 inode 锁；V4 彻底 Share Nothing，又碰到了 Dirty Pages 暴涨。五次优化，五次瓶颈都不在预想的地方。

DDIA 里也是同样的模式：单机存储解决了持久化问题 → 数据增长需要分片 → 分片后需要复制保证可用性 → 复制引入一致性问题 → 一致性需要共识算法 → 共识算法受限于网络和时钟的不可靠性。**每一层的解决方案都是下一层问题的来源。**

### 3. "先定角色，再选实现"的架构思维

这个认知不完全来自 DDIA，但 DDIA 奠定了基础。后来读《架构整洁之道》的时候，Bob Martin 说"架构决策应该尽可能推迟具体实现的选择"，我立刻想到了 DDIA 里对各种技术方案的分析方式——它从来不是说"你应该用 X"，而是说"这类问题有 A、B、C 三种解法，各自的代价如下"。

后来做高并发日志系统设计的时候，我直接用了这个思路：先确定架构里需要"削峰缓冲层""实时计算层""分级存储层""查询层"四个角色，再根据实际量级选具体实现。小规模用内存队列就够了，京东体量才需要 Kafka。**不是每个系统都需要 Kafka，但每个高并发写入系统都需要一个削峰缓冲层。**

### 4. 曙光实习的"恍然大悟"

很感谢去年暑期在曙光的实习。当时读了很多实践代码——分布式一致性的内存级别缓存、kvdb 的 LSM-Tree 实现、Java 管理侧的任务调度逻辑。当时很多东西看不懂，只能照着写。读了 DDIA 之后那些代码的设计意图才逐渐清晰：

- kvdb 为什么用 LSM-Tree 而不是 B-Tree？因为写入场景远多于读取（日志存储）
- 管理侧为什么用内存缓存 + 异步刷盘？因为分布式环境下同步写盘的延迟会被网络放大
- 为什么节点间通信要有心跳和超时机制？因为"不可靠的网络"是分布式系统的基本假设

**先实践再读书，书中的每一个概念都有落脚点。** 这是我读 DDIA 体验最好的原因，也是我反对超前学习的原因。

## 还差什么

说了这么多收获，也要承认差距。第三部分的批处理和流处理，我只能算"知道概念"，没有实操经验。MapReduce 的 Shuffle 过程、Spark 的内存计算优化、Flink 的 Checkpoint 机制——这些在书里读着觉得"哦原来如此"，但真到设计一个流处理系统的时候，肯定还有大量细节要填。

另外，DDIA 讲的是 2017 年的技术图景。这几年变化很大：ClickHouse 在分析场景崛起、eBPF 改变了内核观测方式、io_uring 重新定义了异步 I/O、向量数据库成了新热点。书里的原理没过时，但具体方案需要结合当下的技术栈来更新。

最后，这本书最让我感慨的一点是：**系统设计没有银弹。** 每一个看起来优雅的方案，放到不同的约束条件下可能就是灾难。理解约束、理解 trade-off、在约束中找到当前最合适的平衡点——这大概就是 DDIA 想教的东西。
