# 服务器性能对比，到底在比什么

B 站刷到个视频，对比 Go Gin、Rust Actix、Java Spring Boot 的性能。Actix 一骑绝尘，Go 紧随其后，Java 被甩在后面。

![web-backend](/images/web-backend.png)

评论区在吵谁快。但很少有人问为什么快。

## 先交代一下背景

大学的时候只接触过 Java Spring。线程模型、运行时开销这些概念，课上讲过，但没什么体感。也没参与过上生产的项目，对 trade-off 完全没有感触。

当时跟着黑马学，什么流行往上堆。MySQL、Redis、ES、Kafka、MinIO，技术栈越多越觉得牛逼。不考虑维护成本，不考虑服务器压力，也不清楚这些中间件各自发挥作用的场景是什么。

后来慢慢有了一些转变。

我以前觉得线程越多越高效。后来发现 Python FastAPI 是单线程的。单线程能撑高并发？那一个线程不就只能服务一个请求吗？想不通。再后来才明白，一个线程可以在等待 IO 的时候去服务其他请求。Web 后端大多数情况是 IO 密集型的，CPU 大部分时间在等——等数据库、等网络。单线程只要不在等的时候闲着，就够用了。

然后又冒出一个问题：Python 有 GIL，同一时间只有一个线程跑 Python 代码。那它怎么成了机器学习的主流语言？机器学习不是 CPU 密集型的吗？后来发现，PyTorch、NumPy 这些库底层是 C/C++ 写的。Python 把计算外包给了别的语言，自己专注于胶水逻辑。

这时候我才知道，有 CPU 密集型和 IO 密集型之分。

再后来听人诟病 Spring 的内核级线程——并发低、内存占用高。我当时想，那有什么办法呢，线程不是已经比进程还省资源了吗？后来才知道有协程，有用户态线程，有 Loom 虚拟线程，有 IO 多路复用。Spring 没用这些的时候性能确实不高。但它也能用，因为生态好。而且实际项目里，对延迟和吞吐有高要求的部分，逐渐都外包给了中间件——Redis、数据库、网关，这些全是高性能语言写的。Spring 负责业务逻辑，中间件负责扛性能。

大概这就是业务层和基础设施层的区别。业务方追求快速迭代和上线，不太关心底层实现。基础设施层追求极致性能，不太关心业务方怎么用。

这些想法不是学语言学来的。后来学 Go、学 Rust，语言层面的差异当然存在，但真正让我对这些东西有感触的，是经历了实际上线和 trade-off。对于语言本身，我不是很感冒。每个语言有自己解决问题的方式，关注的层面不同，没什么可比性。

但性能差异背后的因素，可以拆开看。

---

看完视频之后我跑去问了 Kimi 一连串问题，从 WebFlux 问到 Go，又从 Go 问到 Rust，越问越深。[完整对话在这里](https://www.kimi.com/share/19d16517-3e12-85f7-8000-000038048774)。

后来我才慢慢意识到，我真正想弄明白的，其实不是“哪门语言更快”，而是这些 benchmark 到底在奖励什么。影响高并发性能的因素很多，但它们的重要程度差别很大。有些只是入场券，有些才真的决定你最后能跑到哪。我下面这套拆法，不是严格实验，只是我现在比较能自洽的一条理解路径。

## 我后来才知道，性能不是一个数字

我以前看到“性能好”这三个字，脑子里只有一个很模糊的印象：快。但后来才知道，这种说法太糊了。对 Web 服务器来说，至少得拆成几个维度看。

吞吐量，也就是 QPS，一秒能处理多少请求。

延迟，一个请求从进来到响应要多久。别看平均值，看 P99 和 P999。平均延迟 5ms 但 P99 飙到 200ms，上了生产会被投诉。尾部延迟才暴露真问题——比如 GC 造成的毛刺。

并发连接数，同时能端着多少个连接。这个和吞吐量不是一回事。你可以同时挂着 100 万个 WebSocket，但大部分在空闲。吞吐量看活跃请求的处理速度，并发连接数主要看内存够不够。

资源占用，同样的 QPS，一个框架吃 2GB 内存，另一个吃 200MB，差距很大。

我以前有个想不通的地方：协程为什么能提高并发？后来明白了，协程不是更快，是更省内存。一个 OS 线程综合开销 1~2MB，10 万并发就要上百 GB。goroutine 2KB 起步，100 万并发只要几 GB。CPU 没变快，但能同时端着的连接数上去了。后面详细讲。

这几个指标之间是会互相打架的。有些框架吞吐高，但尾延迟抖得厉害；有些延迟稳，但吞吐一般。不看清楚在比什么，benchmark 很容易变成一堆让人兴奋、但没法落地的数字。

## 我现在会先拆这几层

![并发模型对比](/images/simple_concurrency.png)

视频里的性能差距，不太可能是某一个原因单独造成的。我现在会先想到这几层：

1. IO 模型——阻塞还是非阻塞
2. 并发单元——OS 线程还是协程/虚拟线程
3. GC 还是无 GC
4. 编译方式——JIT、AOT、解释执行
5. 锁还是无锁

我当时就是顺着这几层一层层往下想，才开始不那么容易被“某某框架完胜”这种标题带跑。

## 非阻塞 IO：入场券

Java Spring MVC 默认一个请求一个线程。线程在等数据库的时候干等着，1000 并发还行，10000 并发线程池就不够用了。

WebFlux 换了思路，用少量线程跑事件循环，IO 等待时不阻塞。Node.js 和 Python asyncio 也是这个路子。Go 的 netpoller 底层也是 epoll。

大家都有非阻塞 IO。这是门票。没有它进不了高并发的场，但光有它也分不出高下。

## 协程：主要省内存

我一开始以为协程是提速用的。后来想明白了，协程主要解决的是内存问题。

传统 OS 线程的栈空间默认 1MB（Java 的 `-Xss1m`）。但这里有个容易搞混的地方——这 1MB 是虚拟内存，不是物理内存。物理内存按需分配，刚创建时可能只占几十 KB。

不过架不住量大。线程对象要占堆内存，ThreadLocal 要占空间，内核的 task_struct 大概 16KB，每个线程还是一个 GC Root，增加扫描压力。综合算下来一个 Java 线程的全部开销在 1~2MB 左右。10 万并发就要 100 多 GB，不现实。

Go 的 goroutine 初始栈 2KB，按需增长，按需收缩。100 万个 goroutine 大概 2~4GB。Java 虚拟线程（Loom）更小，几百字节起步。

所以协程的核心收益是用 1/100 的内存持有同样数量的并发任务。CPU 吞吐量不一定提升多少——真正干活的线程数约等于 CPU 核数，不管你是线程还是协程。但能同时端着的连接数上去了，I/O 密集型场景下这是关键。

这里有个我当时很想问明白的问题：MVC + Loom 虚拟线程，不就等于 WebFlux 了吗？

差不多，但不完全。Loom 的好处是写同步代码就行，JDBC、Hibernate、RestTemplate 全兼容。WebFlux 有个独门武器：背压。下游可以告诉上游"慢点发"，流式场景需要这个。纯 CRUD，Loom 更简单。

## 到 Rust 这里，差距才开始拉开

到这里，非阻塞 IO 大家都有，协程也不算稀缺了。Go 天生带，Java 21 也补上了。那视频里 Actix 那种明显甩开的感觉，到底是从哪来的？

两个东西。无 GC，和零成本抽象。

Go 的 GC 已经很快了，STW 通常不到 1ms。但高并发下，100 万个 goroutine 产生的对象需要扫描、标记、回收。GC 不怎么影响平均延迟，但会制造尾部延迟毛刺——P99 可能 10ms，P999 跳到 50ms。

Rust 没有 GC。内存的分配和释放在编译期就确定了，靠所有权系统保证。运行时没有扫描，没有 STW。代价是写代码的时候得跟借用检查器搏斗。

零成本抽象是另一个。Rust 的 `async/await` 编译后变成状态机，在不 `spawn` 的情况下不需要堆分配。Java 的泛型有类型擦除，对象有对象头，装箱拆箱有开销。这些在 Rust 里都不存在。

不过这里也不能说得太满。Rust 不是“没有运行时”。用了 Tokio 就有异步运行时，线程池、任务调度器、IO reactor 都在。区别只是它没有 GC，没有 JIT 预热，调度开销通常也更小一点。

## 我后来越来越在意的一层：锁到底多伤

还有一个因素容易被忽略。Actix 用了 Actor 模型——每个 Actor 有自己的状态，消息传递代替共享内存，不需要锁。

锁的开销不只是等待时间。现代 CPU 的瓶颈在内存墙：

```
L1 缓存访问:  1ns
L2 缓存访问:  4ns
L3 缓存访问:  15ns
主内存访问:   100ns
跨核缓存同步: 100~300ns
```

一把锁导致缓存线失效。100 个线程抢一把锁，CPU 大部分时间不在算东西，在等内存同步。

Actor 模型下，每个 Actor 的状态只有自己能碰，天然 cache-friendly。消息传递是异步的，不触发缓存同步。

这个因素在讨论里经常被跳过。但我后来越来越觉得，它可能比很多人想的还重要。因为到最后，瓶颈常常不是“算不动”，而是大家在互相等。

## 如果硬要排个先后

我一开始很想像做消融实验那样，把这些因素排个序。后来发现我手里并没有一套严格控制变量的数据，所以这里更准确地说，是按我现在的理解，给这些因素排一个大概的先后。

先说非阻塞 IO。

Java Tomcat 200 线程 vs WebFlux。并发量从几百跳到 10 万。这是从 0 到 1 的变化。但过了这道坎之后，大家都在同一个起跑线上。

再说协程。

同样是非阻塞 IO，用 OS 线程 vs 用协程。并发连接数差 10~100 倍，因为内存瓶颈。但实际吞吐量差距没那么大——干活的线程数都约等于核数。

然后是我后来越来越不敢忽略的无锁化。

Rust + Mutex vs Rust + Actor 无锁。这个差距可能比想象中大。CPU 30% 的时间花在锁竞争和缓存同步上的情况并不少见。

最后才是 GC 和运行时开销这类事。

Go vs Rust。吞吐量 Rust 大概高 50~60%。尾延迟差距更明显。但 Go 的 GC 对绝大多数场景够用。

如果先按这个思路粗排一下：

| 因素 | 去掉后的影响 | 性质 |
|------|------------|------|
| 非阻塞 IO | 并发量归零 | 入场券，大家都有 |
| 协程/虚拟线程 | 并发连接数降 10~100 倍 | 放大器，大家也快有了 |
| 无锁化/状态隔离 | 吞吐量降数倍 | 架构决策，差异大 |
| 无 GC | 尾延迟恶化，吞吐降 ~50% | Rust 独有 |
| 零成本抽象 | 吞吐降 20~30% | 边际优化 |

我现在更愿意先记住一句话：非阻塞 IO 是门票，协程是放大器，真正决定天花板的，很多时候反而是后面那层锁竞争、GC 毛刺和运行时开销。

这套排序我不敢说很硬，因为它没有严格控制变量的 benchmark 来兜底，更像是我顺着原理往下推出来的一条理解线。如果有人真做过比较干净的实测，我会很想看。

## Actor 模型

说到无锁化就得聊 Actor。Actix 名字里的 Act 就是 Actor。

说实话我对 Actor 模型还没完全搞明白。大概的意思：传统并发用锁保护共享内存，Actor 干脆消灭共享。每个 Actor 有自己的状态，别人只能通过发消息来请求修改。

```rust
impl Handler<Transfer> for BankAccount {
    fn handle(&mut self, msg: Transfer, ctx: &mut Context<Self>) {
        if self.balance >= msg.amount {
            self.balance -= msg.amount;
            msg.to.do_send(Deposit(msg.amount));  // 发消息，不是直接调用
        }
    }
}
```

每个 Actor 同一时间只处理一条消息，内部天然串行，不需要锁。多个 Actor 并行跑，但彼此不共享状态。

我一开始觉得这不就是 Go 的 channel 吗。后来发现不一样。Go 的 channel 是匿名管道，你自己写 `for` 循环收消息，自己起 goroutine 跑。Actor 有框架管理的邮箱，有类型安全的地址 `Addr<T>`，有生命周期管理。Go 给你砖头让你自己搭，Actor 给你一套精装方案。

还有一点：Actor 是 Actix 框架提供的，不是 Rust 语言自带的。没有 Actix，你用 Tokio 的 channel 也能模拟，但得自己写一堆样板代码——消息枚举、事件循环、地址封装。Erlang/Elixir 是语言级别支持 Actor 的，那是另一回事。

---

回头看，这篇其实只是帮我把脑子里那团线先理顺了一点。Web 服务器的性能因素，大概可以先分成两类。

一类是基础能力，比如非阻塞 IO、协程。大家迟早都会有。Java 出了 Loom，Go 天生有 goroutine。这些更像入场券，能决定你有没有资格上桌，但不太决定谁最后甩开谁。

另一类才是实现细节：有没有锁竞争，有没有 GC 停顿，抽象有没有运行时开销。Actix 也不是什么突然冒出来的黑科技，只是它在很多环节上都选了更轻的那条路。单看每个都只快一点，叠起来差距就会越来越明显。

大学学 Java 的时候，`synchronized`、`volatile`、`Lock`、`Condition`，把我搞得头大。后来学 Go，一个 `go` 关键字就能起一个并发任务。再后来碰 Rust，跟编译器吵了一个月。有天突然发现，编译器不让写的代码，大概率确实有问题。

所以现在回头看这种性能榜单，我会更想先问：它到底在奖励哪一种并发模型，哪一种运行时，哪一种对锁和内存的处理方式。只盯着“谁第一”，其实没什么意思。真正有意思的是，你终于能看出来，为什么它会第一。
