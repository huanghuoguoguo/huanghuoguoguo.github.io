在 JVM 的世界里，垃圾回收（Garbage Collection）是一场永恒的博弈：是在**高吞吐量（Throughput）**和**低延迟（Latency）**之间寻找平衡。

随着硬件的发展，GC 已经从最初的单线程“串行时代”，演进到了如今能处理 TB 级内存、停顿时间控制在毫秒级的“极致并发时代”。

---

## 一、核心博弈：吞吐量 vs 延迟

在深入具体的收集器之前，我们需要明确两个核心指标：
- **吞吐量**：用户代码运行时间 /（用户代码运行时间 + GC 时间）。适合侧重计算的后台任务。
- **停顿时间 (STW)**：垃圾回收导致的应用程序暂停时间。适合对实时响应有要求的 Web/交互系统。

## 二、传统收集器：经典组合

经典的 GC 组合通常遵循“分代回收”理论，将堆分为新生代和老年代。

### 1. 吞吐量派系 (Throughput First)
- **Parallel Scavenge (新生代) + Parallel Old (老年代)**
  - **特点**：多线程并行收集，利用多核 CPU 优势，追求最高吞吐量。
  - **算法**：复制算法（轻）+ 标记-整理（老）。
  - **场景**：离线报表分析、科学计算。

### 2. 串行派系 (Resource Limited)
- **Serial (新生代) + Serial Old (老年代)**
  - **特点**：单线程收集，简单直接，不消耗多余内存和线程上下文。
  - **场景**：Client 模式、单核 CPU 容器环境。

---

## 三、关注响应：低延迟收集器 (CMS)

**CMS (Concurrent Mark Sweep)** 是 JVM 历史上第一款真正的并发收集器。

- **算法**：标记-清除。
- **阶段**：
  1. **初始标记 (STW)**：仅标记 GC Roots 直接关联对象（极快）。
  2. **并发标记**：遍历对象图，与用户线程并发运行。
  3. **重新标记 (STW)**：修正并发标记期间产生的变动。
  4. **并发清除**：清除死亡对象。
- **缺点**：会产生**内存碎片**；无法处理浮动垃圾（Floating Garbage）。

---

## 四、现代演进：区域化与全并发

现代收集器打破了严格的分代物理边界，转而采用 **Region（区域化）** 管理内存。

### 1. G1 (Garbage First) - 均衡之王
- **核心理念**：将堆化为多个 Region，追踪每个 Region 的回收价值，优先回收垃圾最多的区域。
- **特点**：可预测的停顿（`-XX:MaxGCPauseMillis`），避免了 CMS 的内存碎片问题。
- **阶段**：初始标记 -> 并发标记 -> 最终标记 -> 筛选回收。

### 2. ZGC & Shenandoah - 毫秒停顿
| 收集器 | 核心技术 | 停顿目标 | 主要阶段 |
| :--- | :--- | :--- | :--- |
| **ZGC** | 染色指针、读屏障 | < 10ms | 并发标记、并发预备重定位、并发重定位 |
| **Shenandoah** | 读/写屏障、一致性屏障 | 亚毫秒级 | 并发标记、并发疏散、并发更新引用 |

- **ZGC 特点**：几乎所有阶段都并发执行，支持 TB 级内存，停顿时间不随堆大小增长。
- **Shenandoah 特点**：类似 ZGC，通过屏障技术实现在回收阶段也能与用户线程并发执行。

## 五、收集器对比汇总

| 收集器 | 分代 | 算法 | 线程模式 | 目标 |
| :--- | :--- | :--- | :--- | :--- |
| **Serial** | 新生代 | 复制 | 单线程 | 简单高效 |
| **ParNew** | 新生代 | 复制 | 并行 | 配合 CMS |
| **Parallel Scavenge** | 新生代 | 复制 | 并行 | 吞吐量优先 |
| **CMS** | 老年代 | 标记-清除 | 并发 | 响应速度优先 |
| **G1** | 整堆 | 复制+标记整理 | 并发+并行 | 平衡性能与延迟 |
| **ZGC/Shenandoah** | 整堆 | 标记-整理 | 全并发 | 极致低延迟 |

## 六、深度延申：STW 与 GC 进化的底层逻辑

在 JVM 的 GC 演进史中，无论是 CMS、G1 还是 ZGC，它们共同的终极敌人只有一个：**STW (Stop The World)**。

### 1. 什么是 Stop The World？
STW 是指在执行垃圾收集算法时，Java 应用程序的所有用户线程都被强制挂起（冻结）。想象一下，你正在公路上超车，交警突然按下了所有汽车的暂停键来打扫路面，这就是 STW。对于高并发、低延迟系统（如股票交易、实时推荐），哪怕几十毫秒的 STW 都可能导致请求超时或用户体验卡顿。

### 2. 为什么 GC 的进化史就是消灭 STW 的历史？
早期 GC（如 Serial、Parallel）采用的是**全暂停、全清理**的策略，简单粗暴。但随着堆内存从 MB 级迈向几十 GB 甚至 TB 级，全量扫描和移动对象的时间成正比增长，动辄秒级的 STW 变得不可接受。
因此，GC 的进化方法论可以总结为：**“化整为零，将重量级操作与用户线程并发执行”**。

### 3. 黑科技：如何实现“边跑边打扫”？
为了在不暂停用户线程的情况下并发分析、甚至**并发移动**对象（解决内存碎片），ZGC 等现代收集器引入了以下极其精妙的黑科技：
- **染色指针 (Colored Pointers)**：ZGC 将对象的一些状态信息（如是否被移动过、是否被标记）直接存储在**引用的指针本身**（占据了 64 位指针的高几位），而不是存储在对象头中。这使得 GC 不需要去解引用访问对象本身，只需要看指针的值就能知道对象状态，极大提高了遍历效率，降低了缓存未命中（Cache Miss）带来的惩罚。
- **读屏障 (Load Barrier) 与懒访问**：当用户线程尝试通过指针读取一个对象时，会触发 JVM 植入的一小段汇编代码（读屏障）。如果染色指针提示该对象已经被 GC 移动到了新位置，但当前指针还没来得及更新，读屏障会**拦截这次访问，根据映射表把指针更新到新地址，然后再返回新对象（这被称为“自愈修复”）**。
- **懒移动 (Lazy 思想)**：GC 线程只负责把存活对象复制到新区域，然后更新一张转发映射表（Forwarding Table），但不着急去全堆遍历、修改所有指向老对象的指针。真正的指针修改工作，被巧妙地推迟到了**下一次用户线程访问该对象时（触发上述的读屏障）顺手完成**。这种把大规模的计算成本分摊给每一次访问的思路，就是这套设计的精髓。

## 七、横向跨界：其他语言的内存管理哲学

跳出 Java 的圈子，不同的编程语言在如何对付“拉圾”这件事上，走出了截然不同的道路：

| 语言 | 内存管理方式 | 核心机制 | 特点与局限 |
| :--- | :--- | :--- | :--- |
| **C / C++** | 手动内存管理 | `malloc/free`, `new/delete`，智能指针 | **优**：极致的性能表现，零 GC 附加开销（无 STW）。<br>**缺**：极易产生内存泄漏、悬垂指针等恶性 Bug，开发心智负担极重。 |
| **Rust** | 无 GC 的自动管理 | 所有权 (Ownership) + 生命周期 + 借用检查器 | **优**：在**编译期**运用严格的规范解决内存安全问题，运行时零代价。<br>**缺**：学习曲线极其陡峭，初写时感觉一直在与编译器“搏斗”。 |
| **Go** | 现代垃圾回收 | 三色标记 + 混合写屏障 | **优**：极度追求低延迟，STW 极短，工程上即装即用体验极佳。<br>**缺**：为了低延迟牺牲了部分吞吐量，且缺乏 JVM 这样丰富的分代调优选项。 |
| **Python** | 引用计数为主，<br>标记清除为辅 | 维护对象的 `refcount`，归零即销毁 | **优**：实时性好，对象一旦不用立刻触发回收逻辑。<br>**缺**：高频更新引用计数开销大，需额外运行标记清除来解决**循环引用**问题。 |

**总结：**
- **Java / C#** 的哲学是**“把脏活累活丢给运行时（Runtime）”**，用极其复杂的虚拟机生态换取开发者的解放。
- **C / C++ / Rust** 崇尚**“自己的事情自己做”**或**“把风险扼杀在编译期”**，追求硬件级的运行时效率和确定性。
- **Go** 则在开发效率和极致并发之间取了一个非常有现代工程质感的平衡。

---

## 八、总结：大道至简

1. **如果你追求吞吐量**：首选 `Parallel Scavenge + Parallel Old`。
2. **如果你的内存 4G-8G 且在乎停顿**：`G1` 是大部分 JDK 8/11 环境的最优选。
3. **如果是超大堆（几十G甚至TB级）或对停顿无法容忍**：果断选用 `ZGC` (JDK 11+) 或 `Shenandoah` (RedHat 系 JVM)。

> **道**：**没有最好的机制，只有适合的取舍。** 理解 GC 的演进史，横看不同语言的内存管理哲学，其核心都是在寻找资源分配的最优解。这些底层的方法论，如“化整为零”、“时间换空间/空间换时间”、“读写屏障的拦截兜底代理思想”，同样也是我们在做大规模高并发架构设计时，最宝贵的灵感来源。
