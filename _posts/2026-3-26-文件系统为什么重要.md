我以前一直把文件系统当成操作系统课里那个“放在后面再看也行”的章节。

前面那些东西更容易让我上头。系统调用很具体，`read()`、`write()`、`fsync()` 一看就跟程序行为连着；虚拟内存、进程调度、锁，这些也都很快能在性能问题里撞上。文件系统不一样。它看起来像一个已经被封装好的背景板。平时写服务，顶多就是开文件、写文件、关文件，好像没什么值得单独拎出来想的。

后来我才发现不是这么回事。为什么 `write()` 返回了，进程一挂数据还是可能丢？为什么数据库总和 page cache 拧着来？为什么块存储已经够“直接”了，上面还要再垫一层文件系统？为什么对象存储反过来又故意不要 POSIX 那套细语义？这些问题最后都会撞回同一个地方。

所以这篇我想问的，其实就是一个问题：文件系统为什么重要。对我们这种平时写服务、碰数据库、做性能测试的人来说，它到底卡在了哪一层。

## 文件系统不是背景板

后来读 OSTEP 和《Linux 内核设计与实现》的时候，我总会看到文件系统被单独放在后面一大块。我那时其实有点不服气。前面的进程、虚拟内存、多级存储介质，我都能比较快地对上工程场景。文件系统凭什么也占这么大一块？

我最早对它的理解很粗：不就是帮我把数据写进磁盘吗？这个理解在刚写点业务、写点小程序的时候勉强够用，因为那时候我也确实不会去碰“第 42 号块”。我碰到的是文件名、目录、权限、路径，还有 `open`、`read`、`write`、`close` 这些系统调用。

后来才慢慢意识到，文件系统麻烦的地方，不只是在“把数据存下去”。它得先在裸块之上建一层秩序，再把这层秩序包装成程序员习惯的接口。程序员以为自己面对的是文件，内核其实一直在替你收底下那堆乱七八糟的硬件差异。

书里有句话叫“Linux 下一切皆文件”。我以前对这句话理解得很浅，只知道 socket 也能拿文件描述符表示。后来翻《性能之巅》，又顺着 eBPF 和内核对象往下看，才稍微明白这句话的分量。它不是一句修辞。它是在说，内核想用尽量统一的一套抽象，把完全不同的资源都收进来。

| 矛盾点 | 文件系统 / VFS 的做法 | 结果 |
| :--- | :--- | :--- |
| 硬件设备千差万别 | 统一抽象成文件和文件描述符 | 用户态只管 `open` / `read` / `write` |
| 底层格式各不相同 | 加一层 VFS | 应用不用关心底下是 ext4 还是 xfs |
| 内核要管很多资源 | 都塞进“文件”这套抽象 | 管道、socket、设备都能共用一套接口 |

从这个角度看，文件系统的重要性更多在于它让程序员不用每次都重新理解底层设备。教科书会把它讲成一组数据结构和分配策略，工程里你更容易先撞到的是另一面：抽象边界、崩溃恢复、缓存语义、以及不同上层系统为什么总想和它重新谈条件。

我也是到这里，才开始愿意认真看它到底在解决什么矛盾。

## ext2、ext3、ext4 和 WAL 有什么区别？

如果顺着这个问题继续往下问，一个最直接的入口其实是：文件系统到底在对付什么麻烦？

以前我看 ext2、ext3、ext4 的演进，总觉得只是“新版本加了点特性”。后来从系统矛盾的角度再看，才觉得它们每次升级都很具体。

**Ext2** 没有日志。写一个文件，不是一次原子动作，而是拆成很多步：改 inode、改位图、写数据块。中间只要断一次电，磁盘上就可能留下一个半拉子状态。重启之后只能靠 `fsck` 慢慢扫。

这件事在小磁盘时代还能忍。盘一大，恢复时间就开始吓人。在曙光那边做监控的时候，我们最怕的就是节点异常重启之后恢复太久。要是底下文件系统一上来先扫很久，整个节点就会掉在线服务。

**Ext3** 引入日志，思路其实跟数据库的 WAL 很像：先记要改什么，再去改真正的数据结构。崩溃之后不用全盘扫，只看那一小块 journal 区域，就能把结构收回来。

但日志不是白拿的。多了一层写放大，也就多了新的权衡。所以 ext3 才会有 `writeback`、`ordered`、`journal` 这些模式。说到底，还是在问一个老问题：我要更快，还是要更稳？

**Ext4** 再往前走，动的就不只是恢复策略了，而是整体布局。

一个典型变化是 **extents**。以前是一个块一个块去记，现在改成“从哪开始，连续多长”。这样元数据少得多，连续空间也更好分配。另一个变化是 **delayed allocation**，先别急着分配物理块，攒一攒再一起安排，这样碎片少，顺序写也更容易做出来。

这些设计放一起看，会发现文件系统一直在对付几组老矛盾：

1. 内存快，磁盘慢。
2. 随机写灵活，但顺序写更便宜。
3. 用户要原子性，但硬件只会一块一块写。
4. 小文件和大文件的最优存法不一样。

page cache、journal、extents、delayed allocation，本质上都在同这些问题周旋。

## 然后我才开始看懂 page cache 到底在卡什么

很多人第一次觉得“文件系统有点意思”，往往是从 page cache 开始的。但是一般来说，初学者在认识page cache之前，会更早认识数据库。

数据库说，要有事务，持久性，原子性，隔离性，一致性，数据库说它保证数据肯定落在盘上才返回用户提交成功。所以我当时理所当然的认为，数据库每次写数据都会刷到磁盘上，我知道有缓存，但是缓存在哪我不知道。

因为 page cache 一下就把“程序眼里的 IO”和“磁盘眼里的 IO”分开了。程序 `write()` 完了，并不代表磁盘已经写完；很多时候只是数据先进了内核缓存。真正什么时候刷盘，由内核来决定。

这对大多数程序来说是巨大的性能福利。小写入可以合并，随机写可以拖一拖再整理，读过的数据还能留在内存里，下次直接命中。

但也正因为这层缓存太强，后面数据库才会开始跟它打架。

前一篇里我讲 `fsync()`，本质上就是在跟这一层打交道。你以为自己在控制“写入”，其实很多时候控制的是“什么时候逼 page cache 把东西吐出去”。不过page cache对大多数程序员来说是无感知的。

## 再往下看，文件系统下面其实还是块

在曙光实习的时候，我第一次更清楚地知道，文件系统下面还有块存储这一层。

在那之前，我对“存储”的理解基本停留在文件系统：打开文件，读写，关闭。程序员平时看到的都是路径、文件名、目录，很少会真去想“底下那一块一块的数据到底长什么样”。块存储把这个视角往下硬拽了一层。

我一开始其实有点不适应。因为块设备给人的感觉太“反人类”了。它不认文件名，不认目录，也不认“这是日志文件还是图片文件”。你给它一个块号，它就还你一段固定大小的数据，通常是 512B 或 4KB。就这些。那些我已经习惯了的东西，比如目录、权限、路径，都是文件系统后来往上搭出来的。

这个接口简单得近乎粗暴。但也正因为它粗暴，很多底层问题在这一层反而特别好看清。

曙光做的是分布式闪存存储系统。一个逻辑卷的数据散在多台物理服务器的闪存盘上。上层应用看到的是一块“大磁盘”，读写请求经过路由、转发，最后落到具体节点。我那时做性能指标采集和监控告警，虽然很多计算发生在内存里，但盯的其实就是这条分布式 IO 路径：哪个盘框的 IOPS 在抖，哪条链路的延迟飙了，哪块盘快出故障了。

我也是那时候才第一次比较具体地意识到，原来操作系统眼里的“磁盘”，未必真插在你机器上。iSCSI 把 SCSI 命令封进 TCP/IP，让操作系统通过以太网访问远端块设备。对内核来说，它看到的还是 `/dev/sdb` 这种本地块设备，并不知道数据其实在另一台机器上。光纤通道走专网，延迟更低，带宽更稳，但也更贵。

到这里我才有点明白，为什么很多存储系统最后讨论的都是 IOPS、吞吐、延迟。因为在块这一层，事情真的被压得很干净。上面写进去的是 redo log、视频文件，还是一张猫的照片，它根本不关心。它只负责把块读出来、写进去。

块存储只解决“块怎么存”的问题。至于“这个东西叫什么名字”“它属于哪个目录”“谁能访问它”，这些都还是文件系统继续往上补的。块存储性能高，不代表它能直接代替文件系统。它只是把问题收得更窄了。

## 再往另一边走，就到了对象存储

在 OB 的共享存储架构里，我又碰到了 IO 栈的另一端：对象存储。

我一开始对对象存储也有点误解。我以前会下意识觉得，文件系统已经能存文件了，那对象存储到底新在哪？后来才慢慢发现，它干脆把问题重新定义了一遍。

对象存储跟文件系统的思路很不一样。它没有那么重的目录树语义，也不鼓励你对一个文件中间随手改几个字节。它更像一套极简的 PUT / GET / DELETE 接口：给一个 key，存一个对象，取一个对象。

这个模型刚看会觉得有点笨。怎么连“改文件中间一小段”这种事都不太支持？后来我才觉得，它不是笨，它是故意不接那些麻烦语义。对象存储不需要太关心目录树一致性，不需要太在乎随机写，不需要暴露 POSIX 那套细碎语义。代价就是它很难覆盖数据库这种需要细粒度随机读写的场景。

三种抽象放一起看，就很清楚了：

| | 块存储 | 文件存储 | 对象存储 |
|---|---|---|---|
| 访问单位 | 固定大小块 | 文件，可 seek | 对象，整体读写 |
| 元数据 | 很少 | 丰富 | 自定义键值对 |
| 随机读写 | 支持 | 支持 | 基本不支持 |
| 常见协议 | iSCSI、FC、NVMe | NFS、SMB、POSIX | S3、Swift |
| 典型场景 | 数据库、虚机盘 | 通用文件 | 备份、归档、海量非结构化数据 |

我后来才慢慢习惯，不要老拿这三种东西做“谁更高级”的比较。它们更像三种把问题收窄的方式。

对象存储下面还是文件系统，文件系统下面还是块设备，一层层套起来。

在 OB 测冷热缓存的时候，这种分层体会特别直观。热数据放本地闪存，冷数据沉到共享对象存储。热数据命中了，本地读就完了；没命中，就得过网络去对象存储拿。性能差别一下就出来了。

我那时候才对“为什么云服务这么爱对象存储”稍微有点感觉。因为很多数据根本不值得一直放在最贵、最近的那一层。这个逻辑跟 CPU cache、内存、磁盘其实是一样的：高频数据尽量放近一点，低频数据往远处沉。

## 数据库为什么总和文件系统拧着来

后来在 OB 那边做存储相关测试，我慢慢觉得数据库和文件系统的关系其实挺拧巴。

我当时有个很直接的问题：文件系统已经有 journal 了，数据库为什么还要自己搞 WAL？这不是重复了吗？

刚看表面，你会觉得两者很像，都是“先记日志，再改数据”。但后来我才慢慢意识到，它们保护的东西根本不是一个层次。

| 维度 | ext4 journal | 数据库 WAL |
| :--- | :--- | :--- |
| 保护对象 | 文件系统元数据 | 事务和数据页 |
| 粒度 | 物理块 | 页、记录、事务 |
| 恢复目标 | 文件系统结构别坏 | 已提交事务别丢，未提交事务能回滚 |
| 额外用途 | 崩溃恢复 | 崩溃恢复、复制、备份、回放 |

如果硬要打个不太严谨的比方，文件系统的 journal 更像是在保护“容器别裂开”。数据库的 WAL 保护的是“容器里的业务语义别乱”。

我后来才觉得，数据库总想绕开文件系统，或者至少不完全信任文件系统，其实是有道理的。文件系统关心结构别坏，数据库关心事务别乱、已提交的东西别丢、没提交的东西能撤回。它们两个都觉得自己得兜这一层。结果就是日志套日志，缓存套缓存。

我也是到这里，才慢慢把前一篇里那条线重新接回来。为什么会有 O_DIRECT，为什么数据库要自己搞 buffer pool，为什么 redo log 和 `fsync()` 总是绑在一起。前一篇从系统调用看，看到的是接口；这一篇从文件系统往下看，才发现这些接口最后都落在了不同的抽象边界上。

---

## 最后：文件系统为什么总卡在中间

我后来越来越觉得，文件系统重要，主要就是因为它刚好卡在一个很要命的位置上。课本愿意单独给它开一章，是有道理的。

往下，它要面对块设备、缓存、日志、刷盘、随机与顺序这些硬约束。往上，它又要给程序员提供一个足够统一、足够好用的接口。

所以它一边在替应用隐藏复杂度，一边又不断被数据库、对象存储、分布式系统这些更上层的东西反过来挑战。

我以前会觉得，文件系统可能就是操作系统课里一个偏后面的章节，平时写业务代码好像也碰不到太多。后来才发现，事情不是这样。只要你开始碰性能、碰持久化、碰崩溃恢复、碰数据库和消息队列的区别，最后迟早还是会撞回文件系统这一层。

所以这篇写到最后，我现在对“文件系统为什么重要”这件事的理解，大概比以前具体了一点。它不是背景板，也不只是一个帮你把数据写进磁盘的工具层。很多系统行为的拐点，最后都要从这里经过。至少对我自己来说，前面那些零散的问题，到这里才真的开始有点连起来了。
