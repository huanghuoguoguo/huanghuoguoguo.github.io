最近在看 Brendan Gregg 的《性能之巅》，同时也在学 eBPF。光看书总觉得隔了一层，于是给自己找了个活——用 Rust 写一个 CSV 数据生成器，目标是尽可能快地造出 2000 万行数据。

听起来很简单对吧？我也这么觉得。结果这个"简单任务"让我折腾了五个版本，中间还把系统跑 OOM 了一次。

<!-- more -->

这篇文章就是这个过程的记录。不是什么方法论，就是一个踩坑实录。

Brendan Gregg 在书里说过一句话我印象很深：*"Performance issues can originate from anywhere, including areas of the system that you know nothing about and didn't suspect."* 后来我发现这话一点没夸张——五次优化，五次瓶颈都不在我预想的地方。

## 先说结论

| 版本 | 思路 | 速度 | 死在哪 |
| :--- | :--- | :--- | :--- |
| v1 | Channel 多生产者单消费者 | 934K rows/s | futex 锁竞争 |
| v2 | 各写各的，最后合并 | 5.4M rows/s | 合并阶段串行，双倍 I/O |
| v3 | 多线程 lseek 写同一文件 | 13.0M rows/s | ext4 inode 锁 |
| v4 | 彻底 Share Nothing | 17.3M rows/s | Dirty Pages 暴涨至 1.1GB |
| v5 | 加背压，每 64MB sync 一次 | 9.9M rows/s | 磁盘带宽（物理极限） |

先跑两个基准，摸清天花板在哪：

```
# CPU 端（纯内存生成，不写盘）
>>> 纯 CPU 极限测试...
    CPU 生成带宽: 2.35 GB/s
    生成速度: 42625814.27 rows/sec

# 磁盘端（dd + fio 顺序写）
$ dd if=/dev/zero of=test bs=1M count=1024 oflag=direct
1073741824 bytes copied, 0.913229 s, 1.2 GB/s

$ fio --name=seq_write --rw=write --bs=1M --size=256M --numjobs=4 --direct=1 --group_reporting
  WRITE: bw=551MiB/s (578MB/s)

$ dd if=/dev/zero of=test bs=1M count=1024 conv=fdatasync
1073741824 bytes copied, 3.0175 s, 356 MB/s
```

CPU 能生成 2.35 GB/s，SSD 裸写上限 1.2 GB/s（direct I/O），加上 fdatasync 等待刷盘只有 356 MB/s。**CPU 比磁盘快了 2~6 倍**，这个差距决定了后面所有故事的走向。

从 v1 到 v5，大约 10 倍。下面一步步说怎么走到这的。每个版本都可以独立编译运行，你也可以在自己机器上复现。

## V1：从单线程开始

### 最初的想法

拿到这个任务，第一反应很朴素：这就是一个 CPU 密集型任务嘛——生成随机数、拼字符串、写文件。先不想那么多，单线程写一版跑起来看看。

单线程版本就是一个循环：生成随机 ID、随机值、格式化时间、拼成 CSV 行、写入文件。跑出来大概几十万行每秒。

### 自然而然想到多线程

单线程跑不满多核 CPU，很浪费。第一个优化方向很直觉——用多线程。

但多线程就有一个问题：多个线程生成的数据怎么汇总到一个文件里？不能让它们同时 `write` 同一个文件（会乱），需要一个协调机制。

Rust 标准库提供了 `mpsc::channel`——多生产者单消费者通道。正好：多个线程当生产者，一个线程当消费者写盘。这是教科书上最经典的生产者-消费者模型。

```
线程1 ──┐
线程2 ──┤──→ mpsc::channel ──→ 消费者线程 ──→ write(file)
线程3 ──┘
```

每个生产者线程干的事：生成随机数据，拼成 CSV 行的字符串，`tx.send()` 丢进 Channel。消费者那边 `BufWriter` 逐条写盘。

### 多线程的效果

跑起来：

```
>>> V1: MPSC 模型 - 多生产者 + 单消费者写入...
    启动 4 个生产者线程
>>> 完成！写入 20000000 行
    耗时: 21.41s
    吞吐: 934204.67 rows/sec
```

不到 100 万行每秒。4 个生产者线程，按道理应该比单线程快不少，但实际提升远没有 4 倍。

哪里不对？

### 观测：syscount

这时候最朴素的性能分析原则派上用场了——别猜，量。用 `syscount-bpfcc` 看系统调用分布。`syscount` 是基于 eBPF 的工具，它在内核里插桩，统计指定进程在一段时间内调用了哪些系统调用、各多少次。

```bash
sudo syscount-bpfcc -p $(pidof v1_mpsc) -d 5
```

```
SYSCALL                   COUNT
futex                     22241
sched_yield                2096
write                        16
```

`futex` 22241 次，5 秒内。`futex` 是 Linux 用户态互斥锁的底层实现——大量的 `futex` 调用说明线程们在疯狂抢锁。而 `write` 才 16 次（BufWriter 在攒批），说明磁盘 I/O 根本没跑满。

瓶颈完全在 Channel 的锁竞争上。

回过头想，`mpsc::channel` 在多个 sender 同时 `send` 的时候，内部需要加锁保护共享队列。每行数据都要在堆上分配一个 `String`，然后 `send` 进 Channel（一次锁），消费者 `recv` 出来再写。2000 万行就是 2000 万次堆分配 + 2000 万次锁操作。线程越多，锁竞争越激烈。

顺便一提，在这个阶段我还试过手写 SIMD 指令加速字符串拼接。之前参加 OceanBase 数据库大赛做向量索引优化的时候，手写 SIMD 取得了很好的效果（向量距离计算这种纯数值密集型运算确实适合 SIMD）。所以这次也想用同样的套路。结果发现在字符串拼接这个场景下，LLVM 编译器在 `release` 模式下自动做的向量化比我手写的还快。但那次 OB 大赛的经历让我对软硬件协同优化有了很深的认识——也正是那段经历驱动我去读《性能之巅》，然后有了这个 db_gen 项目。

> 可运行版本：`cargo run --release --bin v1_mpsc`

## V2：各写各的，最后合并

### 思路

V1 的问题很清楚：**多线程共享一个 Channel，锁竞争严重**。

怎么消灭锁？别共享就行了。让每个线程写自己的临时文件 `part_0.csv`、`part_1.csv`... 完全并行，互不干扰。写完之后主线程把它们合并成一个文件。这是分布式系统里 Share Nothing 思想的一个应用——消灭共享状态，就消灭了锁。

合并用什么？一开始用 `sendfile`，后来换成 Linux 的 `copy_file_range` 系统调用。这是内核提供的"零拷贝"文件复制——数据在内核态直接从一个文件描述符搬到另一个，不需要经过用户态缓冲区。

```
线程0 ──→ part_0.csv ─┐
线程1 ──→ part_1.csv ──┤──→ copy_file_range ──→ final.csv
线程2 ──→ part_2.csv ─┘
```

### 效果

跑起来：

```
>>> 阶段 1：多线程并行造数 (Zero-Allocation)...
>>> 造数完成！耗时: 1.25s，速度: 15971619.54 rows/sec
>>> 阶段 2：合并文件 (copy_file_range)...
>>> 合并完成！耗时: 2.46s
=== 总流程结束 ===
总耗时: 3.71s
总体有效吞吐: 5386073.51 rows/sec
```

造数阶段 16M rows/s，快了十几倍——没有锁之后多线程终于跑满了。但总吞吐只有 5.4M，因为合并阶段是串行的（一个文件一个文件追加写），而且产生了双倍 I/O：数据先写到临时文件，再读出来写到最终文件。等于同一份数据过了两遍磁盘。

造数 1.25 秒，合并 2.46 秒——合并比生成还慢。

> 可运行版本：`cargo run --release --bin v2_merge`

## V3：多线程写同一个文件

### 思路

V2 的瓶颈在合并。合并本质上是一次额外的全量读写，能不能彻底跳过这一步？

我想到一个办法：让所有线程打开**同一个文件**，但每个线程用 `lseek` 定位到不同的偏移量区域，各写各的。线程 0 写 0~300MB，线程 1 写 300MB~600MB，互不干扰。用 `fallocate` 预先分配好文件空间，就不用合并了。

这里需要补充一点 Linux I/O 的背景：应用程序调用 `write()` 的时候，数据并不是直接写到磁盘上的。Linux 在中间加了一层叫 **Page Cache** 的内存缓冲。`write()` 只是把数据从用户态拷贝到 Page Cache 里就返回了，实际的磁盘写入由内核在后台异步完成。这种 Buffered I/O 模式是 Linux 的默认行为，好处是 `write` 很快（基本是内存拷贝速度），坏处是数据实际上没有真正落盘——后面 V4 的 OOM 就跟这个有关。

还有一种叫 Direct I/O（`O_DIRECT` 标志），绕过 Page Cache 直接写磁盘，保证数据真正落盘，但速度受限于磁盘物理带宽。前面基准测试的 dd `oflag=direct` 就是这种模式，跑出来 1.2 GB/s。我在这个阶段也考虑过用 Direct I/O，但它对内存对齐有严格要求（通常要 4KB 对齐），而且在我们这个场景下 Buffered I/O 的吞吐更高，所以最终没用。

### 效果

```
>>> 优化版：多线程直接写入同一文件（无合并阶段）...
    检测到 4 个 CPU 核心，启动 4 个线程
>>> 写入完成！耗时: 1.52s，速度: 13139272.20 rows/sec
    实际写入: 1130.80 MB
总体有效吞吐: 12982070.76 rows/sec
```

13M rows/s，比 V2 的总吞吐快一倍多，而且没有合并阶段。看起来不错？

但我心里还是觉得不够快——V2 的造数阶段（纯粹的 Share Nothing 写独立文件）能跑到 16M rows/s，V3 写同一个文件反而更慢了。同样是多线程并行写，为什么写同一个文件会比写不同文件慢？

### 观测：perf

`perf` 是 Linux 内核自带的性能分析工具，通过采样 CPU 的程序计数器（PC），统计程序在哪些函数上花了多少时间。相当于每秒拍几十张"CPU 在干什么"的快照，汇总之后就是一张热点图。

```bash
sudo perf record -F 99 -a -g -- sleep 5
sudo perf report --stdio --no-children --comm v3_seek
```

```
# Overhead  Shared Object      Symbol
# ........  .................  ................................

     7.47%  [kernel.kallsyms]  [k] rwsem_spin_on_owner
            |
            ---rwsem_spin_on_owner
               rwsem_optimistic_spin
               rwsem_down_write_slowpath
               down_write
               ext4_buffered_write_iter
               ext4_file_write_iter
               new_sync_write
               vfs_write
               ksys_write
               __x64_sys_write
```

**7.47% 的 CPU 时间花在 `rwsem_spin_on_owner`**。

调用栈很清晰：用户态 `write()` → 内核 `vfs_write` → `ext4_buffered_write_iter` → `down_write` → `rwsem_down_write_slowpath` → `rwsem_spin_on_owner`。

`rwsem` 是读写信号量，`spin_on_owner` 说明线程在自旋等待锁的持有者释放锁。这是 ext4 文件系统的 **inode 级别锁**——即使你写的是同一个文件的不同位置，ext4 为了维护元数据（文件大小、Block 分配表、extent tree 等），在内核层面对同一个 inode 加了读写信号量。

所以 V3 的多线程看似在写不同位置，实际上在内核里排队等锁。用户态精心设计的"并行"，到了内核就变成了串行。

这也解释了为什么 V3 (13M) 比 V2 的纯造数阶段 (16M) 还慢——V2 造数阶段每个线程写独立文件，不同 inode，不存在内核级锁竞争。

> 可运行版本：`cargo run --release --bin v3_seek`

## V4：真正的 Share Nothing

### 思路

走到这里，三次碰壁，一个规律越来越清晰：**只要多线程共享任何东西——Channel（V1）、最终文件（V2 的合并）、inode（V3）——就一定有锁。**

那就什么都不共享。每个线程写完全独立的文件，不合并，结束。下游如果需要一个文件，那是下游的事。Hadoop 和 Spark 也是这么干的——分区输出。

### 效果

```
>>> V4: Share Nothing 极速版 (无背压！)
    ⚠️  警告：此版本可能导致 Dirty Pages 暴涨！
    启动 4 个线程
>>> 完成！
    耗时: 1.15s
    吞吐: 17337055.07 rows/sec
    写入带宽: 980.24 MB/s
    Dirty:           1157944 kB
    Writeback:             0 kB
```

17.3M rows/s，1.15 秒写完 2000 万行。但注意最后一行——**Dirty Pages 1.16GB**。

### 发生了什么

前面说了，`write()` 只是把数据写进 Page Cache，不等磁盘。这些在 Page Cache 里被修改过但还没写入磁盘的页面就叫 **Dirty Pages（脏页）**。内核有一组后台线程（pdflush / writeback）负责异步地把脏页刷到磁盘上。

问题是 CPU 生成数据太快了。用 `/proc/meminfo` 采样 Dirty Pages 的变化：

```
时间         Dirty(KB)    Writeback(KB)
08:35:07.08  320          0            ← 启动
08:35:07.45  395,848      0            ← 0.4秒后，已积压 386MB
08:35:07.86  893,636      0            ← 0.8秒后，873MB
08:35:08.29  1,158,132    0            ← 1.2秒后，1.1GB!
08:35:08.62  1,159,400    0            ← 程序结束，内核还没刷完
08:35:09.28  1,159,400    0            ← 仍然 1.1GB 脏页
```

不到一秒，Dirty Pages 从 320KB 涨到 1.1GB。`Writeback` 一直是 0——说明内核后台刷盘线程根本还没来得及启动。

前面的基准测试已经说了——CPU 生成数据 2.35 GB/s，SSD 裸写上限 1.2 GB/s，fdatasync 等刷盘只有 356 MB/s。CPU 灌数据比磁盘消化快了一倍多，没有背压，脏页越积越多，严重时会吃光物理内存触发 OOM killer。

我第一次跑 V4 的时候就是这么把系统搞崩的。而且实际情况比想象的更惨——当时忘了在 `.gitignore` 里排除 `output_*.csv`，IDE 的文件索引服务发现工作目录下突然冒出好几个几百 MB 的 CSV 文件，开始疯狂建索引。造数程序在拼命写，IDE 在拼命读，两个一起抢内存，系统直接 OOM 了。后来加了 `.gitignore` 才解决这个插曲。

> 可运行版本：`cargo run --release --bin v4_nosync`
> **注意：跑之前先开好内存监控，随时准备 Ctrl-C**

## V5：加上背压，回到人间

V4 证明了架构没问题，问题是**太快了**。CPU 全速往内存灌数据，磁盘跟不上。需要在应用层手动给自己踩刹车。

两个改动：

**1. 时间池（干掉 `clock_gettime`）**

`perf` 显示之前版本里 `clock_gettime` 占了大量 CPU——因为每行数据都要获取时间。

解法：启动时预生成 65536 个随机时间字符串放在数组里，运行时用位运算 `index & 0xFFFF` 随机取。一次系统调用变零次。

```rust
const TIME_POOL_SIZE: usize = 65536; // 2^16
let pool_mask = TIME_POOL_SIZE - 1;
// 热循环里：一次位运算代替一次 syscall
let time_idx = rng.gen::<usize>() & pool_mask;
batch_buf.extend_from_slice(&time_pool[time_idx]);
```

**2. 背压（每 64MB sync 一次）**

每写入 64MB 数据，调用 `sync_data()` 强制把 Page Cache 里的脏页刷到磁盘，并等待完成。CPU 被迫停下来等 I/O，Dirty Pages 不会失控。这就是所谓的**背压（Backpressure）**——快的一方主动等慢的一方。

```rust
if bytes_since_sync >= 64 * 1024 * 1024 {
    writer.flush().unwrap();
    writer.get_ref().sync_data().unwrap();  // 等磁盘
    bytes_since_sync = 0;
}
```

跑起来：

```
>>> V5: 最终版 - 批量生成 + 背压控制
    时间池: 65536, 批量: 1024, Sync间隔: 64MB
    启动 4 个线程
>>> 完成！
    耗时: 2.02s
    吞吐: 9903745.78 rows/sec
    写入带宽: 559.96 MB/s
    Dirty Pages: 1404 kB
```

9.9M rows/s，Dirty Pages 只有 1.4MB。看 Dirty Pages 的采样：

```
时间         Dirty(KB)    Writeback(KB)
08:35:10.63  9,932        0            ← 启动
08:35:11.09  85,336       4,188        ← 在写，开始刷盘
08:35:11.53  112,232      9,984        ← 峰值 ~110MB
08:35:11.92  128,680      25,352       ← sync 在工作
08:35:12.36  96,980       32,016       ← 开始回落
08:35:12.70  1,404        0            ← 程序结束，干干净净
08:35:13.03  1,404        0
```

对比 V4 那个 1.1GB 的脏页，V5 的峰值只有 128MB，结束后立刻回落到 1.4MB。

V4 到 V5 速度从 17.3M 降到 9.9M，看起来是"退步"了。但 V4 结束后 1.1GB 脏页还挂在内存里慢慢刷，V5 跑完就是干净的。这就是背压的意义：**跑得快不如跑得稳**。

V5 的 559 MB/s 写入带宽已经接近 fio 4 线程测出的 578 MB/s 了——基本上就是这块 SSD 在有 fdatasync 的情况下能跑到的极限。

> 可运行版本：`cargo run --release --bin v5_final`

## 番外：io_uring 的尝试

V5 稳定之后我还不死心——既然瓶颈在磁盘 I/O，能不能用更高效的 I/O 接口？

Linux 5.1 引入的 `io_uring` 是目前最先进的异步 I/O 接口。传统 `write()` 每次调用都是一次系统调用（用户态→内核态切换、保存寄存器、可能刷 TLB）。db_gen 靠 BufWriter 把写入攒到 2~4MB 一次，但每次 flush 还是一次 syscall。

io_uring 的核心思路是用**共享内存队列**代替系统调用：

- 用户态和内核共享两个环形缓冲区：SQ（提交队列）和 CQ（完成队列）
- 用户态往 SQ 里放 I/O 请求——纯内存操作，不进内核
- 内核从 SQ 取请求执行，结果放进 CQ——用户态直接从共享内存读
- 只有"通知内核来看一眼"时需要一次 `io_uring_submit()` syscall，而且可以**批量提交**

```c
// 纯内存操作，零 syscall
sqe = io_uring_get_sqe(&ring);
io_uring_prep_writev(sqe, fd, &iov, 1, offset);

// 一次 syscall 批量提交所有排队请求
io_uring_submit(&ring);

// 等结果（也是从共享内存读）
io_uring_wait_cqe(&ring, &cqe);
```

在 SQPOLL 模式下甚至连 `io_uring_submit()` 都可以省——内核有个专门的线程轮询 SQ。

我在 db_gen 里实际试了 io_uring。效果？**几乎没有提升**。一是 V5 每次 write 已经是 2~4MB 的大块写入，syscall 开销在总耗时里占比已经很低。瓶颈在磁盘物理带宽，不在 syscall 数量。io_uring 真正大放异彩的场景是高并发小 I/O——比如网络服务器每秒处理几万个小请求，每个请求一次 syscall 的开销加起来就很可观了。

这也是一个教训：**工具再先进，如果瓶颈不在它解决的问题上，就没用。** 得先量清楚瓶颈在哪。

## 自己写 eBPF 观测工具

整个优化过程用的都是现成工具（`syscount`、`perf`、`/proc/meminfo`）。但我总觉得这些通用工具给的信息太散——我想要的是**针对 db_gen 这个场景定制的观测**：每次 write 写了多少字节？延迟分布是什么样的？fdatasync 调了多少次？

于是用 bpftrace 写了一个自定义追踪脚本 `trace_db_gen.bt`，在内核的 `sys_enter_write` 和 `sys_exit_write` tracepoint 上插桩：

```c
// 进入 write() 时记录开始时间和请求大小
tracepoint:syscalls:sys_enter_write {
    @write_start[tid] = nsecs;
    @write_size[tid] = args->count;
}

// write() 返回时计算延迟
tracepoint:syscalls:sys_exit_write /@write_start[tid]/ {
    $latency_us = (nsecs - @write_start[tid]) / 1000;
    @write_latency_us = hist($latency_us);
    @write_count = @write_count + 1;
}
```

跑起来的效果（用 bpftrace 追踪 V5 运行时的写入行为）：

```
@write_count: 2275

@write_latency_us:        ← 每次 write() 的延迟分布（微秒）
[32, 64)              40 |@                                                    |
[64, 128)             40 |@                                                    |
[128, 256)            21 |                                                     |
[256, 512)            20 |                                                     |
[512, 1K)             20 |                                                     |
[1K, 2K)            1735 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@|
[2K, 4K)             351 |@@@@@@@@@@                                           |
[4K, 8K)              25 |                                                     |

@write_size_hist:         ← 每次 write() 写入的字节数分布
[64K, 128K)          109 |@@                                                   |
[2M, 4M)            2141 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@|
```

两个直方图一看就清楚了：

- **写入大小**：绝大多数 write 是 2~4MB 的大块写入（BufWriter 4MB 缓冲区攒满 flush），少量是 64~128KB 的小写入（sync 间隔点的 flush）
- **写入延迟**：大部分在 1~2 毫秒（对应 2~4MB 写入 Page Cache 的内存拷贝时间），少部分在 32~512 微秒（小块写入更快）

2275 次 write，0 次 fdatasync——说明压力测试版没开 sync（只有 V5 final 才有）。这些数据比 `syscount` 的一个"write: 2275"丰富太多了。

在 `perf_experiments` 目录下我还用 Python + BCC 写了两种 eBPF 数据传输的对比实现：
- `my_ebpf_tool.py`：用 BPF_HASH（Map 快照模式），每秒轮询清空，TUI 实时显示 top writers
- `write_ringbuf.py`：用 BPF_RINGBUF（Ring Buffer 模式），零拷贝事件流，逐事件回调

Ring Buffer 版本的开销更低、实时性更好，也是现代 eBPF 工具的主流选择。写完之后回去看 `syscount` 和 `bpftrace`，感觉完全不一样了——知道数据是怎么从内核的 kprobe 点，经过共享内存环形缓冲区，到我终端上的。

## 回头看

五个版本，每次都觉得自己的方案挺聪明，每次都被现实教育。几个比较深的印象：

**瓶颈是会迁移的。** 解决了 Channel 锁竞争，瓶颈移到了合并 I/O；去掉合并，撞上了 inode 锁；绕开 inode 锁，又碰到了内存爆炸；加了 io_uring 却发现 syscall 根本不是瓶颈。Gregg 在《性能之巅》里说：*"The performance of a system under increasing load is a migration of bottlenecks."* 完全就是这么回事。

**每个关键发现都来自工具，不是猜的。** futex 竞争是 `syscount` 看到的，inode 锁是 `perf report` 看到的，Dirty Pages 是 `/proc/meminfo` 看到的，write 延迟分布是自己写的 bpftrace 脚本看到的。之前在 OB 大赛的时候，我也是靠 perf + 火焰图才发现向量距离计算的热点可以用 SIMD 加速。观测能力就是性能优化的基本功。

**最快不等于最好。** V4 跑出 17.3M rows/s 但留下 1.1GB 脏页；V5 的 9.9M rows/s 跑完只剩 1.4MB。工程里选后者。

### 后续的一些想法

优化做到 V5 基本到了这块 SSD 的物理极限。如果还想更快，出路不在"写得更快"，而是"写得更少"。

这个思路其实在很多高性能系统里都有体现：
- OB 大赛里做向量索引，用 PQ（乘积量化）和 SQ（标量量化）把 float32 向量压缩到 int8，精度损失可控，但数据量减少 75%，CPU 一次 cache line 能装 4 倍的数据，IPC（每周期指令数）直接翻倍
- Redis 的 ziplist、listpack 这些紧凑结构，本质也是牺牲一点访问便利性换取更小的内存占用和更好的缓存命中率
- 数据库的列式存储 + 压缩（如 LZ4、ZSTD），减少 I/O 量，吞吐自然就上去了

回到 db_gen，如果把 CSV 的文本格式换成二进制格式（比如把 `u64` 直接写 8 字节而不是 20 字节的十进制字符串），数据密度能提升 2~3 倍。同样的磁盘带宽下，吞吐量就能翻倍。这是一个通用的性能优化思路：**降低数据密度（或者说提高信息密度），让有限的带宽传输更多有效信息。**

不过这就不是这个项目的重点了——db_gen 的目的是学习观测和调优，不是造一个最快的数据生成器。

## 怎么复现

```bash
cd db_gen

# 编译所有版本
cargo build --release

# 逐个运行
cargo run --release --bin v1_mpsc     # V1: MPSC
cargo run --release --bin v2_merge    # V2: 合并
cargo run --release --bin v3_seek     # V3: 同文件 lseek
cargo run --release --bin v4_nosync   # V4: 无背压（注意内存！）
cargo run --release --bin v5_final    # V5: 最终版
cargo run --release --bin bench_cpu   # 纯 CPU 基准
```

观测命令（需要 sudo）：

```bash
# 磁盘基准
dd if=/dev/zero of=./bench_disk bs=1M count=1024 oflag=direct && rm -f bench_disk
dd if=/dev/zero of=./bench_disk bs=1M count=1024 conv=fdatasync && rm -f bench_disk

# V1 运行时：看 futex 锁竞争
sudo syscount-bpfcc -p $(pidof v1_mpsc) -d 5

# V3 运行时：看 inode 锁
sudo perf record -F 99 -a -g -- sleep 5
sudo perf report --stdio --no-children --comm v3_seek

# V4/V5 运行时：看 Dirty Pages
watch -n 0.5 'grep -E "Dirty:|Writeback:" /proc/meminfo'

# 自定义 eBPF 追踪 (需要 bpftrace)
sudo bpftrace trace_db_gen.bt -p $(pidof db_gen)
```

所有运行日志和观测数据保存在 `logs/` 目录下。
