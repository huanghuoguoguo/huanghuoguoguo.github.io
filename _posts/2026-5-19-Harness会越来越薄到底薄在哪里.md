前几天面试被问到一个问题：

> 大模型能力越来越强，外面那层 harness 会不会被吸收掉？

我当时答得很顺。大概意思是，harness 承担了很多工作，LLM 的真实责任是稀疏的、有边界的。模型会变强，边界会移动，但状态持久化、容错恢复、安全沙箱、协议互操作这些东西不会消失。最后还补了一句很像面试答案的话：

> The harness is the chassis; the model is the engine.

讲完以后我觉得还行。至少没跑偏。

这两天又聊了一轮意图识别、tool calling、Claude Code SDK、Pi Agent 这些东西，才发现那段回答不完全对。不是结论错了，是我把两个东西混在一起叫 harness。

<!-- more -->

---

我一开始问的是一个很老的问题：现在做 agent 的，意图识别到底是什么？

以前做 chatbot，意图识别挺具体。

用户说“我要退货”，系统识别成：

```json
{
  "intent": "return_product",
  "slots": {
    "product": null,
    "order_id": null
  }
}
```

然后状态机接管：

```text
缺订单号 -> 追问订单号
有订单号 -> 查订单 API
符合条件 -> 进入退款流程
不符合条件 -> 套模板解释
```

这套东西有一堆名字：intent classifier、slot filling、对话状态机、固定工作流、关键词路由、模板回复。把它们放回 Rasa、Dialogflow、LUIS 那个时代，就不突兀了。模型没那么强，语言理解只能拆成一个个小模块。先分类，再抽槽位，再走流程，再填模板。

但现在你拿 Claude Code 这种 agent 看，就很奇怪。

用户说：

```text
帮我把这个 repo 跑起来，顺便看看测试为什么挂了
```

你不会先把它拆成：

```json
{
  "intent": "run_repository",
  "slots": {
    "repo": "current",
    "task": "test_failure"
  }
}
```

Claude Code 直接读文件、看 README、装依赖、跑测试、看报错、改代码、再跑测试。这里当然也有任务理解，也有工具选择，也有状态推进。但它没有以传统 NLU pipeline 的方式存在。

我之前答面试时说“harness 承担了大部分工作”，里面有一部分其实说的是这种旧 pipeline。它已经不像一个单独模块了。

---

以前那些 intent classifier、slot filling、固定状态机，并不是 agent runtime 的必要组成部分。它们是 pre-LLM 时代为了弥补模型理解能力不足写出来的胶水代码。

胶水代码这个词挺准确。它不直接提供业务价值，也不直接保证系统安全。它只是把用户那句自然语言，硬掰成程序能处理的几个字段。

以前没办法。

用户说“我买的那个手机坏了，钱还能退吗”，你要把它归到“退款”类，要抽出“手机”，还要判断它缺订单号。规则写得太死，泛化差；分类模型训练得不够，漏召回；状态机画得太细，维护到后面像在修水管。

LLM 出来以后，这一层开始松了。

用户说得再绕一点：

```text
上周那个东西到手就不能用了，我是不是只能认栽？
```

现在把这句话丢给强模型，它大概率能知道用户在问售后、退款、维权。关键词里没有“退款”两个字，也不妨碍它往那个方向理解。tool calling 还能把这种理解转成结构化参数。模板回复也松了，模型可以根据工具返回结果解释给用户听。

这时候再听“harness 被 LLM 吃掉”，我脑子里终于有了具体画面。

被吃掉的是：

```text
手写 router
手写 planner
关键词分流
固定 chatbot flow
模板拼接
复杂 prompt chain
为了弱模型准备的一堆中间步骤
```

这些东西会越来越薄。甚至很多时候会薄到你感觉不到它存在。

---

但另一层东西没有被吃掉。

Claude Code 能一把梭，不是因为它没有 harness。它自己就是一个做得很好的 harness。只是你不需要自己写那堆东西。

它给模型准备了文件系统、shell、编辑器、搜索、任务列表、上下文管理、权限边界。代码环境还有几个天然优势：

```text
文件可读
错误可观察
测试可跑
diff 可审查
git 可回滚
执行环境可隔离
```

这些东西让模型有机会试错。跑挂了，看报错。改错了，看 diff。测试不过，再修。人不放心，还能 review 一遍。

业务 agent 就没这么舒服。

退款、打款、审批、删客户数据、发正式邮件，这些动作不能靠“先试试”。错一次就是事故。这里需要的东西和模型聪不聪明关系没那么大：

```text
权限
审计
确认
沙箱
限流
回滚
状态持久化
数据隔离
错误恢复
```

这部分我还是坚持面试时的判断。模型再强，也不能替代数据库事务。模型再会理解用户，也不能自己变成权限系统。它可以决定“我想调用退款工具”，但真正能不能退、退多少、要不要二次确认，应该由系统兜住。

这就把 harness 拆成了两层。

| 层 | 以前我混着叫 harness | 现在我会怎么说 |
|---|---|---|
| 认知编排层 | intent、router、planner、workflow、prompt chain | 会被强模型吃掉很多 |
| 系统运行时层 | tool runtime、state、permission、sandbox、eval、audit | 不会消失，还会更重要 |

我以前说“harness 承担了大部分工作”，现在看，这句话太粗。

在弱模型时代，认知编排层确实承担了很多工作。你要帮它分类，帮它补槽位，帮它选流程，帮它决定下一步。那时候 harness 很厚，厚到很多人以为自己在做 agent，其实是在写一套很复杂的 chatbot framework。

模型变强以后，这部分会被吸进去。不是一点点消失，是整块塌掉。你再给 GPT-5.5 写 200 个 intent，可能只是在挡它。

---

这也解释了我前一篇 [《Harness 是错的》]({% post_url 2026-5-10-Harness是错的 %}) 里那种不舒服。

OpenSpec 生成一堆英文 proposal、design、tasks，看起来很完整，但我不想看。Superpowers 把流程拆得很细，agent 跑得很认真，但我跟不上。那时候我说“有些 harness，连当早餐都不配”。这句话有点狠，但问题确实在这里。

有一类 harness 不是在保护系统，也不是在保存人的判断。它只是在替模型做一堆模型已经能做的认知动作。

比如：

```text
先让 Router Agent 判断任务类型
再让 Planner Agent 拆步骤
再让 Executor Agent 执行
再让 Critic Agent 反思
再写一堆固定 workflow
再塞一堆 prompt 约束
```

对弱模型，这可能有用。对强模型，它就开始变成摩擦。模型被框住，人也被挡在外面。最后产物是一堆 AI 写给 AI 看的文档。人不读，模型也未必需要。

我之前以为 LLM 会吃掉大部分 harness，就是被这类东西影响了。

现在我会收窄这个判断：LLM 会吃掉大部分“认知编排型 harness”。它吃不掉运行时。更准确一点，它会让中间那层越来越薄，把真正该留在代码里的东西逼出来。

---

这件事对做垂直 agent 也有影响。

以前我可能会默认先列 intent：

```text
查询订单
申请退款
修改地址
投诉
转人工
```

然后给每个 intent 设计 slots、状态机、流程图。

现在我不想这么起手了。更好的入口是列真实任务：

```text
用户想完成什么
需要哪些信息
要查哪些系统
哪些动作能自动做
哪些动作要确认
哪些动作必须拒绝
失败了怎么退回来
```

工具也不能直接把后端 API 全丢给模型。

不好的是：

```python
get_user()
get_order()
update_order()
query_table()
call_api()
```

好一点的是：

```python
search_user_orders(user_id, time_range)
check_refund_eligibility(order_id)
create_refund_request(order_id, reason)
get_refund_status(refund_id)
```

这个变化不只是在换好听名字。系统边界应该放到工具里，别塞在 prompt 里。

`check_refund_eligibility` 只检查，不提交。`create_refund_request` 可以要求确认。金额计算走后端规则。权限校验走服务端。模型负责理解用户和选择工具，系统负责控制后果。

这才是我现在更愿意要的 harness。

薄一点。硬一点。

---

这也能接上 [《从 0 到 1 很容易，从 1 到 N 才是第一堵墙》]({% post_url 2026-5-3-从0到1很容易从1到N才是第一堵墙 %})。

0 到 1 的 demo，强模型确实越来越猛。给它工具，给它文件系统，给它长上下文，它能直接冲过去。传统 NLU 那套东西在这里看起来很笨，像是给自行车装铁轨。

从 1 到 N 就不一样了。维护阶段要的是可追踪、可回滚、可验证、可审计。不能只看模型每次都“像是理解了”，还得看它改完之后，系统能不能留下证据。

所以 harness 变薄，不是 agent 工程变少。

工程量会换地方。

少写一点 router，少画一点状态机，少搞一点多 agent 花活。多做一点工具边界，多做一点 eval，多做一点日志，多做一点权限和回滚。以前写在 prompt 里的软约束，尽量沉到代码里。以前让模型背的规矩，尽量变成工具返回值、测试失败、权限拒绝、人工确认。

我写 [《vibe coding 一定要懂 coding》]({% post_url 2026-5-8-vibe-coding一定要懂coding %}) 时讲过那个小说 CLI 的想法：不让 AI 直接乱改 md 文件，给它一个程序化入口。那时候的问题很具体，多份角色设定散在不同目录，agent 不知道该信哪份。CLI 的目标不是让模型更聪明，它只是让模型少碰不该碰的地方。

不要用一堆 prompt 教模型“请不要乱改人物设定”。给它一个 CLI，让它只能按结构化命令改。

prompt 会变薄。工具会变硬。

---

再改一下我那天的面试答案。

如果面试官现在再问我，模型强了以后 harness 会不会被吸收，我大概会这么答：

会，但要看你说的是哪层。

旧式 chatbot 里的 intent classifier、slot filling、关键词路由、模板回复、固定状态机，会被吃掉很多。Agent 框架里一部分 router、planner、workflow graph、多 agent 编排，也会被更强的模型和厂商 SDK 吸收。至少在代码 agent 这类环境里，好模型不需要你在外面给它搭那么多认知脚手架。

留下来的，是运行时边界。工具怎么执行，权限怎么给，状态怎么存，危险操作怎么确认，失败怎么恢复，日志怎么审计，评测怎么做。这些东西不会因为模型更聪明就消失。

所以 harness 会越来越薄。薄在中间那层认知编排。

底下那层会留下来，甚至会更硬。

我以前把这两层混成一坨，然后说“LLM 会吃掉大部分”。这句话太糙。现在我更想说：LLM 会吃掉那些为了弱模型存在的胶水。剩下的 harness，会更像一个运行时，不太像一套对话流程图。

这篇也不是结论。再过几个月，模型和 SDK 可能又往前拱一截。到时候也许我还会改口。

但至少现在我知道，“harness 会越来越薄”这句话该怎么讲了。
