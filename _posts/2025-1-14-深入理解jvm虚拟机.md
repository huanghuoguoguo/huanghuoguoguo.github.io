## 第一部分：自动内存管理

### 第 2 章 Java 内存区域与内存溢出

#### 运行时数据区

| 区域 | 线程共享 | 存放内容 | OOM 可能 |
|------|---------|---------|---------|
| 程序计数器 | 否 | 当前线程执行的字节码行号 | **唯一不会 OOM 的区域** |
| 虚拟机栈 | 否 | 栈帧（局部变量表、操作数栈、动态连接、返回地址） | StackOverflowError / OOM |
| 本地方法栈 | 否 | Native 方法的栈帧 | 同上 |
| 堆 | 是 | 对象实例、数组 | OOM（最常见） |
| 方法区 | 是 | 类信息、常量、静态变量、JIT 编译后的代码 | OOM |

补充：
- **TLAB**（Thread Local Allocation Buffer）：堆中每个线程预分配的小块内存，避免多线程分配时的竞争
- JDK 7 起，字符串常量池从永久代移到堆中
- JDK 8 起，永久代被元空间（Metaspace）取代，使用本地内存
- 直接内存（Direct Memory）不属于运行时数据区，通过 `Unsafe` 或 NIO 的 `DirectByteBuffer` 分配

#### HotSpot 对象探秘

**对象创建流程**：类加载检查 → 分配内存 → 零值初始化 → 设置对象头 → 执行 `<init>`

**内存分配方式**：
- 指针碰撞：堆内存规整时（用过的在一边，空闲的在另一边），移动指针即可
- 空闲列表：堆内存不规整时，维护一个空闲块列表

**对象内存布局**：对象头（Mark Word + 类型指针）+ 实例数据 + 对齐填充（8 字节对齐）

**对象访问方式**：
- 句柄访问：栈 → 句柄池 → 对象。GC 移动对象时只改句柄，栈中引用不变
- 直接指针访问（HotSpot 采用）：栈 → 对象。少一次间接寻址，更快

#### OOM 实战

| 区域 | 触发方式 | 关键参数 |
|------|---------|---------|
| 堆溢出 | 不断创建对象 | `-Xms` `-Xmx` `-XX:+HeapDumpOnOutOfMemoryError` |
| 栈溢出 | 递归过深 / 不断创建线程 | `-Xss` |
| 方法区溢出 | CGLib 等动态生成大量类 | `-XX:MaxMetaspaceSize` |
| 直接内存溢出 | NIO 分配过多 DirectBuffer | `-XX:MaxDirectMemorySize` |

### 第 3 章 垃圾收集器与内存分配策略

#### 判断对象存活

- **引用计数法**：简单，但无法处理循环引用。Python 用这个，Java 不用
- **可达性分析**：从 GC Roots 出发，不可达的对象判定为可回收

**GC Roots 包括**：虚拟机栈中引用的对象、方法区中静态变量和常量引用的对象、本地方法栈中 JNI 引用的对象、同步锁持有的对象等。

**四种引用**（从强到弱）：强引用 → 软引用（内存不足时回收）→ 弱引用（下次 GC 回收）→ 虚引用（仅用于回收通知）

**对象死亡的两次标记**：第一次可达性分析不可达 → 判断是否需要执行 `finalize()` → 放入 F-Queue → Finalizer 线程执行 → 第二次标记。`finalize()` 只会被调用一次，且不推荐使用。

#### 垃圾收集算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记存活对象，清除未标记的 | 简单 | 内存碎片 |
| 标记-复制 | 存活对象复制到另一块，原区整体清除 | 无碎片，速度快 | 浪费一半空间 |
| 标记-整理 | 存活对象向一端移动，清除边界外的 | 无碎片 | 移动对象需要 STW |

**分代假说**：弱分代（大多数对象朝生夕灭）→ 新生代用复制算法；强分代（熬过多次 GC 的对象难以消亡）→ 老年代用标记-清除/整理。

**Appel 式回收**：Eden (80%) + 两个 Survivor (各 10%)，每次只用 Eden + 一个 Survivor。Minor GC 后存活对象复制到另一个 Survivor，空间不够则通过担保机制进入老年代。

#### HotSpot 算法细节

- **OopMap**：记录栈和寄存器中哪些位置是引用，GC 时不需要全栈扫描
- **安全点**：只有在安全点才能停下来 GC。位置选取：方法调用、循环跳转、异常跳转等"长时间执行"的指令
- **安全区域**：线程处于 Sleep 或 Blocked 时无法走到安全点，安全区域解决这个问题
- **记忆集 / 卡表**：解决跨代引用问题。卡表是记忆集的一种实现，一个卡页对应一块内存区域，通过写屏障维护
- **三色标记**：白色（未访问）、灰色（已访问但引用未扫完）、黑色（已访问且引用已扫完）
- **"对象消失"问题**：并发标记时，黑色对象新增了对白色对象的引用，且灰色对象删除了对该白色对象的引用。两种解决方案：增量更新（CMS）、原始快照 SATB（G1）

#### 垃圾收集器对比

这是我重点关注的部分。核心思路是理解**每个收集器解决了什么问题，又引入了什么新问题**。

从串行到并行再到并发，每一代收集器都在减少 STW 的范围。关键转折点在于**移动对象时能否并发**——这直接决定了最后一个阶段是否需要 STW：

| 收集器 | 算法 | 移动对象？ | 最终阶段并发？ | 关键技术 |
|-------|------|----------|-------------|---------|
| CMS | 标记-清除 | 否 | 是（并发清除） | 增量更新处理漏标 |
| G1 | 标记-整理（Region） | 是 | **否（筛选回收 STW）** | SATB + Region + 停顿预测模型 |
| Shenandoah | 标记-整理 | 是 | 是（并发整理） | Brooks Pointer 转发指针 |
| ZGC | 标记-整理 | 是 | 是（并发重分配） | 染色指针 + 多重虚拟内存映射 |

**共同点**：GC Root 扫描都 STW，并发标记阶段都与用户线程并行，都要处理并发标记期间引用关系变化的问题。

**CMS 四个阶段**：初始标记（STW）→ 并发标记 → 重新标记（STW）→ 并发清除。三个缺点：CPU 敏感、浮动垃圾（并发清除期间产生的新垃圾要等下次 GC）、内存碎片。

**G1**：分 Region，Region 也分代。新生代用标记-复制，老年代用标记-整理。引入了停顿预测模型（衰减均值），可以指定期望停顿时间，按回收价值选择 Region。但筛选回收阶段涉及对象移动，没有提供并发手段，需要 STW。

**Shenandoah**：与 G1 的三个不同——支持并发整理、不使用分代、用连接矩阵替代记忆集。通过 Brooks Pointer（每个对象前加一个转发指针）实现并发移动对象。代价是读屏障开销。

**ZGC**：JDK 21 后支持分代。染色指针把 GC 信息（标记、重映射等）编码在指针的高位（44-48 位），通过多重虚拟内存映射让不同标记状态的指针指向同一物理内存。指针自愈能力：读到旧指针时自动修正。优化了 cache line 伪共享问题。STW 时间控制在亚毫秒级。

#### 内存分配策略

- 对象优先在 Eden 分配
- 大对象直接进老年代（`-XX:PretenureSizeThreshold`）
- 长期存活对象晋升老年代（对象年龄计数器，`-XX:MaxTenuringThreshold`）
- 动态年龄判定：Survivor 中相同年龄对象大小总和超过 Survivor 空间一半，则 >= 该年龄的直接晋升
- 空间分配担保：Minor GC 前检查老年代可用空间是否够，不够则触发 Full GC

### 第 5 章 调优案例

这一章是实战案例集，记录几个有印象的：

- **大内存单 JVM 实例**：Full GC 停顿时间很长。可考虑拆成逻辑集群（多个小 JVM），但要处理会话亲和性和全局资源问题

## 第二部分：虚拟机执行子系统

### 第 6 章 Class 文件结构

Class 文件是平台无关性的基石——JVM 只认 Class 文件，不关心源码是 Java、Kotlin 还是 Scala。

Class 文件是紧凑的二进制流，数据项严格按顺序排列。主要结构：

```
魔数 (0xCAFEBABE) → 版本号 → 常量池 → 访问标志 → 类/父类/接口索引
→ 字段表 → 方法表 → 属性表
```

**常量池**：存放字面量（字符串、final 常量）和符号引用（类名、字段名、方法名）。17 种常量类型。

**属性表**中最重要的是 **Code 属性**：存放方法的字节码指令、操作数栈深度、局部变量表大小、异常表等。

**字节码指令**：操作码 1 字节（最多 256 条指令），面向操作数栈设计。指令集按功能分为：加载存储、运算、类型转换、对象创建与访问、控制转移、方法调用与返回、异常处理、同步。

这一章细节很多，粗看。需要时用 `javap -verbose` 反编译查看即可。

### 第 7 章 类加载机制

#### 类的生命周期

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
        |___ 连接 ___|
```

**加载**：读取字节码 → 转为方法区的运行时数据结构 → 创建 `Class` 对象。

**验证**：文件格式验证 → 元数据验证 → 字节码验证 → 符号引用验证。

**准备**：为类的**静态变量**分配内存并设零值（不是赋初值）。`static final` 的编译期常量在这阶段直接赋值。

**解析**：符号引用 → 直接引用。符号引用是字符串描述，直接引用是内存地址/偏移量。

**初始化**：执行 `<clinit>()` 方法（编译器收集所有静态变量赋值和 static 块合并而成）。

**触发初始化的情况**（主动引用）：`new`、`getstatic`、`putstatic`、`invokestatic` 四个指令、反射调用、子类初始化触发父类、主类、接口 default 方法相关、`MethodHandle`。

**不触发初始化的情况**（被动引用）：通过子类引用父类的静态字段、数组定义、访问 `final` 常量（常量内联，编译期已经放入调用方的常量池）。不过被动引用应该还是会触发加载的。

#### 类加载器与双亲委派

三层类加载器：Bootstrap（C++ 实现，加载 `JAVA_HOME/lib`）→ Extension / Platform（加载 `lib/ext`）→ Application（加载 classpath）。

**双亲委派**：收到加载请求时先委派给父加载器，父加载器无法加载时才自己加载。保证核心类库不被篡改。

**三次被破坏**：
1. JDK 1.2 前，自定义加载器重写 `loadClass()` 破坏委派链
2. SPI 机制（如 JDBC），核心类需要加载厂商实现 → 线程上下文类加载器
3. OSGi 热部署，类加载器不再是树形结构而是网状

JDK 9 引入模块化，扩展类加载器改名为平台类加载器，不再派生自 `URLClassLoader`。

### 第 8 章 字节码执行引擎

#### 栈帧结构

每个方法调用对应一个栈帧，包含：
- **局部变量表**：以变量槽（Slot）为单位，32 位以内类型占 1 个 Slot，long/double 占 2 个。实例方法的第 0 个 Slot 是 `this`
- **操作数栈**：字节码指令的工作区，类似 CPU 寄存器的作用
- **动态连接**：指向运行时常量池中该方法的符号引用，支持运行时绑定
- **方法返回地址**：正常返回或异常退出的处理

#### 方法调用

五条调用指令：
- `invokestatic`：静态方法
- `invokespecial`：构造器、私有方法、`super` 调用
- `invokevirtual`：虚方法（运行时根据实际类型分派）
- `invokeinterface`：接口方法
- `invokedynamic`：动态调用点（lambda、方法引用）

**静态分派**（编译期确定）：方法重载，按参数的**静态类型**（声明类型）匹配。
**动态分派**（运行时确定）：方法重写，按对象的**实际类型**查找虚方法表（vtable）。

Java 是**静态多分派、动态单分派**的语言：重载看多个参数的静态类型（多分派），重写只看接收者的实际类型（单分派）。

### 第 9 章 类加载案例

- **Tomcat 类加载器**：Web 应用之间需要隔离（同名类可以共存），所以每个 Web 应用有独立的类加载器
- **OSGi**：通过类加载器实现模块热部署，加载器之间不是树形而是网状结构
- **字节码生成与动态代理**：`java.lang.reflect.Proxy` 在运行时生成代理类的字节码

## 第三部分：编译与优化

### 第 10 章 前端编译（javac）

javac 的编译过程：词法分析（源码 → Token 流）→ 语法分析（Token → AST）→ 填充符号表 → 注解处理（可能触发新一轮编译）→ 语义分析（标注检查 + 数据流分析）→ 解语法糖 → 生成字节码。

**语法糖**：
- **泛型**：Java 使用类型擦除实现伪泛型（编译后泛型信息被擦除），与 C# 的真泛型不同。擦除是为了二进制向后兼容。`Signature` 属性保留了泛型信息供反射使用
- **自动装箱/拆箱**：`Integer.valueOf()` / `intValue()`
- **条件编译**：`if (常量表达式)` 中不可达的分支在编译期被消除

### 第 11 章 后端编译（JIT）

#### 即时编译器

解释器与编译器共存：解释器启动快，编译器运行快。**分层编译**把编译分为 5 个层次（从纯解释到 C2 完全优化），逐步升级。

**热点探测**：基于计数器（方法调用计数器 + 回边计数器），超过阈值触发 JIT 编译。方法调用计数器有**衰减**机制（半衰周期），避免不常用的方法也被编译。

**栈上替换（OSR）**：循环体触发编译时，不等方法退出，直接替换当前栈帧中的解释代码为编译代码。

#### 提前编译（AOT）

与 JIT 相比，AOT 启动快但无法做运行时优化。JIT 的三个独特优势：
- **性能分析制导优化**：根据实际运行的热点路径优化
- **激进预测性优化**：大胆假设 + 逆优化退路
- **链接时优化**：跨模块内联

#### 主要优化技术

| 优化技术 | 说明 |
|---------|------|
| 方法内联 | 消除调用开销，是其他优化的基础。虚方法通过 CHA（类型继承关系分析）判断是否只有一个实现 |
| 逃逸分析 | 分析对象是否逃逸出方法/线程。未逃逸的可以：栈上分配、标量替换、同步消除 |
| 公共子表达式消除 | 表达式已计算过且变量未变，直接用之前的结果 |
| 数组边界检查消除 | 编译期能证明不越界的，去掉运行时检查 |
| 隐式异常处理 | 用 SIGSEGV 信号替代 null 检查，正常路径零开销 |

## 第四部分：高效并发

### 第 12 章 Java 内存模型与线程

**核心问题**：多核 CPU 各有缓存（MESI 协议），对共享变量的操作存在可见性和有序性问题。

**Java 内存模型（JMM）**：每个线程有自己的工作内存（CPU 缓存的抽象），只操作副本，操作完毕后刷回主内存。定义了 read、load、use、assign、store、write、lock、unlock 八种原子操作，以及它们之间的顺序约束。

**volatile 的两个语义**：
1. **可见性**：写 volatile 变量时，通过 lock 前缀指令让其他处理器的缓存失效。但 volatile **不保证原子性**——单条字节码翻译成机器码后可能不是原子的
2. **禁止指令重排序**：通过内存屏障实现

**long/double 的非原子性协定**：64 位数据的读写可能被拆成两次 32 位操作（实际上现代 JVM 都保证了原子性）。

**先行发生原则（Happens-Before）**：判断数据竞争和线程安全的核心规则。包括程序顺序规则、监视器锁规则、volatile 变量规则、线程启动/终止规则、传递性等。

**Java 线程实现**：早期用用户线程，后来改为 1:1 映射到操作系统内核线程。JDK 19+ 引入虚拟线程（协程）。

一个补充：虽然 Tomcat 8.5 后使用了 NIO，但 NIO 只在接收和发送数据时使用非阻塞 I/O，请求的处理逻辑仍然是单线程执行的，所以 ThreadLocal 仍然可用，不会出现上半部分是一个线程、下半部分变成另一个线程的情况。

### 第 13 章 线程安全与锁优化

**synchronized 的缺点**：
- 线程进入等待队列后只能等待，无法中断或设置超时
- 非公平锁
- 不能绑定多个条件（Condition）

**ReentrantLock** 解决了上述三个问题（支持中断、可选公平锁、多 Condition），但默认也是非公平锁。

**锁升级**（JDK 6+，对象头 Mark Word 中记录锁状态）：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

- **偏向锁**：只有一个线程访问时，在对象头记录线程 ID，后续该线程进入无需 CAS。出现竞争时升级
- **轻量级锁**：CAS 尝试获取，适合锁竞争不激烈、持有时间短的场景。CAS 失败后自旋，自旋超限后升级
- **重量级锁**：操作系统互斥量（Mutex），线程阻塞/唤醒涉及用户态和内核态切换

锁只能升级不能降级（偏向锁除外，偏向锁可以被批量撤销）。
