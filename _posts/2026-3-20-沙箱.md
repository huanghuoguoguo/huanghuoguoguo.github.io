前段时间，OpenClaw 这类 Agent 很火。我一开始没有太在意，觉得无非是工具调用做得更完整一些。后来翻了文档，才意识到它们和普通聊天机器人已经不是一个层级的东西了。它们不只是回答问题，还会读写文件、执行命令、调用浏览器、持续跑工作流。

能力强，风险也跟着上来了。只要 Agent 开始碰文件系统、进程和网络，它就是在真实系统里执行动作了。有些动作还是它刚刚自己写出来的代码。问题就变成了：它能不能被约束住。

我最近正好在给 LangBot 想沙箱方案，所以顺手去看了 OpenClaw 的文档。官方在 [Security](https://docs.openclaw.ai/gateway/security) 页面里也讲得很直接，它更偏 personal assistant 场景：一个人自己用还好，到了多人共享、外部消息混入、第三方工具接入这些场景，边界就会复杂很多。

更直接的一点是，OpenClaw 已经专门做了一套 [Sandboxing](https://docs.openclaw.ai/gateway/sandboxing) 配置。`exec`、`read`、`write`、`edit`、`process` 这些工具可以放进 Docker 容器里跑，默认网络是 `none`，工作区权限也分了几档。文档同时也写得很清楚：这不是完美边界；如果你不开沙箱，那这些工具就是直接在宿主机上执行。

看到这里，我才意识到自己以前把沙箱这个词想得太笼统了。Docker 是沙箱，浏览器是沙箱，云函数背后那一套也常被叫沙箱，但它们显然不是一回事。

所以这篇文章不是综述，更像是我这次查资料时顺手整理的一份学习笔记。很多地方我也是查到哪写到哪。先把现在理解到的东西记下来，后面如果发现哪里想错了，再回头改。

你的 AI Agent 想在你的机器上跑一段 Python 脚本。这段代码是大模型生成的，你看了一眼，大概没问题。但如果模型下次生成了一句 `os.system("rm -rf /")`，或者往 `~/.ssh/authorized_keys` 里写了一行公钥呢？

这不是假设。当 Agent 能调用工具执行代码的时候，不可信代码执行就是一个现实问题。不管代码来自模型生成、用户上传、还是第三方插件——只要它不是你自己写的、自己审过的，就不应该直接在宿主机上裸跑。

沙箱就是干这个的。但沙箱这个词被用得太泛了，从浏览器的 JS 沙箱到 AWS Lambda 的 Firecracker，都叫沙箱。它们的隔离强度、性能代价、适用场景差了十万八千里。

## 沙箱的核心思想

我现在会先把沙箱理解成：**默认拒绝，显式放行。**

沙箱把被执行的代码关进一个受限环境——文件系统看不到宿主机的、网络默认不通、资源有上限、进程跑不出去。你想让它访问什么，必须显式地开一个口子。

类比的话，普通子进程就像让一个陌生人在你家客厅随便走动，沙箱是把他关进一间只有桌椅的房间，门从外面锁着，需要什么东西你递进去。

这个想法不复杂。复杂的是实现。从几乎没有隔离的 `chroot`，到接近硬件级隔离的 microVM，差别非常大。它们都叫沙箱，但不是同一种东西。

## 实现方案：从弱到强

### 1. 语言级限制

我一开始先想到的，是这种最轻量的做法：不做操作系统隔离，直接在语言运行时里做限制。

Python 的 `RestrictedPython`、Java 以前的 `SecurityManager`、JavaScript 的 `vm`，都算这个思路。说白了，就是不去隔离一个完整的进程环境，而是在语言自己那一层想办法把危险动作拦下来。比如限制导入 `os`、`subprocess`，或者不让你碰某些敏感 API。

但查了一圈之后，我基本不再把它当成沙箱候选。原因也不复杂：进程还是宿主机进程，内存空间还是共享的。只要找到一个绕过 AST 检查的方式，比如动态导入、C 扩展、子进程，整个限制就形同虚设。`RestrictedPython` 在社区里有大量已知的逃逸方式。Java 的 SecurityManager 在 JDK 17 被标记为废弃，官方也明确说它很难提供可靠的安全边界。

拿来防低级误操作也许可以。比如在线教育平台限制学生代码不要调 `exit()`。

### 2. chroot

再往下一层，就是 `chroot`。这个东西很老，也很 Unix。它干的事情非常直接：不去限制语言，不去虚拟化内核，只是把进程看到的根目录改掉。进程以为 `/` 是整个世界，实际上只是宿主机上的一个子目录。

```bash
# 创建一个最小的 chroot 环境
mkdir -p /sandbox/{bin,lib,lib64}
cp /bin/bash /sandbox/bin/
cp -L /lib/x86_64-linux-gnu/libc.so.6 /sandbox/lib/
chroot /sandbox /bin/bash
# 现在这个 bash 只能看到 /sandbox 下的文件
```

我看到这里的时候，才更明显地意识到，`chroot` 和“真正的沙箱”之间其实还差很远。它只隔离文件系统视图，不隔离网络、PID、用户、IPC。而且 root 用户本来就很容易逃出去，所以它更像一个很粗的文件系统视图切换，不太像一个你敢放心拿来跑不可信代码的边界。

性能开销几乎为零，就是改了一个路径指针。Gentoo 的 `emerge` 和 Arch 的 `makepkg` 用 `chroot` 来隔离编译依赖，防止污染宿主系统，这个用途很合理。

### 3. Linux Namespaces + Cgroups

以前我也知道 Docker 底下有 namespace、cgroup 这些词，但基本停留在背过名词的程度。这次翻文档的时候，我才第一次比较像样地把 Docker 拆开看。说起来挺土的，我之前一直以为 Docker 有什么自己的黑魔法，后来才发现，它用的其实都是 Linux 内核已经给好的能力。Docker 更像是把这些零件拼成了一个大家能顺手用的产品。

我现在会先把它们粗暴地分成两类来看：

- namespace 解决“这个进程能看到什么”
- cgroup 解决“这个进程能用多少”

这样一分，脑子就清楚多了。

Namespaces 把内核资源切成独立的视图：

| Namespace | 隔离什么 | 效果 |
|-----------|---------|------|
| PID | 进程 ID | 沙箱内看不到宿主机的进程 |
| NET | 网络栈 | 独立的网卡、IP、端口、路由表 |
| MNT | 挂载点 | 独立的文件系统视图 |
| UTS | 主机名 | 独立的 hostname |
| IPC | 进程间通信 | 隔离共享内存、信号量 |
| USER | 用户 ID | 容器内 root ≠ 宿主机 root |
| Cgroup | Cgroup 视图 | 看不到其他 cgroup 的资源信息 |

Cgroups（Control Groups）做资源限制：CPU 配额、内存上限、IO 带宽、进程数上限。简单说，namespace 管“能看到什么”，cgroup 管“能用多少”。

我后来对 namespace 比较能记住的一点是：它不是虚拟出一台新机器，也不是复制整个内核，它更像是给同一个内核切出几套不同的“视图”。进程发系统调用的时候，内核先看它在哪个 namespace 里，再决定它看到的是哪一份 PID 表、挂载树、路由表。你可以把它理解成，同一套内核底下，很多进程戴着不同的眼镜在看世界。

几个例子：

- `PID namespace`：容器里的进程看到自己是 PID 1，但在宿主机上它其实是另一个 PID
- `MNT namespace`：给进程一棵独立的挂载树，在里面挂载或卸载文件系统，不影响外面
- `NET namespace`：隔离整套网络栈——网卡、IP、路由表、端口空间、iptables 规则都独立
- `USER namespace`：做 UID/GID 映射，容器内的 root 可以映射成宿主机上的普通用户

这套机制底下主要靠几个系统调用串起来：

- `clone()` 创建子进程时顺手创建新的 namespace
- `unshare()` 让当前进程脱离原来的共享上下文
- `setns()` 把进程切进另一个已有 namespace

用户态看到的 `/proc/<pid>/ns/*` 就是这些 namespace 对象的句柄。Docker、Podman 这些容器工具，说到底也都是在调这些内核能力。

```bash
# 用 unshare 手动创建一个隔离环境
unshare --mount --pid --net --fork --map-root-user bash
# 现在这个 bash 在独立的 PID/NET/MNT namespace 里
```

查到这里我才慢慢明白，为什么容器这套东西性能能这么接近裸机。因为 namespace 更像是在切内核数据结构，cgroup 更像计数器和限流器，它不像虚拟机那样要真的多跑一层 Guest OS。至少按我现在的理解，这也是 Docker 能又轻又快的根子之一。

Docker、Podman、bubblewrap、nsjail 这些工具都踩在这层地基上，区别主要在上层怎么封装。

### 4. bubblewrap / nsjail / firejail

这一类工具，我现在更愿意把它们看成介于“手动玩 namespace” 和 “上完整容器”之间的轻量沙箱。

bubblewrap（bwrap）给我的感觉特别像“把 namespace 这些零件用顺手一点”。它不管镜像管理，也不管网络编排。你告诉它哪些目录可见、哪些只读、要不要网络，它就给你一个最小化的沙箱。Codex 现在就是用 bubblewrap。Chrome 那套渲染进程隔离，本质上也有类似思路，只不过会再叠上 seccomp-bpf 这种 syscall 过滤。seccomp-bpf 这个词第一次看也挺唬人，其实粗暴理解就是：给进程一张 syscall 白名单，不在上面的直接拒绝。

```bash
bwrap --ro-bind /usr /usr \
      --dev /dev \
      --proc /proc \
      --tmpfs /tmp \
      --unshare-net \
      --die-with-parent \
      python3 untrusted_script.py
```

nsjail、firejail 也差不多，都是把 Linux 原生这些隔离能力封起来，变成更容易直接拿来用的工具。看到这里我其实有点感慨，Linux 生态在“单进程隔离”这件事上早就做得挺成熟了，只是平时写业务代码的人很少会主动碰到这层。

它们的性能开销和裸 namespace 差不多，因为本质上还是在配 namespace/cgroup 这些东西，不会凭空再多一层重运行时。所以如果你只是想隔离一个进程，或者隔离一次构建、一次判题，它们会挺顺手。但如果场景开始涉及镜像、依赖缓存、环境复用、生命周期管理，那你很快就会想念 Docker 这种更完整的工程化包装。

### 5. 容器（Docker / Podman）

如果压成一句，容器大概就是 namespace + cgroup，再加上联合文件系统（把多个目录层叠在一起，看起来像一个完整文件系统）、镜像管理和生命周期管理。

技术上，容器的隔离并不比裸 namespace 强多少。Docker 容器跑的进程在宿主机用 `ps` 还是能看到，只是在容器内的 PID namespace 里看不到外面。但容器把隔离做成了工程上好用的产品，这个差距很大：

- 镜像把运行环境打包成不可变的层，任何机器上 `docker run` 就能复现
- 分层存储，改一行代码不需要重新下载整个环境
- Docker Hub 上有几十万个现成镜像
- 创建、启动、暂停、恢复、销毁，API 一条龙

```bash
# rootless Podman —— 不需要 root 权限
podman run --rm --network=none \
  --memory=512m --cpus=1 \
  --read-only --tmpfs /tmp \
  python:3.12-slim python3 untrusted_script.py
```

Podman 的 rootless 模式会更让我安心一些。容器引擎本身就不以 root 运行，即使真发生逃逸，拿到的也只是普通用户权限。

性能开销方面，IBM 那篇经典论文（没看）测了 CPU、内存、网络、磁盘 IO，容器在所有维度上都接近裸机。CPU 密集型任务开销不到 1%，网络延迟多了几微秒。磁盘 IO 要注意：用 bind mount（直接把宿主机目录映射进容器）几乎无损，但 overlay2 文件系统会有 10-30% 的写放大。overlay2 是 Docker 默认的存储驱动，它把镜像的只读层和容器的可写层叠在一起；写文件时需要先把文件从只读层复制到可写层再修改，这就是写放大的来源。冷启动（镜像已拉取）一般在 500ms-2s。

> 下面关于 gVisor、Firecracker、microVM、V8 Isolates 和 WebAssembly 的内容，主要来自论文、官方文档和技术博客。我自己没有在生产环境真正跑过这些方案，所以这里只能尽量用“和 Docker 比，它多做了什么、少做了什么”来理解；如果哪里说错了，后面再改。

### 6. gVisor

如果拿 Docker 当参照物，gVisor 最核心的区别是：普通容器里的进程，系统调用基本是直接打到宿主机内核；gVisor 中间多垫了一层。进程的 syscall 先被它的 Sentry 组件截住，在用户态处理一轮，不直接碰真实宿主机内核。

```
普通容器:   应用 → syscall → 宿主机内核
gVisor:    应用 → syscall → Sentry(用户态) → 少量 syscall → 宿主机内核
```

这样一来，它和 Docker 的差别就很清楚了。Docker 更轻，也更接近原生 Linux；gVisor 则是在“共享宿主机内核”这件事上再补一道防线。你可以把它理解成：还是容器那条路，但不再那么信任应用能老老实实调 syscall。

代价也很直接：syscall 变长了。每个 syscall 都多走一层用户态拦截，IO 密集型场景尤其明显。论文里会把差距写得很夸张，比如某些 open/close 微基准比原生容器慢很多倍。但我现在更愿意把它理解成一句更朴素的话：如果你的程序天天在高频 syscall，gVisor 会比 Docker 更容易让你感觉到“这层保护确实在那”。

反过来说，如果你的任务更像是跑业务逻辑、做计算、偶尔读写文件，而不是 Redis 这种 syscall 特别密的东西，它相比 Docker 的差距未必会大到不能接受。

所以按我现在的理解，gVisor 适合那种还想保留容器工作流，但又觉得 Docker 共享宿主机内核这件事不够放心的场景。尤其是多租户代码执行、共享运行环境、你明显更怕“隔离不够”而不是“多几毫秒开销”的地方。

### 7. Firecracker / microVM

Firecracker 走的是另一条路。它和 Docker 的差别，不是在容器上再补一道 syscall 防线，而是干脆不再共享内核。

VMM（Virtual Machine Monitor）就是负责创建和管理虚拟机的程序，QEMU 是最常见的一个。Firecracker 本质上是一个特别瘦的 VMM，基于 KVM，把传统虚拟机里很多又重又老的设备模拟砍掉了，只保留最小的一套 virtio 设备。

```
容器:        应用 → 宿主机内核（共享）
Firecracker: 应用 → Guest 内核 → KVM → 宿主机内核（不共享）
```

如果把它和 Docker 放在一起看，我脑子里最先冒出来的其实不是启动速度，而是边界感。Docker 里的进程还是在同一个宿主机内核上活动；Firecracker 里面有自己的 Guest 内核，这一层就硬很多了。

它和完整 VM 相比又轻很多，启动快很多，内存壳也小得多。所以它比较像夹在 Docker 和完整虚拟机中间的一层：比 Docker 隔离更硬，比传统 VM 更轻。

所以它更适合那种“比 Docker 更怕逃逸，但又比完整 VM 更在意启动速度”的场景。比如代码执行平台、短生命周期任务、一次请求拉起一个隔离环境这种。如果只是自己机器上单用户跑点代码，或者启动时间根本不是问题，那它相对 Docker 的优势就没那么显著。

### 8. 完整 VM（KVM / QEMU / Hyper-V）

完整 VM 就比较好理解了。它也是最传统的一种隔离方式。和 Docker 比，它不只是“不共享 namespace”，而是连内核、虚拟硬件、内存空间都彻底拆开了。这个边界最硬，也因此最重。

- 启动时间：最小化 Linux 直接内核引导约 2 秒，完整发行版 10-18 秒
- 内存开销：256MB-1GB+（Guest OS 本身消耗）
- 运行时 CPU 开销：3-5%（KVM + VT-x 硬件辅助虚拟化），磁盘 IO 开销 10-15%

所以它和 Docker 的取舍也很直接。你拿更高的隔离强度，换更长的启动时间和更重的资源占用。如果场景是每秒钟要拉起几百个执行环境，它就很笨重；但如果是起一个环境跑几小时，或者你压根不想和宿主机共享太多东西，那完整 VM 反而更省心。

### 9. V8 Isolates / WebAssembly

沙箱也不一定非得走操作系统那条路。V8 Isolates 和 WebAssembly 走的是另一套思路，不靠 namespace，也不靠传统虚拟化。和 Docker 最大的不同，是它们不试图隔离一个“完整 Linux 进程世界”，而是从语言运行时或字节码层面就把执行环境收窄了。

V8 Isolates 可以理解成：不拉起容器，不起 Guest 内核，甚至连独立进程都不一定给你，而是在同一个 V8 进程里切出很多互相隔离的小执行单元。它比 Docker 轻得多，启动也快得多，但前提是你接受语言和运行时边界都被收死了。它天然就不是给“随便跑一段 Linux 程序”准备的。

WebAssembly 也有点这个意思。它和 Docker 的最大区别，不是“更安全一点”这么简单，而是执行模型都换了。Docker 里跑的是普通 Linux 程序；Wasm 里跑的是受约束的字节码，内存模型和系统接口都被收得很窄。这样换来的就是启动快、隔离边界清晰，但代价是生态和语言支持远没有 Docker 那么通用。

所以如果你的场景本来就愿意被语言/runtime 限制，比如插件系统、边缘逻辑、极端在意冷启动，那这些东西会比 Docker 更自然。但如果你就是想“给我一段 Python/bash/apt-get 能跑起来的环境”，那它们和 Docker 根本不是一类东西。

## 全景对比（ai绘制）

| 方案 | 隔离强度 | 冷启动时间 | 运行时开销 | 内存占用（每实例） | 语言限制 |
|------|---------|-----------|-----------|-----------------|---------|
| 语言级限制 | 极弱 | 0 | 0 | 0 | 单语言 |
| chroot | 弱 | <1ms | ~0 | ~0 | 无 |
| bubblewrap/nsjail | 中 | <100ms | <1% | <5MB | 无 |
| 容器（Docker/Podman） | 中 | 500ms-2s | <1% CPU, 10-30% overlay IO | 约 1MB 容器壳 + 业务进程 | 无 |
| gVisor | 中强 | ~200ms | <3% 生产典型, IO 密集可达 30% | 20-50MB | 无 |
| Firecracker/microVM | 强 | ~125ms（快照 ~28ms） | <5% | ~5MB + Guest OS | 无 |
| 完整 VM | 最强 | 2-18s | 3-5% CPU, 10-15% 磁盘 | 256MB+ | 无 |
| V8 Isolates | 中 | <5ms | <5% | 1-3MB | JS/Wasm |
| WebAssembly | 中 | <1ms | 5-25% | <5MB | 需编译目标 |

表里有个有意思的地方：隔离强度和冷启动时间并不总是正相关。Firecracker 砍掉了 BIOS、ACPI、PCI 总线模拟这些历史包袱，做到了 VM 级别的隔离但只要 125ms 启动。而 gVisor 隔离比它弱，IO 密集场景下的运行时代价反而更高。选型的时候不能只看一个维度。

## 性能影响到底多大

我自己最关心的其实就一件事：加了沙箱之后，到底会慢多少？一个 AI Agent 每次对话要执行 3-5 次代码，每次几百毫秒到几秒，额外开销大概落在哪？

### 冷启动开销

冷启动是最容易被用户感知到的。用户说"帮我跑一下这段代码"，结果等 10 秒才开始跑，体验就已经变差了。

| 方案 | 冷启动 | 对用户的感知 |
|------|--------|------------|
| 语言级/bubblewrap | <100ms | 无感 |
| 容器（镜像预热） | 500ms-1s | 可接受 |
| 容器（未预热，需拉镜像） | 10s-60s | 不可接受，首次体验很差 |
| gVisor | ~200ms | 无感 |
| Firecracker | ~125ms（快照 ~28ms） | 无感 |
| 完整 VM | 2-18s | 仅适合长期使用场景 |

容器的冷启动慢，主要慢在拉镜像和准备环境，容器本身启动其实不算慢。所以实际部署时，提前拉好基础镜像几乎是基本操作。Firecracker 如果用快照恢复，28ms，这个数字已经接近无感。

### 运行时开销

代码跑起来之后的开销：

| 方案 | CPU 密集型（如矩阵运算） | IO 密集型（如文件读写） | 网络密集型 |
|------|----------------------|---------------------|-----------|
| 容器 | <1% | 10-30%（overlay2 写放大） | <2% |
| gVisor | <3%（生产典型） | 15-30%（微基准可达数十倍） | 20-40% |
| Firecracker | <5% | <5% | <5% |

至少在我关心的 Agent 执行代码场景里，大部分任务都不是 CPU 密集型的。更多时候是跑个 Python 脚本、执行几条 Bash、装几个依赖。

容器方案下，如果用 bind mount 而不是 overlay，IO 开销也会小很多。说起来我一开始还担心 overlay2 的写放大会不会很要命，后来想了想，Agent 场景下写的文件通常也就几 KB 到几 MB，这个量级根本感知不到。

### Session 复用

这个点是我后来才慢慢意识到的：如果 Agent 一次对话要执行多次代码，每次都新建一个沙箱环境，开销会被放大。更实用的做法往往是 session 级复用，一次对话共享一个沙箱，多次执行在同一个环境里进行。

| 模式 | 5 次执行的总开销 | 优势 | 劣势 |
|------|----------------|------|------|
| 每次新建 | 5 × 冷启动 + 执行时间 | 每次干净环境 | 慢，依赖不缓存 |
| Session 复用 | 1 × 冷启动 + 5 × 执行时间 | 快，依赖可缓存 | 前一次执行可能影响后一次 |

差别会很明显。第一次执行 `pip install numpy` 之后，后续执行直接就能 `import numpy`，不用重新安装。我在本地试的时候，同一个容器里第二次跑脚本比第一次快了好几倍，主要就是省了装依赖的时间。

## 沙箱的硬件成本

还有一个问题也得聊：沙箱本身到底有多重？如果加了沙箱之后运行成本高到没法默认开启，那也不行。

有个容易踩的坑：镜像体积和运行时内存是两码事。镜像标签写着 `110MB`，不代表运行时就要占 110MB RAM。沙箱的真实硬件成本，更多取决于容器里实际跑了什么进程。

按 2026-03-21 的一组本机实测，宿主环境是 Linux + Docker + cgroup v2，基础镜像用的是和目标场景很接近的 `python:3.12.7-slim`，结果大致是：

- 只启动一个长期存活的 `sleep infinity` 容器，`docker stats` 稳态内存约 `916 KiB`
- 启动一个空闲 Python 进程 `python -c 'import time; time.sleep(3600)'`，稳态内存约 `5.199 MiB`
- 启动一个更接近"裸解释器"的 `python -S -c 'import time; time.sleep(3600)'`，稳态内存约 `4.578 MiB`

这组数字至少说明两件事：

- "容器壳"本身的额外内存成本非常低，量级接近 `1 MiB`
- 一个空闲 Python 沙箱的基础运行成本通常只有几 MiB，不应该按镜像体积去脑补成上百 MiB 的常驻内存

所以沙箱值不值得引入，该看的是几个更实际的问题：

- 沙箱里跑的是 `sleep`、空闲 Python、轻量 MCP，还是带数据库驱动、向量库、浏览器 SDK 的重进程
- session 能不能按 request 复用，而不是每个 tool call 都新起一个环境
- 空闲 session 有没有明确回收策略，避免长时间挂着无业务价值的进程

从资源规划角度，我现在更倾向于这样估算：

- 只执行轻量 Bash / Python 命令时，沙箱机制本身通常不是主要成本项
- 轻量 MCP server 的预算，更该按"解释器 + 依赖导入"估，而不是按镜像大小估
- 浏览器、数据库、向量检索这类重依赖场景，即使不做沙箱，进程本身也已经是主要成本，沙箱更多只是额外加了一层相对可控的隔离管理开销

所以我现在觉得沙箱在工程上是成立的。隔离成本通常远小于业务进程本身，而且可控。换来的是宿主机保护和统一的执行环境，后续接 MCP 或其他插件时也有一致的底座。

## 怎么选

> 这部分选型建议，主要基于我自己实际用过的技术（Docker、Podman、nsjail、bubblewrap）和查到的资料。gVisor、Firecracker、V8 Isolates、WebAssembly 这些，我没有生产环境的实战经验，下面的判断可能不够准确。

如果让我现在做一个很粗的选型，我大概会先看信任级别和启动频率。

自己的代码跑测试，我大概不会专门上沙箱。半可信插件，容器通常就够了。完全不可信的代码，比如用户上传、模型生成，我会把容器当基线；安全要求再高，再去看 gVisor 或 microVM。

启动频率也很关键。每次请求都要现起一个环境的话，冷启动就必须快，容器预热或 Firecracker 会更合适。长期驻留的服务倒没那么敏感。边缘节点这类场景，则更容易走向 V8 Isolate 或 Wasm。

然后是环境复杂度。只跑简单脚本，bubblewrap 可能就够了。要安装依赖、管理文件、复用环境，我会更偏向容器，镜像和缓存生态都成熟。要完整 OS 体验，那还是 VM。

还有一个容易被低估的是运维。gVisor 和 Firecracker 的隔离更强，但运维门槛也更高，调试更难，兼容性问题更多，社区资料也少。容器生态最成熟，真遇到问题，至少能搜到一堆现成经验。

以我目前的理解，对于大多数需要在服务端执行不可信代码的场景，容器大概还是最实际的起步方案。隔离不够，再往上叠 gVisor；性能不够，再做 session 复用和镜像预热。至少这条路线现在看起来最稳妥。

## 业务能不能接受

我自己最开始也担心这个问题：引入沙箱之后，业务体验和成本到底会不会一下子变差？

### 延迟影响

以一个 AI Agent 执行代码的场景为例：

- 不使用沙箱：`subprocess.run("python script.py")` → 开销约 10-50ms（进程创建）
- 使用容器沙箱（session 复用）：首次 500ms-1s（创建容器），后续每次 `docker exec` 约 50-100ms
- 使用 Firecracker：首次 125ms，快照恢复 28ms，后续执行接近裸机

首次调用多出来的通常是几百毫秒到 1 秒。放在 Agent 场景里，用户本来就在等模型思考，大模型推理本身就要几秒到十几秒，所以这点额外延迟通常还在可接受范围内。

### 资源成本

容器方案下，每个沙箱 session 额外占用：

- 内存：轻量容器壳本身接近 `1 MiB`，空闲 Python 沙箱通常是几 MiB 起步；真正拉高占用的往往是导入的依赖和常驻进程
- CPU：空闲 session 的 CPU 消耗通常接近零
- 磁盘：镜像层共享，每个容器的独有写层通常 <100MB

如果同时跑 100 个 session，而且每个 session 都只是轻量 Bash / Python 环境，额外内存通常也就是几百 MiB 到 1GB 左右。真正会把预算拉高的，不是"有 100 个容器"这件事本身，而是这 100 个容器里到底各自导入了什么依赖、挂了什么服务。

### 和不做沙箱比

后来我慢慢觉得，更应该问的是另一边：不做沙箱的风险代价到底有多大？不可信代码逃逸可以拿走整台机器，一个 fork bomb 能把服务拖垮，数据泄露的合规代价更不用说。这种事出一次就够受的。沙箱带来的几百毫秒延迟和几十 MB 内存，跟这些比起来也就没什么了。

## 还没想清楚的

写到这里，有几个点我自己也在想：

Wasm 会不会成为通用沙箱的最终形态？它的安全模型确实漂亮——内存安全、能力安全、语言无关。但现实是 Python 在 Wasm 上跑得很勉强，依赖管理更是一团糟。AI Agent 执行 Python 是最高频的场景，Wasm 在这里还不成熟。也许等 WASI 和 Component Model 再迭代几个版本？

安全边界应该在哪一层？容器共享宿主机内核，如果内核有 0-day，理论上可以逃逸。但现实中，容器逃逸的已知案例绝大多数需要特权容器 + 宿主机配置失误。对于不是在做公有云多租户的场景，rootless 容器加上合理的 seccomp profile 够不够？我觉得大概率是够的，但这话不敢说得太死。

这些问题我现在也没有标准答案。先用容器把隔离做起来，再一点点看哪里不够，我觉得已经比什么都不做好很多了。
