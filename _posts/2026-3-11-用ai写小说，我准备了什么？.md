我想让 AI 帮我写一部长篇小说。网上的教程说：写好背景设定、人物设定、大纲、明暗线，然后让 AI 逐章输出就行了。我照做了，结果让我大失所望。

---

## V1：按教程来，准备了一切"该准备的"

我的准备工作：

- **世界观设定**：几千字的背景文档，地理、政治、魔法体系
- **人物设定**：主要角色的性格、动机、关系网
- **大纲**：32 章的章节大纲，每章的关键事件
- **明暗线**：主线剧情 + 两条暗线的节点标注

看起来挺齐全的。然后我开始让 AI 逐章写，每章要求至少 4000 字。

### 翻车现场

**问题一：AI 数不了数。** 我要求 4000 字，AI 写了 2000 字告诉我"以上大约 5000 字"。大模型的 tokenizer 和人类的"字数"是两回事，它对自己输出了多少内容几乎没有准确感知。

**问题二：上下文遗忘。** 到第 10 章左右，AI 已经记不住前面的细节了。角色性格开始漂移，前面埋的伏笔被忘得一干二净，开始"天花乱坠"地自由发挥。即使是 200K token 的窗口，32 章小说的全部细节也塞不进去。

**问题三：模板化写作，这是最致命的。** AI 的输出有一套肉眼可见的"模板"：

| 位置 | AI 的惯性行为 |
|------|-------------|
| 每章开头 | 必定回顾上一章情节，"上回说到……" |
| 每章结尾 | 要么高视角总结，要么悬念钩子："xx 不知道的是，一个巨大的阴谋笼罩了整片大陆" |
| 情绪描写 | 车轱辘表达反复用："他浑身一抖""他回头，笑得灿烂""她的眼眶微微泛红" |
| 转折处 | "然而""殊不知""命运的齿轮开始转动" |

这些表达单独出现一次没问题。但当它在 32 章里反复出现，读者一看就知道——这是 AI 写的。

**问题四：人物立不住。** 大纲规定了角色 A 是冷静理性的，但到了具体场景里，AI 让他跟其他角色一样"浑身一抖""眼眶泛红"。性格描写停留在标签层面，没有落到行为和语言的差异上。

### V1 的根本问题

回头看，V1 的失败不在于准备得不够多，而在于我对 AI 写作能力的预期有误。大纲和设定解决的是"写什么"的问题，但 AI 的短板在于"怎么写"——它在训练中见过太多"安全"的表达方式，倾向于用最高概率的模板来填充内容。

## V2：造工具，用程序检测 AIGC 痕迹

既然 AI 自己发现不了问题，那让程序来发现。

核心思路：**把 AI 不擅长的"检测"工作交给程序，AI 只负责"创作"和"润色"。**

### 方法一：N-gram 分析

提取文本中连续 n 个字/词的序列，统计出现频率。纯程序化，O(n) 时间复杂度，不需要 AI 参与。

能做什么：
- 跨章节检测重复短语
- 发现模板化表达（如"想起……心里"出现了 8 次）
- 找到硬编码正则遗漏的模式

```python
def find_repeated_ngrams(texts, n=10, min_count=3):
    all_ngrams = Counter()
    for text in texts:
        ngrams = extract_ngrams(text, n)
        all_ngrams.update(set(ngrams))  # 每篇去重，统计跨文档频率
    return {k: v for k, v in all_ngrams.items() if v >= min_count}
```

### 方法二：TF-IDF 分析

TF（词频）× IDF（逆文档频率）。每章当一个文档，IDF 惩罚在多数章节中都出现的词。角色名几乎每章都有，IDF 极低，TF-IDF 得分自然被压下去——这是合理的，因为它本来就不是需要关注的重复。但"命运的齿轮"如果在 32 章里出现了 20 次，它的 IDF 同样很低，TF-IDF 也会把它标出来。再结合词性过滤（专有名词放行，动词/形容词短语报警），就能把"正常的重复"和"模板化的重复"区分开。

### 方法三：余弦相似度

把段落转成向量，计算向量夹角：

$$\cos(\theta) = \frac{A \cdot B}{\|A\| \times \|B\|}$$

用来检测结构性重复——比如每章结尾是不是都在用相似的句式做总结。

### 方法四：词汇多样性分析

两个关键指标：
- **TTR（Type-Token Ratio）**：独特词数 / 总词数。越低说明用词越贫乏
- **Hapax Ratio**：只出现一次的词 / 独特词数。越低说明表达越单调

拿新章节和前 20 章的均值做对比，偏差超过 10% 就报警。

### 方法五：向量数据库 + 语义检索

用 Embedding 模型把文本转成向量，存进 DuckDB 的 VSS 扩展。比字面匹配更强——"他感到一阵恐惧"和"恐惧像蛇一样爬上他的脊背"字面不同，但语义相似度很高。

### 数据库设计

核心就两张表——`paragraphs` 做细粒度索引，`ngrams` 做跨章节统计：

```sql
CREATE TABLE paragraphs (
    id INTEGER PRIMARY KEY,
    chapter_id INTEGER,
    para_index INTEGER,
    text TEXT,
    word_count INTEGER
);

CREATE TABLE ngrams (
    id INTEGER PRIMARY KEY,
    ngram VARCHAR,
    n INTEGER,
    count INTEGER,
    chapter_count INTEGER  -- 出现在多少个章节
);
```

风格指标（TTR、平均句长等）和向量表（embedding）也各一张，结构比较直白，不展开了。

### 实时检测流程

```python
def realtime_check(new_paragraph):
    """每写完一段就检测"""
    issues = []

    # 1. 与历史段落的语义相似度
    similar = find_similar_in_db(new_paragraph, threshold=0.8)
    if similar:
        issues.append({
            'type': 'similar_paragraph',
            'similar_to': similar,
            'suggestion': '与第{}章第{}段相似，建议改写'
        })

    # 2. 是否命中高频 N-gram
    ngrams = extract_ngrams(new_paragraph, n=10)
    repeated = check_ngrams_in_db(ngrams)
    if repeated:
        issues.append({
            'type': 'repeated_ngram',
            'ngrams': repeated,
            'suggestion': '检测到模板化表达'
        })

    # 3. 词汇多样性是否偏低
    ttr = compute_ttr(new_paragraph)
    if ttr < reference_ttr * 0.9:
        issues.append({
            'type': 'low_diversity',
            'ttr': ttr,
            'suggestion': '词汇多样性低于参考值'
        })

    return issues
```

### V2 的效果

| 指标 | V1（纯 AI） | V2（程序检测） |
|------|------------|--------------|
| 模板表达检测 | 靠 3 条硬编码正则 | N-gram 统计，自动发现 |
| 重复段落检测 | 字符串相似度 | 余弦相似度 + 语义向量 |
| 风格检测 | 无 | TTR / Hapax 量化 |
| 检测时间 | AI 读全文，慢 | 程序秒级完成 |
| Token 消耗 | 高（全文读取） | 低（只传问题片段） |

确实好了一些。模板化表达能被批量发现，跨章节重复能被自动标记，每章结尾的"高视角总结"被余弦相似度精准命中。

**但是，输出仍然不像人写的。**

## V2 的瓶颈：去掉 AIGC 痕迹 ≠ 写得像人

这是我在 V2 之后想明白的最关键的一个点。

统计方法能检测**表面特征**，但 AI 写作不像人的根源不在表面。打个比方：N-gram 能发现"他浑身一抖"出现了 12 次，TTR 能发现某章词汇贫乏，余弦相似度能标记结尾雷同——然后呢？把"他浑身一抖"替换成"一股寒意从脚底窜上头顶"，问题就解决了吗？

没有。因为问题从来不是**用了什么词**，而是**为什么用这个词**。

### 人和 AI 写作的差距在哪

人类作家写"他浑身一抖"，是因为在这个具体场景下他觉得这个表达最准确。AI 写"他浑身一抖"，是因为在训练数据里，"恐惧"后面接这个表达的概率最高。一个是精确匹配，一个是统计平均。统计平均的结果就是——永远不出错，永远平庸。

但这还不是最要命的。更要命的是 AI 不懂**不写什么**。好的写作有大量留白。一个冷静理性的角色面对冲击，可能只是"他沉默了三秒，把茶杯放下了"——没有浑身一抖，没有心如刀绞。正是因为不写这些，读者才感受到这个角色的克制。AI 做不到这种减法，它的本能是把每个情绪都写满、写透。

还有节奏。人类作家知道紧张场景之后要接一段舒缓的，对话密集之后来一段环境描写让读者喘口气，高潮之前要有铺垫让情绪慢慢积累。AI 写出来的东西是匀速的——每一段描写密度差不多，情绪强度差不多，句子长度差不多。这种"均匀感"本身就是最明显的 AIGC 特征，偏偏它不是任何统计指标能抓到的。

最后是角色的"语言指纹"。现实中每个人说话都不一样，有人话多有人话少，有人爱反问有人爱省略，有人口头禅是"算了"有人是"行吧"。AI 笔下的角色共享同一个语言模型，对话口吻都一样，只是内容不同。我让朋友盲读 V2 修复后的章节，他们还是一眼看出来是 AI 写的——给出的理由已经不是"用词重复"了，而是"所有人说话的感觉都一样""读起来太顺了，太平了"。

### 统计检测能做什么，不能做什么

| 能做 | 不能做 |
|------|-------|
| 发现重复用词 | 判断一段描写是否有张力 |
| 标记模板化句式 | 评估叙事节奏是否合理 |
| 量化词汇多样性 | 分辨角色语言是否有个性 |
| 检测段落结构雷同 | 理解留白和减法的艺术 |

**V2 解决了"检测"问题，但没解决"生成"问题。** 程序能告诉你哪里有毛病，但不能告诉 AI 怎么写得更好。

## V3 方向：从"事后修补"转向"事前引导"

V1 是让 AI 自由写，V2 是写完之后检测修补。V3 的思路应该反过来——**在 AI 动笔之前就约束它的写法**。

### 思路一：场景级控制，而不是章节级

V1 的问题之一是控制粒度太粗。"请写第 7 章"这个指令给了 AI 太大的自由度。改成按场景写：

```
场景：酒馆密谈
参与角色：A（冷静，话少），B（急躁，口头禅"你他妈在开玩笑"）
情绪曲线：平静 → 紧张 → 爆发
节奏要求：前半段对话为主，短句；后半段 A 的内心独白，长句
字数：800-1000
禁止使用：浑身一抖、命运的齿轮、殊不知
```

粒度越细，AI 胡来的空间越小。

### 思路二：角色语音卡

不是只告诉 AI "角色 A 性格冷静"，而是给出语言层面的具体约束：

```yaml
角色A:
  说话风格: 陈述句为主，极少用感叹号
  常用词: "所以""也就是说""无所谓"
  禁用词: "天哪""怎么会""不可能"
  句子长度: 偏短，10字以内居多
  情绪表达: 内敛，用动作暗示而非直接描写
  示例对话:
    - "知道了。" （同意时）
    - "你说完了？" （不耐烦时）
    - 沉默，转身走了。（愤怒时）
```

### 思路三：风格锚定（Few-shot）

与其告诉 AI "不要写得像 AI"，不如直接给它看"应该像什么"。从已经写好的、质量满意的章节中提取片段作为风格样本：

```
参考风格（取自第3章）：
"雨停了。街上的水洼映着歪歪扭扭的灯光，像是谁把一整条街的霓虹灯揉碎了
丢进去。李昭踩过去，鞋袜湿透，他没在意。他在想刚才老周最后那句话——
'你以为你在选择，其实你早就没得选了。'"

请用类似的风格写下一个场景。
```

### 思路四：叙事节奏曲线

在大纲层面为每一章定义情绪和节奏曲线，不只是"发生什么事"，而是"读者在读这一章时的体验应该是什么样的"：

```
第12章 节奏设计：
  开头：日常场景，节奏慢，大量环境描写（让读者放松）
  中段：对话推进，信息密度逐渐增大（制造不安）
  转折点：一个细节揭示（短句，留白，不解释）
  结尾：角色沉默，场景描写收束（情绪留给读者）

  不要在结尾做任何总结性叙述。
  不要用"xx不知道的是……"这种上帝视角钩子。
```

### 思路五：程序辅助的实时引导

把 V2 的检测系统从"事后检测"改为"实时引导"——AI 每写一段，程序就反馈：

```
┌─────────────────────────────────────────┐
│  AI 正在写第12章第3段...                  │
│                                           │
│  [实时提示]                               │
│  ⚠ "他浑身一抖" 已在第3、7、9章出现      │
│  ⚠ 当前段落与第8章第5段相似度 0.82       │
│  ⚠ 本章 TTR 低于均值 15%                 │
│  ℹ 角色A在本章说了12句话，均为陈述句     │
│    → 参考语音卡，适当加入他的口头禅       │
│  ℹ 参考：第3章有类似场景的优质描写       │
└─────────────────────────────────────────┘
```

这五个方向里，我打算先从**场景级控制**和**角色语音卡**入手——它们改动最小，能直接塞进现有流程的 prompt 里，不需要额外的工程开发。风格锚定也可以同步做，本质上就是在 prompt 里多贴一段参考文本。叙事节奏曲线和实时引导涉及更多工程量，放后面。

实现路线：

1. **基础设施**（已完成部分）：DuckDB 建表、章节段落索引、N-gram 统计
2. **程序化检测**（V2，进行中）：N-gram 重复、余弦相似度、风格偏差
3. **事前引导**（V3，下一步）：场景级 prompt 模板、角色语音卡、风格锚定
4. **实时集成**（V3+）：写作时实时检测 + 引导反馈

## 写在最后

回头看这个过程，每一轮的瓶颈其实都在指向同一个事实：**AI 写作的上限不取决于模型本身，而取决于你给它的约束有多精确。**

V1 失败，约束太粗——一份大纲管不住具体怎么写。V2 改善了，但只是事后修补——发现问题不等于解决问题。V3 想把约束细化到场景、角色语言、叙事节奏，在 AI 动笔前就框住它。

但说实话，我不确定 V3 就够了。写作里最核心的东西——什么时候该来一个出人意料的比喻，什么时候该让角色说出不符合人设但完全合理的话，什么时候该故意打破节奏——这些本质上是"反模式"的，而 AI 就是一个模式机器。

所以我现在的想法是：别指望 AI 从头到尾写出一部能看的小说。工具链的价值不在于替代人，而在于把人从体力活里解放出来——AI 出初稿、程序做质检、人来改灵魂。如果 V3 之后还是不行，那下一步可能就不是继续优化工具链了，而是反过来，人写初稿，AI 做扩写和润色。谁当主笔谁当助手，取决于哪种分工出来的东西更像人写的。


### 3-12 更新：从"统计异常"到"写作手法诊断"

写完上面那段话之后，我发现自己其实一直在绕一个弯：V2 做的是**统计检测**——N-gram 发现重复、TF-IDF 标记高频词、余弦相似度抓结构雷同。这些方法的共同点是：它们只知道"这里不太对"，但说不出来"为什么不对"。

"他浑身一抖"出现了 12 次，N-gram 会告诉你。但"她终于学会了放下，带着希望和勇气，踏上了新的旅程"只出现一次，N-gram 不会报警——可是一个活人看到这句话，立刻就知道这是 AI 写的。

问题在哪？不是重复，而是**写法本身就有问题**。

所以我换了一个思路：不再只做统计异常检测，而是直接建一个**AI 写作坏习惯的 case 库**——把"AI 喜欢怎么写"这件事结构化、可查询、可匹配。

项目地址：[ai-writing-bad-cases-app](https://github.com/huanghuoguoguo/ai-writing-bad-cases-app)（应用层）+ [ai-writing-bad-cases](https://github.com/huanghuoguoguo/ai-writing-bad-cases)（数据层）

#### 从统计到诊断：9 个维度

V2 的检测方法是 bottom-up 的——先看到数据里有异常，再去猜为什么。这次我反过来，先从 top-down 定义了 9 个**诊断维度**，每个维度回答一个问题："AI 写出来的东西为什么像 AI？"

| 维度 | 核心问题 | 典型表现 |
|------|---------|---------|
| 结构对称性 | 句子太工整，像模板拼的 | "不是X，而是Y""没有X，没有Y，只有Z" |
| 收束冲动 | 每段都急着写成结论 | "真正重要的是""归根结底" |
| 过度显式性 | 把组织过程说出来而不是直接做 | "不绕弯子，直接说重点"（然后才开始说） |
| 元话语密度 | 不断解释自己在干什么 | "换句话说""本质上""底层逻辑" |
| 抽象替代具体 | 大量概念词，不落到场景和动作 | "认知迭代""底层思维" |
| 姿态先于内容 | 先摆态度，再补内容 | "强行升华""段尾拔高到人生意义" |
| 情绪服务化 | 像在照顾读者情绪 | "稳稳接住你""给你满满的情绪价值" |
| 连接词驱动 | 段落靠连接词推进，不靠内容 | "此外""再者""与此同时"连用 |
| 平均风格 | 全文没有作者偏置 | 每段差不多长、差不多密、差不多稳 |

这 9 个维度不是我拍脑袋想的。写了十几章小说之后，让朋友盲读，他们说"像 AI"的那些段落，我逐句拆解、分类，最后聚合出这些维度。它们和 V2 的统计指标有一个根本区别：**统计指标描述"是什么"（重复了、相似了），诊断维度回答"为什么"（因为结构太对称了、因为收束太急了）。**

#### Case 库：40 条坏习惯，3 种体裁

每条 case 不只是一个正则或关键词，而是一个完整的"诊断单元"：

```json
{
  "id": "zh.nar.growth_ending",
  "label": "成长式结尾",
  "severity": 0.86,
  "diagnostic_dimensions": ["closure_impulse", "posture_before_content"],
  "matchers": [
    {"type": "regex", "pattern": "(终于|从此)(学会了|明白了|懂得了|变得)", "weight": 0.9},
    {"type": "regex", "pattern": "(带着|怀着).{0,8}(希望|勇气|信心).{0,8}(继续|走向|迈向|踏上)", "weight": 0.95}
  ],
  "examples": ["她终于学会了放下，带着希望和勇气，踏上了新的旅程。"],
  "counter_examples": ["后来的事情她没再跟我提过。我也没问。"],
  "rewrite_hint": "结尾不必总结人物的成长。可以停在一个具体的动作或场景上。"
}
```

注意 `counter_examples`——这是我认为比 case 本身更有价值的字段。它回答的是"那应该怎么写"。上面那个成长式结尾的反例是"后来的事情她没再跟我提过。我也没问。"——没有成长声明，没有希望和勇气，就停在一个具体的动作上。这才是人写的感觉。

目前 40 条 case 覆盖了三种体裁：

| 体裁 | Case 数 | 典型问题 |
|------|---------|---------|
| 议论文 | 22 | 时代开场、三段式、强行升华、递进模板、冒号展开结论 |
| 叙事文 | 10 | 时间凝固、嘴角上扬、暖光描写、眼神套路、成长式结尾 |
| 学术文 | 8 | 空洞意义声明、模糊论据、引言脚手架、模板化结论 |

叙事文那 10 条对写小说最有用。我列几个自己反复中招的：

- **时间凝固式描写**（severity 0.88）："那一刻，时间仿佛凝固了"——AI 用这句话来跳过情感描写，因为它写不出具体的身体感受
- **嘴角微微上扬**（severity 0.83）：AI 写微笑几乎只会写嘴角，而且每次都是"微微上扬"或者"勾起一抹弧度"
- **成长式结尾**（severity 0.86）：AI 给每个故事都安排一个"她终于学会了"的结局，真人写的故事经常是暧昧的、没说完的

#### 评分算法：为什么不用均值

V2 里做相似度检测用的是阈值——超过 0.8 就报警。这次段落评分遇到了一个坑：最初用 hit confidence 的均值，结果**命中越多反而分越低**。

一条 confidence 0.93 的 hit，分数 0.93。再加一条 0.68 的，分数变成 (0.93+0.68)/2 = 0.805——降了。这不合理：一个段落同时命中"强行升华"和"综上所述"，说明问题更严重，不应该降分。

改成了补集乘积公式：

$$score = 1 - \prod_{i=1}^{n}(1 - c_i)$$

数学上和概率论里"至少发生一次"的计算一样——每加一条 hit 只会增加 score，不会降低。再叠加一个维度多样性奖励（命中的诊断维度越多，说明问题越广，额外加分）。

跑了一组对比测试：13 段 AI 风格文本 vs 12 段人类写作文本。AI 段落平均分和人类段落平均分的**分离度 ≥ 0.3**，F1 ≥ 0.7。数字不算惊艳，但考虑到这只是规则匹配（还没加语义召回），已经能用了。

#### 和 V2 的关系

这套东西不是替代 V2，而是补上 V2 缺的那块：

```
V2（统计层）                    新增（写作手法层）
─────────────                  ─────────────────
N-gram 重复检测                 模板化句式诊断
余弦相似度                      段落级 AI 风格评分
TF-IDF 高频词                   9 维度 + 40 case 匹配
TTR / Hapax                    反例 + rewrite_hint

发现"哪里不对"                  解释"为什么不对" + "可以怎么改"
```

V2 告诉你"第 8 章结尾和第 3 章结尾相似度 0.85"，新系统告诉你"这是收束冲动 + 过度显式性，建议去掉总结标签，让结论从前文细节自然生长"。

#### 下一步

这套 case 库最终的用法可能不是事后检测，而是**直接喂给 AI**。每条 case 里有 `prompt_rule` 字段（英文写的，方便直接塞 prompt），比如：

> "Do not wrap up with explicit growth statements. End on a concrete image or action."

理论上可以把匹配到的 case 的 prompt_rule 打包成一个 skill，让 AI 在写作时主动遵守，或者写完后自己 review。从"程序检测 → 人类修改"变成"程序检测 → AI 自查自纠 → 人类终审"。这个还在想，没做。

但至少现在有了一个可以量化、可以迭代的 case 库。每次发现新的 AI 坏习惯，往里面加一条 case 就行，不用改代码。
