## 从一个 map 到一个数据库

内存中的 map 可以被视为最简单的数据库：存键值对，根据键快速检索值。但内存是易失的，断电数据消失，所以需要持久化。最直接的方式是序列化到磁盘。

但顺着这个思路往下走，问题一个接一个：

- **何时持久化？** 每次操作后都序列化，延迟不可忍受；只在启停时序列化，中间断电怎么办？
- **全量还是增量？** 未修改的部分有必要重复写吗？
- **并发安全？** 多线程下的覆盖问题？操作失败了怎么回滚？（undo log）
- **持久化时阻塞吗？**（double write、disk buffer pool）
- **数据量大了怎么办？** 分片、分散存储后怎么检索和整合？分片下的索引？
- **冗余？** 像 RAID 一样做数据复制，增强安全性？

以 InnoDB 为例，它用 Disk Buffer Pool 做页面缓存（局部性原理），用 redo log 做崩溃恢复。redo log 是顺序写的环形结构，写入性能远高于 Buffer Pool 的随机换入换出——这就是一种典型的"用高性能日志弥补随机 I/O 短板"的折中方案。

从一个 map 到一个功能完备的数据库，每一步都是为了满足新的需求。而当数据不再集中于单机，分片、复制、事务又会带来新的挑战。这本书就是沿着这条路展开的。

## 第 1 章 可靠性、可扩展性、可维护性

三个核心目标。书中用推特的例子说明，混合架构往往比纯粹的某种方案更合适，关键在于将资源合理利用。数据存储可以被当成消息队列使用（Redis），消息队列则带有类似数据库的持久化保证（Kafka）。数据库更倾向纵向扩展。

## 第 2 章 数据模型与查询语言

每一层通过提供明确的数据模型来隐藏底层复杂性。关系型、文档型、图形数据库各有适用场景：

- **文档型**：数据自包含，文档间关系稀少。JSON 具有更好的局部性
- **图形数据库**：任意事物都可能与任何事物关联
- **关系型**：结构化数据，强一致性需求

一个有意思的现象：很多高性能系统都采用"粗查询定位范围 → 细查询顺序扫描"的策略。MySQL 页内扫描可能直接顺序遍历，Redis 小数据量用 ziplist/quicklist 顺序扫描。核心原因是**小数据范围内顺序扫描的缓存友好性和简单性优于复杂索引结构**，本质上还是局部性原理和性能-复杂度权衡的体现。

## 第 3 章 存储与检索

这是第一部分最核心的一章。数据库要做两件事：存起来，还给你。为此需要设计存储引擎。书中研究两类：日志结构的存储引擎和面向页面的存储引擎。

### 世界上最简单的数据库

```bash
#!/bin/bash
db_set () {
    echo "$1,$2" >> database
}
db_get () {
    grep "^$1," database | sed -e "s/^$1,//" | tail -n 1
}
```

写入用 append（最快的写操作），读取用线性查找 O(n)。数据量增大后查找变慢，所以需要索引。索引提升读性能，但降低写性能——这是贯穿所有存储引擎的核心权衡。

### 哈希索引

在内存中维护哈希表，key 是数据的 key，value 是文件偏移量。Bitcask 存储引擎就用这种方式：写入时追加到数据文件，查询时通过内存哈希表定位。

为防止文件无限增长，将数据文件分成固定大小的段，对旧段做**压缩**（compaction）——同一个 key 只保留最后一次写入的值。进程重启时需要重建哈希表，可以将哈希表也持久化到磁盘加速恢复。

哈希索引的两个缺点：**key 太多时内存放不下**；**范围查询效率低**（必须逐个查哈希表）。

### SSTable 与 LSM-Tree

SSTable（Sorted String Table）解决了范围查询的问题：每个数据文件中的 key 是排序的。

**写入流程**：数据先写入内存中的平衡树结构（memtable，如红黑树或跳表），达到阈值后将有序数据写入磁盘的 SSTable 文件。后台进程用归并排序持续合并旧的 SSTable 文件。

**查询流程**：先查 memtable → 再查最新的 SSTable → 依次向前查旧文件。不存在的 key 可以用**布隆过滤器**快速排除。内存中不需要保存所有 key 的索引，因为数据有序，只需**稀疏索引**即可定位到一个范围，再在范围内顺序扫描。

**崩溃恢复**：写 memtable 前先追加一个 WAL 文件，重启时用 WAL 重建 memtable。memtable 写入 SSTable 后，对应的 WAL 就可以删除了。

整个方案写入只涉及 memtable 和顺序追加的 WAL，写性能极高；数据有序，范围查询高效。这就是 **LSM-Tree**（Log-Structured Merge-Tree）。LevelDB、RocksDB、Cassandra、HBase 都用了 LSM-Tree。

### B-Tree

B-Tree 是目前最广泛使用的索引结构。数据存在固定大小的页（通常 4KB）中，分层组织。根页按序保存 key 和指向子页的指针，叶子页保存实际数据。分支因子通常几百，4 层 B-Tree（4KB 页，分支因子 500）可存储 256TB 数据。

添加 key 时，找到所属叶子页，空间不够则分裂成两页。写入时先写 WAL（write-ahead log），再写数据页，保证崩溃安全。

### LSM-Tree vs B-Tree

| 维度 | LSM-Tree | B-Tree |
|------|---------|--------|
| 写入 | 快（顺序写 memtable + WAL） | 较慢（WAL + 数据页两次 IO） |
| 读取 | 较慢（可能查多个 SSTable） | 快（树结构直接定位） |
| 空间利用 | 高（压缩紧凑） | 较低（页分裂导致半空页） |
| 后台开销 | 压缩进程占 IO，可能跟不上写入速度 | 无额外后台开销 |
| 适用场景 | 写密集（日志、时序数据） | 读密集（关系型数据库） |

### 二级索引

实现方式和主索引一样（B-Tree 或 LSM-Tree），区别在于数据存在哪里：

- **聚集索引**：数据直接存在索引中。InnoDB 主键索引就是这种方式
- **非聚集索引**：索引中只存数据的指针。InnoDB 二级索引存的是主键值，需要回表
- **覆盖索引**：索引中只存部分列，折中方案，避免回表

## 第 4 章 编码与演化

粗看。核心问题：应用演化时新旧数据格式要共存，需要双向兼容性——新代码能读旧数据（向后兼容，较易），旧代码能读新数据（向前兼容，较难）。

常见编码格式：JSON、XML、Protocol Buffers、Thrift、Avro。其中 Avro 通过作者模式（Writer's Schema）和读者模式（Reader's Schema）实现灵活的模式演化。通信方式：REST、RPC、消息队列。
