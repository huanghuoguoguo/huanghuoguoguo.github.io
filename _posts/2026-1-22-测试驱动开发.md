前几天写了一篇 [《测试驱动 AI Coding》]({% post_url 2026-1-16-测试驱动开发与vibe-coding %})，核心观点是：测试用例可以给 AI 划边界，至少别让 Vibe Coding 一路散掉。

写完之后总觉得差点意思。我讲的都是怎么用测试约束 AI，但测试驱动开发本身到底是怎么回事？光靠二手概念搭出来的框架，地基不太牢。

于是翻开了 Kent Beck 的《测试驱动开发》。读完才发现，我之前对 TDD 的理解确实浅了——TDD 的重点根本不在测试，而在设计。

---

## 1. 从 Money 例子说起

这本书的 Part I 用了一个多币种算术的例子，贯穿了大概十几章。问题很简单：怎么处理不同货币之间的运算？比如 5 美元 + 10 瑞士法郎，在某个汇率下应该等于 10 美元。

Kent Beck 没有先画 UML 图，没有先设计类层次结构。他写了一个测试：

```java
public void testMultiplication() {
    Dollar five = new Dollar(5);
    five.times(2);
    assertEquals(10, five.amount);
}
```

这时候 `Dollar` 类还不存在。编译都过不了。这就是"红"。

然后他做了一件让我意外的事——用最笨的方式让测试通过：

```java
class Dollar {
    int amount = 10; // 直接硬编码
    void times(int multiplier) {}
}
```

没错，`amount` 直接写死成 10。`times` 方法是空的。测试绿了。

这就是 Kent Beck 说的 **Fake It**——先骗过测试再说。

我第一反应是：这不是自欺欺人吗？我记得当时看到这页，翻回去确认了一遍，以为自己漏看了什么。但继续读下去才理解他的用意。测试一旦通过，他马上就回头重构：

```java
class Dollar {
    int amount;
    Dollar(int amount) { this.amount = amount; }
    void times(int multiplier) {
        amount *= multiplier;
    }
}
```

一个红→绿→重构的循环就这么走完了。步子非常小，小到你几乎觉得多此一举。

---

## 2. 测试怎么把设计逼出来的

后面的演进过程比第一个例子更有意思。

当 Kent Beck 要加入瑞士法郎的支持时，他没有一上来就设计一个 `Money` 基类。他先写了一个 `Franc` 类，代码跟 `Dollar` 几乎一模一样——是的，就是赤裸裸的复制粘贴。

这又让我不舒服了。任何一个程序员看到两段重复代码都会手痒。但 Kent Beck 的态度是：**先让测试通过，再消除重复**。重构是在绿灯保护下做的事，不是在红灯的时候急着做。

然后他才开始抽取公共基类 `Money`。先把 `amount` 提上去，再把 `equals` 提上去，最后把 `times` 的返回类型统一。每一步都有测试保护。中间也不是一帆风顺——`times` 到底应该返回 `void` 还是返回新对象，他来回改了一次，最终选了返回新对象（Value Object 模式），因为 side effect 让测试变得不可靠。

到了处理"5 美元 + 10 瑞士法郎"这个需求时，事情变得有趣了。直接相加行不通，因为涉及汇率。Kent Beck 引入了 `Expression` 接口和 `Bank` 类来处理汇率转换。而这个设计——不是他提前想好的，是测试一步步逼出来的。

整个过程大概是这样：

```
Dollar 类   →  复制出 Franc 类  →  提取 Money 基类
     ↓              ↓                    ↓
  测试乘法       测试乘法            测试相等性
                                         ↓
                              Money.times() 返回 Money
                                         ↓
                              需要跨币种相加 → Expression 接口
                                         ↓
                              Bank.reduce(Expression, 目标货币)
```

回头看，`Expression` 这个抽象层是被需求（跨币种加法）硬生生拽出来的。如果一开始就画架构图，大概率会设计出一套更优雅但也更笨重的东西。

## 3. 三个实用的策略

书的 Part III 讲了一堆 TDD 的模式，最让我觉得有用的是三个：

**Fake It**：先硬编码让测试过。上面的 `amount = 10` 就是例子。好处是你能立刻进入绿灯状态，心理负担变小，再从具体值反推出通用逻辑。

**Triangulation**：当你不确定怎么泛化时，再加一个测试用例。比如光有 `5 * 2 = 10` 这一个用例，`amount = 10` 就能骗过去。但加上 `5 * 3 = 15`，你就不得不写出真正的乘法逻辑了。两个点确定一条线。

**Obvious Implementation**：如果实现太简单，比如一个 getter 方法，就别 Fake 了，直接写。不是所有代码都值得走完整个仪式。

这三个策略的切换很灵活。Kent Beck 自己说，他大部分时间在 Obvious Implementation 和 Fake It 之间切换，只有在真的不确定怎么泛化的时候才用 Triangulation。

## 4. 读完再看之前写的东西

读完这本书，回头看我 [那篇关于 TDD + Vibe Coding 的文章]({% post_url 2026-1-16-测试驱动开发与vibe-coding %})，发现有个地方想简单了。

我之前一直把测试当作给 AI 的行为边界——写好测试，让 AI 去填实现，跑不过就把报错丢回去。这个工作流本身没错，但我忽略了一件事：写测试这个动作本身就在做接口设计。

Kent Beck 在书里反复强调这一点。你在写 `assertEquals(10, five.amount)` 的时候，其实已经站在调用方的视角了——`times` 该怎么调、`Dollar` 怎么构造，都在测试里定了。后来他把 `times` 从 `void` 改成返回新 `Dollar` 对象（Value Object 模式），也是测试先改、实现跟着变。

这对 AI Coding 意味着什么？如果我只是写好测试然后让 AI 补实现，我确实能得到能跑的代码。可我要是先在写测试的时候把接口来回推敲清楚，再把这个接口丢给 AI，补出来的代码质量通常会好很多。因为大量歧义在前面就已经被排掉了。

我之前把 TDD 当事后检验用。Kent Beck 讲的是事前设计。

另外，书里 Fake It 策略也让我重新理解了 AI 生成代码的过程。AI 第一次生成的代码，很多时候就是某种意义上的 Fake It——它能跑通你给的那个 case，但可能是在用某种巧合蒙混过关。你需要更多的测试用例去 triangulate，把 AI 逼到真正泛化的实现上去。

这跟之前在 [《发现规则，应用规则》]({% post_url 2026-1-17-发现规则应用规则 %}) 里写的认知飞轮是一回事——红灯就是发现问题，绿灯加重构就是应用规则。

## 5. 没想清楚的地方

读完有些东西我还是没想明白。

第一个是**系统级代码的 TDD 怎么做**。Kent Beck 的例子大多是业务逻辑，比如货币运算、测试框架，输入输出很明确，断言写起来也自然。但我平时接触的东西不太一样。比如实习期间我做过一个内部项目。它的困难不在于断言不好写，更多是接口规范经常变。那时我也尝试写测试来约束代码，但边界没先想清、依赖装配又是硬编码，测试很难隔离，反馈周期也很难压到 TDD 需要的红绿节奏。结果这些测试更像是事后补救，不太像事前把接口逼出来。

第二个是**TDD 和性能优化之间的张力**。TDD 鼓励你先写最简单的实现，再重构。但性能敏感的代码，最简单的实现和最终实现之间的距离可能非常远。一个 naive 的排序跟一个经过 SIMD 优化的排序，接口一样但内部完全不是一回事。测试能保证正确性，但没法驱动你走向高性能实现——那是 profiling、benchmark、hotspot 分析的地盘。

这两个问题我还没有答案。也许后面读到别的书，或者在项目里踩够坑了，会有新的理解。
