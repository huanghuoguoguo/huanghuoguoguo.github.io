上一篇[沙箱科普]({% post_url 2026-3-20-沙箱 %})讲了沙箱有哪些实现方式、性能代价多大。这篇聊一个具体问题：LangBot 为什么要引入沙箱，以及我们是怎么做的。

先说结论：LangBot 新增了一个叫 LangBot Box 的沙箱执行层。它给 local agent 提供了一个容器级的受限执行环境，让不可信代码不再直接跑在宿主机上。后来在这个底座上又接了 MCP stdio 的隔离运行，所有本地 stdio MCP server 也默认跑在 Box 里了。

## 问题出在哪

LangBot 对扩展代码的信任其实分三层：

1. **LangBot 核心代码**：项目维护者自己写的，默认可信。
2. **插件**：有审核、有上架流程，算半可信。
3. **本地可执行对象**：默认不可信。

第三层是这次要解决的问题。它包括的东西比想象中多：

- local agent 在工具调用阶段让模型生成 Bash / Python 代码，然后执行
- 模型把代码写入工作目录后再运行
- 执行过程中安装依赖、下载数据、跑测试

这些代码以前怎么跑？`subprocess` 开一个子进程，直接在宿主机上执行。

这意味着：

- 代码继承了宿主机的文件系统和环境变量可见性——`.env` 文件、数据库凭据、SSH 密钥，全看得到
- 没有资源限制——一个 `while True` 或者 fork bomb 就能把机器搞挂
- 网络不受控——代码可以往任意地址发请求
- LangBot 主进程和不可信代码跑在同一个平面上，出了事很难止损

说直白点：以前的执行模型约等于让用户（通过模型）在你的服务器上拿到了一个没有任何限制的 shell。

如果 LangBot 只是一个个人开发者的玩具，这问题不大——反正出事了也只影响自己。但 LangBot 要面对的场景是：多用户、部署在服务端、插件生态越来越开放。这时候"裸执行"就是一个定时炸弹。

## 为什么不够用

在沙箱之前，我们也能做一些事情：

| 能做的 | 实际效果 |
|--------|---------|
| 超时控制 | 能杀掉跑太久的进程，但不阻止恶意代码在超时前干坏事 |
| 工作目录切换 | 换了个目录，但 `../` 就能跳出去 |
| 环境变量裁剪 | 不传就看不到，但进程可以自己读 `/proc/self/environ` |
| 黑名单命令过滤 | 防君子不防小人，`eval`、动态导入、编码绕过方式太多 |

这些措施有一个共同的问题：它们是在进程内部做限制，被执行的代码仍然是宿主机上的一个普通进程，拥有宿主机进程该有的一切能力。限制和隔离是两回事。

要做到真正的隔离，必须在操作系统层面建立边界。这就是沙箱。

---

## 为什么不做语言级沙箱

一个自然的想法是：既然 local agent 主要跑 Python，能不能用 RestrictedPython 之类的方案，在 Python 层面做限制？

不行。原因在[上一篇]({% post_url 2026-3-20-沙箱 %})里讲过了，这里简单说：

- 只能限制一种语言。模型生成 Bash 脚本的时候就失效了
- 覆盖不了子进程、动态导入、C 扩展、系统调用
- 不构成真正的宿主隔离边界
- 已知逃逸方式太多，社区早就不推荐用于安全目的

LangBot Box 要解决的是进程和宿主环境的隔离问题，不是限制 Python 代码能调哪些函数。

## 为什么选容器

在各种沙箱方案里，我们选了容器（rootless Podman 优先，Docker 兼容）作为 V1 的底层。

选型逻辑：

1. 隔离够用。容器提供了文件系统、网络、PID、资源的全面隔离。配合 rootless 模式和 seccomp profile，隔离强度足以应对不可信代码执行场景。LangBot 不是公有云多租户平台，不需要内核级隔离，不需要上 Firecracker。

2. 生态成熟。这是最重要的理由。Docker/Podman 的镜像管理、依赖缓存、生命周期 API 都是现成的。agent 要跑 Python 就用 Python 镜像，要跑 Node 就用 Node 镜像，要装 numpy 就在容器里 pip install——这些场景容器生态已经打磨了十年。

3. 后续扩展自然。不管是 MCP server、Skill 还是用户的自定义项目，最终都要映射成"在一个受控环境里启动进程"。容器对这些场景的适配都很自然。如果 V1 选了 bubblewrap 或 nsjail，后续要做镜像管理、依赖缓存、环境复用的时候，还得把这些轮子重新造一遍。

4. 性能可接受。上一篇的数据：容器 CPU 开销不到 1%，磁盘 IO 用 bind mount 接近无损，冷启动（镜像预热后）500ms-1s。agent 的一次对话里，模型推理本身就要好几秒，沙箱带来的几百毫秒完全可以接受。

不选的方案：

- bubblewrap / nsjail：隔离够，但缺少镜像管理和环境封装能力。适合隔离单条命令，不适合管理持续的执行环境
- gVisor / Kata：隔离更强，但 V1 过重，引入的运维复杂度和性能开销不划算
- Firecracker / VM：同上，过重
- Wasm：Python Wasm 支持不成熟，agent 的首要场景就是跑 Python

## 设计思路

具体的模块设计和数据模型不在这里展开，只说设计层面的几个关键决策。

### 一个独立的执行服务

引入 Box 之后，LangBot 的执行架构变成三层：

```
LangBot Core     ── 核心业务、调度、API
Plugin Runtime   ── 插件生态，保持不变
Box Runtime      ── 新的不可信代码执行平面（独立进程）
```

Box Runtime 是一个独立的服务进程，架构上参考了现有的 Plugin Runtime——LangBot 负责业务控制面与调用编排，Runtime 负责具体运行面，两者通过 RPC 通信。差别在于 Plugin Runtime 管的是插件组件与插件生命周期，Box Runtime 管的是沙箱 session、命令执行、资源限制与回收。

其实一开始 Box Runtime 不是独立服务，是直接嵌在 LangBot 主进程里的。后来发现 LangBot 自己在容器里部署的时候，主进程要拿到宿主机的容器引擎控制权，路径视角和权限都很别扭，才拆出来做成独立进程。

LangBot 不直接持有容器控制权。职责划分是：LangBot 决定"执行什么"，Box Runtime 决定"在哪里、以什么限制执行"，底层 backend 决定"如何真正创建沙箱"。这样做的好处是 LangBot 在 Docker / Kubernetes 中部署时，不必直接暴露宿主机容器引擎 socket——这件事交给 Box Runtime 所在的环境来处理。

**LangBot 侧与 Box Runtime 侧的边界**

为了把"执行控制面"和"运行面"拆开，LangBot 引入了 `BoxRuntimeClient` 抽象：

- `BoxRuntimeClient`：抽象接口，定义了与 Runtime 通信的契约
- `ActionRPCBoxClient`：基于 Action RPC 的具体实现（复用 Plugin Runtime 的通信框架）
- `BoxRuntimeConnector`：负责解析配置、决定连接方式、管理本地 runtime 生命周期

LangBot 侧保留：
- `BoxSpec` / `BoxProfile` / `BoxExecutionResult` 等模型定义
- profile 合并与请求级参数校验
- `exec` 工具、审批/提权策略与业务胶水层
- 输出截断与错误翻译

Box Runtime 侧负责：
- session registry 与生命周期管理
- backend 抽象与实现（Podman/Docker）
- 状态跟踪、失败回收、空闲回收
- 暴露 Action RPC 接口

这样 Box Runtime 可以在不改动 LangBot 主进程的前提下替换 backend，也可以像 plugin runtime 一样独立部署、重启、扩容。

### 通过内置工具暴露能力

agent 不是直接调用容器 API，而是通过一个内置工具来执行代码。这个工具对模型来说，和其他工具（搜索、查数据库）没有区别——它在工具列表里平级注册，模型决定调用时，系统把请求路由到 Box Runtime 而不是宿主机。

这里有个关键的抽象层次问题：**对 agent 暴露的是能力语义，不是基础设施术语**。

LangBot 最终采用的 agent-facing 工具名是 `exec`，而不是 `sandbox_exec`。原因是 agent 不需要知道 Box Runtime 是什么，不需要知道 mount 和 host_path 是什么，也不需要知道 session 是如何创建和复用的。agent 只需要知道：

- 有一个执行能力叫 `exec`
- 默认在受控工作区里运行
- 可能受工具策略、网络策略、审批策略限制
- 某些场景下可请求提权，但未必会被允许

这和 OpenClaw 的 `exec` / `system.run`、OpenCode 的 `bash` 是一类抽象——agent 看到的是"能力"和"限制"，而不是"沙箱"、"容器"、"runtime"这些基础设施名词。

执行流程：

```
agent 决定执行代码
    → 调用 exec 工具
    → LangBot 构造 BoxSpec，应用 profile 约束
    → 通过 Box Runtime Client 发 RPC 给独立 Box Runtime
    → Box Runtime 在沙箱内执行
    → 返回 stdout / stderr / 退出码
    → agent 基于结果继续决策
```

LangBot 在工具调用上下文中决定：当前 query 所属 session、当前可见工作区、默认工作目录。这些都不暴露给 agent。

### Session 复用与基础设施隐藏

一次 agent 请求里，模型可能调多次 `exec`——先写个文件，再执行，再看输出，再改，再执行。如果每次都新建一个容器，体验会很差（重复安装依赖、文件丢失）。

V1 的做法是 request 级 session 复用：同一次 agent 请求共享一个沙箱 session，多次执行在同一个工作目录里进行。请求结束或空闲超时后回收。不同请求之间默认不共享——避免环境污染。

**关键点是：session 管理对 agent 完全不可见**。agent 看不到 `session_id`，看不到 `host_path`，看不到 `mount_path`，也不感知 profile。这些基础设施术语只在 LangBot 内部和 Box Runtime 之间传递。agent 看到的只是：
- 调用 `exec` 执行命令
- 命令在同一个工作目录里保留了状态
- 工作目录在请求结束后被清理

### 默认拒绝，三层策略

两个原则不能妥协：

- **默认拒绝**：文件系统只读（除了指定的工作目录）、网络默认关闭、环境变量不透传、资源有上限。要开放什么，显式配置
- **fail closed**：如果容器引擎不可用，拒绝执行，不静默回退到宿主机。这是安全边界要求，不是体验优化选项

为了尽量与 OpenClaw / OpenCode 的成熟抽象靠齐，LangBot 把策略明确分成了三层：

**1. Sandbox Policy（执行位置策略）**

决定工具默认在哪里运行，以及默认能看到哪些工作区。至少支持：
- `off`：关闭隔离，直接在宿主执行
- `all`：所有 agent 执行都先进入 Box

**2. Tool Policy（工具可用策略）**

决定当前 agent / pipeline / session 允许调用哪些工具。这是硬边界——tool deny 优先于 tool allow，`elevated` 也不能绕过工具禁用。

**3. Approval / Elevated Policy（提权策略）**

只影响单次执行请求，不改变工具是否存在。`elevated` 表示"从默认受限执行位置临时切换到更高权限执行位置"，它不授予额外工具，也不覆盖 tool policy。

这三层不要混成一个概念。Sandbox policy 回答"在哪执行"，tool policy 回答"能用什么"，approval policy 回答"这次能不能破格"。

对 `exec` 而言，如果容器引擎不可用（Docker 没装、daemon 没起来），拒绝执行，不静默回退到宿主机。MCP stdio 的策略稍有不同：Box Runtime 可用时默认走 Box，不可用时降级到宿主机直跑。两者的差异在于信任级别——MCP server 是管理员配置的可信服务，LLM 生成的代码则完全不可信。

## V1 做了什么，没做什么

已完成：

- Linux 平台，rootless Podman 优先，Docker 兼容
- local agent 内置 `exec` 工具
- 通用命令执行（Bash / Python / 任意命令）
- Request 级 session 生命周期管理
- 基础的资源限制、网络控制、日志采集、错误转换
- stdio MCP server 默认走 Box（runtime 可用时自动隔离，不需要额外配置）
- 完整的 MCP session 生命周期管理（创建、重试、shutdown、失败均有 cleanup）
- 细粒度错误分层（7 个阶段，可精确定位哪一步出了问题）

没做：

- 插件沙箱化（插件是第二层，暂时不动）
- Skill 的标准化和生态导入
- sse / http MCP 的沙箱化（当前只有 stdio 走 Box）
- Sub-agent 编排
- 跨平台（Windows / macOS）
- 细粒度网络策略（比如只允许访问 pypi.org）

最初的计划里，MCP 接入是排在 V1 之后的。但沙箱底座做完之后发现，stdio MCP 映射到 Box 上很自然——一个 MCP server 就是一个长期 session 加一个 managed process，底座已经有了 session 管理和进程托管的能力，适配起来并不需要改底层抽象。所以就顺手做了。回头看，当初担心"先做 MCP 会把 Box 做成协议专用层"，这个担心是对的——正因为先做了通用底座，MCP 接入才没有污染底层的 session 和执行模型。

## 已经解决的和还没想清楚的

**容器嵌套**已经有了落地方案。LangBot 自己可能就部署在容器里，这时候 Box Runtime 要创建沙箱容器，就涉及容器里创建容器的问题。最终的做法是让 Box Runtime 独立部署在最接近容器引擎的位置，通过宿主引擎创建同级容器，而不是在 LangBot 容器内部嵌套。Docker Compose 下 `langbot` 和 `langbot_box_runtime` 是两个 service，后者持有宿主机容器引擎 socket；Kubernetes 下两者是两个 Deployment。本地开发时，LangBot 会自动托管启动一个本地的 Box Runtime 子进程，省去手动配置。不推荐 Docker-in-Docker 特权模式。

**Profile 粒度**也定下来了。V1 内置了四个 profile：`default`（默认离线、资源受限、rootfs 只读）、`offline_readonly`（更严格的低资源只读档位）、`network_basic`（允许外网但仍受资源和挂载限制）、`network_extended`（为调试场景放宽网络和资源限制）。MCP server 因为需要网络和可写 rootfs（pip install），默认用的是比 `sandbox_exec` 更宽松的配置——这两者的信任级别不一样，策略也该不一样。

还没解决的主要是**细粒度网络策略**。"允许访问 pypi.org 但不允许访问其他地址"这种需求一定会来，但在容器层面实现不简单，暂时没做。

**镜像预热**也还在打磨。目前 Box Runtime 初始化时会预热基础镜像，但预热哪些镜像、什么时候更新，还没有特别好的自动化方案。
