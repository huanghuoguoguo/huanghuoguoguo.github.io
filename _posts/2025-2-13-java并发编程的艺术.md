## 第 1 章 并发编程的挑战

频繁的上下文切换可能比单线程更慢——切换会使高速缓存失效，带来更多的缓存未命中。减少上下文切换的几种思路：

| 手段 | 原理 | 实例 |
|------|------|------|
| 无锁并发 | hash 取模分散写压力，降低锁粒度 | ConcurrentHashMap 分段锁、Redis 哈希槽分片 |
| CAS | 乐观策略，争抢时间极短时比加锁高效 | AtomicInteger、AQS 的 state 更新 |
| 最少线程 | 避免创建不必要的线程 | 线程池复用 |
| 协程 | 单线程内多任务调度，避免内核态切换 | Go goroutine、JDK 19+ 虚拟线程 |

## 第 2 章 并发机制的底层原理

这章进入 JMM 的范围。

### volatile

共享变量面临三个问题：工作内存不可见、指令重排、多线程下读到未初始化的值。

volatile 通过在字节码中嵌入 **lock 前缀指令**保证可见性。lock 前缀在多核处理器下做两件事：

1. 将当前处理器缓存行的数据写回系统内存
2. 使其他 CPU 中缓存了该内存地址的数据无效（总线嗅探协议）

volatile 的使用优化要考虑**缓存行填充**——避免多核处理器对同一个缓存行（64 字节）并发操作引起频繁失效。得益于之前 OB 比赛中对 sq4 的优化，对缓存行已经提前理解了一些。

### synchronized 与对象头

对象头的 Mark Word 中嵌入了锁状态、线程 ID、GC 年龄等信息。这种在一个 long/int 内紧凑编码的方式很常见——OB 比赛中有人把 c1 数据嵌入高 32 位、ZGC 的染色指针，都是类似思路。

### 锁升级

| 锁级别 | 适用场景 | 加锁方式 | 升级条件 |
|--------|---------|---------|---------|
| 偏向锁 | 单线程反复进入同步块 | CAS 设置线程 ID 到 Mark Word | 出现第二个线程竞争 |
| 轻量级锁 | 短暂、低竞争 | 栈帧 Lock Record + CAS 替换 Mark Word | 自旋超过阈值（默认 10 次） |
| 重量级锁 | 高竞争、长持有 | OS 互斥量，未获取锁的线程挂起 | — |

锁只能升级不能降级。后续 JDK 版本默认关闭偏向锁，就是因为编码复杂、收益在现代多核场景下不明显。

这和 JVM 的锁升级本质上是同一件事——偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（OS 互斥量）。在操作系统内核设计中也能看到这种思想的对应：内核的自旋锁适合持有时间短的场景（不让出 CPU），信号量/互斥体适合可能睡眠的场景，底层的取舍思路是一致的。

### 原子操作

处理器层面通过**缓存锁定**或**总线锁定**实现原子操作。跨总线、跨多个缓存行、跨页表的访问只能用总线锁。

Java 层面的 CAS 有三个已知问题：ABA、自旋开销大、只能保证单个变量的原子操作。

## 第 3 章 Java 内存模型

### 核心结构

JMM 的抽象：每个线程有自己的本地内存（涵盖缓存、写缓冲区、寄存器），线程间通信必须通过主内存。线程之间不是直接发消息，而是让对方的缓存失效，迫使其从主内存重新 load。

这个模型和业务中 Redis 缓存的失效策略很像——我不更新缓存，只是将其删除，等下一个查询的时候才从数据库 load 出来。JMM 做的事情本质上是一样的。

### 重排序

三个层面的重排序：编译器优化 → 指令级并行 → 内存系统。JMM 通过四种内存屏障（LoadLoad、StoreStore、LoadStore、StoreLoad）禁止特定重排序。

核心原则：**as-if-serial**——不管怎么重排序，单线程的执行结果不能被改变。但多线程下，重排序可能让线程读到未初始化的值。

### happens-before

happens-before 不意味着物理上一个操作在另一个之前执行，而是保证前一个操作的结果对后一个操作可见。关键规则：

- volatile 写 happens-before 后续的 volatile 读
- 锁释放 happens-before 后续的锁获取
- Thread.start() happens-before 该线程的 run()

### volatile、synchronized、final 的内存语义

三者本质上都是在回答同一个问题：怎么保证一个线程对共享变量的修改对其他线程可见。

- **volatile**：写时刷回主内存 + 读时从主内存加载，禁止重排序。适合状态标志位
- **synchronized**：进入时清空工作内存，退出时刷回主内存。既保证可见性也保证互斥
- **final**：构造函数内的 final 赋值不会被重排到构造函数之外，保证对象构造完成后 final 字段一定可见

### 双重检查锁

经典的延迟初始化问题：不加 volatile 可能因为重排序，导致其他线程拿到一个已分配内存但未初始化的对象。两种解决方案：

1. **volatile**：禁止对象创建的重排序
2. **类初始化**：利用 JVM 类加载的线程安全性，只在首次加载时初始化（静态内部类持有实例）

### JMM 的定位

从大到小：顺序一致性模型（理论参考）→ 处理器内存模型（硬件级）→ JMM（语言级）。JMM 屏蔽了不同处理器内存模型的差异，在不同平台上为 Java 程序员提供一致的内存可见性保证。

## 第 4 章 并发编程基础

线程不会被强行终止——只能自己检查退出标志，或者在 wait 状态下收到中断信号抛出异常。

### 等待/通知机制

轮询不是好的通知方式。wait/notify 是 Object 上的方法，经典范式：

- **等待方**：获取锁 → 条件不满足则 wait → 被唤醒后 **while 循环**重新检查条件 → 条件满足执行逻辑
- **通知方**：获取锁 → 改变条件 → notify/notifyAll

为什么用 while 而不是 if？因为被唤醒后条件可能已经被其他线程改变了，必须重新检查。

### ThreadLocal

绑定在当前线程上的数据，底层是 ThreadLocalMap。

## 第 5 章 Java 中的锁

### Lock 接口

Lock 接口相比 synchronized 多了四个能力：公平/非公平选择、可中断获取、超时获取、Condition 条件队列。

### AQS（AbstractQueuedSynchronizer）

AQS 是 Java 并发包的核心基础设施。ReentrantLock、Semaphore、CountDownLatch 都基于它。

核心设计：**volatile int state + FIFO 双向队列 + CAS**。

- `state` 的含义由具体同步器定义：ReentrantLock 中表示重入次数，Semaphore 中表示可用许可数
- 线程获取 state 失败时，封装成 Node 加入队列尾部，通过 `LockSupport.park()` 阻塞
- 前驱节点释放资源时，通过 `LockSupport.unpark()` 唤醒后继节点

调用链路：**Lock → AQS → CAS → Unsafe → CPU 原子指令（cmpxchg）**。

### AQS vs synchronized

| 维度 | AQS（Lock） | synchronized（Monitor） |
|------|------------|----------------------|
| 同步队列 | 显式 FIFO 双向队列，AQS 框架管理 | 隐式 Entry Queue，JVM 管理 |
| 等待队列 | Condition 对象维护，可以有多个 | Wait Set，只有一个 |
| 阻塞方式 | `LockSupport.park()`，不释放锁 | `Object.wait()`，释放锁 |
| 灵活性 | 可扩展为各种同步器 | 功能固定 |
| 适用场景 | 复杂同步控制（公平锁、读写锁、信号量） | 简单的同步块 |

两者的本质区别：synchronized 是语言层面的关键字，行为固定；AQS 是并发库的底层实现类，允许自定义锁行为。

### ReentrantLock

公平锁与非公平锁的区别在于首次获取：非公平锁直接 CAS 尝试，成功就插队；公平锁先检查队列中是否有等待线程。非公平锁性能更高，因为减少了线程切换。

### 读写锁

用一个 int 的高低位分别表示读状态和写状态，保证判断操作的原子性。支持**锁降级**：持有写锁时可以获取读锁，然后释放写锁。

### Condition

一个 Lock 可以创建多个 Condition，每个 Condition 有自己的等待队列。线程调用 `await()` 进入等待队列，被 `signal()` 后转移到同步队列去竞争锁。

## 第 6 章 并发容器和框架

### ConcurrentHashMap

几个关键点：
- JDK 7 的 Segment 段锁 → JDK 8 的数组 + 链表/红黑树 + CAS + synchronized，锁粒度更细
- 渐进式 rehash：扩容时多线程并发转移，同时有标识标志是否正在扩容。先查旧表，查不到再查新表（COW 思想）
- 早期版本有 bug，链表头插法导致成环

数据分段降低冲突——这是《设计数据密集型应用》中讲的分区思想，和 Redis 哈希槽分片是同一个道理。

### Fork/Join 框架

工作窃取算法——空闲线程从其他线程的队列尾部偷任务。在 Caffeine 的 common 线程池中见过这个实现。

有点类似 MapReduce 的分治思想，但 Fork/Join 的任务是有状态的，可以获取异常和结果。

## 第 8 章 并发工具类

### CountDownLatch vs CyclicBarrier

| 特性 | CountDownLatch | CyclicBarrier |
|------|---------------|---------------|
| 复用 | 一次性 | 可循环使用 |
| 角色 | 线程职责不同：有的倒计时，有的等待 | 线程职责相同：都在等彼此 |
| 场景 | 主线程等待多个子任务完成 | 多线程到达同步点后一起继续 |

### Semaphore

信号量，维护一组许可证。`acquire()` 获取许可（许可不足则阻塞），`release()` 归还许可。用于控制并发访问资源的线程数。

### 用六种方式解同一道题

LeetCode 1115「交替打印 FooBar」，用不同的同步原语解同一个问题，很好地展示了各工具的适用场景。

**CyclicBarrier + volatile**：屏障同步 + 标志位控制顺序。

```java
class FooBar {
    private int n;
    private CyclicBarrier cb = new CyclicBarrier(2);
    volatile boolean fooExec = true;

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            while (!fooExec) { /* 自旋等待 */ }
            printFoo.run();
            fooExec = false;
            try { cb.await(); } catch (BrokenBarrierException e) { e.printStackTrace(); }
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            try { cb.await(); } catch (BrokenBarrierException e) { e.printStackTrace(); }
            printBar.run();
            fooExec = true;
        }
    }
}
```

**Semaphore**：两个信号量交替释放，最直观。

```java
class FooBar {
    private int n;
    private Semaphore fooSema = new Semaphore(1);
    private Semaphore barSema = new Semaphore(0);

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            fooSema.acquire();
            printFoo.run();
            barSema.release();
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            barSema.acquire();
            printBar.run();
            fooSema.release();
        }
    }
}
```

**Thread.yield() + volatile**：让出 CPU 时间片，等待条件满足。

```java
class FooBar {
    private int n;
    volatile boolean fooExec = true;

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        for (int i = 0; i < n; ) {
            if (fooExec) { printFoo.run(); fooExec = false; i++; }
            else { Thread.yield(); }
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        for (int i = 0; i < n; ) {
            if (!fooExec) { printBar.run(); fooExec = true; i++; }
            else { Thread.yield(); }
        }
    }
}
```

**ReentrantLock + volatile**：n=5 时 LeetCode 就 TLE 了，因为公平锁的获取 / 释放开销太大，不适合这种高频交替场景。

```java
class FooBar {
    private int n;
    private ReentrantLock lock = new ReentrantLock(true);
    volatile boolean fooExec = true;

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        for (int i = 0; i < n; ) {
            lock.lock();
            try {
                if (fooExec) { printFoo.run(); fooExec = false; i++; }
            } finally { lock.unlock(); }
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        for (int i = 0; i < n; ) {
            lock.lock();
            try {
                if (!fooExec) { printBar.run(); fooExec = true; i++; }
            } finally { lock.unlock(); }
        }
    }
}
```

**BlockingQueue**：两个队列交替 take/add，利用阻塞语义天然实现同步。

```java
class FooBar {
    private int n;
    private BlockingQueue<Integer> fooQueue = new LinkedBlockingQueue<>() { { add(0); } };
    private BlockingQueue<Integer> barQueue = new LinkedBlockingQueue<>();

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            fooQueue.take();
            printFoo.run();
            barQueue.add(0);
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            barQueue.take();
            printBar.run();
            fooQueue.add(0);
        }
    }
}
```

**synchronized + wait/notify**：最基础的方式。注意 notify 不会立即释放锁，要等退出同步块。

```java
class FooBar {
    private int n;
    private Object obj = new Object();
    private volatile boolean fooExec = true;

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            synchronized (obj) {
                if (!fooExec) { obj.wait(); }
                printFoo.run();
                fooExec = false;
                obj.notifyAll();
            }
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        for (int i = 0; i < n; i++) {
            synchronized (obj) {
                if (fooExec) { obj.wait(); }
                printBar.run();
                fooExec = true;
                obj.notifyAll();
            }
        }
    }
}
```

**LockSupport**：`park()` 阻塞当前线程，`unpark(thread)` 唤醒指定线程。许可是 (0,1) 信号量，上限为 1。和 Semaphore 不同，不会累加。

```java
class FooBar {
    private int n;
    private Map<String, Thread> map = new ConcurrentHashMap<>();
    private volatile boolean fooExec = true;

    public FooBar(int n) { this.n = n; }

    public void foo(Runnable printFoo) throws InterruptedException {
        map.put("foo", Thread.currentThread());
        for (int i = 0; i < n; i++) {
            while (!fooExec) { LockSupport.park(); }
            printFoo.run();
            fooExec = false;
            LockSupport.unpark(map.get("bar"));
        }
    }

    public void bar(Runnable printBar) throws InterruptedException {
        map.put("bar", Thread.currentThread());
        for (int i = 0; i < n; i++) {
            while (fooExec) { LockSupport.park(); }
            printBar.run();
            fooExec = true;
            LockSupport.unpark(map.get("foo"));
        }
    }
}
```

六种方案的对比：

| 方案 | 核心机制 | 优点 | 缺点 |
|------|---------|------|------|
| CyclicBarrier | 屏障同步 | 语义清晰 | 需要额外的 volatile 标志位 |
| Semaphore | 许可证交替 | 最简洁 | — |
| yield | 让出时间片 | 无锁 | 自旋浪费 CPU |
| ReentrantLock | 公平锁 | 可控 | 开销大，LeetCode TLE |
| BlockingQueue | 阻塞队列 | 天然同步 | 多一层队列抽象 |
| synchronized | wait/notify | 基础 | 只有一个等待队列 |
| LockSupport | park/unpark | 可指定唤醒线程 | 需要持有线程引用 |

## 第 9-10 章 线程池与 Executor 框架

线程池的几个核心参数（corePoolSize、maximumPoolSize、keepAliveTime、workQueue、handler）和 Executors 提供的几种预置线程池，这里不再罗列。

Executor 框架这一章对我有一些启发，因为我正在模仿着自己做一个，把曙光的单体任务调度框架迁移到分布式集群（最后失败了^^）。几个准则：

**任务隔离**：根据优先级或互斥关系，对任务的执行顺序做隔离。比如公司的有状态任务调度中，下电操作优先于风扇风速调整——这是显而易见的优先级需求，在线程池层面需要用不同的队列或优先级策略来保证。

**重试策略**：在单体上实现的 phase 级重试、重试抓取等策略，在分布式环境下不直接生效。如果发生宕机和主从转移，需要重新设计重试的粒度和范围。

**任务无状态**：分布式调度要求任务尽量无状态。但公司的 job 框架是有状态的——比如调整风扇转速必须落到特定机器上，跟硬件绑定，不好做成无状态。引入分布式调度时需要区分：纯计算任务可以无状态调度，硬件绑定的任务需要亲和性调度。

---

看完全书，对 JMM、并发编程面临的问题、如何一步步从底层原理到应用层工具有了更完整的认识。从 volatile 的 lock 前缀到 AQS 的 state + FIFO 队列，再到 Executor 框架的任务调度，每一层都是在前一层的基础上做封装。理解了底层，上层的 API 设计就不再是黑盒。
