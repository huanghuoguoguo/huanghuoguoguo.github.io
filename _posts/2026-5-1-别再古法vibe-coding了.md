上礼拜天下午跟几个小伙伴开了个会，我把最近啃的那一坨东西——Claude Code、Superpowers、OpenSpec、Hermes、OpenClaw——拼成一份 PPT 讲了一遍。本来想讲架构，结果讲到一半气氛就拐了。大家最在意的根本不是哪个项目怎么实现，而是一个很糊的问题：

> "为什么明明能用，还是不舒服？"

我一开始没准备答这种问题。但一路被追问下来，等讲完那份 PPT，我自己也被那个问题搞得不太舒服了。

这篇是那场分享会的过程记录，顺带把最后一个没答上来的问题也写下来。

## 一、那些说不清的"不舒服"

我让大家先把自己用 Claude Code / Cursor / Copilot 时最烦的瞬间讲出来。我不想从工具讲起，我想从痛点讲起。

凑出来的清单大概是这样：

- 上次跟它解释过的项目背景，这次开新会话又要重讲一遍
- 需求聊着聊着就漂了，最后写出来的东西和最初想做的不是一回事
- 它说"做完了"，但你不敢直接合，要从头看一遍它干了啥
- 同样的坑反复踩，上礼拜被坑过的命令这礼拜还会忘加 `--no-cache`
- 跨天的任务接不上，昨天那个 todo 列表今天找不回来
- 名词一堆——记忆、RAG、Hooks、Skills、Specs、MCP——但你也说不上它们到底解决了什么

这些抱怨听起来很零散。但我们一条条对下来，发现它们不是同一类问题。

有的是模型本身的局限——它就是没记忆。
有的是工具的缺口——CC 没有内置的 spec 系统。
有的是工作流没建——你自己没把"踩过的坑"沉淀成可以被下次会话拿到的东西。

这三类东西我之前一直混着用同一个词喊"AI 不好用"。摊开看才发现，它们要不同的方式补。

---

## 二、Harness：补的是哪一层

讲到这里我把那张 PPT 翻出来。

大模型本身是一个无状态的纯推理器。它有的是预训练知识 + 当前上下文窗口；它没有的是持久记忆、文件系统、命令执行、跨会话的状态。**Harness 就是套在模型外面、专门补这些洞的那层东西**。

Claude Code 自己就是一个 harness。它给了：

- 工具：Read / Edit / Write / Bash / Glob / Grep
- 扩展点：MCP、Skills、Hooks
- 记忆载体：CLAUDE.md、Auto Memory
- 隔离：git worktree

这些已经够你用一阵了。但用久了你会发现下面几个洞它没补——或者说，补得很浅：

| 洞 | CC 现状 | 体感 |
|---|---|---|
| 长期记忆 | CLAUDE.md 要你手写，Auto Memory 是模型自己判断存什么 | 你以为很重要的决策它没存 |
| Plan 进度 | Plan 文件没有打勾，存在 `~/.claude/` 不在项目里 | 跨天就接不上 |
| TodoWrite | 单会话有效，会话结束就丢 | 第二天从零讲 |
| Spec | 没有结构化规范存储 | 早期约束聊着聊着就忘了 |
| 经验沉淀 | 记忆 ≠ 回归测试 | 同一个错重复犯 |

讲到这一栏的时候有个朋友插一句："所以 Auto Memory 不是已经能记了吗？"

能记，但记不准。Auto Memory 是模型自己判断"这条值不值得记"，它的优先级和你的优先级常常对不上。你以为最重要的是"这次为什么否掉了 Memcached"，它存下来的可能是"用户喜欢用 Tailwind"。这不是 bug，是设计上的必然——靠模型自由判断，就一定会有偏差。

所以痛点 1 不是 CC 不行，是**它给的记忆机制还没到"项目级可维护资产"的程度**。

## 三、Hooks 是那个口子

CC 自己留了一个让你往里加东西的口子，叫 Hooks。

类比 Git Hooks 就行——pre-commit、post-commit。CC 这边就是 SessionStart、PreToolUse、PostToolUse、SubagentStop。在这些时机自动跑你写的脚本，可以注入上下文，可以捕获变更，也可以做质量门禁。

最直观的例子是 claude-mem 这种自动记忆方案。它做的事并不复杂：

```
PostToolUse Hook 拦住每次 Edit / Write / Bash
        ↓
Worker Service 把变更内容压缩、分类
        ↓
存进 memory/ 下的 markdown 文件
        ↓
SessionStart Hook 下次会话开始时把这些 memory 注入系统提示
```

你看，从头到尾没有什么"长期记忆"的黑魔法。就是 hook 拿到事件 → 写 markdown → 下次启动时读回来。听起来很朴素，但它能让 CC 隔天还记得你昨天改过什么。

Skill 框架（Superpowers 那一套）和 Spec 系统（OpenSpec 那一套）也是建在同样的口子上。Skills 是把方法论做成 markdown 文件，被 description 字段触发；Specs 是把需求/设计/任务做成项目目录里的文件，进入版本控制。

讲完这一段我说了一句让大家有点出戏的话：

> 记忆、RAG、网络搜索、Skills、Specs，这些名词去掉包装，做的是同一件事——**在合适的时机，把合适的内容塞进上下文窗口**。

RAG 不就是"有个文件库 + 用关键词搜 + 把结果拼进 prompt"吗？记忆不就是"把对话里有用的部分写成文件，下次会话读回来"吗？Skills 不就是"一个有触发条件的 markdown 文件"吗？

把名词拆掉之后，你会发现你只是在不同的时机、用不同的检索方式，往那个上下文窗口里灌不同的内容。

但这只是事情的一半。**上下文负责"让模型知道"，工具/脚本/测试/Guard 负责"让系统稳定执行"**。两者都要在。光把内容塞进窗口，模型还是会乱跑；只靠脚本和测试，模型又不知道该往哪个方向跑。

## 四、Superpowers 重，OpenSpec 轻

讲到这里其中一个朋友——他这段时间一直在啃这套——开始皱眉头。我猜到他要问什么。

> Superpowers 我也在用。它确实把 TDD、systematic-debugging、verification-before-completion 这些方法论做成了 Skills，agent 能自己触发，自己遵守。但它太重了，开一次就一堆 Skill 加载进上下文，token 烧得心疼。
>
> 然后我去用 OpenSpec。它够轻，proposal/design/tasks 三个 markdown 文件就开干，可学习曲线低。但它考虑不全，没有方法论那一层，agent 干起来还是凭感觉。
>
> 我试过两个一起用，结果发现两套文档并存——specs/ 那边一套，skills/ 那边一套，agent 也搞不清该听哪个。然后我就一直在反复横跳。

他说完之后会议室没人接话。因为这是一个会议室里几乎每个人都遇到过的问题：你试了 A，觉得太重；换 B，觉得太轻；想搭一起用，发现它们不是为彼此设计的。

我当时给的回应大概是这样：

SP 和 OpenSpec 不是同一层的东西。SP 在补"agent 该怎么干活"，OpenSpec 在补"项目里有哪些待办契约"。它俩并存其实不冲突——只要你愿意承认 specs 是契约层、skills 是执行层，前者描述边界，后者描述步骤。

听起来好像挺通的。但我自己讲完都觉得，这只是一个"听起来工整"的答案。它没有解决他真正的痛点：

**他想要的不是一个分层模型，他想要的是一套自己用着顺手的工作流。**

## 五、最后那个没答上来的问题

会快散的时候他又补了一刀：

> 我已经折腾过很多次了。每次都是先选一个框架，用一阵觉得不对，再让 AI 帮我改改改，最后改出一个四不像。下次再换个新框架，又重来一遍。
>
> 不断的尝试，不断的让 AI 修修补补，最终怎么沉淀为自己的工作流资产呢？

我没答上来。

这个问题之所以扎手，是因为它把前面那些干干净净的分层模型给拆穿了。你可以画一张很漂亮的图——specs 在上、skills 在中、scripts 在下、tests 兜底——但这张图是事后画的。**实际过程是你东试一个西试一个，每次都会留下一点 markdown、一点配置、一点临时脚本，它们散在各处，互相不知道对方存在**。所谓"工作流资产"听起来很正经，但你打开自己的 `.claude/` 目录，里面有几个是你真的在复用的？大部分是某次试了一下就再也没动过的草稿。

我现在能想到的也就这几条，写在这里不是答案，是我自己也还在试的方向：

**1. 先承认大部分尝试不会沉淀。** 大多数 Skill / Spec / 配置都是一次性的。你写了一个 `web-sqli-triage.md`，下次解 SSRF 题它根本用不上。把"每次试都要留下东西"当成压力源，反而会让你写一堆没用的文档。

**2. 沉淀的触发条件是"同一个摩擦出现 ≥ 3 次"，不是"我学到了一个新概念"。** 第一次卡住，你可能是状态不好；第二次，可能是这类题的共性；第三次再卡，那就该把它写成一个有触发条件的 Skill，或者下沉成一个脚本。一次性的卡顿配不上一份资产。

**3. 资产宁可少，别贪多——前提是它们真的被触发到。** 一个写得很烂、但每次都被自动加载的 Skill，比十个写得很漂亮、agent 永远想不起来的 Skill 有用。CSO（Claude Search Optimization）那一套讲的就是这个：description 字段决定它会不会被找到，写不好它就是死文档。

**4. 留一个 backlog，但别把 backlog 当资产。** 把"这次卡住了但还没成型的东西"先扔 backlog，等它在 backlog 里出现第二次、第三次再升级成 Skill。backlog 是观察池，不是仓库。

**5. 进化要看 specs 那一层会不会动。** 新做一个 Skill 是补能力，改一个 Skill 是修方法。但你的 specs（这个项目应该能干什么、不能干什么）一旦发生变化，才意味着你的工作流自己学到了新东西。如果用了半年 specs 一次都没改，要么你这半年没在长，要么你的 specs 写的是废话。

但这五条还远远不是答案。

老实说，我自己手里那个 ctfagent 就卡在这一步。Skills 写了一堆，knowledge/ 也有一堆，但要说它形成了"我的工作流资产"，我心里没底。下次开新题、新场景，我大概率还是会从头试一遍，再骂一句 AI 修修补补好烦，再回到这个循环里。

那场分享会最后是怎么散的我已经记不清了。大概就是大家一边收东西一边随口说"那就先试着养一养看吧"。没人给出答案，因为没人有答案。

如果你也卡在 SP 和 OpenSpec 中间反复横跳，那至少我们卡在同一个地方。这一篇没办法帮你跳出来——它只是把那个让你不舒服的问题，原样写下来了一遍。

下一步我想试的是一件笨事——挑一个用了一阵的项目，把 `.claude/` 下的 Skill、Spec、scripts 全摊开，统计一下过去三个月里每个文件的命中次数。看完那张表，"什么算工作流资产"这个问题大概会比现在清楚一些。如果到时候有结论，再写一篇接上。
