这两天偷学一下 nodejs，说不定有用。

我前面写后端，手感大多还是 Java 那边养出来的。实习里写过 Spring Boot 管理后端，接口、告警、指标计算这些东西都碰过；再往后做项目，脑子里默认走的也还是那条路：Controller、Service、DAO 怎么分，配置放 `application.yml` 还是配置中心，接口前面要不要挂网关，服务之间是 HTTP 调还是 Feign，数据后面接 MySQL 还是 Redis。

所以转到 Node.js 这里，我反而没先盯 TypeScript 语法。真正先把我拽住的是另外几件事：单线程的事件循环怎么扛住 Web 请求？内存会怎么涨？CPU 密集型任务来了怎么办？

再往外一点，就是 AWS 这套 Serverless。它跟我前面熟的那种“先起服务，再谈注册发现、调用链和扩缩”的开发手感，到底差在哪。

我拉了一个任务管理的 Demo，顺着这些问题把这套技术栈过了一遍。

这次折腾下来，我更强的感觉其实不是 “Node.js 到底好不好用”，而是前面那套后端经验没有白长。它不会直接告诉我 Express 怎么写、DynamoDB 怎么建表，但会让我少走很多弯路。

---

## 第一版：本地内存存储

最开始我想快速跑通接口，存储直接用的内存数组。

```typescript
// 最早的版本，简单到有点简陋
const tasks: Task[] = [];

app.get('/tasks', (req, res) => {
  res.json(tasks);
});
```

这个版本活了大概 10 分钟。写完增删改查后，我对语法就没什么兴趣了。真正让我停下来看的是另一件事：这种写法太轻了，轻到数据就在进程里，服务一重启就没了。

这跟我熟的 Java 后端很不一样。平时写 Spring Boot，数据库、缓存、配置这些东西很快就会进来。Node.js 这边不是没有这些东西，但我一开始上手时，最直观的感觉就是它先把骨架压得很薄，后面很多东西都要自己决定往哪加。

所以我很快就不再盯着语法看了。真正让我别扭的是，东西虽然很快能跑起来，但后面该补什么、补到哪一层，不像以前那样有一条现成路径。

---

## 第二版：想加 Redis，但没加

最初的计划里，第二版是用 Redis 做缓存层。

想法很标准：本地内存不够持久，先加个 Redis，数据存在内存但能跨进程共享。等这一版跑通了，再迁到 DynamoDB。

但这地方我卡了一下。在 Java 世界里，多级缓存是很自然的架构设计：Caffeine 做本地缓存，Redis 做分布式缓存，数据库做持久层。可如果我在 AWS 上部署，Redis 要不要自己搭集群？ElastiCache 是不是又贵又麻烦？我对 DynamoDB 还一无所知，只知道是个 NoSQL 数据库，它能不能直接扛住？

想来想去，干脆直接上 DynamoDB，先看 Serverless 数据库自己能跑到哪一步。Redis 这一版先不写，不然问题一下子会变成三件事：Node.js 怎么写，Redis 怎么接，DynamoDB 到底是什么脾气。东西一多，最后反而不知道自己到底在验证什么。

---

## 第三版：DynamoDB 与单表设计

现在代码里跑的是 DynamoDB。

先把客户端接起来不难：

```typescript
// lib/dynamoDb.ts
const client = new DynamoDBClient({
  region: env.awsRegion,
  endpoint: env.awsEndpoint,  // 本地用 LocalStack
});
```

但写查询的时候，我卡住了。

DynamoDB 是 NoSQL，但它和 MongoDB 那种随手塞字段的感觉还不太一样。它的查询能力很受限：没有 JOIN，没有复杂的 WHERE，排序也主要围着主键或索引转。我最初想给任务加用户隔离，第一反应还是像关系数据库那样建个 `user_id` 外键关联，后来才发现 DynamoDB 的设计思路完全不同。

单表设计远不是一句“因为 DynamoDB 没有 JOIN”就能概括的。它真正的难点在于：你必须先想清楚系统会按哪些模式访问数据，再去设计主键、排序键和索引。

关系数据库里，我习惯先把实体拆开，后面再靠 JOIN 补回来。DynamoDB 不是这条路。你要先问自己：我要按用户查任务，还是按状态查任务，还是按时间倒序翻最近更新？这些访问模式如果一开始没想明白，表结构后面会越改越别扭。

我现在的代码还没做单表设计，只是最简单的单用户任务存储。这个地方先把我绊住的，其实是建模习惯。数据库一换，习惯也得跟着换。以前在 Spring Cloud 那套里，我默认后面站着关系数据库；到了 DynamoDB，这个默认前提先没了。

## Express vs Spring Boot：轻量级的体感

写 Express 的第一感受是：轻。

Spring Boot 一个启动类加各种注解，依赖管理用 Maven 或 Gradle，配置文件有 `application.yml`、profile 隔离、配置中心。Express 这里，`package.json` 几行依赖，一个 `index.ts` 就把路由挂起来了。

```typescript
const app = express();
app.use(cors());
app.use(express.json());

app.get('/tasks', getTasks);
app.post('/tasks', createTask);
// ...
```

没有注解，没有 IOC 容器，中间件就是函数嵌套。上手确实快，但约束也少。项目小的时候会觉得很爽，东西一多就会开始追问：这层逻辑到底该放哪，团队准备靠什么把结构稳住。

我最初习惯把 Express 中间件类比成 Spring 的拦截器，但越写越发现不对。Express 中间件更像一串按顺序往下走的处理函数，`req` 和 `res` 在里面一层层传下去，谁处理、谁返回，链路很直。而 Spring 那边的“切面”是分层的：Servlet `Filter` 在最外层，`HandlerInterceptor` 靠近 Controller，AOP 则切入方法调用。三者都叫“中间处理”，但抽象层次完全不同。

## AWS Serverless：另一种架构思维

我之前的经验主要在 Spring Cloud 那套：Eureka 服务发现、Gateway 网关、配置中心、Feign 调用。看到 AWS 这边这些东西时，我第一反应当然还是类比。

API Gateway 我会先往网关上想，Lambda 我会先往一段后端执行逻辑上想，DynamoDB 就先当成一个托管的 NoSQL 数据库。这些类比不能说错，至少刚开始够用。

但写着写着，我又会觉得不太对。

Spring Cloud 那套默认前提是，你先有一堆服务在那里跑着，后面再去想它们怎么注册、怎么调、怎么扩。AWS 这边不是这个手感。它更像是先把入口、计算、存储这些东西分别摆在你面前，你再想怎么把它们接起来。

这个顺序一换，脑子里的很多默认动作也得跟着换。以前我会先想服务怎么拆，接口怎么挂，调用链怎么走。到了这里，前面那层“先起服务”本身就淡了，注意力会先被引到函数、事件和存储模型上。

---

## 事件循环与 Promise：从其他语言迁移

这部分我其实没遇到太多障碍，但值得记录一下视角。

我一开始就是拿 Python 的 `asyncio` 去对它的。至少在我现在碰到的这一层，这个类比是顺的：单线程，I/O 不阻塞，靠事件循环把回调和任务往前推。Python 里是 `async/await`，Node.js 里也是。

所以这块我上手没太费劲。Node.js 这里的 `async/await`，按我现在的理解，就是把很多等 I/O 的活挂在事件循环上往前推，不是说写了 `await` 就自动并行了。

所以我当时会继续追一个很自然的问题：那 CPU 活怎么办？图像处理、压缩、大块 JSON 计算，总不能也全塞在这一条线上。

这个问题我现在也还没真做实验，只是先知道 Node 不是完全没别的路。可以用 `worker_threads`，也可以走多进程。只是这些东西我还没亲手写过，所以现在先记到这里，不往下讲太深。

---

## 内存模型与并发模型：先记我现在能确定的部分

这部分我现在其实只抓住了一个很粗的轮廓。

并发这块，我还是先拿 Python 的 `asyncio` 去对。这个类比到目前为止是顺的：一条事件循环往前推，I/O 不在原地傻等，`async/await` 主要是把这些等待过程写得更直一点。

但类比到这里差不多也该停了。再往下像 CPU 活怎么处理，`worker_threads` 到底值不值得上，多进程和线程之间真实开销差多少，我这次都还没做实验。现在硬讲，只会讲成一堆自己还没验证过的话。

内存这块我现在也只敢先记一个很粗的印象。Node.js 后面站的是 V8，它不是手动管内存那条路，运行时会把对象分配和回收这层兜住。这个我现在能确定。

但再往下像 GC 具体怎么跑，长时间服务的内存曲线会不会抖，什么场景下会顶得比较难看，这些我现在都没有实测。前面先把问题记在这，后面真做压测了再回来说。

## 先写到这里

这次把 Demo 跑下来，我对 Node.js 的陌生感主要不在 TypeScript。

真正让我停下来想的，还是那些老问题：服务骨架有多薄，数据模型怎么跟着存储方式变，事件循环把 I/O 扛起来以后，CPU 和内存到底往哪里掉。

所以这次我自己的感觉更像是，上手会快一点，但快的不是语法，还是前面那些判断。先跑一个很小的东西，把链路打通；哪块最先把我绊住，我就先看哪块；能类比的先类比，类比到不顺的地方就停。很多时候这样反而比一上来系统学一遍更省时间。

后面如果继续折腾，我大概会直接去做压测，看内存曲线，再把 `worker_threads` 和多进程各试一遍。现在这些类比先够我往下走，但它们到底能撑到哪一步，我还得再碰几次壁。
