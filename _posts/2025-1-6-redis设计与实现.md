## 第一部分：数据结构与对象

这部分是整本书的地基。Redis 对外暴露的五种数据类型（String、List、Hash、Set、Sorted Set），底层各有多种编码实现，会根据数据特征自动切换。理解这一层，才能理解 Redis 为什么快、为什么省内存。

### 第 2 章 简单动态字符串（SDS）

Redis 没有直接使用 C 字符串，而是自己实现了 SDS。核心改进：

| 对比维度 | C 字符串 | SDS |
|---------|---------|-----|
| 获取长度 | O(n)，遍历到 `\0` | O(1)，`len` 字段直接读 |
| 缓冲区溢出 | 不检查，容易溢出 | 修改前自动检查空间 |
| 内存分配 | 每次修改都要重新分配 | 空间预分配 + 惰性释放 |
| 二进制安全 | 不安全，`\0` 会被截断 | 安全，可存任意二进制数据 |
| C 函数兼容 | — | 兼容部分 `<string.h>` 函数 |

**空间预分配策略**：修改后长度 < 1MB 时，分配 `2 * len` 的空间；>= 1MB 时，多分配 1MB。减少连续增长时的内存重分配次数。

**惰性空间释放**：缩短字符串时不立即回收多余空间，留给后续操作使用。

### 第 3 章 链表

Redis 的链表是**双向无环链表**，`list` 结构持有 `head`、`tail` 指针和 `len` 计数器。

- 头尾插入删除：O(1)
- 获取长度：O(1)
- 节点值用 `void*` 指针存储，可存任意类型数据
- 通过函数指针实现多态（`dup`、`free`、`match`）

### 第 4 章 字典

Redis 字典 = **两个哈希表** + **渐进式 rehash**。

**哈希表结构**：`dictht` → `table` 数组 → 每个槽位挂 `dictEntry` 链表（链地址法解决冲突）。

**哈希算法**：MurmurHash2，分布均匀、计算快。

**渐进式 rehash**：

这是字典最精巧的设计。当负载因子过高/过低时需要 rehash，但一次性迁移所有键值对会阻塞服务器。Redis 的做法是：

1. 为 `ht[1]` 分配空间
2. 设置 `rehashidx = 0`，标记 rehash 开始
3. 每次对字典执行增删改查时，**顺便**把 `ht[0]` 在 `rehashidx` 位置的所有键值对迁移到 `ht[1]`
4. rehash 期间，查找在两个表上都做；新增只往 `ht[1]` 写
5. 迁移完成后，释放 `ht[0]`，把 `ht[1]` 设为 `ht[0]`

### 第 5 章 跳跃表

有序集合（Sorted Set）的底层实现之一。通过多层索引实现 O(log n) 的查找、插入、删除。

**节点结构（zskiplistNode）**：
- `score`（分值）+ `obj`（成员对象）
- 多层 `level`，每层有 `forward` 前进指针和 `span` 跨度
- `backward` 后退指针（只有一层，用于反向遍历）

**跳跃表结构（zskiplist）**：
- `header`、`tail` 指针
- `length` 节点数量
- `level` 最大层数

层数随机生成（幂次定律），大部分节点层数较少，少数节点层数较多。

### 第 6 章 整数集合（IntSet）

当集合只包含整数且元素数量较少时使用。底层是一个**有序的整数数组**。

**编码升级**：当插入的整数超出当前编码范围（int16 → int32 → int64），整个数组升级。步骤：分配新空间 → 转换所有现有元素 → 插入新元素。**不支持降级。**

优点：灵活适应数据范围 + 节省内存（不用一开始就按最大类型分配）。

### 第 7 章 压缩列表（ZipList）

列表和哈希在元素少时的底层实现。**内存连续**的紧凑结构。

每个节点由三部分组成：
- `previous_entry_length`：前一节点长度（1 或 5 字节），支持反向遍历
- `encoding`：数据类型和长度
- `content`：实际数据

**连锁更新**：插入/删除节点时，如果后续节点的 `previous_entry_length` 需要从 1 字节扩展到 5 字节，会引发链式更新。最坏 O(n^2)，但实际发生概率极低。

### 第 8 章 对象

Redis 不直接暴露底层数据结构，而是通过 `redisObject` 封装，包含 `type`（类型）、`encoding`（编码）、`ptr`（数据指针）、`refcount`（引用计数）、`lru`（空转时长）。

**五种类型与编码的对应关系**：

| 类型 | 编码方式 | 切换条件 |
|------|---------|---------|
| String | int / embstr / raw | 整数用 int；≤ 44 字节用 embstr；否则 raw |
| List | ziplist / linkedlist | 元素数 < 512 且每个元素 < 64 字节时用 ziplist |
| Hash | ziplist / hashtable | 键值对数 < 512 且每个键值 < 64 字节时用 ziplist |
| Set | intset / hashtable | 全是整数且元素数 < 512 时用 intset |
| Sorted Set | ziplist / skiplist+dict | 元素数 < 128 且每个元素 < 64 字节时用 ziplist |

**为什么有序集合同时使用跳跃表和字典？** 跳跃表支持范围查询（ZRANGEBYSCORE），字典支持 O(1) 的成员查找（ZSCORE）。两者取长补短。

**内存优化**：
- 引用计数法回收内存，`refcount` 归零时释放
- 共享对象：Redis 启动时预创建 0-9999 的整数字符串对象，多个键可共享
- 不共享包含字符串的对象（验证相等性的时间复杂度太高）

## 第二部分：单机数据库的实现

### 第 9 章 数据库

**结构**：`redisServer` → 多个 `redisDb`，每个 `redisDb` 包含键空间（dict）和过期字典（expires）。

**过期键删除策略**：

| 策略 | 做法 | 优缺点 |
|------|------|--------|
| 定时删除 | 设置定时器，到期立即删除 | CPU 不友好，占用时间事件 |
| 惰性删除 | 访问时才检查是否过期 | 内存不友好，过期键可能长期占用 |
| 定期删除 | 每隔一段时间扫描一批键 | 折中方案 |

**Redis 实际使用：惰性删除 + 定期删除。** 两者配合，兼顾性能和内存。

**过期键与持久化/复制的交互**：
- 生成 RDB：过期键不写入
- 载入 RDB：主服务器忽略过期键，从服务器全部载入（由主服务器的 DEL 来同步删除）
- AOF 写入：键过期时追加一条 DEL 命令
- AOF 重写：过期键不写入
- 主从复制：从服务器不自主删除过期键，等主服务器的 DEL 命令

### 第 10 章 RDB 持久化

将内存中的数据库状态**快照式**保存为二进制文件。

**两种触发方式**：
- `SAVE`：阻塞主进程，完成后才能处理请求
- `BGSAVE`：fork 子进程执行，主进程继续服务

**自动触发**：通过 `save` 配置项（如 `save 900 1` = 900 秒内至少 1 次修改），serverCron 定期检查 `dirty` 计数器和 `lastsave` 时间戳。

**RDB 文件结构**：头部信息 → 各数据库的键值对（区分有/无过期时间）→ EOF → 校验和。值的编码方式取决于数据类型。

### 第 11 章 AOF 持久化

将每条写命令**追加**到文件末尾，恢复时重新执行所有命令。

**写入同步策略（appendfsync）**：

| 选项 | 行为 | 数据安全 | 性能 |
|------|------|---------|------|
| always | 每条命令都 fsync | 最高 | 最低 |
| everysec | 每秒 fsync 一次 | 最多丢 1 秒 | 折中 |
| no | 由 OS 决定何时 fsync | 最低 | 最高 |

**AOF 重写**：文件越来越大怎么办？后台 fork 子进程，读取当前数据库状态，用最少的命令重新生成 AOF 文件。重写期间的新写命令先进**重写缓冲区**，子进程完成后追加到新文件，最后原子替换旧文件。

### 第 12 章 事件

Redis 是**单线程事件循环**，事件分两类：

**文件事件**（I/O）：
- 基于 I/O 多路复用（select/epoll/kqueue，编译时自动选择最优实现）
- READABLE 事件 → 客户端连接、命令请求
- WRITABLE 事件 → 命令回复
- 三个处理器：连接应答处理器、命令请求处理器、命令回复处理器

**时间事件**：
- 定时事件（一次性）和周期性事件
- 最重要的时间事件：`serverCron`（默认 100ms 执行一次）

**事件调度**：文件事件和时间事件在同一个循环中交替执行，不会抢占，所以单条命令不能太慢。

### 第 13 章 客户端

每个连接对应一个 `redisClient` 结构，关键字段：

- **套接字描述符**：标识连接
- **输入缓冲区**：存放客户端发来的命令
- **命令参数**：解析后的命令和参数
- **输出缓冲区**：存放待发送的回复（固定大小 + 可变大小两个缓冲区）
- **标志位**：标识客户端状态（事务中、监视器模式等）
- **空转时长**：用于超时清理

**输出缓冲区限制**：
- 硬限制：超过立即关闭
- 软限制：超过一定时间后关闭

**伪客户端**：Lua 脚本执行和 AOF 载入时创建，不关联真实套接字。

### 第 14 章 服务器

**命令执行完整流程**：
1. 客户端发送命令 → 2. 服务器读取并解析 → 3. 查命令表找到实现函数 → 4. 执行预检（参数校验、权限、内存检查等）→ 5. 调用实现函数 → 6. 后续处理（慢查询日志、AOF 追加、命令传播给从服务器）→ 7. 回复客户端

**serverCron 的主要工作**：
- 更新时间缓存和 LRU 时钟
- 统计每秒命令数、内存峰值
- 处理 SIGTERM 信号（优雅关闭）
- 清理超时客户端
- 检查并触发 BGSAVE / BGREWRITEAOF
- 将 AOF 缓冲区写入文件

**服务器启动流程**：initServerConfig（加载默认配置）→ 载入配置文件 → initServer（初始化数据结构、创建事件循环）→ 创建共享对象 → 载入 RDB/AOF 还原数据 → 进入事件循环。

## 第三部分：多机数据库的实现

### 第 15 章 复制

**旧版（SYNC）**：全量复制，每次断线重连都要重新传 RDB，非常浪费资源。

**新版（PSYNC）**：支持**部分重同步**。核心机制：

- **复制偏移量**：主从各自维护，用于判断数据差异
- **复制积压缓冲区**：主服务器维护的固定大小 FIFO 队列，保存最近的写命令
- **服务器运行 ID**：判断从服务器之前连的是不是同一个主服务器

断线重连时：如果偏移量之后的数据还在积压缓冲区内 → 部分重同步；否则 → 全量重同步。

**复制完整流程**：设置主服务器地址端口 → 建立连接 → PING → 身份验证 → 发送端口信息 → PSYNC 同步 → 命令传播。

**心跳检测**：从服务器每秒发送 `REPLCONF ACK <offset>`，用于检测网络状态、辅助 min-slaves 配置、检测命令丢失。

### 第 16 章 Sentinel

Redis 高可用方案，自动监控 + 故障转移。

**核心流程**：

1. **监控**：Sentinel 向主/从服务器定期发 `INFO`，获取拓扑信息
2. **主观下线（SDOWN）**：单个 Sentinel 认为服务器不可达（超过 `down-after-milliseconds`）
3. **客观下线（ODOWN）**：询问其他 Sentinel（`SENTINEL is-master-down-by-addr`），多数同意则确认
4. **选举领头 Sentinel**：基于 Raft 协议的领导选举
5. **故障转移**：
   - 从可用从服务器中选新主（优先级 → 偏移量最大 → 运行 ID 最小）
   - 向其他从服务器发 `SLAVEOF` 指向新主
   - 旧主恢复后变为从服务器

**Sentinel 之间通过频道通信**（`__sentinel__:hello`），互相发现、共享信息。命令连接用于发送命令，不创建订阅连接。

### 第 17 章 集群（Cluster）

Redis 分布式方案，数据分片 + 去中心化。

**核心概念**：

- **16384 个槽（slot）**：每个节点负责一部分槽，通过 `CRC16(key) % 16384` 计算键属于哪个槽
- **MOVED 错误**：客户端请求的键不在当前节点 → 返回 `MOVED <slot> <ip:port>`，客户端重定向
- **ASK 错误**：槽正在迁移中 → 返回 `ASK`，客户端临时重定向（下次还是先问原节点）

**重新分片**：将槽从源节点迁移到目标节点。源节点标记 MIGRATING，目标节点标记 IMPORTING，逐键迁移。

**故障转移**：
1. 节点间通过 Gossip 协议（PING/PONG）交换状态
2. 某节点超时 → 标记为 PFAIL（疑似下线）
3. 超过半数主节点标记 PFAIL → 标记为 FAIL（确认下线）→ 广播 FAIL 消息
4. 下线主节点的从节点发起选举 → 当选后成为新主节点 → 接管槽

**MOVED vs ASK 的区别**：MOVED 表示槽已经永久转移，客户端应更新本地路由；ASK 表示槽正在迁移中，只是本次请求临时重定向。

## 第四部分：独立功能的实现

### 第 18 章 发布与订阅

**频道订阅**：`SUBSCRIBE` 订阅具体频道，信息存在 `pubsub_channels` 字典中（频道名 → 客户端列表）。

**模式订阅**：`PSUBSCRIBE` 用通配符订阅（如 `news.*`），存在 `pubsub_patterns` 链表中。

**发布消息**：`PUBLISH` 时，先查 `pubsub_channels` 发给频道订阅者，再遍历 `pubsub_patterns` 发给匹配的模式订阅者。

注意：Pub/Sub 消息不持久化，客户端离线期间的消息会丢失。

### 第 19 章 事务

**流程**：`MULTI` 开始 → 命令入队（不执行）→ `EXEC` 批量执行 → 按顺序返回结果。

**WATCH 乐观锁**：`WATCH key` 监视键，如果在 `EXEC` 前被其他客户端修改 → 事务取消。本质是 CAS（Compare-And-Swap）思想。

**ACID 特性**：

| 特性 | Redis 事务的表现 |
|------|----------------|
| 原子性 | 命令要么全部入队执行，要么全部不执行（但单条命令失败不回滚其他命令） |
| 一致性 | 入队错误拒绝执行，执行错误跳过但不影响后续命令 |
| 隔离性 | 单线程执行，天然串行隔离 |
| 持久性 | 取决于持久化配置（AOF always 模式最高） |

注意：Redis 事务**不支持回滚**。如果 EXEC 后某条命令执行出错，前面已执行的命令不会撤销，后面的命令继续执行。这是 Redis 的设计选择——认为事务中的错误都是编程错误，不应该在生产环境出现。

### 第 21 章 排序（SORT）

`SORT` 命令功能丰富，选项包括：

| 选项 | 作用 |
|------|------|
| `ALPHA` | 按字典序排序（默认按数值） |
| `ASC / DESC` | 升序 / 降序 |
| `BY *-field` | 按外部键的值排序 |
| `GET *-field` | 获取外部键的值作为返回结果 |
| `LIMIT offset count` | 分页 |
| `STORE key` | 将结果存储到新键 |

**执行顺序**：排序（BY）→ 限制（LIMIT）→ 获取（GET）→ 存储（STORE）→ 返回。

### 第 22 章 二进制位数组

Redis 用**字符串对象**表示位数组，提供四个命令：

- `SETBIT key offset value`：设置指定位
- `GETBIT key offset`：读取指定位
- 定位方式：`byte_index = offset / 8`，`bit_mask = 1 << (offset % 8)`

- `BITCOUNT key [start end]`：统计 1 的个数（汉明重量）
  - Redis 实现使用 **variable-precision SWAR 算法**，比逐位遍历和查表法都快

- `BITOP op destkey key [key ...]`：按位运算（AND/OR/XOR/NOT），结果存入 destkey

### 第 23 章 慢查询日志

- `slowlog-log-slower-than`：超过该阈值（微秒）的命令被记录
- `slowlog-max-len`：日志最大条数
- `SLOWLOG GET [n]`：查看最近 n 条慢查询
- `SLOWLOG RESET`：清空日志
- 每条记录包含：唯一 ID、时间戳、耗时（微秒）、命令及参数

### 第 24 章 监视器

`MONITOR` 命令让客户端实时接收服务器处理的所有命令，用于调试。输出格式：时间戳 + 客户端地址 + 命令。生产环境慎用，会影响性能。
