## 背景

面试中被问到一个问题：设计一个告警日志框架，支持高并发写入和时间窗口查询，假设在京东这样的体量下使用，涉及多服务器、多告警类别。面试官还额外提了一条：要有缓存自动清理机制，1 小时内清理，不能内存泄漏。

当时第一反应是"选个数据库存日志"的事。后来回去越想越觉得不对。写入端是突发的，故障时每秒百万条往里灌；查询端是时间窗口式的，最近 1 小时、今天、本周。这两个方向对存储引擎的要求几乎相反。面试时答得比较散，回来重新理了一遍。

先梳理告警日志的场景特征：

```
告警日志场景特征：
├── 写入特征：爆发式高并发（故障时每秒百万条）
├── 查询特征：时间窗口为主（最近1小时/今天/本周）
├── 数据特征：结构化但字段不固定（不同告警类型字段不同）
├── 时效特征：热数据（24h）查询频繁，冷数据（30天+）归档即可
└── 分析特征：需要聚合统计（错误率趋势、TopN 告警类型）
```

## 1. 架构设计：先定角色，再选实现

之前看过《架构整洁之道》，里面有个思路我觉得挺好：先把架构中每个位置需要的"角色"定下来，再去选具体用什么组件填充。架构定义的是边界和职责，具体技术栈可以晚点再定。

比如面对这个问题，我不会一上来就想"用 Kafka"，而是先意识到这里需要一个削峰缓冲层。至于用 Kafka、Redis Stream、还是进程内的内存队列，取决于量级和可靠性要求。日志量每秒几千条的话，一个 `BlockingQueue` 加异步批量写入就够了，没必要上分布式消息队列。量级到每秒百万条、要求不丢数据，Kafka 才有上的理由。

同样，持久化层需要一个时序数据库，但具体是 InfluxDB、ClickHouse 还是别的，得看团队技术栈和运维能力。不要为了不存在的需求盲目追求某个实现。

按照这个思路，整个系统可以拆成四个角色：

```
┌──────────────────────────────────────────┐
│            削峰缓冲层                      │
│   角色：吸收写入突发流量，保证不丢数据       │
│   候选：Kafka / Redis Stream / 内存队列    │
└──────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────┐
│            实时计算层                      │
│   角色：流式聚合、告警分级、窗口统计        │
│   候选：Flink / 自研消费者聚合             │
└──────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────┐
│            持久化存储层（分级）             │
│   角色：按数据温度分层存储                  │
│   热层 → 内存/缓存（实时查询）              │
│   温层 → 时序数据库（窗口分析）              │
│   冷层 → 对象存储（归档审计）               │
└──────────────────────────────────────────┘
                    ↓
┌──────────────────────────────────────────┐
│            查询层                          │
│   角色：统一查询接口，屏蔽底层存储差异       │
│   候选：各存储自带查询 + Grafana 可视化     │
└──────────────────────────────────────────┘
```

架构定好之后，如果最终评估发现某个产品（比如某个 All-in-One 的数据库）能同时覆盖多个角色，那完全可以合并，这是实现层面的简化，不影响架构层面的清晰性。

## 2. 各层选型推导

架构角色定下来之后，再逐层往下推选型。对我来说，后面这一步最值钱的地方，是把为什么别的方案不合适讲明白，而不是只报一个名字出来。

### 2.1 持久化层：为什么是时序数据库？

先排除不合适的方案。这一步在面试时其实很快，MySQL 和 Redis 的局限一想就知道：

**MySQL** — B+ 树索引写入性能上限约 1 万 TPS，告警高峰期差了两个数量级。时间范围查询随数据量增长会退化。排除。

**Redis** — 写入吞吐够，但容量受限、持久化能力弱。告警日志不能丢，Redis 只能当热数据缓存层，不能当核心存储。

排除这两个后，方向基本就收敛到时序数据库了。告警日志这类数据，天然就是按时间写、按时间查，写多读少，查询也大多围着时间范围转。时序数据库在这些地方正好做过专门优化：时间分区、列式压缩、降采样。

具体选 InfluxDB 还是 ClickHouse，取决于团队现有技术栈。角色定了就行：这里需要一个时序存储引擎。

### 2.2 ES 的定位：不当核心存储，但不能没有

我一开始答的时候没提 ES，觉得时序数据库就够了。面试官追问了一句：实际场景中告警日志经常需要复杂条件联查怎么办？

想了想，ES 当核心存储确实有硬伤：写入吞吐有限（单集群约 10 万 TPS），突发流量会出现 GC 抖动；存储成本也高，需要同时维护正排索引、倒排索引和 doc values。但按关键字搜索、多字段组合过滤、全文检索——这些恰恰是 ES 的强项，时序数据库反而不擅长。

这样一来，ES 更适合放在辅助检索层。核心数据先写时序数据库，再异步同步一份到 ES，专门接复杂查询。

### 2.3 削峰缓冲层：不一定是 Kafka

告警日志的写入是极度不均匀的：平时可能每秒几千条，故障时瞬间飙到每秒百万条。如果应用服务器直连存储层，突发流量会直接打垮数据库。所以中间需要一个缓冲层来削峰。

但削峰层不一定要上 Kafka。选择取决于量级和可靠性要求：

| 量级 | 可靠性要求 | 方案 |
|------|----------|------|
| 每秒千级 | 允许少量丢失 | 进程内内存队列（`BlockingQueue`） |
| 每秒万级 | 不丢数据 | Redis Stream / RocketMQ |
| 每秒百万级 | 不丢数据 + 多消费者 | Kafka |

京东体量下，Kafka 是合理的选择。但中小型系统用 Redis Stream 甚至内存队列就够了。没必要为了架构图好看引入不需要的东西。

## 3. 以京东体量为例的完整架构

确定了各层角色和选型原则后，以京东的量级做一个具体化：

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  应用服务器   │     │  应用服务器   │     │  应用服务器   │
│  (日志埋点)   │     │  (日志埋点)   │     │  (日志埋点)   │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           ↓
              ┌─────────────────────────┐
              │    削峰层：Kafka         │
              │    (100+ partitions)    │
              │   按 server_id 哈希分区  │
              └────────────┬────────────┘
                  ↓                 ↓
        ┌────────────────┐  ┌─────────────────┐
        │ 计算层：流式聚合 │  │ 异步同步         │
        │ 实时窗口/告警   │  │ → ES（复杂检索）  │
        └───────┬────────┘  └─────────────────┘
                ↓
  ┌──────────────────────────────────────────┐
  │          存储层（分级）                    │
  │  ┌────────┐  ┌──────────┐  ┌──────────┐  │
  │  │ 热数据  │  │  温数据   │  │  冷数据   │  │
  │  │ Redis  │  │ 时序数据库 │  │   OSS   │  │
  │  │ (24h)  │  │  (30天)  │  │  (归档)  │  │
  │  └────────┘  └──────────┘  └──────────┘  │
  └──────────────────────────────────────────┘
                    ↓
  ┌──────────────────────────────────────────┐
  │          查询层                           │
  │    各存储自带查询接口                     │
  │    Grafana（可视化大盘）                   │
  └──────────────────────────────────────────┘
```

和之前的纯架构图相比，这里多了一条到 ES 的异步同步链路。告警日志先落时序数据库保证写入吞吐，再异步写一份到 ES 支持关键字搜索和多条件联查。

## 4. 高并发写入优化

写入链路每一环都可以优化。说起来这些策略单独看都不复杂，但组合起来效果挺明显的：

| 环节 | 策略 | 说明 |
|------|------|------|
| 客户端聚合 | 100ms 或满 1000 条批量发送 | 减少网络 IO，提升吞吐 |
| 分区策略 | 按 `server_id` 哈希 | 同一台服务器的日志落在同一分区，保证时序 |
| 批量写入 | 5000 条/批写入时序数据库 | 批量写入性能远优于逐条插入 |
| 故障降级 | MQ 堆积 → 本地磁盘缓冲 → 恢复后补录 | 保证不丢数据 |
| 背压机制 | 写入超过处理能力时客户端降速 | 防止雪崩 |

这里有个设计思路是在客户端就开始做聚合，不把逐条日志直接扔给 MQ。每台应用服务器在本地先做 100ms 的微批（micro-batch），凑够一批再发送。这一步就能把网络请求量降低两个数量级。

## 5. 时间窗口查询设计

不同的查询场景对延迟的容忍度不同，需要分别对待：

### 场景 A：实时大盘（最近 1 分钟）

```
数据源：流式消费者实时聚合后写入 Redis
延迟：< 100ms
```

实时大盘查询频率极高（每秒刷新）但只需要聚合值，不需要查原始日志。直接从实时聚合结果里读。

### 场景 B：故障排查（某服务器最近 1 小时原始日志）

```sql
SELECT * FROM alarm_logs
WHERE server_id = 'srv-001'
  AND ts > NOW() - INTERVAL 1 HOUR
ORDER BY ts DESC;
```

典型的定位式查询：知道具体的服务器和时间范围，需要原始日志。时序数据库的时间分区在这种场景下表现很好。

### 场景 C：关键字搜索 + 多条件联查

```
"找出最近 1 小时内，order-service 中包含 'timeout' 关键字的所有 error 级别告警"
```

这种场景就是 ES 发挥作用的地方。时序数据库不擅长全文检索和复杂条件组合，而 ES 的倒排索引天然适合这类查询。

### 场景 D：趋势分析（本周错误率，按小时聚合）

大量数据的聚合查询，如果每次都从原始数据算会很慢。优化手段是物化视图预聚合：数据写入时就按小时/天粒度预先计算好聚合值，查询时直接读。

### 场景 E：跨月审计

```
数据源：时序数据库过期数据自动转 Parquet 存 OSS
查询：按需加载，或者用支持 Parquet 的查询引擎直接扫
延迟：秒级（可接受）
```

审计查询频率低但时间跨度大，冷数据用 Parquet 格式存对象存储，省成本。需要查的时候加载出来就行，不用常驻索引。

## 6. 关于缓存自动清理：我觉得可以先不做

面试官的第四条要求是：缓存自动清理机制，1 小时内清理，不能内存泄漏。

我当时犹豫了一下，觉得这个需求需要推敲。

告警日志的查询以时间窗口为主，缓存命中率天然不高。原因在于：

1. **时间窗口是滑动的**。"最近 1 小时"这个查询，每一秒执行的结果都不一样。缓存一个时间窗口的结果，下一秒就失效了。
2. **如果做时间粒度切分**（比如按分钟粒度缓存，查询时汇总），就引入了粒度对齐和聚合的复杂度，缓存管理的成本不一定比直接查时序数据库低。
3. **如果只缓存完全相同参数的查询**（精确匹配），那命中率取决于用户查询的重复度——告警排查场景下，用户查的通常是不同服务器、不同时间段，重复率很低。

还有一点我后来想得更清楚：既然核心存储已经选了时序数据库，那时间窗口查询这件事本来就是它擅长的。时间分区、列式压缩、预聚合这些优化都已经在里面了。大部分查询本来就在毫秒到秒级，再额外罩一层缓存，收益不一定高，倒是会把一致性问题一并带进来。

所以我的建议是：先不做查询缓存，上线后观察实际查询延迟。如果确实发现某些高频查询延迟不达标，再针对性地加。这比一开始就设计一套通用的缓存清理机制务实得多。实时大盘那种场景，前面已经用流式聚合 + Redis 解决了，本身就是一种针对性的缓存。

如果一定要做通用缓存，可以用带 TTL 的 LRU 缓存（比如 Caffeine 设置 `expireAfterWrite(1, HOURS)`），key 用查询参数的哈希，过期自动淘汰。实现不复杂，但问题在于收益不确定——这就是为什么我觉得应该等真实场景的性能数据出来再决定。

## 7. 多级存储与生命周期

```
原始日志（高并发写入）
    ↓
┌──────────────────────────────────────────────┐
│  MQ 缓冲层 (Kafka, 保留 3 天防消费延迟)       │
└──────────────────────────────────────────────┘
    ↓                         ↓
┌──────────┐           ┌─────────────┐
│  流式聚合 │           │  时序数据库   │
│ (实时窗口) │           │ (热/温存储)   │
│  告警分级  │           │  原始日志     │
└────┬─────┘           └──────┬──────┘
     ↓                        ↓
┌──────────┐           ┌─────────────┐
│  Redis   │           │ 自动过期策略  │
│ (热查询)  │           │  7天 → 转 OSS │
│ 秒级延迟  │           │ 30天 → 删除   │
└──────────┘           └──────┬──────┘
                              ↓
                       ┌─────────────┐
                       │ OSS (Parquet)│
                       │  冷存储/审计  │
                       └─────────────┘
```

存储成本估算（假设日增 10 亿条，每条 500 字节）：

| 层级 | 日增量 | 说明 |
|------|-------|------|
| 原始数据 | 500 GB/天 | 未压缩 |
| 时序数据库存储 | 50 GB/天 | 约 10:1 压缩 |
| OSS 归档 | 25 GB/天 | 约 20:1 压缩 |
| **30 天总计** | **热存储 ~1.5 TB + 冷存储 ~750 GB** | — |

分层存储其实是按访问频率选存储介质。最近 24 小时的聚合数据放内存（Redis），最近 30 天的原始数据放时序数据库（SSD），更早的数据放对象存储。每一层的存储成本差一个数量级。

## 8. 高可用与容灾

```
架构冗余设计：
├── 缓冲层：MQ 多副本，自动故障转移
├── 计算层：流式消费者做 checkpoint，失败自动重启
├── 存储层：
│   ├── 时序数据库：多节点集群，数据自动同步
│   ├── Redis：主从 + 哨兵，或 Cluster 模式
│   ├── ES：多副本分片
│   └── OSS：11 个 9 持久性，跨可用区复制
└── 查询层：多实例 + 负载均衡
```

告警日志系统有个特殊的高可用要求：**系统自身不能成为告警的盲区。** 系统出故障的时候，恰恰是告警日志量最大的时候。日志系统这时候也挂了，就成了"最需要日志的时候没有日志"。这个场景面试官没有明确问，但我觉得是做日志系统绕不开的问题。

容灾设计里，我会特别盯极端场景：

| 故障场景 | 处理策略 |
|---------|---------|
| 某机房网络中断 | 心跳超时 → 客户端路由至备用机房 MQ → 恢复后断点续传补录 |
| MQ 集群不可用 | 客户端降级写本地磁盘 → MQ 恢复后自动补录 |
| 时序数据库写入超时 | MQ 消息堆积不丢数据 → 恢复后消费积压消息 |
| 数据重复写入 | 按时间戳 + 唯一 ID 去重，幂等写入 |

## 9. 关键设计决策总结

| 决策点 | 架构角色 | 京东体量下的选择 | 说明 |
|-------|---------|----------------|------|
| 削峰缓冲 | 消息队列 | Kafka | 量级决定，小规模可用 Redis Stream |
| 实时计算 | 流式消费/聚合 | Kafka Consumer + 窗口聚合 | 量大可以上 Flink，我没深入用过 |
| 核心存储 | 时序数据库 | InfluxDB / ClickHouse 等 | 根据团队技术栈选择 |
| 复杂检索 | 搜索引擎 | Elasticsearch | 全文检索 + 多条件联查 |
| 热数据缓存 | 内存存储 | Redis | 实时大盘，亚秒级延迟 |
| 冷数据归档 | 对象存储 | OSS + Parquet | 低成本，按需查询 |
| 统一查询 | 各存储自带接口 | Grafana 做可视化 | 不一定需要联邦查询引擎 |

这张表左两列是架构层面的决策，在需求分析阶段就能定下来。右两列是实现层面的选择，可以推迟到技术评估阶段。

## 10. 和商品 RAG 系统放一起看

这两个系统的数据特征差得挺远，架构自然也不一样：

| 维度 | 商品 RAG | 告警日志框架 |
|------|---------|------------|
| 数据特征 | 半结构化（文本 + 属性） | 结构化时序数据 |
| 核心操作 | 向量相似度检索 | 时间范围扫描 |
| 写入模式 | 批量更新（小时级） | 流式写入（秒级，突发百万级） |
| 查询模式 | 探索式（语义模糊） | 定位式（时间精确） |
| 存储优化 | 向量索引 (HNSW) | 时间分区 + 列式压缩 |
| 一致性要求 | 最终一致即可 | 强一致（不能丢数据） |
| 核心矛盾 | 语义理解 vs 精确匹配 | 写入吞吐 vs 查询延迟 |

两个系统倒是有个共同点：都做了分层。但分层的维度不同。RAG 系统围绕检索精度分层（粗排 → 精排 → 重排），日志系统围绕数据温度分层（热 → 温 → 冷）。分层这个动作不稀奇，搞清楚"按什么维度分"才是关键。
