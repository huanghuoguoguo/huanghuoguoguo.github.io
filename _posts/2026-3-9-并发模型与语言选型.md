看完《Java 并发编程的艺术》之后，我脑子里先冒出来的是另一个更别扭的问题：如果 Java 这一套已经讲得这么细了，那 Go 的 goroutine、Rust 的 async/await、Node.js 的事件循环、Python 的 GIL，到底该放到一张什么图里看？它们真的是在解同一道题吗？

我当时还会顺手往下追一层。Python 和 Node.js 都老被说成"单线程"，怎么一到 Web 服务这里又没那么弱？Redis 靠单线程加 epoll 就能扛住很高的 QPS，这是不是在提醒我，很多时候我盯错地方了？我以前总觉得线程模型是第一位，后来越看越怀疑，真是这样吗？

再往后看，问题就更乱了。Java 21 有了虚拟线程，字节这些年也在大规模用 Go，Rust 在基础设施层的声量越来越高。那 Java 现在到底是在退，还是只是不再通吃所有场景了？生态和服务器成本，真到了要二选一的时候吗？

这些问题我当时没有一口气想通。这里只是把后来慢慢串起来的一点理解记下来。

## 六种语言的并发模型

先把各语言的并发模型摊开来看：

| 语言 | 并发单元 | 调度模型 | 单个并发单元内存 | GC | 典型 Web 框架 |
|------|---------|---------|----------------|-----|-------------|
| Java（平台线程） | Thread | 1:1（一个线程对应一个内核线程） | ~1MB（栈默认 1MB） | 有（G1/ZGC） | Spring Boot |
| Java（虚拟线程） | VirtualThread | M:N（多个虚拟线程映射到少量平台线程） | ~几 KB（初始栈，按需增长） | 有 | Spring Boot 3.2+ |
| Go | goroutine | M:N（GMP 调度器） | ~2KB（初始栈，按需增长到 1GB） | 有（低延迟 GC） | Gin、Echo |
| Rust | tokio task | M:N（tokio 异步运行时） | ~几百字节（Future 状态机） | **无** | Actix-web、Axum |
| C++ | std::thread | 1:1 | ~1-8MB（取决于 OS 配置） | **无** | 无主流框架，通常自建 |
| Python | coroutine（asyncio） | 1:N（单线程事件循环） | ~几 KB | 有（引用计数 + 分代 GC） | FastAPI、uvicorn |
| Node.js | Promise/callback | 1:N（单线程事件循环 + libuv） | ~几 KB | 有（V8 GC） | Express、Fastify |

几个关键差异值得展开。

### 1:1 vs M:N

Java 的平台线程和 C++ 的 std::thread 是 1:1 模型。你每创建一个线程，操作系统就得分一个内核线程出来。在 Linux 上，这一步通常就是走 `clone()`。地址空间是共享的，但每个线程还是有自己的栈。内核栈虽然只有 8-16KB，用户态栈默认却常常有 1MB，这才是线程数量上不去的硬约束。

Go 和 Java 虚拟线程走的是 M:N 模型——大量用户态并发单元映射到少量内核线程上。Go 的 GMP 调度器（Goroutine-Machine-Processor）自己管理调度，遇到系统调用时能把阻塞的 goroutine 从 M（内核线程）上摘下来，换一个可运行的上去。Java 虚拟线程做的事情本质相同，只是实现在 JVM 里。

Rust 的 tokio 也是 M:N，但用的是 async/await 编译为状态机的方式——编译器把异步函数转换成一个 `Future` 状态机，每个 `.await` 点就是一个状态切换，不需要保存完整的栈帧。所以 tokio task 的内存开销可以做到几百字节级别。

### 1:N（单线程事件循环）

Python 的 asyncio 和 Node.js 的事件循环是 1:N 模型——只有一个线程，通过事件循环调度所有协程/回调。这看起来最"简陋"，但在 IO 密集型场景下性能出奇地好。原因在下一节展开。

## 为什么单线程也能有好的 Web 性能？

我后来会先盯住一个很朴素的事实看：**典型的 Web 请求，很多时间都耗在等 IO 上**。

一个常见的 Web 请求链路：

```
接收 HTTP 请求 → 解析参数 → 查数据库（等 IO）→ 调远程服务（等 IO）
→ 组装响应 → 返回
```

真正让 CPU 忙起来的，往往只有"解析参数"和"组装响应"这两步，可能加起来不到 1ms。可数据库一查就是 5ms，远程服务一调又是 10ms。那线程中间这十几毫秒在干什么？很多时候就是挂着，等。

如果还是老老实实一个请求配一个线程，1000 个并发请求就得先准备 1000 个线程。每个线程按 1MB 栈算，光栈空间就先吃掉 1GB。可这些线程大部分时间并不干活，只是在等数据库，等网络。那我为什么还要给它们一人发一张长期床位？

### Redis 的启示

Redis 的单线程模型加上 IO 多路复用（epoll）能抗住几十万 QPS。这里最让我对上号的是下面几件事：

1. **所有操作都在内存中**，单次操作微秒级，CPU 不是瓶颈
2. **epoll 让一个线程同时监听数千个连接**，哪个连接有数据就处理哪个，不需要为每个连接分配一个线程
3. **避免了多线程的锁竞争和上下文切换开销**

我一开始看到这里，直觉其实是抗拒的：单线程怎么会快？可回头看，如果请求大部分时间都耗在等 IO 上，那一个线程盯着很多连接轮着处理，反而很合理。一定能说它就是最优解吗？我不太敢这么写死。至少在 Redis 这种内存操作为主、单次处理很短的场景里，这套模型确实非常顺。

### Node.js 和 Python 的 Web 性能

Node.js 和 Python 的 asyncio 走的路子其实很像：

- **Node.js**：V8 引擎 + libuv 事件循环。所有 IO 操作（网络、文件、DNS）都通过 libuv 异步化，回调/Promise 注册到事件循环，IO 完成后触发回调。单线程处理所有请求，CPU 密集型任务用 Worker Threads 分离
- **Python asyncio**：单线程事件循环 + 协程。`await` 挂起当前协程，让出控制权给事件循环处理其他协程。uvicorn（ASGI 服务器）+ FastAPI 的组合在 IO 密集型场景下性能不输多线程方案

所以 Node.js 和 Python 在 Web 场景里跑得不差，我现在更愿意先从这里理解：很多服务主要是等得太久。Python 解释器当然慢，可如果请求大部分时间都卡在数据库和网络上，这个慢到底会不会马上变成决定性问题？很多时候还真不会。

我后来读到 AIO/BIO/NIO、再往下看到 select/poll/epoll，慢慢才觉得这些东西能连起来看。它们不一定长得像一家人，但都在绕同一个弯子：别让 CPU 傻等 IO。事件循环只是把这件事做得更彻底一点。

### 那什么时候单线程不够用？

那单线程什么时候开始不对劲？通常就是请求里真的混进了**CPU 密集型计算**。比如图片处理、加密解密、复杂规则引擎、机器学习推理。只要有一个慢计算把事件循环卡住，后面的请求就只能排队。

Node.js 会拉 Worker Threads，Python 会开多进程，或者把热点计算塞进 C 扩展里。我以前会把这些手段理解成"也能解决"。后来越看越觉得，它们更像是在给事件循环模型补边界。要是业务从头到尾就是算力活儿，那为什么不一开始就选更容易把多核吃满的语言？

## Java 21 虚拟线程解决什么问题？

### 平台线程的代价

Java 的线程是 1:1 映射到内核线程的，每个线程的用户态栈默认 1MB。算一笔账：

| 并发连接数 | 线程数 | 栈内存占用 | 实际 CPU 利用 |
|-----------|--------|-----------|-------------|
| 1,000 | 1,000 | ~1GB | < 5%（大部分时间在等 IO） |
| 10,000 | 10,000 | ~10GB | < 5% |
| 100,000 | 理论需要 100,000 | ~100GB（不可能） | < 5% |

一台 16GB 内存的服务器，光是线程栈就会占掉大半内存，真正留给业务堆的空间反而不多。而且这些线程绝大部分时间都在 `park()` 等 IO。`LockSupport.park()` 干的事很直接，就是先把线程挂起来，等内核以后再唤醒它。一万个线程这么挂着，内核调度本身也会开始变重。

所以 Java 社区后来才会折腾 WebFlux、Vert.x 这些响应式框架，想用更少的线程扛更多并发。我对这条路一直有点别扭。它当然能解问题，可一个简单 CRUD 一旦改成响应式写法，代码怎么就突然拧起来了？异常栈不好读，调试也难，写的人和接手的人都要重新适应。

### 虚拟线程：同步的写法，异步的性能

我现在更愿意把 Java 21 的虚拟线程（Project Loom）理解成一件很具体的事：尽量别逼开发者把同步代码全改成异步回调，但又别继续背着平台线程的成本往前跑。

```java
// 传统方式：每个请求占用一个平台线程
ExecutorService executor = Executors.newFixedThreadPool(200);

// 虚拟线程：每个请求一个虚拟线程，底层只用少量平台线程
ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
```

虚拟线程的初始栈只有几 KB，按需增长。当虚拟线程执行到 IO 操作（比如 JDBC 查询、HTTP 调用）时，JVM 会自动把它从平台线程上**卸载**（unmount），让平台线程去执行其他虚拟线程。IO 完成后，再**挂载**（mount）回某个平台线程继续执行。

写到这里，我脑子里最自然的对照物还是 Go 的 goroutine。两边最后都走向了 M:N，只是 Go 一开始就这么设计，Java 是绕了一大圈才回来。Java 这一步对我最有吸引力的地方，是它尽量没逼你把老代码推倒重来。很多同步代码换个 Executor 就能先跑起来，这一点很现实。

| 方案 | 写法 | 并发能力 | 心智负担 | 生态兼容 |
|------|------|---------|---------|---------|
| 线程池（平台线程） | 同步 | 千级 | 低 | 完全兼容 |
| WebFlux / Vert.x | 响应式 | 万级+ | **很高** | 需要专用库 |
| 虚拟线程 | 同步 | 百万级 | 低 | 几乎完全兼容 |
| Go goroutine | 同步 | 百万级 | 低 | 语言原生 |

虚拟线程当然也有边界。它主要解决的还是 IO 密集型场景下的扩展性，不会把 CPU 密集型任务凭空变快。还有一些现实限制，比如 `synchronized` 块里的 IO 可能会 pin 住平台线程，这时候还得回头改锁。可即便这样，我还是会觉得这条路值得看。thread-per-request 这套老办法的问题，Java 总算肯自己正面接了，而不是继续让业务代码去消化。

## Go 和 Rust 的 Web 性能为什么高？

### Go：我为什么总觉得它很适合微服务

我以前看 Go，总容易把它理解成"语法简单的 C"。后来看并发这一层才慢慢意识到，它很多地方从一开始就是朝着服务端场景去长的：

- **goroutine 初始栈 2KB**，按需增长到 1GB。一台 16GB 的机器理论上可以跑几百万个 goroutine
- **GMP 调度器**（Goroutine-M-Processor）：用户态调度，goroutine 切换只需要保存/恢复几个寄存器，不需要陷入内核。系统调用阻塞时自动把 goroutine 从 M 上摘下来
- **编译为原生机器码**，没有 JVM 的启动开销和 JIT 预热
- **GC 延迟极低**：Go 的 GC 设计目标是 STW（Stop-The-World）不超过 1ms，牺牲一定的吞吐量换极低延迟
- **编译速度极快**：几十万行代码几秒编译完，开发和部署效率高

Go 那套"少即是多"我一开始并不完全买账，甚至会觉得它有点抠。但真放到大团队里看，语法简单、概念少、没有太多花样，反而容易把代码风格压平。微服务一多，这种朴素有时比精巧更值钱。

### Rust：零成本抽象的极致

Rust 我到现在也不敢说自己吃透了，所以这里只记我比较能对上号的部分。它老讲零成本抽象，放到 Web 并发里，大概可以先这样理解：

- **没有 GC**：所有权系统在编译期管理内存，运行时零 GC 暂停。对于延迟敏感的场景（P99 延迟），这是巨大的优势
- **tokio 异步运行时**：async/await 编译为状态机，每个 task 只有几百字节。M:N 调度，work-stealing 算法（和 Java 的 Fork/Join 工作窃取、Caffeine 的设计思路一脉相承）
- **零成本抽象**：泛型单态化、迭代器链编译后展开成单循环、trait 的静态派发。抽象层不引入运行时开销
- **编译期内存安全**：不需要运行时检查空指针、数组越界，少了一层开销

Rust 在 TechEmpower Web Framework Benchmarks 里常常排得很靠前。Actix-web、Axum 这些框架的吞吐量也确实很猛。可我每次看到这种表，都会多问一句：这里测出来的到底是框架本身，还是业务系统最后会遇到的瓶颈？

Rust 的难也是真的。所有权、生命周期、借用检查器，这些东西第一次碰的时候很难不烦。为什么很多人最后又认可它？可能就是因为它把很多运行时会炸的问题，提前拦在编译期了。只是这笔账到底划不划算，还是得看团队愿不愿意把心智负担前置。

### 数据说话

在相同硬件上（典型的 4 核 8GB 云主机），不同语言 Web 框架处理简单 JSON 响应的吞吐量大致范围：

| 框架 | 语言 | 吞吐量（req/s）| 内存占用 |
|------|------|-------------|---------|
| Actix-web | Rust | 400,000+ | ~10MB |
| Axum | Rust | 350,000+ | ~10MB |
| Gin | Go | 200,000+ | ~20MB |
| Fastify | Node.js | 60,000+ | ~80MB |
| Spring Boot（虚拟线程） | Java | 50,000+ | ~300MB |
| Spring Boot（平台线程） | Java | 30,000+ | ~500MB |
| FastAPI + uvicorn | Python | 15,000+ | ~100MB |

这些数字会跟场景一起飘。JSON 序列化、数据库查询、文件上传，结果都会变。要是只先记一个量级印象，我会记 Rust、Go 在吞吐量和内存占用上通常比 Java 轻不少。

不过 benchmark 看到这里也该收一收。裸框架跑分和真实业务不是一回事。数据库、缓存、网络延迟，这些东西一进来，框架之间的差距很容易被淹掉。一个 Spring Boot 服务后面挂着 Redis 和 MySQL，最后先卡住的，常常还是那条慢 SQL。

## 大厂语言选型：不是技术问题，是组织问题

### 字节选 Go

字节跳动在 2016 年前后大规模从 Python 转向 Go。核心原因不只是"Go 比 Python 快"（虽然确实快），更在于下面这些工程因素：

- **微服务数量爆炸**：抖音、头条、飞书这些产品线加起来，微服务已经是万级规模。Go 编译快，部署镜像也小，在容器化环境里这个优势非常实在
- **语言简单，上手快**：字节的工程师流动性大，新人多。Go 的语法极简，一周上手，一个月能写生产代码。Java 的 Spring 全家桶新人要学几个月
- **goroutine 天然适合微服务**：微服务之间大量 RPC 调用，本来就是 IO 密集型，goroutine 的轻量并发刚好合适
- **自研基础设施**：字节自己做了 Kitex（RPC 框架）、Hertz（HTTP 框架），Go 在内部也就越用越顺手

### 阿里选 Java

阿里是中国 Java 生态的最大推手。原因也不纯粹是技术层面的：

- **中间件生态**：Dubbo（RPC）、RocketMQ（消息队列）、Nacos（配置中心）、Sentinel（流控）、Seata（分布式事务）——这套中间件经过双十一验证，是阿里技术体系的核心资产。这些中间件都是 Java 写的，用 Java 接入最自然
- **人才池**：中国的 Java 开发者数量远超其他语言。阿里每年校招几千人，用 Java 意味着最大的候选人池
- **历史积累**：淘宝从 PHP 转 Java 是 2008 年的事情，十几年的代码资产和运维经验不可能轻易丢弃
- **JVM 的成熟度**：G1、ZGC 等 GC 的调优经验，JFR/JMC 的监控能力，jps/jstat/jmap/jstack 这套完整的诊断工具链——这些是其他语言生态目前无法匹敌的

### 腾讯选 C++

腾讯的 C++ 传统可以追溯到 QQ 时代：

- **游戏引擎**：腾讯是全球最大的游戏公司，游戏服务器对延迟要求极高（几毫秒级），GC 暂停不可接受。C++ 的确定性内存管理是刚需
- **QQ/微信的底层通信**：亿级用户的即时通讯，协议层和网络层用 C++ 实现，追求极致的内存效率和延迟控制
- **历史和人才**：早期腾讯的技术栈就是 C++，培养了大量 C++ 工程师，形成了惯性

但腾讯内部也在大量使用 Go（微服务层）和 Java（企业级业务），不是纯 C++ 栈。

### 语言选型的本质

把三家放在一起看：

| 公司 | 主力语言 | 核心诉求 | 语言特性匹配 |
|------|---------|---------|------------|
| 字节 | Go | 大量微服务、快速迭代、新人多 | 简单、编译快、部署小、goroutine |
| 阿里 | Java | 中间件生态、大规模系统、稳定性 | 生态成熟、JVM 可观测、人才池大 |
| 腾讯 | C++ | 游戏引擎、即时通讯、极致延迟 | 无 GC、确定性内存、硬件级控制 |

把这几家放在一起看，我现在不太想直接下"谁更先进"这种判断了。更像是在看另一件事：你的组织到底被什么东西卡着？团队规模、招人难度、历史代码、业务类型、基础设施投入，这些约束往往比语言跑分更先落到地上。

## Java 架构转型：生态重要还是成本重要？

### Java 的成本问题是真实的

在云原生和 Kubernetes 的时代，Java 的"重"变成了实实在在的成本问题：

- **启动慢**：Spring Boot 应用启动通常 5-15 秒，Serverless 场景中冷启动延迟不可接受
- **内存占用高**：一个简单的 Spring Boot 服务至少 200-300MB RSS，同等功能的 Go 服务可能只要 20-30MB。10 倍的内存差距意味着同等硬件跑的实例数差 10 倍
- **镜像大**：JDK 基础镜像 200MB+，Go 的静态二进制几 MB。在大规模容器调度下，镜像拉取时间是问题

当微服务数量从几十涨到几千，这些"小问题"就变成了大账单。

### Java 的应对

Java 社区并没有坐以待毙：

| 方案 | 解决什么 | 代价 |
|------|---------|------|
| GraalVM Native Image | 编译为原生二进制，启动 < 100ms，内存减少 50%+ | 编译慢、反射受限、生态兼容性问题 |
| Quarkus / Micronaut | 编译期依赖注入，减少运行时反射 | 需要学习新框架，和 Spring 不兼容 |
| Spring Boot 3 + AOT | 预处理 Bean 定义，减少启动时反射开销 | 改进有限，仍然比 Go/Rust 重 |
| 虚拟线程 | 解决并发扩展性问题 | 不解决内存和启动问题 |
| CRaC（Coordinated Restore at Checkpoint） | 快照恢复，秒级启动 | 需要 OS 支持，状态管理复杂 |

GraalVM Native Image 算是里面最激进的一条路。把 Java 直接编译成原生二进制，启动和内存都能压下来，看起来很诱人。可问题马上也来了：Java 那些好用又麻烦的运行时能力怎么办？反射、动态代理、ClassLoader，这些东西一收紧，Spring 那套熟悉的魔法还能剩多少？

### 要不要换语言？

我现在的判断还是偏保守：**大部分场景未必需要急着换。**

1. **很多性能问题并不先发生在语言层**。先观测、再定位，这个顺序我现在反而更信。多数 Java 服务真跑慢了，先看到的往往是数据库查询、网络延迟、缓存命中率，不是框架本身。把 Spring Boot 换成 Gin，MySQL 的慢查询也不会自己变快
2. **生态经常比语言本身更难替换**。Spring Cloud、MyBatis、中间件客户端、监控工具链，这些东西平时不显山不露水，真要迁的时候才发现成本很重。是不是很多人低估了重建一套配套设施的代价？
3. **最后拍板的常常还是人**。一个 20 人的 Java 团队转 Go，培训、招聘、代码评审、线上值班经验，这些都要重新积累。服务器账单当然是真的，但组织切换的摩擦也是真的

但在特定场景下，转型是值得的：

- **Serverless / FaaS**：冷启动延迟是硬约束，GraalVM 或 Go 更合适
- **超大规模微服务**（数千个服务）：内存和镜像的成本累积到不可忽视
- **基础设施层**（代理、网关、消息队列）：Rust/Go 的性能和内存优势在这里最明显
- **边缘计算**：资源受限环境下 Java 的 JVM 开销不可接受

我不太想把这件事写成"Java 不行了"。更像是场景在分化。Java 还是很强，只是没法再理所当然地覆盖所有层。以后会不会越来越像这样：业务层继续偏 Java，基础设施层更多用 Go 或 Rust，数据和 AI 场景还是离不开 Python？我现在觉得这个方向是有可能的。

## 回到开头的问题

回到开头那几个问题，我现在大概会这样回答自己。

Web 多半是 IO 密集型任务吗？至少我现在遇到的大多数业务服务，是。CPU 真正忙的时间没我一开始想得那么长，所以 Python、Node.js 这种事件循环模型才会在 Web 场景里显得很能打。

Java 虚拟线程到底解决了什么？我会把它理解成：Java 终于在尽量保住同步写法的前提下，去补 thread-per-request 这套模型的扩展性缺口。它不像 Go 那样从第一天就把并发当中心思想，但至少现在开始补课了。

Go 和 Rust 为什么快？前者是把轻量并发、原生编译、低 GC 延迟捏得很顺；后者是把运行时成本继续往下压，把很多约束提前放到编译期。可这些优势一落到真实系统里，究竟还能剩下多少？还是要看你的瓶颈是不是就在语言和框架这一层。

生态重要还是成本重要？我现在更愿意先问规模，再问组织。几十个服务的时候，生态和人手常常比那点机器成本更要命。几千个服务的时候，成本账才会越来越难装作没看见。

我现在真正记住的，是并发模型背后那套取舍终于比以前清楚了一点。至于最后选哪个，可能还是老问题：你手里的系统，最先痛的是哪里？
