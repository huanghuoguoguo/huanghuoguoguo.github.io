面试的时候被问到在 OB 实习期间做的可观测 Agent 的存储层。我说为了不引入重量级依赖，自己手写了一个小型 LSM-Tree。面试官追问了 memtable 的数据结构、布隆过滤器、compaction 策略，我发现自己有些地方说得不对，有些地方做的设计现在回头看也不合理。

这篇文章把当时的设计、面试时说错的地方、以及事后的反思整理一下。

## 背景：为什么要自己写存储

那个可观测 Agent 的核心功能是：挂在 QA 自动化 Job 上，采集运行时指标（CPU、内存、请求延迟等），做时间窗口聚合（最近 1 分钟平均值、最近 5 分钟 P99 等），然后输出报告。

存储需求其实很简单：

```
写入：每秒几百到几千条指标数据，追加写，没有更新和删除
查询：
  1. 点查：某个 metric 在某个时刻的值
  2. 窗口查询：某个 metric 在最近 N 分钟内的所有数据点
聚合：对窗口内的数据做 avg / max / min / p99
生命周期：Job 跑完就结束，通常几十分钟到几小时
```

第一反应是用 SQLite 或者 DuckDB。但当时的环境只有 Python 3.6，DuckDB 要求 Python 3.10+。SQLite 能用，但时序数据的聚合查询写起来比较别扭，而且 B+ 树对纯追加写入场景有不必要的开销。

当时想的是：需求这么简单——只有追加写入，没有删除，查询就两种模式——不如自己写一个轻量的存储引擎。LSM-Tree 天然适合写多读少的场景，而且核心逻辑不复杂，砍掉不需要的功能应该能写得很小。

现在回头看，这个判断的前半段没问题，后半段太乐观了。

## 标准 LSM-Tree 长什么样

先理清标准 LSM-Tree 的组件，再说我做了哪些简化。

```
                    写入
                     │
                     ▼
              ┌─────────────┐
              │  MemTable   │  ← 内存中的有序结构（跳表/红黑树）
              │  (可变)      │
              └──────┬──────┘
                     │ 达到阈值，冻结为 Immutable MemTable
                     ▼
              ┌─────────────┐
              │  Immutable  │  ← 只读，等待刷盘
              │  MemTable   │
              └──────┬──────┘
                     │ flush 到磁盘
                     ▼
         ┌───────────────────────┐
         │   Level 0 (SSTable)   │  ← 可能有重叠的 key 范围
         ├───────────────────────┤
         │   Level 1 (SSTable)   │  ← key 范围互不重叠
         ├───────────────────────┤
         │   Level 2 (SSTable)   │
         └───────────────────────┘
                     │
              Compaction 线程
              持续合并、去重、整理
```

几个关键组件：

| 组件 | 作用 | 典型实现 |
|------|------|---------|
| MemTable | 内存中缓存写入，保持有序 | 跳表（LevelDB/RocksDB）或红黑树 |
| WAL | 写前日志，防止 MemTable 数据丢失 | 顺序追加的日志文件 |
| SSTable | 磁盘上的有序数据文件 | 排好序的 key-value 对 + 稀疏索引 |
| 布隆过滤器 | 快速判断某个 key 是否可能存在于某个 SSTable | 位数组 + 多哈希函数 |
| Compaction | 合并多层 SSTable，回收空间，减少读放大 | 分层合并（Leveled）或大小分层（Size-Tiered） |

## 我的简化版：砍了什么，改了什么

### MemTable：跳表 + HashMap

面试时我说内存缓存用的 HashMap。这是不准确的——实际上用了**跳表和 HashMap 两个结构**。

为什么需要两个？因为有两种查询模式：

```
点查：metric_name + timestamp → value
      → HashMap，O(1)

窗口查询：metric_name + [t_start, t_end] → [value1, value2, ...]
      → 跳表，按 timestamp 排序，O(log n) 定位起点，然后顺序扫描
```

HashMap 以 `(metric_name, timestamp)` 为 key，返回单个值。跳表以 timestamp 为排序键，支持范围扫描，用来返回一个时间窗口内的所有数据点。

```python
class MemTable:
    def __init__(self):
        self.point_index = {}          # {(metric, ts): value}  点查
        self.range_index = SkipList()  # 按 ts 排序，窗口查询

    def put(self, metric, ts, value):
        self.point_index[(metric, ts)] = value
        self.range_index.insert(ts, (metric, value))

    def point_get(self, metric, ts):
        return self.point_index.get((metric, ts))

    def range_get(self, metric, t_start, t_end):
        # 跳表定位到 t_start，顺序扫描到 t_end
        results = []
        for ts, (m, v) in self.range_index.range(t_start, t_end):
            if m == metric:
                results.append((ts, v))
        return results
```

面试时只提了 HashMap，漏掉了跳表。如果只有 HashMap，窗口查询就得遍历所有 key 去过滤时间范围，复杂度退化到 O(n)，对于聚合计算来说不可接受。

标准 LSM-Tree 的 MemTable 通常只用一个有序结构（跳表或红黑树），因为它需要整体有序来支持高效的 flush 和 merge。我这里多维护了一个 HashMap，是因为点查在我的场景里频率很高（实时展示当前指标值），跳表的点查是 O(log n)，HashMap 是 O(1)，多一个索引换查询性能。代价是双倍的内存占用和写入时的双写开销，在数据量不大的情况下可以接受。

### 没有 WAL

标准 LSM-Tree 写入前先追加 WAL（Write-Ahead Log），这样即使进程崩溃，MemTable 里未刷盘的数据也能从 WAL 恢复。

我没做 WAL。原因是这个 Agent 跟着 QA Job 跑，Job 失败了整个重跑就行，不需要存储层自己做崩溃恢复。丢几秒的指标数据不影响最终报告。

这个简化在当时的场景下是合理的。

### 没有布隆过滤器

标准 LSM-Tree 在每个 SSTable 上挂一个布隆过滤器，查询时先过布隆过滤器，如果返回"不存在"就跳过这个文件，避免无用的磁盘读取。这在有删除操作（tombstone）的场景下尤其重要——删除不是真删，而是写入一个墓碑标记，查询时要往下找到墓碑才能确定数据已删除。没有布隆过滤器，每次查询可能要读多个 SSTable 才能确认结果。

我的场景只有追加，没有删除，也没有更新。同一个 `(metric, timestamp)` 不会出现两次。所以不需要布隆过滤器来加速"不存在"判断——如果某个时间段的数据在 Level 0 的某个 SSTable 里找到了，就不用再往下找了。

### Compaction：基于时间文件夹的合并

这是改动最大的部分。标准 LSM-Tree 的 compaction 是按 key 范围做多路归并排序。我的做法完全不同——按时间组织目录结构，**每小时一个文件夹，每分钟一个文件**：

```
data/
├── 2024-01-15/
│   ├── 14/                     ← 每小时一个文件夹
│   │   ├── 14-00.dat           ← 每分钟一个文件
│   │   ├── 14-01.dat
│   │   ├── 14-02.dat
│   │   └── ...
│   ├── 15/
│   │   ├── 15-00.dat
│   │   └── ...
│   └── merged/
│       ├── 14.dat              ← 合并后的小时级文件
│       └── full_day.dat        ← 合并后的天级文件
```

MemTable 达到阈值（或者每分钟定时触发），就把内存数据排序后写入当前分钟对应的文件。文件内部的数据是有序的（按 timestamp），不同文件之间通过文件名天然有序——这利用了时序数据的特性：**时间本身就是天然的排序键，而且只会单调递增。**

读取的粒度是**文件级别**——每次读一个分钟文件，反序列化后得到这一分钟的全部数据点。

还有一个关键设计：**内存索引通过扫描文件夹重建。** 进程启动时（或者 MemTable 刷盘后），遍历 `data/` 下的目录结构，根据文件夹名和文件名构建一个内存中的索引，记录每个时间段的数据落在哪个文件里：

```python
class DiskIndex:
    """通过扫描文件夹结构重建的内存索引"""
    def __init__(self, data_dir):
        # {(date, hour, minute): file_path}
        self.file_map = {}
        self._scan_and_rebuild(data_dir)

    def _scan_and_rebuild(self, data_dir):
        for date_dir in sorted(os.listdir(data_dir)):
            for hour_dir in sorted(os.listdir(f"{data_dir}/{date_dir}")):
                hour_path = f"{data_dir}/{date_dir}/{hour_dir}"
                # 优先用合并后的文件
                merged = f"{data_dir}/{date_dir}/merged/{hour_dir}.dat"
                if os.path.exists(merged):
                    self.file_map[(date_dir, hour_dir)] = merged
                else:
                    for minute_file in sorted(os.listdir(hour_path)):
                        minute = minute_file.replace('.dat', '')
                        self.file_map[(date_dir, hour_dir, minute)] = \
                            f"{hour_path}/{minute_file}"

    def locate_files(self, t_start, t_end):
        """根据时间范围定位需要读取的文件列表"""
        # 解析时间，找到对应的 (date, hour, minute) 组合
        # 如果有合并文件就返回合并文件，否则返回分钟级文件列表

```

这样查询时不需要遍历文件系统——直接从内存索引定位到文件路径，然后读文件。索引的构建成本在启动时一次性付出，之后的增量更新（新文件写入、合并完成）只需要更新 `file_map` 里的几个条目。

Compaction 的逻辑简化为：

```python
def compaction_worker():
    while running:
        sleep(60)  # 每分钟检查一次

        for date_dir in scan_date_dirs():
            for hour_dir in list_hour_dirs(date_dir):
                minute_files = list_minute_files(hour_dir)

                # 如果同一小时的分钟文件超过 30 个，合并为小时文件
                if len(minute_files) > 30:
                    merge_to_hour_file(minute_files, hour_dir)
                    disk_index.update(...)  # 更新内存索引

            # 如果同一天的小时文件超过 12 个，合并为天文件
            hour_merged = list_merged_hour_files(date_dir)
            if len(hour_merged) > 12:
                merge_to_day_file(hour_merged, date_dir)
                disk_index.update(...)
```

合并操作本身很简单——因为每个文件内部已经有序，且文件之间的时间范围不重叠（每个文件严格对应一个时间段），所以合并就是简单的拼接，不需要多路归并。合并完成后更新内存索引，把多个分钟级条目替换为一个小时级条目。

这和标准 LSM-Tree 的 compaction 有本质区别：

| | 标准 LSM-Tree Compaction | 我的时间文件夹合并 |
|---|---|---|
| 触发条件 | Level 大小超过阈值 | 同一时间段的文件数量过多 |
| 合并方式 | 多路归并排序，去重 | 简单拼接（时间范围不重叠） |
| 去重需求 | 必须（同 key 可能多次写入） | 不需要（只有追加，无更新） |
| 索引维护 | SSTable 自带稀疏索引 | 扫描文件夹重建内存索引 |
| 复杂度 | O(n log k)，k 为路数 | O(n)，纯拼接 |

这个设计在当时的场景下工作得还行，但后来暴露了问题——后面再说。

## 查询是怎么走的

把整个读路径画出来：

```
查询：metric_name 在 [t1, t2] 的数据

  1. 先查 MemTable
     ├── 点查 → HashMap
     └── 窗口查询 → 跳表 range scan
           │
  2. MemTable 没有完整覆盖 [t1, t2]？
     │
     ▼
  3. 通过内存索引（DiskIndex）定位文件
     ├── t1 落在 14:05，t2 落在 14:08
     ├── 内存索引查到 14/ 有合并文件 → 直接读 merged/14.dat
     └── 没有合并文件 → 读 14/14-05.dat, 14-06.dat, 14-07.dat, 14-08.dat
           │
  4. 按文件粒度读取、反序列化，合并结果
           │
  5. 做聚合（avg / max / p99）
```

查询时不走文件系统——内存索引已经记录了每个时间段对应的文件路径，直接定位、直接读。文件夹目录结构的扫描只在启动时做一次，之后靠增量更新维护。这是这个设计的优点——**查询路径和时间范围直接对应，通过内存索引 O(1) 定位到文件，不需要逐层查找。**

## 后来出了什么问题

Agent 最初设计给短期 QA Job 用，跑几十分钟就结束。但后来有人拿它去监控长时间运行的服务，一跑就是几天。问题逐渐暴露：

**问题一：小文件过多。**

每分钟一个文件，一天就是 1440 个文件分散在 24 个小时文件夹里。compaction 线程每分钟检查一次，合并速度跟不上生成速度，尤其是在磁盘 I/O 比较慢的测试机上。某些小时文件夹里堆了几十个未合并的分钟文件，内存索引条目暴涨，启动时的扫描重建也变慢。

**问题二：Python 单线程做聚合，太慢。**

文件读取本身其实还好——第一次从磁盘读过的数据会被缓存在内存里，后续查询同一时间段不需要再碰磁盘。真正的瓶颈是 Python 的 GIL。聚合运算（遍历数据点算 avg / p99 / 百分位）是纯 CPU 密集型操作，Python 单线程执行，数据量一大就慢得明显。

更要命的是，我还设计了一套"用户自定义字段提取"的机制——允许用户传入自己的逻辑去从原始数据里提取和组装字段，类似一个简易的 map 函数。初衷是想提供灵活性，但实际效果是：每条数据都要过一遍用户的 Python 回调，聚合一个小时的数据可能要在 Python 层循环几万次。这等于把本来就慢的聚合又套了一层解释执行的开销。

现在想来这是一个失败的设计。灵活性给了用户，但代价是性能完全不可控——用户写了一个低效的提取函数，整个查询都被拖慢，而且你在存储层根本没法优化它。正确的做法应该是在存储层定义好固定的聚合操作（avg / max / min / percentile），用户只能选择用哪个，不能自己写逻辑。把"可编程性"放在这一层是错误的抽象级别。

**问题三：应用层补丁越来越多。**

为了解决上面的问题，在应用层做了很多妥协：

- 加了内存中的预聚合缓存：每分钟的 avg/max/min 在写入时就算好，窗口查询时直接读预聚合值，不再读原始数据——但这意味着查询灵活性变差了（讽刺的是，之前费力做的"用户自定义提取"在这里被绕过了），新加一个聚合函数就要改写入逻辑。
- 加了文件合并的优先级队列：热数据（最近 1 小时）优先合并——但这又引入了调度复杂度。
- 加了查询结果的 LRU 缓存——但缓存失效策略写得很粗糙，偶尔返回过期数据。

每个妥协都在应用层加了一坨代码，存储层的"简洁"被应用层的复杂度完全吃掉了。

## 应该用 DuckDB

事后看，正确的选择是 DuckDB。

DuckDB 是一个嵌入式的列式分析数据库，对聚合查询天然友好。如果用 DuckDB，上面那些问题全都不存在：

```sql
-- 窗口聚合，一行 SQL
SELECT
    metric_name,
    avg(value),
    max(value),
    percentile_cont(0.99) WITHIN GROUP (ORDER BY value) as p99
FROM metrics
WHERE ts BETWEEN '2024-01-15 14:00:00' AND '2024-01-15 15:00:00'
GROUP BY metric_name;
```

不需要自己写 MemTable、不需要自己管 compaction、不需要在应用层做预聚合。列式存储对时序数据的压缩效率也远好于我的按行存储。

但当时没用 DuckDB，理由是：**环境只有 Python 3.6，DuckDB 要求 Python 3.10+。**

这个理由现在看来站不住脚。当时不愿意升级 Python 版本，是因为测试环境上跑着其他依赖 3.6 的服务，怕升级影响到它们。但解法很多——比如用 pyenv 装一个独立的 3.10 环境，或者用 conda，或者后来我们确实做了的：把 Python 应用打包成单文件可执行程序（用 PyInstaller），根本不依赖系统 Python 版本。

问题是**打包这件事是后来为了解决别的问题才做的**。在做存储层选型的时候，我被"环境只有 3.6"这个约束框住了，没有去想"这个约束能不能绕过"，而是直接接受了约束，然后在约束内找方案。

这是一个典型的**把约束条件当成了不可变的前提**。工程中很多看似硬性的约束，其实是可以通过额外的工作来绕过的。关键是要评估"绕过约束的成本"和"在约束内凑合的成本"哪个更高。我的手写 LSM-Tree 加上后来的一堆应用层补丁，花的时间和维护成本，远远超过了搞定 Python 版本问题的成本。

## 复盘：哪些判断是对的，哪些是错的

**对的判断：**

- 选择 LSM-Tree 的思路。写多读少、追加为主的时序场景，LSM-Tree 确实比 B+ 树更合适。
- 砍掉 WAL。短生命周期的 Job，崩溃恢复没有意义。
- 砍掉布隆过滤器。只有追加没有删除，不需要加速"不存在"判断。
- 用时间做天然排序键。时序数据的时间单调递增，利用这个特性简化了合并和查询。

**错的判断：**

- 低估了"简单需求"的增长速度。一开始是短期 Job，后来变成长期服务监控。存储层的设计没有留余量。
- 把 Python 3.6 当成不可变约束。没有评估绕过这个约束的成本，直接在约束内做了一个更复杂的方案。
- 在应用层补存储层的不足。预聚合缓存、合并优先级、查询缓存，这些本应是存储引擎内部的功能，被分散到了应用层各处，增加了维护成本。

**面试时说错的地方：**

- MemTable 说的是 HashMap，实际是跳表 + HashMap。面试时可能是紧张，只说了一半。关键区别是窗口查询需要有序结构，纯 HashMap 做不了高效的范围扫描。

## 几个更一般的教训

**第一，"不引入重量级依赖"不等于"自己写就更轻"。** 手写的代码初始确实轻量，但随着需求变化，应用层的补丁越打越多，最终的总复杂度可能远超引入一个成熟依赖的成本。DuckDB 是一个单文件嵌入式数据库，引入它的成本其实很低，"重量级"是我的错误认知。

**第二，存储和计算的边界要想清楚。** 我的设计把聚合计算完全放在应用层，存储层只负责存取。这在数据量小的时候没问题，大了以后数据搬运的开销就成了瓶颈。DuckDB、ClickHouse 这类分析型数据库之所以快，核心就是把计算推到存储层（pushdown），减少数据搬运。这不是"优化"，是架构上的根本差异。

**第三，约束条件是可以被挑战的。** "环境只有 Python 3.6"看起来是硬约束，实际上可以通过 pyenv、conda、打包等方式绕过。在接受一个约束之前，先问一句："绕过这个约束需要多大代价？" 如果代价比在约束内凑合的代价低，那就应该去绕过它。
