GC 日志里又刷出一屏 STW 停顿，监控面板上采集延迟的毛刺已经从偶发变成每个周期都有。我打开 MAT 一看——分配最多的对象是 `MetricsSnapshot`，其次是 `java.lang.Double`。

一个存储性能指标的采集系统，5 秒一个周期，每轮给数万个存储对象算 IOPS、吞吐、延迟……每轮都 new 一堆 `Double`、`Long` 出来，用完就扔。GC 不炸才怪。后来把包装类型干掉、对象复用做上去，计算延迟从 100ms 降到 40ms。

但这不只是"少 new 几个对象"那么简单。`Double` 比 `double` 到底贵在哪？为什么包装类型一上量就拉胯？这篇文章从底层内存布局讲起，用 benchmark 和 `perf stat` 实证，把"基本类型比包装类型快"从面试八股变成可量化的认知。

## 先画个图：int[] 和 Integer[] 在内存里长什么样？

这是理解性能差异的关键。

**`int[]` —— 连续平铺，和 C 数组一样：**

```
int[] arr = new int[6];

堆内存:
┌────────────────┬────┬────┬────┬────┬────┬────┐
│  array header  │ i0 │ i1 │ i2 │ i3 │ i4 │ i5 │
│   (16 bytes)   │ 4B │ 4B │ 4B │ 4B │ 4B │ 4B │
└────────────────┴────┴────┴────┴────┴────┴────┘
                  ← 连续排布，一条 cache line(64B) 装 16 个 int →
```

**`Integer[]` —— 指针数组 + 散落的对象：**

```
Integer[] arr = new Integer[6];

堆内存 (默认开启压缩指针 UseCompressedOops, 堆 < 32GB):
┌────────────────┬──────┬──────┬──────┬──────┬──────┬──────┐
│  array header  │ ref0 │ ref1 │ ref2 │ ref3 │ ref4 │ ref5 │
│   (16 bytes)   │  4B  │  4B  │  4B  │  4B  │  4B  │  4B  │
└────────────────┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┴──┬───┘
                    │      │      │      │      │
                    ▼      ▼      ▼      ▼      ▼
               ┌────────┐  ...  ┌────────┐  ...  ┌────────┐
               │ header │       │ header │       │ header │
               │  12B   │       │  12B   │       │  12B   │
               │ value  │       │ value  │       │ value  │
               │  4B    │       │  4B    │       │  4B    │
               └────────┘       └────────┘       └────────┘
               ← 每个 Integer 对象 16 字节，散布在堆的各个位置 →
```

区别一目了然：

1. **空间膨胀**：`int` 4 字节，`Integer` 对象本身 16 字节（Mark Word 8B + Klass Pointer 4B(压缩) + value 4B），加上数组中的压缩引用 4 字节 → 每个元素实际占约 20 字节。实测也印证了这个数字（后面 benchmark 会看到每个 `Integer` 约占 20 bytes），**膨胀近 5 倍**
2. **多一跳指针**：访问 `arr[i]` 要先从数组取引用，再解引用跳到 `Integer` 对象的地址去读 `value` 字段
3. **空间局部性被破坏**：这些 `Integer` 对象在堆上的位置取决于分配时机和 GC 整理结果，不保证连续

所以对 CPU cache 而言：

- `int[]` 顺序遍历时 prefetcher 可以预测下一条 cache line，命中率极高
- `Integer[]` 遍历时每个元素都要跳到一个随机地址，prefetcher 无能为力，L1/L2 cache miss 频发

## 别猜，量：Benchmark 实测

写了一个 benchmark 程序（[完整源码](https://github.com/huanghuoguoguo/huanghuoguoguo.github.io/blob/main/assets/code/PrimitiveVsWrapperBenchmark.java)），400 万元素，对比 `int[]` vs `Integer[]` 的顺序遍历和随机访问性能。JIT 预热 5 轮，正式测 10 轮取中位数。

### 内存占用

```
============================================
  内存占用估算 (数组大小: 4000000)
============================================
  int[]     : 16,819,200 bytes (16.0 MB)
  Integer[] : 83,191,952 bytes (79.3 MB)
  膨胀倍数   : 4.9x
  每个 int  占: 4 bytes
  每个 Integer 约占: 20 bytes (含对象头+对齐)
```

400 万个 `int` 占 16MB，换成 `Integer` 直接涨到 79MB。同样的数据量，**内存占用膨胀近 5 倍**。

### 顺序遍历求和

```
============================================
  测试 1：顺序遍历求和 (体现 cache 局部性)
============================================
  int[] 顺序求和                 :     1.57 ms
  Integer[] 顺序求和             :     7.48 ms
  → Integer[]/int[] = 4.8x 慢
```

顺序遍历求和，`Integer[]` 比 `int[]` **慢 4.8 倍**。计算逻辑完全一样，都是累加，唯一的区别就是内存布局。

> 实验里还跑了一组"故意打散的 Integer[]"（先分配干扰对象再交错分配 Integer），结果反而比顺序分配的略快（6.03ms vs 7.48ms）。这个反直觉的数据说明：TLAB 连续分配的"相邻性"在 `System.gc()` 后被 G1 的 compaction 重新整理了，打散组反而因为 GC 后的布局更紧凑而受益。**对象在堆上的实际分布，GC 策略说了算，不是分配顺序说了算。**

### 随机访问求和

```
============================================
  测试 2：随机访问求和 (体现指针追逐开销)
============================================
  int[] 随机访问                 :    27.47 ms
  Integer[] 随机访问             :    96.28 ms
  → Integer[]/int[] = 3.5x 慢
```

随机访问场景下，`int[]` 本身 cache miss 也不低（随机下标嘛），所以倍数缩小到 3.5x。但看绝对数字：**27ms vs 96ms，差了 70ms**——在延迟敏感的系统里，这 70ms 就是一次采集周期能不能在 deadline 内完成的区别。

### perf stat 硬件观测

用 `perf stat` 观测进程级指标。需要说明：perf stat 是进程粒度的采样，数据里混入了 JVM 启动（类加载、JIT 编译、GC）的开销，所以只能看趋势，不能精确归因。实验跑在 VMware 虚拟机上，硬件 PMU 计数器不可用，只能用软件事件替代：

| 指标 | int[] 顺序 | Integer[] 顺序 | int[] 随机 | Integer[] 随机 |
|---|---|---|---|---|
| cycles | 15.2B | 16.1B | 15.0B | 20.6B |
| page-faults | 122,606 | 137,163 | 123,749 | 138,561 |
| task-clock | 4,231 ms | 4,454 ms | 4,157 ms | 5,605 ms |
| wall time | 2.26 s | 2.44 s | 2.70 s | 3.81 s |

趋势上的几个观察（注意绝对值包含 JVM 启动噪声，重点看组间差异）：

- **page-faults**：`Integer[]` 比 `int[]` 多出约 15,000 次（~12%）。对象散布在更多内存页上，触发了更多缺页
- **cycles**：随机访问 `Integer[]` 消耗 20.6B cycles vs `int[]` 15.0B，多出 37%。多出来的 cycle 大概率消耗在等待内存返回数据（cache miss → pipeline stall）
- **wall time**：随机访问 `Integer[]` 的 wall time 是 `int[]` 的 1.4 倍

> 如果在物理机上跑，可以用 `perf stat -e L1-dcache-load-misses,LLC-load-misses` 直接看 L1/LLC cache miss 计数，数据会更有说服力。VMware 虚拟机不暴露硬件 PMU 给 guest，这是本次实验的局限。

## JVM 会优化吗？

看到这里可能会想：JVM 不是有各种优化吗？逃逸分析、标量替换、JIT 编译，难道不能消除包装类型的开销？

**答案是：能，但场景非常有限。**

| JVM 优化 | 能帮到什么 | 帮不到什么 |
|---|---|---|
| **逃逸分析 + 标量替换** | 局部变量 `Integer x = 42` 可以被拆成栈上的 `int` | 对象放进数组、集合、逃逸出方法 → 失效 |
| **Integer 缓存 (-128~127)** | `Integer.valueOf(50)` 返回缓存对象，减少分配 | 值超出范围 → 照样 new；缓存对象也是堆上的指针 |
| **TLAB 分配** | 连续 new 的对象在 TLAB 中物理相邻 | GC 后对象重新整理，相邻性不保证 |
| **JIT 编译** | 循环体内的拆箱可以被优化成直接读字段 | 减少了方法调用开销，但指针跳转和 cache miss 无法消除 |

说白了，**只要对象在堆上独立存在，就需要指针间接访问，就有 cache miss 的风险**。这不是编译器能解决的，是内存模型层面的根本限制。

Java 的 **Project Valhalla**（Value Types / Primitive Classes）正是为了解决这个问题——引入 value class 后，可以像 `int[]` 一样把用户自定义类型扁平排布在数组中，消除对象头和指针间接层。但这个项目从 2014 年提出至今仍未正式发布。

## 实战：从 100ms 到 40ms 的三板斧

讲完原理回到开头那个让我头疼的场景。

### 背景

系统每 5 秒采集一次所有存储对象（数万个）的性能指标（IOPS、吞吐量、延迟等），需要实时计算各种聚合指标。架构是多线程分片 + `CompletableFuture` 并行计算。

### V1：发现问题

初始版本中，每个采集周期都会为每个存储对象创建新的指标对象：

```java
// 伪代码：每 5 秒执行一次
for (StorageObject obj : allObjects) {
    MetricsSnapshot metrics = new MetricsSnapshot(); // 每次 new
    metrics.setIops(calculateIops(obj));
    metrics.setThroughput(calculateThroughput(obj));
    metrics.setLatency(calculateLatency(obj));
    results.add(metrics);
}
```

数万个对象 × 多个指标字段 × 5 秒一个周期 → **每个周期创建大量短命对象** → GC 压力飙升 → 出现 GC 风暴和服务假死。

通过堆内存分析（MAT）定位到热点代码，发现 `MetricsSnapshot` 和内部的包装类型字段（`Double`、`Long` 等）是分配大户。

### V2：以 ID 索引的缓存复用

核心思路：这些存储对象是相对固定的，ID 不变，不需要每次 new 一个新的指标对象。改为以 ID 索引的缓存，复用已有对象：

```java
// 以 ID 为 key 的缓存 map，对象只创建一次，后续复用
private final Map<String, MetricsSnapshot> metricsCache = new ConcurrentHashMap<>();

for (StorageObject obj : allObjects) {
    MetricsSnapshot metrics = metricsCache.computeIfAbsent(
        obj.getId(), id -> new MetricsSnapshot()
    );
    // 更新字段而不是重新创建
    metrics.updateIops(calculateIops(obj));
    metrics.updateThroughput(calculateThroughput(obj));
    metrics.updateLatency(calculateLatency(obj));
}
```

同时：

- 指标字段从 `Double` / `Long` 改为 `double` / `long`，消除自动装箱
- 用邮戳锁（`StampedLock`）替代 `synchronized`，读多写少场景下减少锁竞争
- 引入不可变集合传递中间结果，避免防御性拷贝

这一步做完，单周期延迟从 ~100ms 降到 ~60ms。GC 日志里 Young GC 的频率和 STW 毛刺明显改善——对象分配速率下去了，Eden 区不再每个周期都被塞满。

但 60ms 还不够。profiler 显示热点转移到了数据结构的查找开销上。

### V3：Guava ArrayTable 替代嵌套 HashMap

瓶颈迁移到了数据结构。系统中有很多"两个维度索引"的场景（比如 `[存储对象ID, 指标类型] → 值`），最初用的是 `Map<String, Map<String, Double>>`——两层 HashMap，每层都是指针追逐，查一次值要跳好几跳。

换成 Guava 的 `ArrayTable`：

```java
// 之前：两层 HashMap，每次查找至少 2 次 hash + 多次指针跳转
Map<String, Map<String, Double>> metrics;
Double value = metrics.get(objectId).get(metricName);

// 之后：ArrayTable，底层是二维数组
// 行列 key → 数组下标的映射只在构造时做一次，后续查找走数组索引
ArrayTable<String, String, Double> metrics = ArrayTable.create(objectIds, metricNames);
metrics.put(objectId, metricName, 42.0);
Double value = metrics.get(objectId, metricName); // 一次 hash 查下标 + 数组访问
```

`ArrayTable` 内部用 `Object[][]` 存储数据，行列索引用 `ImmutableMap` 映射到数组下标。查找时一次 hash 查下标 + 一次数组访问，比两层 `HashMap` 少了大量指针追逐。

> 这里有一个遗憾：`ArrayTable` 的值类型还是 `Double`（包装类型），因为泛型擦除的限制。如果要做到极致，需要用 Eclipse Collections 的 `MutableObjectDoubleMap` 这种原始类型特化集合。

### 效果

| 指标 | V1 (优化前) | V2 (缓存复用) | V3 (ArrayTable) |
|---|---|---|---|
| 单周期计算延迟 | ~100ms | ~60ms | ~40ms |
| Young GC 频率 | 频繁，有 STW 毛刺 | 明显改善 | 进一步降低 |
| 对象分配速率 | 高（每周期大量 new） | 低（缓存复用） | 低 |
| 查找开销 | 高（嵌套 HashMap） | 高（未改结构） | 低（数组索引） |

延迟从 100ms 到 60ms，靠的是消灭对象创建和 GC 压力；从 60ms 到 40ms，靠的是减少指针追逐、改善数据局部性。两步优化的本质是同一件事：**让数据在内存里待在该待的地方，别让 CPU 满世界找它**。

## 什么时候该在意，什么时候不用

那是不是应该把所有 `Integer` 都换成 `int`，所有 `HashMap` 都换成数组？显然不是。过度优化和不优化一样有害——代码可读性也是成本。

**这些场景值得死磕：**
- CPU 密集型的紧密循环（数值计算、指标聚合、排序）
- 大量数据的批量处理（日志分析、ETL）
- 延迟敏感的热路径（交易系统、实时监控）
- 内存受限的场景（嵌入式、大数据集）

**这些场景别折腾：**
- 业务逻辑代码中偶尔的装箱拆箱
- I/O 密集型任务（瓶颈在网络/磁盘，不在 CPU cache）
- 元素数量很少的集合（几十个元素的 `List<Integer>` 无所谓）

如果确实需要在热路径上用集合存基本类型，这些库可以帮你绕过泛型擦除的限制：

| 需求 | 避免使用 | 推荐使用 |
|---|---|---|
| int 列表 | `ArrayList<Integer>` | Eclipse Collections `IntArrayList`、HPPC `IntArrayList` |
| int→object 映射 | `HashMap<Integer, V>` | Eclipse Collections `IntObjectHashMap` |
| 二维表 | `Map<R, Map<C, V>>` | Guava `ArrayTable`、`ImmutableTable` |
| 数值数组 | `Integer[]` | `int[]` |

## 写在最后

这篇文章的起点是一个简单的问题："包装类型做 CPU 密集型任务，是不是会 cache miss 严重？"

答案是 yes，而且可以量化：顺序遍历慢近 5 倍，内存膨胀近 5 倍，随机访问慢 3.5 倍。

但面试里背"拆箱有开销"是没用的——真正的开销不在 `intValue()` 那个方法调用上，而是包装类型破坏了内存的空间局部性，让 CPU 的硬件预取机制彻底失效。这是一个从 Java 语言层穿透到硬件层的问题，JIT 再聪明也优化不掉堆上散落的指针。

在实习的那个项目里，这个认知最终转化成了两个动作：
1. 热路径上消灭不必要的对象创建，字段用 `double` 不用 `Double`
2. 用 cache 友好的数据结构替代嵌套 `HashMap`

Java 的泛型擦除决定了 `List<int>` 不可能存在（至少在 Valhalla 落地之前）。语言层面没法改的东西，就在工程层面绕过去。cache line 64 字节、指针追逐会 miss、对象头 12 字节——这些数字不是用来面试背的，是用来在 MAT 里看到一屏 `java.lang.Double` 时知道该怎么办的。
