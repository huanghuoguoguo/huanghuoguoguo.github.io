上一篇聊了 RAG 怎么解决领域知识注入的问题——把知识库里跟问题相关的内容捞出来，塞进上下文，让模型"带着参考资料答题"。在知识问答场景下，这套机制很好用。

但如果你想让 LLM 做的不只是回答问题呢？

之前我写过一篇用 LLM 做性能诊断的文章。那个场景就不是 Q&A——需要 LLM 去跑 `top` 看 CPU、跑 `syscount` 看系统调用、读源码、定位瓶颈，每一步的决策都依赖上一步的结果。光检索几段文档搞不定。

从"能聊"到"能干"，中间隔着一整套架构演化。这篇文章来聊这个演化是怎么一步步发生的。

<!-- more -->

---

## 起点：Prompt + RAG

最简单的架构：System Prompt 定义行为，RAG 提供知识，模型回答问题。

```
用户提问 → RAG 检索知识片段 → 拼入 System Prompt → LLM 回答
```

这就是上一篇文章讲的模式。优势很明显：开发成本极低，对于"根据文档回答问题"这类场景，ROI 是最高的。

但它有一个根本限制：**模型只能"说"，不能"做"。**

用户问"SSH 连不上怎么办"，它能告诉你排查步骤，然后你去执行，解决不了，把报错再复制粘贴回来让 ai 分析。但如果用户说"帮我查一下这台机器为什么 SSH 连不上"——对不起，它只是一个接收文本、输出文本的函数，它没有手。

当需求从"告诉我怎么做"变成"帮我做"，就需要新的东西了。

---

## Function Calling：给模型装上手

LLM 本质上是一个文本到文本的函数，它没有能力执行任何外部操作。但 2023 年 OpenAI 给 GPT 加了 Function Calling 之后，事情变了——模型可以在回复中声明"我想调用某个函数"，由外部代码去执行，再把结果喂回来。

```
用户："帮我查一下北京今天天气"
  → LLM 输出：{ "function": "get_weather", "args": {"city": "北京"} }
  → 你的代码执行 get_weather("北京")，拿到 "晴，28°C"
  → 把结果喂回 LLM
  → LLM: "北京今天晴，最高温 28°C"
```

注意：**LLM 自己不执行函数，它只是输出一段结构化的"我想调这个"。** 实际执行在你的代码里完成。这个设计很关键——LLM 负责理解意图和选择工具，代码负责安全地执行，两边各做擅长的事。

有了 Function Calling，模型从"只能说"变成了"能调度"。手还是没有，但它拿到了一个遥控器。

这个机制是后面所有架构的基石——不管是 Workflow 里的确定性调用、ReAct 里的自主决策、还是 Multi-Agent 之间的工具协作，底层都是 Function Calling 在驱动。后面聊到 MCP 的时候会看到，当工具越来越多、Agent 框架越来越杂，Function Calling 的**标准化**就成了新的问题。

但拿到遥控器不代表会用。早期的实践很快暴露了一个问题：**你把工具给了 LLM，在 Prompt 里告诉它"需要查数据库就调 query_db，需要发邮件就调 send_email"——然后它就开始乱来了。** 该查数据库的时候它先发了封邮件，该停下来的时候它继续调工具，有时候还会编造根本不存在的函数名。

这种情况很普遍。早期模型（GPT-3.5 时期）的指令遵循能力有限，上下文一长就容易"忘了"自己该干什么。工具一多，选择空间变大，出错概率也随之上升。你在 Prompt 里写得再详细，也没法保证模型每次都按预期行动——概率模型就是这样，你管不住它每一步都按你写的来。

所以最早落地的方案很朴素：**把流程用代码钉死，LLM 只在指定的节点里干活**。

---

## Workflow：当步骤是确定的

这就是 Workflow 的思路：既然 LLM 自己决策不靠谱，那就别让它决策。你来画流程图，LLM 只负责每个节点里的"理解"和"生成"。

比如一个客服场景：

```
用户反馈问题
    → 第一步：LLM 分类意图（退款？咨询？投诉？）
    → 第二步：根据意图查对应数据库
    → 第三步：LLM 基于查询结果生成回复
```

每一步做什么、下一步跳到哪，都是你在代码里用 if/else 写好的。LLM 只负责每一步里的"理解"和"生成"，流程本身由你控制。

用代码表示大概是这样：

```python
intent = llm("判断用户意图", user_input)       # LLM 做分类

if intent == "退款":
    order = db.query_order(user_id)              # 确定性操作
    response = llm("生成退款回复", order)         # LLM 做生成
elif intent == "咨询":
    docs = rag.search(user_input)                # RAG 检索
    response = llm("基于文档回答", docs)
```

本质上就是用代码编排 LLM 调用。流程是硬编码的，LLM 只是被嵌在流程节点里做自然语言处理。

Workflow 的优势很实际：

| 维度 | 说明 |
|------|------|
| **可预测** | 每一步做什么是确定的，不会跑偏 |
| **可调试** | 某一步出错了，直接定位到那个节点 |
| **可监控** | 每个节点的输入输出都可以记录 |

适用场景也很明确：**当你能把任务流程画成一张有限状态机的时候。** 客服工单处理、表单审批、数据管道——这些步骤固定、分支有限的任务，Workflow 是最稳的选择。

但它有一个根本的问题：**刚性太强。**

你能提前预见用户可能走的所有路径吗？如果用户的问题不在你画的状态机里呢？Workflow 的覆盖率取决于你写了多少 if/else。现实场景中，用户的需求千变万化，试图用 Workflow 穷举所有路径，最终会变成一个巨大的面条代码。

当任务的步骤**不能提前确定**，就需要让 LLM 自己决定该做什么了。

---

## ReAct：让 LLM 自己决策

ReAct 的思路很直接：**别替 LLM 规划了，给它工具，让它自己想该怎么做。**

ReAct 是 Reasoning + Acting 的缩写，核心是一个循环：

```
Thought:  "用户说 SSH 连不上，我应该先检查目标机器的网络是否通"
Action:   ping(target_ip)
Observation: "PING 192.168.1.100: 0% packet loss"
Thought:  "网络通的，那看看 SSH 服务是否在运行"
Action:   check_service(target_ip, "sshd")
Observation: "sshd is not running"
Thought:  "找到原因了，SSH 服务没启动"
Answer:   "目标机器的 SSH 服务没有运行，建议执行 systemctl start sshd"
```

Thought → Action → Observation，不断循环，直到 LLM 认为可以给出最终答案。

跟 Workflow 的本质区别：流程的控制权交给了 LLM。下一步调什么工具、传什么参数、什么时候停下来，全由 LLM 在运行时根据上下文决定。

这就是 Agent 最原始的形态——一个能使用工具、自主决策的 LLM。

```
┌────────────────────────────────────────────────┐
│                  Agent Loop                     │
│                                                 │
│  ┌─────────┐   ┌────────┐   ┌─────────────┐   │
│  │ Thought  │ → │ Action │ → │ Observation │──┐│
│  └─────────┘   └────────┘   └─────────────┘  ││
│       ↑                                       ││
│       └───────────────────────────────────────┘│
│                                                 │
│  Tools: [search, ping, read_file, run_cmd...]  │
└────────────────────────────────────────────────┘
```

我在做性能诊断 Agent 的时候，第一版就是这个模式：一个 Agent，给它 `top`、`syscount`、`pidstat`、`read_proc_file` 四个工具，让它自己跑 OODA 循环。效果大方向上对——它会先看 CPU，再看系统调用，最后给出"瓶颈在 write 系统调用"的结论。

但问题也马上出来了。

### Single Agent 的瓶颈

**上下文窗口又爆了。** 上一篇文章说 RAG 是为了解决上下文爆炸，结果到了 Agent 这里，上下文又爆了——只不过爆的原因不一样。RAG 是知识太多塞不下，Agent 是每次工具调用的输入输出都会追加到上下文里。跑几轮 `top`、`syscount` 之后，上下文里全是系统指标的文本，LLM 真正需要关注的信息被淹没了。

**Agent "走神"。** 当一个 Agent 身兼数职（又要诊断、又要读代码、又要改代码），System Prompt 就会变得很长。LLM 在长 Prompt 下很容易"忘了自己该干什么"——这跟人在一个嘈杂的环境里工作容易分心是一个道理，只不过 LLM 分心的原因是注意力被过多的上下文稀释了。

**推理链条太长容易出错。** 每一步决策都有出错的概率，而 Agent 是串行的——一步错，步步错。诊断阶段判断错了方向，后面读代码、改代码都是白费。

回到性能诊断那个项目，这些问题正是我从 V1（单 Agent）升级到 V2（三个 Agent 分工）的原因。

## Multi-Agent：分而治之

思路很自然：一个人做不好的事，拆给几个专家做。

我在性能诊断 V2 里把一个全能 Agent 拆成了三个：

```
Detective（侦探）──→ Architect（架构师）──→ Coder（程序员）
   查系统指标           看代码找原因           改代码修复
```

每个 Agent 只关注自己的领域，Prompt 更短、工具集更小、不容易走神。侦探只管看 `top` 和 `syscount`，不需要知道怎么读 Rust 代码；架构师只管分析代码，不需要理解 `/proc/meminfo` 的输出格式。

这就是 Multi-Agent 的核心价值：**通过职责隔离，让每个 Agent 的上下文保持精简和聚焦。**

Google DeepMind 在论文 *Towards a Science of Scaling Agent Systems* 里把 Multi-Agent 分成了四种模式：

```
Independent（独立）     各干各的，最后汇总
Decentralized（去中心） Agent 之间点对点通信
Centralized（中心化）   一个主 Agent 分配任务，子 Agent 执行
Hybrid（混合）          中心调度 + 局部自治
```

实际项目中最常见的是 Centralized 模式——一个 Orchestrator（编排者）负责理解用户意图、把任务分发给对应的子 Agent：

```
┌─────────────────────────────────────────┐
│            Orchestrator                  │
│         "这个问题该交给谁？"              │
└───┬──────────┬──────────┬───────────────┘
    ↓          ↓          ↓
┌───────┐ ┌───────┐ ┌───────┐
│ Agent │ │ Agent │ │ Agent │
│  ECS  │ │  RDS  │ │ 网络  │
│ 专家  │ │ 专家  │ │ 专家  │
└───────┘ └───────┘ └───────┘
```

每个子 Agent 有自己独立的 Prompt 和工具集，只负责自己擅长的领域。Orchestrator 不需要懂具体业务，只需要做好意图识别和路由。

### Multi-Agent 的代价

分工解决了上下文爆炸和注意力分散的问题，但引入了新的麻烦：

**路由准确率。** 当子 Agent 有十几二十个的时候，Orchestrator 的分类决策压力就很大了。一旦路由错了——把 ECS 的问题分给了 RDS 专家——后面再怎么努力都是错的。而且这种错误是**静默的**，用户不知道发生了误路由，只会觉得"这 AI 怎么答非所问"。

**上下文割裂。** 这是最隐蔽的问题。每个子 Agent 只看到自己那一段对话，不知道其他 Agent 之前做了什么。

举个例子：用户问"服务器连不上"，Agent A 诊断出"CPU 负载过高"。用户追问"为什么负载高"，如果 Agent B 接手了这个问题，它不知道 A 已经查过负载了，可能会重新执行一遍相同的诊断步骤。更糟的是，B 基于自己的局部信息得出的结论可能跟 A 矛盾。

**通信损耗。** 为了解决上下文割裂，你可以在 Agent 之间共享对话历史。但这又会带来两个问题：

- 如果传完整历史，Token 消耗暴涨，上下文又爆了（兜了一圈回到原点）
- 如果做摘要再传，摘要过程是有损的，关键细节可能丢失

这就是 Multi-Agent 的核心矛盾：**隔离带来了聚焦，但也切断了全局视野。** 想要全局视野就要共享上下文，共享上下文又会重新引爆上下文窗口。

我在性能诊断 V2 里用了一个折中方案：三个 Agent 共享同一条消息链（像群聊），侦探说完，架构师接着看。这样不会丢失上下文，但代价是消息链会越来越长——对于一次性的诊断任务来说还好，如果是持续运行的系统，这个方案撑不住。

随着 Agent 数量增加，系统复杂度非线性上升，但效果提升的边际越来越小。打磨每一个 Agent 的 Prompt、设计路由策略、处理通信协议和各种边界情况——这些工程成本往往超出预期。

## 工具标准化：MCP

上面不管是 Single Agent 还是 Multi-Agent，都绕不开一个基础问题：**Agent 怎么调用工具？**

在我做性能诊断 Agent 的时候，每个工具都是手写的 Python 函数：

```python
def get_system_metrics():
    """跑 top -b -n 1，返回系统指标"""
    result = subprocess.run(["top", "-b", "-n", "1"], capture_output=True)
    return result.stdout.decode()
```

然后手动绑定给 LLM：

```python
llm = ChatOpenAI(model="...").bind_tools([get_system_metrics, ...])
```

这种方式对一个项目来说没问题。但如果你想让 Agent 接入更多工具——数据库查询、API 调用、文件操作、浏览器控制——每一个都要手写绑定函数、定义 schema、处理异常，工作量就上去了。而且不同的 Agent 框架（LangChain、AutoGen、CrewAI）对工具的定义格式都不一样，同一个工具要适配多个框架就得写多套代码。

MCP（Model Context Protocol）要解决的就是这个问题。Anthropic 提出这个协议，本质上是给工具调用定了一个**统一接口标准**。

类比一下：USB 出现之前，键盘用 PS/2 口，打印机用并口，手机用各种私有接口。USB 统一了这些，不管什么设备，一个口就够了。MCP 对 Agent 工具做的事情类似——不管什么工具（查数据库、读文件、调 API），只要实现了 MCP 协议，任何支持 MCP 的 Agent 都能直接用。

```
MCP 之前:
Agent A ──专用代码──→ 工具 1
Agent A ──专用代码──→ 工具 2
Agent B ──另一套代码→ 工具 1   （同一个工具，不同的绑定）

MCP 之后:
Agent A ─┐
         ├──MCP 协议──→ 工具 1（MCP Server）
Agent B ─┘
         ├──MCP 协议──→ 工具 2（MCP Server）
         └──MCP 协议──→ 工具 3（MCP Server）
```

MCP 定义了三件事：工具怎么被**发现**（Agent 启动时自动获取可用工具列表）、工具怎么被**描述**（参数、返回值、用途的标准化 schema）、工具怎么被**调用**（统一的请求/响应格式）。

工具的提供方（MCP Server）和使用方（Agent / MCP Client）完全解耦。这意味着：

- 社区可以共享工具：有人写了一个"查 GitHub Issues"的 MCP Server，所有人都能用
- Agent 框架无关：不管你用什么框架开发 Agent，只要支持 MCP，就能接入所有 MCP 工具
- 工具可以独立演化：工具的实现升级了，Agent 端不需要改任何代码

在实际体验中，Claude Code 就是通过 MCP 来接入外部工具的。你配置一个 MCP Server，Claude Code 会自动发现它提供的工具，然后在需要的时候调用。

## Skills：可复用的能力单元

当你用 Agent 做了一段时间之后，会发现一个模式：**很多任务需要的能力是重复的。**

比如"审查代码并提交"这个能力，可能在代码审查 Agent、CI/CD Agent、重构 Agent 中都会用到。每次都从零开始写 Prompt 和工具组合，既浪费又不一致。

Skills 的思路是把这些反复出现的能力封装成标准化的、可复用的单元。一个 Skill 包含：

- 一段针对特定任务优化过的 Prompt
- 需要用到的工具集合
- 输入输出的格式约定

类比编程的话，如果 Agent 是一个程序，Skills 就是它的函数库。你不需要每次都手写排序算法，调用标准库就行。

Anthropic 在 Claude Code 中实践了这个概念——Claude Code 的 Slash Commands（比如 `/commit`、`/review-pr`）本质上就是预定义的 Skills。每个 Skill 封装了一套完整的 Prompt + 工具组合 + 工作流，用户一条命令就能触发一个完整的能力。

Skills 带来的好处是：能力可以独立开发、测试、迭代，然后被不同的 Agent 复用。当你有一个经过充分验证的"代码审查" Skill，任何 Agent 都可以直接调用它，而不是每个 Agent 各自摸索怎么做代码审查。

## 退一步看：每一层都在补偿

把整个演化链条摆在一起：

```
Prompt + RAG
    ↓ 模型只能聊，不能做 → 加工具
Workflow
    ↓ 步骤写死太刚 → 让模型自己决策
ReAct / Single Agent
    ↓ 上下文爆炸，注意力分散 → 拆分职责
Multi-Agent
    ↓ 工具绑定太碎，不可复用 → 标准化
MCP + Skills
```

每一层架构的出现，都是因为上一层碰到了解决不了的问题。而且有一个规律：**越往下走，LLM 的自主权越大，人的控制力越小。**

| 架构 | 谁控制流程 | LLM 的角色 | 可预测性 |
|------|-----------|-----------|---------|
| Workflow | 代码 | 节点里的处理器 | 高 |
| Single Agent | LLM | 决策者 + 执行者 | 中 |
| Multi-Agent | LLM + 代码 | 协作者 | 低 |

这不是说越靠后的架构越好。**选什么架构，取决于你的任务需要多少灵活性、你能容忍多少不确定性。**

- 客服工单分类？Workflow 就够了，稳定可控。
- 代码审查？Single Agent + 几个工具，灵活够用。
- 复杂的多领域问题诊断？可能需要 Multi-Agent。

过度设计是 Agent 领域最常见的坑。一个 Workflow 能解决的问题，没必要上 Multi-Agent。

## 显式的脚手架会消失吗？

上一篇文章结尾留了一个问题：**当基座模型能力越来越强，我们在外部搭建的这些架构，会不会逐渐变得多余？**

回顾一下已经发生的事：

- **2023 年**：需要精心设计的 Chain-of-Thought 提示词才能让模型一步步推理。**2026 年**：Claude、GPT-4o 原生就会做 extended thinking，不需要你提示。
- **2023 年**：工具调用需要在外部框架里做解析和编排。**2026 年**：大部分模型原生支持 function calling，工具调用成了 API 的一等公民。
- **2023 年**：上下文窗口 4K-8K，RAG 是刚需。**2026 年**：窗口到了 128K-1M，很多原来必须用 RAG 的场景，直接塞进上下文就行了。

模式很清晰：**外部显式架构正在被模型的原生能力吸收。**

沿着这个趋势推演：

- 如果上下文窗口继续增长，且 Lost in the Middle 问题被解决，RAG 的"检索"步骤是不是可以省掉？直接把整个知识库塞进去？
- 如果模型的推理和规划能力足够强，预定义的 Workflow 是不是就变成了多余的约束？
- 如果模型能自主管理多个并行任务，Multi-Agent 的显式编排还有没有必要？

当然，这些"如果"目前还没有完全实现。即使技术上可行，工程约束仍然存在——成本、延迟、可控性。你能把 100 万字的知识库塞进 1M token 的上下文吗？技术上可以，但每次查询的成本可能是 RAG 的几十倍。

所以当前更实际的问题是：**这些架构分别在什么条件下才值得上。** 基座能力越强，需要外部脚手架的场景就越少，脚手架本身也越轻。

不过有一个方向值得注意。Anthropic 最近在 Claude Code 上推的 Agent Teams——多个 Claude Code 实例并行工作，通过文件系统共享上下文——跟传统 Multi-Agent 有一个本质区别：它不再需要复杂的通信协议和上下文压缩，每个 Agent 直接读写同一个代码仓库，用文件作为共享内存。这某种程度上就是在用模型的原生能力（理解代码、读写文件）来替代显式的 Agent 通信架构。

**显式脚手架还会存在很长一段时间，只是会被越来越强的基座能力不断压缩。** 对于工程师来说，更值得关注的是每一层架构到底在补偿什么能力缺失。这样当模型原生具备了这种能力时，你知道哪层可以拆掉。
