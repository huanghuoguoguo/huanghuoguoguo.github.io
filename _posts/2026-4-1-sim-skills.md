# sim-skills：一个检查文档格式合规的 skill

写大论文这段时间，最烦的事之一就是改格式。

前一天刚把正文调顺，第二天导进 Word 一看，字体乱了，行距也飘了。尤其是现在会让 AI 帮忙改论文，内容在 Markdown、Word、聊天窗口里来回搬，格式错得很隐蔽。不盯着看，真不一定发现。

学校给了模板，但没给检查器。学长学姐有成品论文，也只能拿来参考，没法一项一项对。手动核对很累，还容易漏。

市面上也有维普、知网这类格式检查服务。去看了一圈，总觉得不踏实。它们往往先让你选学校，可真到学院、课题组、期刊这一层，要求又可能不一样。模板一换，规则就得重来。

所以想做个东西，把”人工查格式”这件事拆开，让程序和 Agent 一起干。

现在先收成两条能力：

- 从模板或成品论文里提取格式规则。学校给的 `.dotm` 模板、学长学姐的合格论文，丢进去，输出自然语言规则文档 `spec.md`，把字体、字号、行距、缩进、页边距这些规则先收出来。
- 基于规则检查论文格式。把待检查的论文和 `spec.md` 丢进去，输出问题列表和 Markdown 报告，告诉你哪一段字体不对，哪一节的行距错了，页边距偏了多少。

改一版，查一次；再改，再查。有时候改完一个段落，手贱也想查一下，反正没成本。

---

## 为什么想做这个

那篇《为什么你 Vibe Coding 的产品没人买单》我在讲伪需求。sim-skills 则是写大论文时天天会碰到的真问题。

最早的想法其实很土：能不能先把模板里的规矩扒出来。字体、字号、行距、页边距、标题层级，老师最后盯的很多就是这些东西。要是能先抽成一份结构化规则，后面每改一版就查一遍，至少不用再肉眼对 Word 了。

往下做的时候，线路就慢慢分成了两条。

---

## 两条主线

sim-skills 收成两条主线。

### 主线一：规则提取

```
输入：模板.docx + 成品.docx（可选）
输出：spec.md
```

从学校提供的模板或学长学姐的成品论文中，提取字体、字号、行距、缩进这些格式规则，输出自然语言规则文档 `spec.md`。用户可以直接审核和修改，下游工具也直接读它。

### 主线二：格式检查

```
输入：待检查.docx + spec.md
输出：report.md
```

基于 `spec.md`，检查待检查文档的格式合规性，输出 Markdown 问题报告。

前一条回答”规矩到底是什么”，后一条回答”你这版到底错哪了”。

## 技能组织

一开始走的是很典型的 V1：把 workflow 直接写死在代码里。

输入模板，吐出规则；输入论文，吐出报告。能跑，但很快就别扭了。

有些规则是死的，程序很好办。可有些规则天生就模糊，比如题注到底算不算合规、页面视觉效果和 XML 描述对不上时该信谁，这些地方很难全靠硬编码写干净。流程一旦写死，Agent 就只剩下”按顺序调工具”这一种活法，碰到分支就很僵。

后来拆成现在这套结构。底层工具负责给事实，Agent 负责做判断：

```
.claude/skills/
├── word/                        # 底层解析库（共享实现）
│   └── scripts/
│       ├── docx_parser.py       # 文档解析器 + 数据模型
│       ├── parse.py             # 解析入口
│       ├── query_style.py       # 样式查询
│       ├── query_text.py        # 文本查询
│       └── render_page.py       # 页面渲染
│
├── parse-word/                  # ─┐
├── query-word-text/             #  │ 工具层：每个能力独立成 skill
├── query-word-style/            #  │ Agent 按需调用
├── render-word-page/            #  │
├── read-text/                   # ─┘
│
├── extract-spec/                # 规则提取 → spec.md
│   ├── SKILL.md
│   └── scripts/
│       └── collect_body_evidence.py
│
├── check-thesis/                # 格式检查（Python + Agent 混合）
│   ├── SKILL.md
│   └── scripts/
│       ├── translate_spec.py    # spec.md → 检查指令
│       ├── batch_check.py       # 批量执行确定性检查
│       ├── summarize_results.py # 聚合失败摘要
│       └── run.py               # workflow 包装层
│
├── compare-docs/                # 文档比对
│
├── evaluate-spec/               # spec.md 质量门禁
│   └── scripts/
│       ├── check_structure.py   # 核心主题覆盖
│       ├── check_conflicts.py   # 内部冲突检查
│       └── check_body_consistency.py
│
└── __libs__/                    # 跨 skill 共享库
    ├── spec_rules.py
    ├── text_sources.py
    ├── thesis_profiles.py
    └── utils.py
```

`word` 还是最底层的解析库，负责把 `.docx`/`.dotm` 文件拆成结构化的 Document IR，里面有段落、样式、布局这些信息。但它现在不再直接面向 workflow，而是作为共享实现，给 `parse-word`、`query-word-text`、`query-word-style`、`render-word-page` 这些独立的工具 skill 提供底层能力。每个工具 skill 只干一件事，Agent 按需组合调用。

`extract-spec` 仍然以 `SKILL.md` 为核心，Agent 自主编排提取流程。但现在多了一个 `collect_body_evidence.py`，用来采样真实正文段落的属性分布。这是因为光看 Word 样式定义不够靠谱——`Normal` 样式写的和正文段落实际用的可能根本不一样。加了这步之后，提取出来的规则有了段落级的统计证据，不再只靠样式定义拍板。

`check-thesis` 变化最大。原来只有一个 `run.py`，现在拆成了四步流水线：先用 `translate_spec.py` 把 `spec.md` 里的规则翻译成结构化检查指令，再用 `batch_check.py` 批量执行字体、字号、行距这些确定性检查，然后用 `summarize_results.py` 把原始失败项聚合压缩。程序搞不定的语义规则——比如摘要格式对不对、参考文献排列对不对——还是交给 Agent 判断。报告里会标注每条结论是 `Python 检查` 还是 `Agent 判断`。

`compare-docs` 不变，比对两份文档的格式差异。

`evaluate-spec` 替代了原来的 `validate-spec`。原来只检查 schema 缺不缺字段，现在做的是对 `spec.md` 的质量评估：覆盖性够不够、表述是不是具体可执行、内部有没有自相矛盾、正文规则有没有段落级证据支撑。它是一个 gate——评估不通过就退回去重提，不是建议性的”看起来不错”。

## 技术组合

sim-skills 用的是「程序 + AI + OCR + LLM Vision」这套组合。

程序负责解析 `.docx`/`.dotm` 的底层 XML，提取段落、样式、字体、间距，执行规则匹配，输出结构化结果。这些是死的，程序干最合适。

但有些事程序干不了。比如”题注应位于图表下方”这句话怎么落成检查规则，或者拿到一堆结构化数据后先查什么后查什么，这些需要 AI 来判断。

还有一种情况更麻烦：有些格式在底层 XML 里看不出来，或者 XML 描述和真实排版效果对不上。这时候就得靠 OCR / Vision 去看页面截图，拿渲染结果做裁决。

Agent 在这里面做的事更像编排。有时候先解析，有时候先截图，发现冲突了再决定往哪边追。

## 使用场景

最想用它干三件事：

- 改一次，查一次。每次修改论文后跑一次，看看有没有引入新的格式问题。
- 提交前总查。真要花钱送审，至少先把低级格式错误查掉。
- 对照模板或成品论文查差异。自己看不出来的时候，直接和”正确答案”比。

不是天天挂着的工具，但在论文修改期会用得很密。改一版查一版，很正常。

## 工程边界

目前支持 `.docx`（标准 Word 文档）和 `.dotm`（Word 模板）。

不支持 `.doc`（旧版二进制格式）。PDF 这块先认怂，页面级的精确判断——比如真实页码、跨页元素——也暂不支持。复杂表格、公式、图形对象的检查能力还比较有限，自动修复功能也还没做。

---

## 为什么叫 sim-skills

sim 一开始是 simulation，想模拟老师人工检查文档格式的过程。

先看整体布局，再看字体字号，再看标题、摘要、目录、参考文献这些部分。人工怎么查，就想怎么把它拆出来。

后来又觉得，sim 也可以顺手理解成 simplification：把一个很烦的人工流程拆成几个细粒度 skills，让 Agent 去组合。

---

## 还没想透的地方

sim-skills 目前只支持 `.docx`/`.dotm`，还想支持其他格式，比如 PDF。

PDF 是另一套问题。它没有样式 XML，很多格式信息都埋在渲染指令里。这个缺口很大，但确实是真需求——很多学校最后交的是 PDF，有些学校给的模板也直接是 PDF。

硕士论文盲审这波已经错过了，后面更现实的目标，大概是本科生论文、课程报告、期刊初稿这些场景。

但现在还是想先把 `.docx` 这条路走通。这个场景缺口最大，也最容易先做出能用的东西。PDF 先放着，不急着在第一版里把坑全跳一遍。
