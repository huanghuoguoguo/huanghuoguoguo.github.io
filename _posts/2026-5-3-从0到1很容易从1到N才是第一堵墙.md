最近冒出来很多 AI 编程测评。让 AI 从零搭一个网页、写一个游戏、做一个 demo，然后比谁快、谁好看、谁 bug 少。刷到这些内容的时候，我总有一种感觉：测是测了，但测错了地方。

不是说 0→1 不重要。是这事已经不缺解法了。Claude Code 能做，Codex 也能做，各种 agent 框架都能做，而且做得越来越快、越来越稳。真正缺解法的，是另一边：**AI 产出的东西，在需求变了之后，能不能不出事。**

维护过一个真实项目的人都知道我在说什么。修一个 bug 牵出来三个，加一个功能把隔壁模块的契约拆烂了，改到第三轮的时候 AI 开始忘掉第一轮做的设计决定。0→1 的 demo 可以靠一次性火力堆出来，但从 1 到 N 的维护，靠的是另一套能力。而这套能力，恰恰是现在的测评几乎不碰的。

## 上下文孤岛：agent 会干活，但不会留脚印

说一个 LangBot 上的具体场景。

有一轮要重构 RAG 模块的插件加载逻辑。我开了一个 agent 窗口，把当前的插件结构、加载顺序、几个已知的兼容性坑都摊给它，让它出一个重构方案。它做得不错，方案里把初始化顺序、依赖解析、fallback 路径都考虑到了。我按这个方案改完了，跑通了，关了窗口。

隔了两天，要在这个重构后的代码上加一个新的检索源。开了新窗口，把设计文档和重构记录都喂给新 agent。它读完了，开始改。改到一半我发现不对劲——它把上一轮我特意绕开的一条加载路径又加回来了。上一轮绕开它是因为某个三方库的兼容性问题，这个问题在当时的设计文档里只记了一行「不走 loader-b 路径」，但为什么不能走、试过哪些替代方案、绕开之后的副作用是什么——这些在当时的调试窗口里都讨论过，但文档没记，新 agent 也不知道。

它不是故意搞破坏。它只是没有那段记忆。而那段记忆在窗口关闭的时候就蒸发了。

这就是我说的上下文孤岛。不是为了省 token，不是窗口不够大——是一个 agent 在调试时长的判断，传不到下一个 agent 手里。

写文档能解决一部分：把架构决定、关键约束、已知的坑都写下来。做完一个大模块，整理一份 PRD，下次开窗口的时候喂回去。但只解决一小部分。开发是写一点、测一下、不对、调方向、再测——很多判断不是在设计阶段做出来的，是在调试的过程里长出来的。一个接口为什么多传了一个参数，一处绕弯的写法是因为踩了什么坑，这些在当时的对话窗口里活得好好的，窗口一关就没了。

这个问题在行业里已经有人给它起了名字——context rot（上下文腐化）。Chroma 的团队在 2025 年对 18 个 LLM 做了系统测试，结论很简单：模型对上下文的使用不是均匀的，上下文越长，推理质量越不可靠。MindStudio 的文章描述得更直白：每当你贴进去一大段代码、一份报错日志、几次失败的尝试，它们全部留在上下文窗口里，模型每次生成回复都要同时处理这些噪音。信息还在，但信号的比重越来越低。

Factory 的团队在他们的 Context Stack 设计里也说了同一件事：一个企业级代码仓可能有几百万 token，而最前沿的模型上下文窗口也就一两百万 token。硬塞塞不进去，塞进去了模型也未必能注意到真正重要的部分。他们的解决方案是多层上下文架构——仓库概览、语义搜索、文件系统命令、企业知识集成、分层记忆——每一层都把上下文当成稀缺资源来管理。

这些方案的方向是对的，但它们现在主要还是平台级的基础设施。对于个人开发者或者小团队来说，你能用的工具还很有限。Claude Code 的 `/compact` 能压缩历史，但压缩本身也会丢信息。你可以在项目里维护 `.claude/` 目录来存架构决策和约定，但这需要你有意识地去维护它——而这个维护动作本身，目前还没有被很好地自动化。

上下文孤岛的问题不在 token 不够用，在信息传递结构。每个 agent 窗口都是一座岛。岛和岛之间没有桥。你可以在岛上建灯塔（文档、PRD、CLAUDE.md），但灯塔照不到水面下的东西——那些调试时长的判断、绕弯的原因、被排除的选项。

## 反馈不仅是纠错，更是喊停

上下文孤岛让 AI 在维护阶段容易漂移。但漂移的背后还有一个更根本的问题：**AI 不知道怎么停下来。**

现在的 AI 编程流程大多是：你给指令 → AI 生成 → 你检查 → 不对 → 再生成。这个循环里没有「停」。AI 不会主动说「这个我改不了，需要你判断」「这个改动会影响三个我没见过的模块，我先不动」「你再确认一下这个方向，我感觉不太对」。

「停」的能力有多重要？维护代码的时候，最贵的不是改一行代码，是改错了方向而不自知。AI 在陌生模块里横冲直撞，改出一个看起来能跑、但契约已经偏了的结果——这时候它不但没有帮你省时间，反而在悄悄制造未来的债务。如果它能在这个时刻停下来，说「我不确定」，你至少有机会在偏掉之前纠正方向。

这件事不在任何 AI 编程测评的评分标准里。没人测「AI 会不会在不该动手的时候动手」。大家都在比谁能写得更快，没有人比谁更懂得沉默。

但我越来越觉得，一个真正好用的编程 agent，应该像一个有经验的同事——不仅知道怎么做，也知道什么时候不该做。这不是「降低产出」，是「提高产出的可靠性」。

## 多模型互审：有用，但还没闭环

我自己的用法是：让 Claude Code Opus 提方案或编码，然后让 Codex 或 GPT 审，审出来的问题回馈给 Opus 改。或者反过来——GPT 写一版，Opus 来审。

这个做法确实能多抓出不少问题。同一个模型审自己的产出，很容易掉进「回声室」——它在写代码时的盲区，审代码时还是同样的盲区。换个模型审，至少能从不同角度扫一遍。Zylos 团队 2026 年 3 月的多模型代码审阅研究把这个问题说得很清楚：单一模型自审存在「一致性偏见」，模型作为审阅者时，对正确反馈的确认率很高，但对错误反馈的拒绝率很低。就是说——它更擅长点头，不擅长摇头。

所以生成和审阅应该由不同的模型实例来做，最好用不同训练偏好的模型。这不是理论推演，是已经被验证过的工程实践。Cursor 的 BugBot 甚至用了 8 个并行审阅通道，每个通道随机打乱文件顺序（因为不同阅读顺序会产生不同假设），最后多数投票，只保留跨通道一致命中的问题。

Zylos 的 8 轮收敛实验里，从第 3 轮 7 个问题一直降到第 8 轮 0 个，收敛是干净的。他们也给出了实用的收敛窗口：交互式审阅 3-5 轮，CI/CD 自动化 1-2 轮，带硬上限。

但核心问题也在这里：逼近不等于闭环。多模型互审能逼近「更少的错误」，但不能保证「没有错误」。两个模型可能在同一个问题上都有盲区。审出来的问题可能是误报（研究数据显示最好的单次配置 F1 分数也只有 19.38%——真 bug 和假警告的比例大概是 1 比 9）。审不出来的时候，你也没法确定是「真的没问题」还是「两个模型都没看出来」。

更麻烦的是，代码量爆炸之后，人力 review 根本跟不上，全是模型审模型。如果你不知道这些模型在什么情况下容易集体失明，那你其实是在用一个黑箱检查另一个黑箱——看起来很闭环，实际上可能只是两个盲人在互相确认方向。

## Token 换正确率：TDD 的新位子

obra 的 Superpowers 框架截至 2026 年 5 月在 GitHub 上拿了 5 万多 star。它的做法是给 AI coding agent 套上一整套工程纪律：强制 brainstorm → 写设计文档 → 拆成 12 个 2-5 分钟的小任务 → 每个任务派独立子 agent → 每个子 agent 必须走 RED-GREEN-REFACTOR 的 TDD 循环——先写会失败的测试，再写能通过的代码，最后重构。

Superpowers 社区做过一次 A/B 对照：同一个需求（一个视频下载+转文章 web 应用），先用 Claude Opus 裸跑，再用 Superpowers 加持的 Opus 跑。裸跑版本能做出来，但第一次就跑不起来（import 路径错了），下载按钮行为也不对（在浏览器里打开视频而不是触发下载）。Superpowers 版本不但一次跑通，还在自审阶段主动发现并修了一个 SSE 解析的 bug。

这就是我说的 **token 换正确率**。

TDD 在这里已经不只是开发方法论了——它是一种买可靠性的方式。你烧 token 跑测试，换一个「至少契约没破」的保证。每一轮的 RED-GREEN-REFACTOR 都在消耗 token，但它也在消耗不确定性。测试就是给未来窗口里的 agent 一个客观信号：「这里改没改坏，你先跑这个。」

Superpowers 的 subagent 设计也有意思。每个任务派独立的子 agent，每个子 agent 有自己的干净上下文。这其实是在用架构解决上下文孤岛问题——不靠更大的窗口，靠把任务切小、切独立、每个子 agent 只负责一件事，然后把结果通过 git commit 和测试持久化下来。

但这里有一个还没被回答好的问题：测试本身也是 AI 生成的。谁来保证测试写对了？

AI 能写测试，也能把测试改歪。前面 Zylos 的数据说单一模型自审存在一致性偏见，那单一模型自测呢？它写的测试会不会刚好只覆盖它已经想对的部分，而绕开它想错了的部分？如果测试也跟着实现在漂移，那 TDD 就不是「兜底」，是「两个人一起编故事」。

这不是否定 TDD。正相反，我觉得 TDD 是现在最靠谱的方向。但它的天花板也很明显：token 能换正确率，但换不出判断力。

## 还没被回答的问题

写到这，我手上有一些观察、一些实践、一些外部研究支撑，但确实没有结论。下面这些问题是我真正关心、也还没想透的：

**上下文怎么从「窗口级」变成「项目级」？** 上一篇写 Hooks 的时候讲过——PostToolUse Hook 拦住每次变更 → 压缩分类 → 写进 markdown → SessionStart 时读回来。这个链路在技术上已经走得通了。但现实是，Hooks 能搭桥，agent 不知道什么时候该过桥。它不会在调试撞墙之后主动把「为什么选了这个数据结构」「试过哪些死路」写进 memory——这些判断要么靠人手动记，要么靠 Auto Memory 碰运气。我把 friction ≥ 3 次作为沉淀触发条件，但那是给人用的规则，不是给 agent 用的。agent 自己能不能学会识别「这条信息下次会用」？这才是我觉得还没被回答的问题。

**「停」的能力能不能被训练或约束出来？** 目前的 agent 几乎不会主动喊停。能不能通过 prompt 设计、奖惩信号、或者类似 Superpowers 的强制工作流，让 agent 在不确定的时候主动收敛、主动升级给人——而不是继续往前冲？

**多模型互审的收敛边界在哪？** 3-5 轮能收敛到 0 个新发现，但这个 0 是「真没有问题了」还是「两个模型都看不出来了」？不同模型组合的盲区有没有办法标定？能不能在人力确实跟不上 review 的时候，至少知道自己不知道什么？

**测试的完整性怎么自证？** TDD 保证了已有的测试要过，但保证不了测试覆盖了该覆盖的东西。有没有办法让 AI 在写代码的同时标记「这段逻辑我没有写对等测试」「这个边界条件我只测了正常路径」——至少在不确定的地方把不确定性亮出来？

这些问题现在没人回答。

但我觉得它们才是真正该被测评的东西。

现在的测评在比谁 0→1 快。等有一天测评开始比谁从 1→N 不出事的时候，上面这些问题就不是我一个人的追问了。
