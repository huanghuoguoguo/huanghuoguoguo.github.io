## 第 1 章 Linux 内核简介

Linux 内核是单内核（monolithic）架构，但通过模块机制吸收了微内核的部分优点——可以在运行时动态加载/卸载内核模块，不需要重新编译整个内核。

内核与用户空间的边界就是系统调用。操作系统中经典的 trap/return-from-trap 机制，在 Linux 中的具体实现是通过 `int 0x80`（x86）或 `syscall` 指令触发，从而切换到内核模式执行相应逻辑。

## 第 2 章 从内核出发

内核开发和用户态开发的几个关键差异：

| 差异 | 用户态 | 内核态 |
|------|--------|--------|
| C 库 | 完整的 glibc | **没有 libc**，内核自己实现必要的函数 |
| 浮点运算 | 随意使用 | **禁止**，保存/恢复浮点寄存器的开销太大 |
| 栈大小 | 通常 8MB | **固定且很小**（通常两页，8KB 或 16KB） |
| 并发 | 可选 | **无处不在**，多处理器 + 中断 + 抢占 |
| 错误后果 | 进程崩溃 | **整个系统崩溃**（kernel panic） |

内核开发中栈只有几 KB 这一点，意味着不能在内核函数里声明大的局部数组，也不能做太深的递归。

## 第 3 章 进程管理

### task_struct

Linux 用 `task_struct` 描述进程，这是一个巨大的结构体，包含进程的所有信息：PID、状态、调度信息、打开的文件、地址空间、信号处理等。所有 `task_struct` 通过双向链表连接。

### 进程状态

```
TASK_RUNNING（就绪/运行）
    ↓ 等待事件
TASK_INTERRUPTIBLE（可中断睡眠）/ TASK_UNINTERRUPTIBLE（不可中断睡眠）
    ↓ 事件到来
TASK_RUNNING
    ↓ 退出
TASK_ZOMBIE（僵尸态，等待父进程 wait）
```

`TASK_UNINTERRUPTIBLE` 状态的进程不响应信号。所以有时候你会看到 `kill -9` 也没法立刻把进程干掉，它还卡在一个不可中断的 I/O 等待里。

### fork 与写时拷贝（COW）

`fork()` 创建子进程时不会立即复制父进程的整个地址空间，而是让父子共享同一份物理页面，标记为只读。只有当其中一方尝试写入时，才会触发缺页异常，内核这时才复制那一页。这就是**写时拷贝**——大幅减少了 fork 的开销。

“进程启动为什么不直接 exec，而是先 fork” 这个问题，放到这里就更具体了。因为有 COW，fork 的代价其实不高，只复制页表，不会立刻复制数据页。所以 fork + exec 这套组合在性能上没有想象中那么浪费。

### Linux 的线程实现

Linux 不区分进程和线程——线程就是共享地址空间的进程。创建线程实际上是调用 `clone()` 时传入 `CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND` 等标志，让新进程共享父进程的虚拟内存、文件系统信息、文件描述符表和信号处理。

这和 JVM 的线程模型直接相关。由于 HotSpot 的传统 Java 平台线程是 1:1 映射到操作系统内核线程的，它的每个 `new Thread()` 最终在底层都会调用 `clone()`去创建一个轻量级进程。相比之下，JDK 19+ 引入的虚拟线程采用了完整的用户态调度，不再 1:1 映射到内核线程，从而极大地减少了内核线程的创建开销。

## 第 4 章 进程调度

### CFS 完全公平调度器

Linux 2.6.23 引入的 CFS（Completely Fair Scheduler）是当前默认的调度器。它一直在维护每个进程的**虚拟运行时间**（vruntime），然后优先挑 vruntime 最小的那个去跑。最近分到的 CPU 越少，越容易先被调度到。

数据结构用**红黑树**，按 vruntime 排序。选择下一个进程就是取红黑树的最左节点，O(1)。进程运行后 vruntime 增加，重新插入红黑树，O(log n)。

优先级通过 nice 值（-20 到 +19）影响 vruntime 的增长速度：nice 值越低（优先级越高），vruntime 增长越慢，相当于获得更多 CPU 时间。

对比《操作系统导论》中的调度算法：

| 调度器 | 思路 | 问题 |
|--------|------|------|
| FIFO | 先来先服务 | 短任务饿死 |
| SJF | 最短优先 | 长任务饿死，且无法预知执行时间 |
| MLFQ | 多级反馈队列 | 参数调优复杂 |
| **CFS** | 虚拟运行时间 + 红黑树 | 公平性好，实时任务用独立的 FIFO/RR 策略 |

### 实时调度

Linux 支持两种实时调度策略：`SCHED_FIFO`（先到先得，不会被同优先级抢占）和 `SCHED_RR`（时间片轮转）。实时进程的优先级永远高于普通进程。

## 第 5 章 系统调用

系统调用是用户空间进入内核的唯一合法入口。流程：

```
用户态调用 read()
  → glibc 包装函数把系统调用号放入寄存器
  → 执行 syscall 指令（x86-64）/ int 0x80（x86-32）
  → CPU 切换到内核态，跳转到系统调用入口
  → 根据系统调用号查 sys_call_table，调用对应的内核函数
  → 返回用户态
```

内核必须验证用户传入的所有指针参数——用户可能传一个非法地址试图读写内核内存。`copy_from_user()` 和 `copy_to_user()` 就是干这个的，它们在复制数据的同时检查地址的合法性。

这就是《操作系统导论》中"系统调用与特权级别"的 Linux 具体实现。

## 第 6 章 中断和中断处理

硬件设备通过中断通知 CPU 需要处理事件（如网卡收到数据包、磁盘完成读写）。中断处理的核心矛盾：**要快速响应（不能让其他中断等太久），但有些工作量又很大**。

Linux 的解决方案是把中断处理分成两部分：

- **上半部（top half）**：中断处理程序本身。在中断上下文中运行，不能睡眠，必须尽快完成。通常只做最紧急的事：确认中断、拷贝数据到内存、调度下半部
- **下半部（bottom half）**：推迟执行的工作。在稍后的安全时机执行，可以处理更耗时的任务

中断处理程序运行时，这条中断线会先在当前处理器上关掉，免得同一中断自己套自己。不过别的中断还是能继续响应。

## 第 7 章 下半部和推后执行的工作

下半部有三种实现机制，各有适用场景：

| 机制 | 执行上下文 | 能否睡眠 | 并发性 | 适用场景 |
|------|-----------|---------|--------|---------|
| 软中断（softirq） | 中断上下文 | 不能 | 同一类型可在不同 CPU 并发 | 对性能要求极高的场景（网络、块 I/O） |
| tasklet | 中断上下文（基于 softirq） | 不能 | 同一类型不会并发 | 大多数中断下半部 |
| 工作队列（work queue） | 进程上下文（内核线程） | **能** | 可调度、可睡眠 | 需要睡眠的场景（如分配大量内存） |

选择原则很简单：**需要睡眠就用工作队列，不需要睡眠就用 tasklet，对性能极端敏感且能自己处理好并发就用 softirq**。

这个上半部/下半部的分离思想在很多地方都能看到。Redis 的事件循环就是类似的思路——主线程（"上半部"）快速接收请求放入队列，耗时的操作（如 RDB 持久化）交给后台线程（"下半部"）。Netty 的 EventLoop 也是：I/O 事件在 EventLoop 中快速处理，耗时的业务逻辑 offload 到业务线程池。

## 第 8-19 章（待读）

后续章节的大致脉络，等重新看一遍再补充：

**同步机制**（第 8-9 章）：原子操作、自旋锁、信号量、互斥体、顺序锁、内存屏障。自旋锁适合持有时间短的场景（不会让出 CPU），信号量/互斥体适合可能睡眠的场景。对应《JVM 虚拟机》中的锁升级：偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（OS 互斥量），思路是一致的。

**内存管理**（第 11 章）：物理页管理（伙伴算法）、slab 分配器（内核对象缓存）、高端内存映射。slab 的思路和对象池类似——预分配固定大小的内存块，避免频繁的分配/释放开销。

**VFS**（第 12 章）：虚拟文件系统是 Linux "一切皆文件"的实现基础。超级块、inode、dentry、file 四个核心对象。这层抽象让上层代码不需要关心底层是 ext4、XFS 还是 NFS。

**块 I/O**（第 13 章）：I/O 调度算法（电梯、最后期限、CFQ、空操作）。和《操作系统导论》中 IO 分层框架的内核层直接对应。

**页缓存和回写**（第 15 章）：page cache 对读写性能很关键。读文件先看缓存，写文件先写缓存，再慢慢刷盘。这个思路和数据库里的 Buffer Pool 很像。你如果看 InnoDB 的 Disk Buffer Pool，也会觉得它是在用户态又搭了一层类似 page cache 的东西。
