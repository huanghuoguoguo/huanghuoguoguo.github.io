最近我读到一篇很有启发的文章：`sysls` 的 `How To Be A World-Class Agentic Engineer`。它讨论的重点很明确：**怎样才能把 agent 真正用稳、用深、用到生产环境里。**

先说明一下：**这一篇的核心内容，主要整理自小红书作者 `ody` 对 `sysls` 那篇文章的转述，我基本保留了原意，只做了 Markdown 化、结构整理和少量表达顺序调整。**

前两篇都以整理为主。不过我也不想把它们写成完全没有自己痕迹的转载，所以文中会穿插少量引用块，补上我自己在实习、写代码和用 AI 过程里慢慢长出来的一些判断。第三篇再专门回到一个具体项目，把这些判断落到实战里。

这篇文章也和我自己的经验高度吻合。过去这段时间里，无论是实习项目，还是后来给开源项目贡献代码，我都大量使用了 AI。回头看，真正起作用的是上下文控制、规则沉淀和工作流设计，不是工具越堆越多。

所以这一篇只讲第一部分：**个人开发者如何从最小配置出发，逐步建立自己的 agent 工作流。**

---

## 1. AI 公司会替你解决问题

你的 `CLAUDE.md` 写了 26000 行。你装了各种 memory（记忆）插件、agentic harness（agent 工具框架）和第三方工具。你每天都在想，是不是还差一个包没装。

但你的 agent 表现依然不稳定，而别人用基础 CLI 就能搭出完整的生产系统。

作者 `sysls` 从 agent 还几乎不会写代码的时代就开始用，试过各种工具和框架，也在生产环境里跑过交易信号系统、基础设施和数据管道。

他的结论很简单：**回到最简配置之后，反而做出了最好的成果。Less is more。**

AI 前沿实验室的员工，本来就是最重度的 agent 用户。他们有几乎无限的 token 预算，用的是尚未公开发布的最新模型。

如果某个第三方工具真的解决了核心痛点，AI 大厂通常会最先感知到，而且很可能会把这类能力逐步做进产品里。对平台型公司来说，把核心体验长期交给外部依赖，往往不是它们愿意接受的状态。

回顾过去一年的产品演进，你会看到这个规律反复出现：

- `Skills` 最早是社区里一些用户发明的“规则模板”，后来成了 Claude Code 和 Codex 的官方能力。
- `Planning Mode` 最早是用户发现“先规划、再实现”能显著提高质量，后来也被做成了核心功能。
- `Memory`、`subagents`、语音输入、远程控制，基本都走过类似路径。

反面例子也有。`stop-hook` 曾经很流行，因为早期 agent 不愿意做长任务，需要外部钩子强制它继续。但在 `gpt-5.2-codex` 更新之后，agent 自己就能处理长任务了，`stop-hook` 很快就变得多余。

所以你唯一真正需要做的，是：

- 定期更新 CLI 工具；
- 阅读更新日志；
- 谨慎引入第三方依赖。

很多插件短期内也许确实有用，但从更长的周期看，它们很容易变成未来版本里的技术债。

## 2. 上下文控制是核心技能

核心原因只有一个：**上下文控制能力，是 agentic 工程的核心技能；工具堆叠，往往是它的反面。**

给 agent 完成任务所需的精确信息量，**多一点都是噪声**。

装太多插件和记忆系统，很容易导致 `context bloat`（上下文膨胀）。你让 agent 写一个猜字游戏，它却在回忆 71 个会话前的进程崩溃记录，还在纠结 26 个会话前的 memory 管理笔记。

这些信息和当前任务没有关系，但它们会占据 agent 的注意力。

最有效的上下文控制方法，是**把计划和实现分离**。

坏指令：

> 搭一个 auth 系统。

这个任务里，agent 需要先调研“auth 有哪些方案”“各自优缺点是什么”，上下文很快被无关的实现细节填满。等真正开始写代码时，它更容易混淆，或者幻觉出不相关的内容。

好指令：

> 用 JWT 认证，`bcrypt-12` 做密码哈希，开启 refresh token rotation，7 天过期。

这时 agent 不需要调研，直接进入实现。

如果你自己也不确定该用什么方案，做法也很简单：

1. 先开一个研究任务，让 agent 调研并给出建议。
2. 拿到结论后，开一个全新的会话。
3. 在干净上下文里只做实现。

这个模式几乎适用于所有复杂任务。关键是：**让实现 agent 的上下文里，只有实现所需的信息。**

这一点其实已经被集成进主流工具里了。比如 Claude Code 的 `Planning Mode` 结束后，就可以清空上下文，再进入自动执行。

> 我后来给自己找了个更顺手的说法：**持续收束 AI 在工程问题上的解空间。**
>
> 写清任务、拆开计划和实现、把规则固定下来，看起来像几套不同动作。真落到项目里，它们都在干一件事：别让模型带着一堆还没裁掉的可能性乱跑。

## 3. 利用 agent 的“讨好型人格”

Agent 的天性，是尽可能满足用户指令。

但这种“讨好”有个明显副作用：**如果你让它“找 bug”，它就会努力找出 bug，哪怕需要自己编一个。**

很多人抱怨 LLM 幻觉，却没意识到，自己的 prompt 本身就在引导幻觉。

### 实操 1：用中性 prompt

“找数据库的 bug”是一个带方向性的 prompt，它预设了“这里一定有 bug”。

更好的写法是：

> 读一遍数据库代码的逻辑流程，汇报所有发现。

这就是中性 prompt。有时候它会发现 bug，有时候它只是如实描述代码如何运行。但它不会为了迎合你的目标，去凭空编造问题。

### 实操 2：三 agent 对抗框架

这是原文里最精彩的方法：把 `sycophancy`（讨好倾向）反过来当作系统性工具使用。

**第一个 agent：Bug Finder**

给它的规则是：

- 低影响 bug 记 1 分；
- 中影响 bug 记 5 分；
- 高影响 bug 记 10 分。

这个 agent 会疯狂找 bug，甚至把一些不是 bug 的东西也报上来。它给你的，是一个“所有可疑问题的超集”（superset）。

**第二个 agent：Adversarial**

给它的规则是：

- 每成功反驳一个 bug，就得到该 bug 对应的分数；
- 如果反驳错了，扣双倍。

这个 agent 会非常激进地反驳前一个 agent，但因为有惩罚机制，它又不敢太草率。它给出的，是“真实 bug 的子集”（subset）。

**第三个 agent：Referee**

给它的规则是：

- 你手上有标准答案；
- 判断正确得 1 分；
- 判断错误扣 1 分。

裁判会认真审查前两个 agent 的每一条分歧。

本质上，每个 agent 都在尽全力“讨好”你，但你把这种倾向拆成了三个互相制衡的角色，于是它们反而变得可靠。

当然，我自己没有用过这么复杂的框架。我的日常做法更简单：**Claude 负责写代码，Codex 负责审查。** 这已经足够有效。

## 4. 让 agent 知道怎么“结束”

Agent 通常知道怎么开始任务，但不知道怎么结束任务。它经常写几个占位函数，就宣布完成。

人类知道什么时候叫“做完了”，agent 需要被显式告知。

### 方法 1：用测试做终止条件

直接写清楚：

> 这 X 个测试全部通过之前，任务未完成。不许修改测试代码。

这样你只需要审查测试本身是否合理。测试通过之后，任务是否完成就有了客观标准。

### 方法 2：截图 + 视觉验证

这对 UI 任务尤其有效。让 agent 实现功能后截图，再由你验证截图是否符合设计要求；不符合就继续迭代。

### 方法 3：任务契约

写一个 `{TASK}_CONTRACT.md`，把所有完成标准列清楚，例如：

- 哪些测试必须通过；
- 哪些截图需要验证；
- 哪些行为必须确认。

让“完成”从模糊感受变成明确契约。

> 这件事我后来踩过一个更具体的坑：测试能保护稳定契约，但保护不了一直摇摆的目标。需求和接口如果还在来回变，测试很多时候也只是在陪着设计一起搬家。

## 5. `CLAUDE.md` 的正确写法

很多人把 `CLAUDE.md` 当知识库来写，堆几千甚至上万行。其实 `CLAUDE.md` 更适合被当成一个**逻辑路由表**。

它应该尽可能精简，只包含“在什么场景下去读哪个文件”的 `if-else` 逻辑。具体知识和偏好，放在独立文件里，通过 `CLAUDE.md` 的条件分支按需引导 agent 去读取。

一个典型结构可以写成这样：

- 如果在写代码，读 `coding-rules.md`；
- 如果测试失败，读 `test-failing-rules.md`；
- 如果需要实现某个功能，读对应的 `skill-x.md`；
- 如果发生 `compaction`，重新读取任务计划和相关文件。

这里有两个实践特别重要。

**Rules 的编码方式**

每当你看到 agent 做了一件让你不满意的事，就写一条规则，告诉它：以后在做这类事情之前，先读这条规则。

只要规则写得具体，agent 基本不会再犯同样的问题。

**Skills 的编码方式**

如果你希望 agent 按特定方法去解决某一类问题，就先让它研究一次可行方案，再把结果写成一个 skill 文件。

这样你可以在 agent 真正遇到这类任务之前，先审查并修改它的方法。

## 6. Compaction 之后的上下文恢复

还有一个经常被忽视的场景：长会话里的 `compaction`（上下文压缩）。

压缩之后，agent 很容易丢掉关键上下文，然后开始用错误假设补空白。而 agent 目前在“连接线索”和“填补空白”这两件事上，依然很弱。

一旦它开始做假设，输出质量通常会断崖式下降。

所以在 `CLAUDE.md` 的最优先位置，最好写一条规则：

> 如果发生 compaction，先重新读取任务计划和相关文件，再继续工作。

不过我自己的实战习惯更激进：**直接避免长对话。** 一个 plan 执行完，我就开一个新对话；即使 bug 还没修完，也尽量切新会话处理。

## 7. 从最简配置开始迭代

如果你今天重新搭建 agentic 工作流，路径其实很清晰：

- 回到基础 CLI；
- 先把第三方插件和工具框架放一边；
- 在真实工作中，遇到 agent 犯错就写一条 rule；
- 遇到需要固定方法解决的问题，就写一个 skill；
- `CLAUDE.md` 只放路由逻辑，具体内容都放在独立文件里。

规则积累到一定程度以后，再做一次系统性清理：

- 合并重复规则；
- 消除矛盾；
- 删除已经过时的内容。

这就是 AI Native 工程师的日常节奏：

> 不断加 rules 和 skills，agent 表现越来越好；
> 到某个临界点，性能开始下降（规则矛盾或信息过载）；
> 做一次清理合并；
> 性能恢复。

这和带一个新助理几乎没有区别。你不会指望助理第一天就知道你的全部偏好，而是会在长期协作里慢慢让它理解你的工作方式。

区别只在于：agent 的“偏好学习”是通过 `rules` 和 `skills` 文件持久化下来的。

所以，第一部分的结论很简单：

- agentic 工程的核心能力是上下文管理；
- 工具堆叠通常是累赘；
- 从最简配置开始；
- 用 rules 和 skills 迭代出自己的工作流；
- 定期清理。

Agent 还在快速进化，今天的最优方案明天可能就会变。但“控制上下文、精确指令、迭代规则”这三个原则，大概率会比任何具体工具活得更久。
