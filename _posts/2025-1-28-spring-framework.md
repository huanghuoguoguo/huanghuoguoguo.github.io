之前我读《Spring Framework》相关书籍时，只看了一半进度，且感触不深。当时看着源码里繁复的 `BeanFactory`、`ApplicationContext` 以及各种后缀为 `Processor` 的类，觉得过于啰嗦——为什么一个 Bean 的创建要经过 `BeanDefinition` 解析、`BeanFactoryPostProcessor` 修改元数据、实例化、属性填充、`BeanPostProcessor` 前后置处理、初始化回调这么长的链条？为什么 `ApplicationContext` 要在 `BeanFactory` 之上又包一层事件机制、资源加载、国际化？这些层层包裹的抽象看上去只是在制造复杂度。

不理解的原因其实很简单：我当时并没有亲自去构建过一个通用的、旨在解决复杂扩展性问题的系统。没有体会过痛点，自然无法体会那些高度抽象和预留钩子的价值。你在家门口散步，自然觉得路灯多余；直到在深山老林迷过路，才懂得路灯的好。

直到后来实习，业务需要我从零尝试构建一个内部的测试系统框架（实习做的观测工具），我才意识到了差距。

---

## 1. 实习中的架构实践与反思

我实习时面对的业务方是 QA 团队。大家知道，不同 QA 同学的测试方法、观测手段和关注的指标各不相同。基于架构设计的一个核心理念——"不变的是框架，变动的是业务"，我最初的期望是比较理想化的：我想构建一个通用的底层架构系统，框架本身保持稳定，而将多变的、定制化的测试逻辑交由作为用户的 QA 自己去配置和实现。

为了达到这个目的，我在系统里尝试手写了统一的启动生命周期、类似 IoC 的依赖管理以及插件化机制。

### 具体做了什么

实习做的观测工具的定位是一个性能测试与观测框架。它的核心流程大致是：**启动 → 加载配置 → 注册任务与插件 → 执行采集 → 汇聚指标 → 输出报告**。任务通过本地文件系统扫描自动注册，插件用 `git clone` 远程拉取再动态装载，核心代码不用动。我当时觉得这种设计已经很"开闭原则"了。

依赖管理方面，我手写了一个简易的容器：用一个全局的 `Map<String, Object>` 存放所有组件实例，通过字符串 key 进行查找和注入。启动时按固定顺序实例化各个组件，组件之间通过容器获取对方的引用。

生命周期管理方面，我定义了 `init()`、`start()`、`stop()`、`destroy()` 四个阶段，每个组件实现对应接口即可。

听起来似乎方向正确。但事与愿违，由于我才疏学浅，实际开发中暴露出了非常多的问题：

### 暴露的问题

1. **扩展点（钩子）设计不合理**：我原以为理顺核心流程就可以，结果当 QA 想要在特定生命周期阶段插入自己的测试逻辑或观测探针时，由于系统缺乏合理的预置机制，最后往往演变成在核心启动流中加入各种 `if-else`。比如有人想在"加载配置之后、注册任务之前"插入一段环境预检逻辑，而我的生命周期里根本没有这个缝隙。原计划的"用户修改变动部分"变成了"我频繁修改底层框架"。

2. **依赖管理的脆弱性**：全局 `Map` 的方案看似简单，实则危机四伏。组件的初始化顺序完全靠手动编排，一旦新增组件有循环依赖或顺序依赖，就得回头调整启动流程。更致命的是，字符串 key 没有编译期检查——拼错一个字符，运行时才会抛 `NullPointerException`，排查起来非常痛苦。

3. **交付困境**：因为一直没理清楚架构，加上自己项目管理经验不足，我的系统遭遇了频繁的重构。代码总是处于一种不稳定的状态，导致经常无法按期交付。正如在[《持续交付》]({% post_url 2026-1-24-持续交付 %})中反思的，"重构比 commit 还勤快"是最可怕的状态。

4. **问题评估的主观化**：由于系统耦合度越来越高，每次遇到一个偶现的缺陷，就会陷入焦虑。我还没有学会如何客观地评估问题的边界和影响范围，总是习惯性地将问题严重化。因为不知道改一处会不会崩全局，所以每个 bug 看起来都像是系统性问题。

5. **"类比 Spring"不等于"理解 Spring"**：我当时以为自己在实践 IoC 和生命周期管理，但其实只是照猫画虎。我画的是形，丢的是神。Spring 的 IoC 容器不是一个存放对象的 `Map`，它是一套完整的对象生命周期治理体系；Spring 的生命周期不只是四个回调方法，而是一条精心编排的、每个节点都开放给外部干预的流水线。当时的我不理解这些区别，因为我还没有被自己的简陋方案折磨够。

## 2. 重新认识 Spring 的设计哲学

带着这些真实的挫败经历，回头重看 Spring Framework 构建通用系统的思路，才真正明白了它的设计意图。Spring 的本质不仅仅是一个存放单例 Bean 的容器，而是一个将"开闭原则"和"扩展点（Hooks）"做到极致的工程典范。

### BeanDefinition：元数据与实例的分离

Spring 并不是直接 `new` 一个对象就完事了。它先将 Bean 的定义信息（类名、作用域、依赖关系、初始化方法等）解析成 `BeanDefinition` 对象，存入 `BeanDefinitionRegistry`。这一步只处理"描述"，不处理"创建"。

为什么要这么做？因为一旦元数据和实例分离，`BeanFactoryPostProcessor` 就可以在实例化之前修改元数据。举个例子，`PropertyPlaceholderConfigurer` 能把 `${db.url}` 替换成真实的数据库地址——这一切发生在任何 Bean 被创建之前。

回想实习做的观测工具，我的组件定义和实例化是混在一起的：代码里写死了 `new SomeComponent()`，如果想在创建前修改配置，只能去改源码。Spring 把这个问题用元数据层彻底解耦了。

### BeanPostProcessor：实例化前后的万能钩子

* **BeanPostProcessor (BPP) 与 BeanFactoryPostProcessor**：当我面对无法优雅整合 QA 的定制化探针时，Spring 其实早就给出了标准答案。它通过暴露这类极高权限的接口，允许开发者在对象实例化前后、甚至元数据解析阶段横插一脚。核心容器对此无需关心，完美解耦了框架的不变性与业务的可变性。

具体来说，`BeanPostProcessor` 的 `postProcessBeforeInitialization` 和 `postProcessAfterInitialization` 两个方法，让任何人都可以在 Bean 初始化前后做手脚。AOP 代理就是通过 `AbstractAutoProxyCreator`（一个 BPP）在初始化后把原始对象替换成代理对象实现的。而 `@Autowired` 的注入则由 `AutowiredAnnotationBeanPostProcessor` 在属性填充阶段完成。

如果实习做的观测工具当时有这样的机制，QA 想在某个组件初始化后插入探针，只需要注册一个 Processor，而不是来找我改框架代码。

### 事件驱动：模块间的松耦合通信

* **事件驱动 (ApplicationEvent)**：各个模块不必死板地等待对方初始化，而是通过注册监听器，在容器到达特定状态时自动执行逻辑。这恰恰是解决插件挂载、解除硬编码依赖的最佳手段。

Spring 内置了 `ContextRefreshedEvent`（容器刷新完成）、`ContextClosedEvent`（容器关闭）等事件。任何 Bean 只要实现 `ApplicationListener` 或用 `@EventListener` 注解，就能监听这些时刻。更关键的是，开发者可以自定义事件——比如"数据源初始化完成"事件，让依赖数据源的组件不再需要关心数据源是谁、什么时候初始化的，只需要监听这个事件然后执行自己的逻辑。

实习做的观测工具 当时的插件加载顺序是硬编码在启动流程里的。如果两个插件之间有先后依赖，我就得在启动代码里手动调整顺序。如果用事件驱动的方式，插件 A 初始化完成后发出一个事件，插件 B 监听到这个事件后自行启动，启动流程就不需要知道它们之间的关系了。

### Aware 接口：可控的上下文注入

Spring 定义了一系列 `Aware` 接口：`BeanNameAware`、`BeanFactoryAware`、`ApplicationContextAware`、`EnvironmentAware` 等。Bean 实现对应接口后，容器会在特定阶段将相应的上下文对象"推送"给它。

这种设计看似简单，实则是一种克制。它没有让每个 Bean 都能随意访问容器的一切，而是通过接口声明式地表达"我需要什么"。这比实习做的观测工具里直接从全局 `Map` 中 `get` 任意对象要安全得多——后者本质上是全局变量，谁都能访问任何东西，依赖关系完全不可见。

### 异常边界与容错隔离

* **异常边界与容错隔离**：Spring 对抽象和生命周期的严格划分，使得组件故障被限制在局部。这种隔离度给开发者提供了客观评估问题的依据，而不是一出 bug 就觉得系统底层要垮塌。

比如 `BeanCreationException` 会精确指出是哪个 Bean 在哪个阶段失败，是构造器出错、属性注入失败还是初始化回调抛异常。错误信息的精确度直接决定了排查的效率。在实习做的观测工具里，由于启动流程是一个大的 `try-catch`，任何组件的失败都表现为同一个笼统的启动失败，定位问题全靠断点和打日志。

## 3. IoC 的本质——控制反转不只是依赖注入

在实习期间，我一直把"IoC"简单等同于"不 `new` 对象，让容器帮你创建"。这个理解太窄了。

依赖注入（DI）只是 IoC 的一种实现手段。IoC 的真正含义是**控制权的反转**：不是你调用框架，而是框架调用你。在传统编程中，应用程序代码是主控方，它决定何时创建对象、何时调用方法、何时释放资源。在 IoC 容器中，这些控制权全部交给了容器——容器决定何时创建你的 Bean、何时注入依赖、何时调用你的初始化方法、何时销毁。

这就是所谓的**好莱坞原则**（Hollywood Principle）："Don't call us, we'll call you."

Spring 的 `AbstractApplicationContext.refresh()` 方法就是这种控制反转的集中体现。它定义了容器启动的完整流程，每一步都是确定的、不可跳过的，但每一步又都开放了扩展点：

```
prepareRefresh()                    // 准备工作，记录启动时间，设置状态标志
  → obtainFreshBeanFactory()        // 创建/刷新 BeanFactory，加载 BeanDefinition
  → prepareBeanFactory()            // 配置 BeanFactory 的标准特性
  → postProcessBeanFactory()        // 子类可重写的钩子
  → invokeBeanFactoryPostProcessors()  // 调用所有 BeanFactoryPostProcessor
  → registerBeanPostProcessors()    // 注册 BeanPostProcessor
  → initMessageSource()             // 初始化国际化
  → initApplicationEventMulticaster() // 初始化事件广播器
  → onRefresh()                     // 子类可重写的钩子（如 Spring Boot 在此启动内嵌 Tomcat）
  → registerListeners()             // 注册监听器
  → finishBeanFactoryInitialization() // 实例化所有非懒加载的单例 Bean
  → finishRefresh()                 // 发布 ContextRefreshedEvent
```

这就是一个典型的**模板方法模式（Template Method）**。整体流程由父类控制，子类通过重写特定方法来注入自己的行为。Spring Boot 的 `ServletWebServerApplicationContext` 正是通过重写 `onRefresh()` 来启动内嵌的 Web 服务器。

对比实习做的观测工具的启动流程：我写了一个 `start()` 方法，里面按顺序调用各个组件的初始化。如果要在"加载配置之后、注册任务之前"插入逻辑，就得去改这个方法的源码。没有钩子，没有事件，没有子类可重写的扩展点。Spring 的 `refresh()` 则在每两个步骤之间都留好了缝隙，外部可以通过多种机制（Processor、Listener、子类重写）插入逻辑而不触碰核心代码。

这也是[《架构整洁之道》]({% post_url 2026-1-25-架构整洁之道 %})中反复强调的：好的框架只暴露行为扩展点，不暴露执行控制权。扩展者可以无限"加料"，却永远动不了"锅"本身。Spring 把"锅"（`refresh()` 的流程）牢牢掌握在自己手里，同时在"锅"的每个关键节点都挂上了钩子，让使用者自由发挥。

### DIP 在 Spring 中的体现

经典的依赖反转原则（DIP）在 Spring 中随处可见。`BeanFactory` 是一个接口，`ApplicationContext` 也是一个接口，它们的具体实现（`DefaultListableBeanFactory`、`AnnotationConfigApplicationContext`）被藏在框架内部。开发者面向接口编程，永远不需要知道底层用的是哪个实现类。

更深层地看，Spring 通过 DI 机制把 DIP 变成了一种全自动的工程实践。在实习做的观测工具里，虽然我也定义了接口，但组件之间的装配是手动的——谁依赖谁、谁先初始化，全靠代码里的显式调用。Spring 把这些"布线"工作全部收归容器，开发者只需要声明依赖（`@Autowired`），容器负责解析和注入。依赖方向在编译期就确定了（面向接口），运行时由容器填充具体实现，源代码依赖方向和运行时控制流方向相反——这正是 DIP 的精确定义。

## 4. 从 Bean 的一生看生命周期设计

Spring 中一个 Bean 从出生到死亡，经历的完整链条大致如下：

```
BeanDefinition 注册
  → BeanFactoryPostProcessor 修改元数据
  → 实例化（构造器）
  → 属性填充（@Autowired 等）
  → BeanNameAware / BeanFactoryAware / ApplicationContextAware 回调
  → BeanPostProcessor#postProcessBeforeInitialization
  → @PostConstruct / InitializingBean#afterPropertiesSet / init-method
  → BeanPostProcessor#postProcessAfterInitialization（AOP 代理在此生成）
  → Bean 就绪，可被使用
  → @PreDestroy / DisposableBean#destroy / destroy-method
```

这条链路初看繁复，但每个节点都有明确的职责，且每个节点都是一个扩展点。这就是为什么 Spring 能支撑如此庞大的生态——AOP、事务管理、安全框架、缓存注解，它们全都是通过在这条链路的某个节点插入 Processor 来实现的，而不是修改 Spring 的核心代码。

回想我在实习做的观测工具里定义的 `init()` / `start()` / `stop()` / `destroy()` 四个阶段，问题不在于阶段太少，而在于这些阶段是封闭的：只有实现了接口的组件自己能执行逻辑，外部观察者无法在这些节点上"搭便车"。如果 QA 想在某个组件 `init()` 之后做一些事情，他要么改组件代码，要么改框架代码。Spring 通过 BPP 机制，让第三方可以在不修改任何已有代码的前提下，介入任何 Bean 的任何生命周期阶段。

这也让我重新理解了 OCP（开闭原则）在框架层面的真正含义：不是"你的代码不用改"，而是"你的框架提供了足够多的缝隙，让别人不需要改你的代码就能实现新功能"。

## 5. 框架与库的边界

在读 Spring 的过程中，一个长期模糊的概念终于清晰了：**框架（Framework）和库（Library）的本质区别**。

库是你调用它：你写 `main()`，你决定程序流程，你在需要的地方调用库提供的 API。比如 `Gson`，你在需要序列化 JSON 时调用 `gson.toJson()`，控制权始终在你手里。

框架是它调用你：框架定义了程序的主体流程（如 Spring 的 `refresh()`、Servlet 的 `doGet()`），你只需要在框架留好的扩展点里填入自己的逻辑。控制权在框架手里，你只是被框架"调用"的一方。

这直接呼应了[《架构整洁之道》]({% post_url 2026-1-25-架构整洁之道 %})中对框架的警告：

> 框架作者想让我们与框架订终身——这相当于我们要对他们的框架做一个巨大而长期的承诺，而在任何情况下框架作者都不会对我们做出同样的承诺。

所以，好的做法是让框架留在"细节层"，用例层保持框架无感。Spring 在这一点上做了一个有趣的平衡：它确实要求你用注解（`@Component`、`@Autowired`）或 XML 来声明 Bean，这算是一种"侵入"；但如果你遵循 DIP，核心业务实体可以是纯 POJO，完全不依赖 Spring 的任何类。业务逻辑面向自己定义的接口编程，Spring 只负责在运行时把实现类注入进来。

实习做的观测工具 在这方面做得不好。由于我的"容器"是一个自定义的全局 `Map`，组件想获取依赖必须直接调用容器的 API（`container.get("someKey")`），这就让每个组件都和我的框架强耦合了。如果哪天要换掉这个容器实现，所有组件都得改。这就违背了模块化设计的初衷：能随时拔插，才算真正"采用"某个模块而非被框架"绑架"。

## 6. Spring Boot 的"约定优于配置"——另一种设计智慧

重读 Spring Framework 的同时，不得不提 Spring Boot。如果说 Spring Framework 展示了"如何设计一个极度灵活的框架"，那 Spring Boot 展示的就是"如何在灵活性之上提供合理的默认值"。

Spring Framework 的问题在于，灵活性太强反而导致初学者面对大量的配置选项不知所措。`ApplicationContext` 有多种实现，Bean 的声明方式有 XML、注解、Java Config 三种，数据源的配置、事务管理器的注册、视图解析器的设置……每一个都需要显式配置。

Spring Boot 的 `@SpringBootApplication` 一个注解就启动了整个应用。它背后做的事情是：通过 `@EnableAutoConfiguration` 触发 `spring.factories` 中注册的自动配置类，这些配置类通过 `@ConditionalOnClass`、`@ConditionalOnMissingBean` 等条件注解，根据 classpath 上存在的类和已有的 Bean 定义来决定是否应用某个配置。

这本质上是一种**"发现规则，应用规则"**的思路——和我在[同名文章]({% post_url 2026-1-17-发现规则应用规则 %})中的方法论不谋而合：扫描环境（发现 classpath 上有 `DataSource.class`），判断条件（发现用户没有自定义数据源 Bean），应用默认值（自动创建 `HikariDataSource`）。

"一个软件项目的重心会根据项目的开发时间和成熟度不断变动"，Spring Boot 正是对这一规律的工程化实践：项目早期用默认配置快速跑起来（重心在交付），项目成熟后通过覆盖默认配置进行定制（重心在可控性和复用性）。

这也给了我一个反思实习做的观测工具的新视角：当时的系统不仅缺乏扩展点，也缺乏合理的默认行为。QA 用户面对一个空壳框架，不知道从何配起。如果我能像 Spring Boot 一样，提供一套"开箱即用的默认采集方案"，同时允许用户通过实现接口来覆盖默认行为，上手门槛和定制灵活度就能同时兼顾。

## 7. 从框架之痛到工程体系的建立

这次实习经历，不仅让我看到了自己脱漏的 Spring 设计精髓，更连带拉动了我后续一系列的知识体系重构。

正是因为频繁的架构变更导致交付延期，且重构时由于没有保护网而战战兢兢，我才迫切地去寻找工程和管理学上的解法。这也直接引出了我后来阅读的几本书籍：

* **[《测试驱动开发 (TDD)》]({% post_url 2026-1-22-测试驱动开发 %})**：我意识到，只有建立了扎实的自动化测试用例，我才能安全地去重构框架，摆脱改一处崩全局的恐惧。TDD 甚至可以反向逼迫我把系统的模块与扩展点设计得更易于测试和接入。

  这和 Spring 的设计是自洽的：Spring 的 `ApplicationContext` 可以在测试中独立启动（`@SpringBootTest`），Bean 可以被 Mock 替换（`@MockBean`），这些能力的前提正是容器对依赖的接管和生命周期的可控。如果实习做的观测工具的组件都是从全局 `Map` 里 `get` 出来的硬编码依赖，测试时根本无法替换。

* **[《持续交付》]({% post_url 2026-1-24-持续交付 %})**：为了解决"经常不能交付"的顽疾，我了解到代码不能只停留在开发环境，必须有一条自动化流水线和持续集成机制，来保障每一次提交的质量。

  Spring 的模块化设计天然适配 CI/CD 流水线：每个模块的 Bean 可以独立测试，模块之间通过接口解耦，集成测试只需要关注接口契约而不是实现细节。反过来说，如果框架设计得像实习做的观测工具那样高度耦合，单元测试的边界都划不清，流水线上每次构建都会因为意料之外的依赖变更而红掉。

* **[《测试驱动开发与 Vibe Coding》]({% post_url 2026-1-16-测试驱动开发与vibe-coding %})**：包括后来我尝试拥抱 AI 辅助编程时，其思路依然一脉相承：用自动化的测试边界去约束 AI 生成的代码，就如同当年应该用清晰的钩子边界去约束系统里不可靠的业务插件一样，核心都是为了将变化置于可控的轨道内。

  从 Spring 的视角来看，AI 生成的代码就像一个"不可预测的 Bean"。如果框架（测试用例）定义了清晰的接口契约和生命周期约束，那么不管这个 Bean 的内部实现多么离谱，只要它通过了契约验证（测试绿了），就是可接受的。这又回到了 OCP 的核心：对扩展开放，对修改关闭。

### 串联起来看

现在回头看，这些书和经历形成了一条清晰的线索：

| 痛点 | 对应的方法论 | Spring 中的体现 |
|------|-------------|----------------|
| 扩展点不够，频繁改框架 | OCP 开闭原则 | BPP、事件机制、模板方法 |
| 依赖关系混乱 | DIP 依赖反转 | IoC 容器、接口驱动 |
| 重构没有安全网 | TDD 测试先行 | 可独立测试的 Bean、Mock 机制 |
| 交付频繁中断 | 持续交付流水线 | 模块化设计适配 CI/CD |
| 用户不知怎么用框架 | 约定优于配置 | Spring Boot 自动配置 |
| 所有问题看起来都像系统性崩溃 | 异常边界隔离 | 精确的错误报告、组件故障隔离 |

只有亲手造轮子时踩过坑，再去看《Spring Framework》或[《架构整洁之道》]({% post_url 2026-1-25-架构整洁之道 %})，那些抽象的术语才会变成看得见摸得着的教训。而顺着痛点去摸索，最终所有的技术认知都在工程交付的诉求下串联了起来。

## 8. 如果重来一次

如果让我带着现在的认知重新设计实习做的观测工具，我会做这些改变：

1. **先交付最小可用版本**。不做任何插件化、不做通用容器，就一个硬编码的采集流程。先让 QA 用起来，验证核心流程是否有价值。正如《架构整洁之道》所说，一个系统的架构设计偏向某种特定的"形状"时，新的变更就会越来越难实施，所以不如在早期保持形状的模糊和简单。

2. **用接口隔离变化点**。存储层、采集策略、输出格式——这些是可预见的变化方向。先定义接口，先用最简单的实现跑通，后续替换时只需新增实现类，不动核心代码。这不是"预留功能"，而是[《架构整洁之道》]({% post_url 2026-1-25-架构整洁之道 %})中说的"预留接缝（seam）"。

3. **在启动流程中预埋钩子**。哪怕第一版只有三个阶段（加载、执行、输出），也要在每两个阶段之间留一个可注册的回调列表。Spring 的 `refresh()` 证明了这种前瞻性设计的价值。

4. **建立最基本的测试和构建流程**。编译通过 + 核心用例测试绿，这是底线。有了这条线，重构才有安全网，交付才有信心。

5. **提供开箱即用的默认配置**。不让用户面对一个空壳。默认采集 CPU、内存、GC 指标，用户需要定制时再覆盖默认实现。

这些做法说到底都不是什么高深的道理，Spring 二十年前就在做了。但只有亲手造过一个蹩脚的轮子，才会真正理解圆的才好用。
