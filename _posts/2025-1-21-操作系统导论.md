操作系统三大主题：虚拟化、并发、持久性。

抽象是计算机科学各个方面的基础。从上往下看，驱动层不关心实现细节，只关注接口。从下往上看，实现层内聚自己的功能，不做额外的事（约束边界）。当一个业务无关的实现层要接轨某个接口时，用组合将其接轨即可。

妥协无处不在，想要好的虚拟化，就要舍弃一些不得不的性能。

我已经学会了怎么让系统复杂，现在主要的学习任务是，怎么让系统简单。这本书描述的是操作系统，不是 Linux 也不是 Windows。在更上层的抽象接口中，我们学习一般性的设计原则。

## 关于设计原则的思考

### 不成熟优化 × 欠考虑泛化

书中有一句："不成熟的优化是所有坏事的根源。让整个应用的某一小部分变快，却没有提高整体性能，其实没有价值。"

每一次泛化都伴随语义膨胀、实现复杂度和概念债。锁、条件变量、信号量、无锁结构，各自只解决一个"最小但完整"的问题——保持"小而正交"的抽象，比追求"大一统"更利于系统正确与演进。

这两种毒性叠加，就是生产事故的常见根源：

|  | 概念膨胀 | 接口膨胀 | 依赖膨胀 |
|--|---------|---------|---------|
| 提前底层化 | 规则引擎编成 BPF 平台，业务改不动 | 暴露裸指针给上层，30% 代码在打补丁 | 多语言插件全拉满，CI 从 3min→45min |
| 过度并行化 | 动态自适应线程框架，上下文切换占 40% CPU | 异步 DSL 50+ 运算符，栈深 300+ 没人看懂 | 所有 IO 走 Netty，glibc 一升级全站崩 |
| 手工微优化 | 泛型特化 8 套原始类型，改一处要改 8 份 | 业务 inline 到公用库，300+ public 方法 | 热点路径上 SIMD+JNI，Docker 镜像 2GB |

共同后遗症：新人 3 个月还在学框架看不懂业务、栈里 70% 是框架代码、接口被下游 10 个团队依赖退不回去、为 10% 提升引入 50% 额外开销。

**解毒剂**："先让代码对，再让代码快，再让代码少——每一步都留下回滚按钮，并且按钮必须能被下个月的你一键按下。"

### 几个具体的反思

关于 dooba 项目：首先，我解决了之前最核心的问题吗？（QPS/TPS 抖动检测、跌零检测、持久化、数学分析）在解决了这些的基础上，再考虑泛化。项目初期，即使能预见后续扩展，仍然应该先用成本低、性能低的方案去尝试。

关于抽象的时机：

- 接口只声明"做什么"，不暴露"怎么做"。早期用 ExecutorService 就够了，别急着把 mmap/sendfile/GPU 抛给调用者
- 具体实现可以脏、可以硬编码，但必须是私有的、可替换的
- 泛化时机 = 第三次出现相似需求，而不是第一次性能瓶颈
- "把优化留在实现层，把熟悉抽象留给调用层；让优化只污染源码，不污染接口"
- 泛化本身不是原罪，关键是封装边界和暴露粒度

我以为足够扩展就能增加它的生命周期，现在看来，维护好核心功能，其他功能在该基础上添加才是正确的。概念膨胀导致业务同学失去表达能力，所以要让其能写符合直觉的代码，将业务逻辑完全留在调用侧，而不是盲目减少代码。

## 第一部分 虚拟化

### 系统调用与特权级别

系统调用和过程调用的关键区别：系统调用通过 trap 指令将控制转移到 OS，同时提升硬件特权级别到内核模式。用户模式下应用不能直接发起磁盘 I/O、访问物理内存或发送网络数据包。完成服务后通过 return-from-trap 返回用户模式。

### CPU 虚拟化

- **进程**：程序代码 + 生命周期。操作系统是绝对懒的——代码、静态数据、初始化栈，不包括动态部分（如动态库的动态链接）
- **fork vs exec**：为什么不直接 exec 而是先 fork？因为 fork 后、exec 前可以做重定向、管道等操作
- **受限直接执行**：让进程直接在 CPU 上运行（性能），但通过特权级别限制其能力（安全）
- **调度**：FIFO → SJF → MLFQ（多级反馈队列，可以让实时任务保持高优先级）→ 比例份额 → 多处理器调度

### 内存虚拟化

地址空间 → 地址转换 → 分段 → 分页 → 多级页表 → swap

多级页表的思路是**做减法而不是做加法**。

### 关于 IO 的理解框架

曾经被问到"你对 IO 有什么理解"。开放性问题，怎么成体系、有条理地回答？操作系统三大块：虚拟化、并发、IO，可以按层次展开：

| 层次 | 关注点 |
|------|--------|
| 编程语言 | 系统调用（open/read/write），AIO/BIO/NIO |
| 应用系统 | 网络 IO（select/poll/epoll），序列化，协议栈 |
| 内核 | 匿名页、缺页（频繁缺页、透明大页）、mmap、page cache |
| 硬件 | HDD/SSD 顺序 vs 随机读写差异，总线延迟，NUMA |
| 冗余 | RAID 0/1/5/10 组合，对上层透明 |

先说广度，再邀请面试官针对某一部分深入提问，回应深度。

### 关于开放性面试问题的思考

怎么在听到开放性问题时快速规划回答的路径和范围？

这类问题的共同特点是没有标准答案，面试官想了解的是你的知识广度和深度，以及你组织表达的能力。回答策略其实是通用的：**先搭框架，再填内容，最后引导深入**。

之前被问到"实习中最有挑战性的问题"时，STAR 法则（Situation → Task → Action → Result）能帮助快速组织叙事结构。IO 问题也类似——先按层次（编程语言 → 应用系统 → 内核 → 硬件）搭出骨架，再在每一层填充自己了解的细节。这个方法可以推广到很多开放性问题：

- "你对 IO 有什么理解？" → 按抽象层次从上到下展开
- "你怎么构建一个高性能 Web 服务器？" → 按请求链路从接入到存储展开
- "如果再让你重新做这个项目，你会怎么做？" → 按时间线回顾决策点，说清当时的选择和现在的反思
- "你怎么测试这个分布式系统？" → 先列验收指标，再说测试手段

关键是在回答的前 30 秒建立起一个面试官能跟着走的结构，而不是想到哪说到哪。说出广度之后，可以主动邀请面试官针对某一部分深入提问，把主动权拿回来。能体系化地表达自己知道的东西，本身就是一种能力。

几年前看不懂的文章，现在也能看懂大部分了。现在的认知都建立在之前所有来时路的收获上，不管好的坏的，只要在做就是提升。回头看这些积累，读书、实习、写项目，每一段经历都在往这个框架里填充内容。框架是可以学的，但填进去的东西只能靠自己一点一点攒。
