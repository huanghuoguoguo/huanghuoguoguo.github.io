2023 年初 ChatGPT 刚火的时候，我跟大部分人一样，注册完就开始疯狂跟它聊。一开始让它帮我写 Java 代码，写出来的东西不太对，我就在同一个窗口里不断纠正它——"不对，这里应该用 ConcurrentHashMap"、"返回值类型错了"、"你上次犯过同样的错误"——纠正了几轮之后，它的回答确实越来越准了。

当时我真的以为是我在"训练"它，觉得这个 AI 好聪明，能从对话中学习。我甚至还特意维护了一个长对话窗口，舍不得关，生怕关了它就"忘了"我教它的东西。直到有一天，我打开一个新窗口问了同样的问题——它又犯了一模一样的错误。

那一刻我才意识到，可能跟我想的不一样。后来搞清楚 LLM 的工作机制之后才明白，完全不是那么回事。在聊 RAG 解决了什么问题之前，有必要先搞清楚这件事：你跟大模型聊天的时候，到底发生了什么？

## 先搞清楚：你跟大模型对话时，到底发生了什么？

### 大模型是无状态的

这可能是最反直觉的一点。你用 ChatGPT 或者 Claude 聊了一下午，感觉它越来越"懂你"，但实际上模型的参数从头到尾**一个都没变**。

大模型在部署上线之后，参数就被**冻结**了。每一次你发消息，对模型来说都是一次独立的函数调用：

```
response = model(input_tokens)  # 纯函数，没有副作用
```

为什么要冻结？原因很实际：

- **成本**：一个大模型服务同时面对几百万用户，如果每个人的对话都去更新参数，那就不是推理，是在线训练，算力成本高几个数量级。
- **安全**：如果用户对话能影响参数，恶意用户就能通过精心构造的输入"污染"模型，影响所有人。
- **一致性**：冻结参数意味着相同输入产生确定性行为（温度为 0 时），这对工程部署至关重要。

那"越聊越好"的感觉到底怎么来的？

### 上下文窗口：聊天的真实机制

聊天产品其实一直在替你保存上下文。

你在 ChatGPT 里发第三条消息时，后端通常不会只把这一句丢给模型。它会把前面那几轮一起打包，再发一遍：

```json
[
  {"role": "system",    "content": "You are a helpful assistant..."},
  {"role": "user",      "content": "帮我解释一下什么是 B+ 树"},
  {"role": "assistant", "content": "B+ 树是一种平衡多路搜索树..."},
  {"role": "user",      "content": "那它跟红黑树比有什么优势？"},
  {"role": "assistant", "content": "B+ 树更适合磁盘存储..."},
  {"role": "user",      "content": "为什么更适合磁盘？"}
]
```

看到了吗？**每一轮对话，所有历史消息都被重新发送给模型。** 模型并没有"记住"之前说了什么——是应用层把对话历史打包成一个完整的输入，喂给一个无状态的函数。

三个角色各司其职：
- **system**：系统指令，定义模型的行为规则（"你是一个客服"、"回答要简洁"）
- **user**：用户的输入
- **assistant**：模型之前的回复

所谓越聊越好，只是模型这一次能看到的信息更多了，不是它的能力自己长出来了。这也解释了为什么大模型没有真正意义上的个性化。你跟它聊了半年，它的参数还是没有变化；换个会话窗口，很多东西又得重头再来。

### 为什么上下文窗口有限制？

既然只是拼接消息，那为什么不能无限拼下去？这涉及 Transformer 的核心计算——Self-Attention。

标准 Self-Attention 中，每个 token 都要跟序列中所有其他 token 计算注意力分数：

```
Attention(Q, K, V) = softmax(QK^T / √d) · V
```

这意味着：

- **计算复杂度 O(n²)**：n 是序列长度（token 数）。上下文从 4K 扩到 128K，计算量涨了 1024 倍。
- **显存消耗 O(n²)**：QK^T 产生的注意力矩阵是 n × n 的，序列越长，显存占用越大。虽然 Flash Attention 等技术可以优化显存，但计算量的平方增长是绕不开的。
- **训练边界**：模型在训练时使用的上下文长度有限。虽然有 RoPE 等位置编码外推技术，但超出训练长度后效果会衰减。

更致命的是，即使硬件撑得住，**长上下文 ≠ 长记忆**。至少按我现在的理解，模型对序列中间位置的信息关注度会明显下降，这就是常说的 Lost in the Middle 问题。标称窗口很长，不等于中间那一大段信息它都能稳稳用好。

这就是为什么你的长对话到后面会变"健忘"：对话一长，上下文窗口的限制和注意力衰减就会一起冒出来。前面的消息可能先被截掉；就算还没截掉，中间那部分信息模型也未必抓得住。

## 理解了机制，问题就清晰了

模型参数冻结、上下文窗口有限——这两个约束决定了一个核心问题：

**怎么让大模型在特定领域做得好？**

比如你想让大模型回答公司内部技术文档、产品手册、业务规则相关的问题。这些知识大模型在训练时从没见过，怎么办？

历史上走过两条路。

### 路线一：改参数——微调

这是从 BERT 时代延续下来的经典范式。拿基础模型当底座，通过 SFT（有监督微调）、RLHF（人类反馈强化学习）等方式，把领域知识"刻"进模型参数里。

听起来是最直接的思路，但实践下来有几个很难绕过的问题：

- **成本高、周期长**：每次微调都要清洗数据、构造训练集、设计评测集，还需要大量 GPU 算力。从准备数据到评测完成，通常以月为单位。
- **灾难性遗忘**：提升了领域 A 的能力，可能同时损害了领域 B 的表现。微调的经典困境——局部最优可能带来全局退化。
- **基座迭代太快**：这是最致命的。你花几个月微调出来的模型，可能还没上线，新一代基座的原生能力就已经超过了你的微调版本。"刚毕业就失业。"

再加上当前最强的模型往往是闭源的（GPT-5、Claude），你根本没有参数可以调。即使用开源模型微调，效果也很难追上闭源模型的原生能力。随着 Scaling Law 生效，顶尖模型的参数量越来越大，传统的单机甚至小规模集群已经难以承担训练任务。

"向内"改参数这条路，要么成本太高，要么根本走不通。

### 路线二：不改参数——在上下文里做文章

既然改不了参数，那就换个思路：**把领域知识塞进上下文窗口**，让模型"带着参考资料答题"。

最直接的做法是把所有领域知识写进 System Prompt。对于知识量不大的场景（核心指令和背景知识在两万 token 以内），这招简单有效。

但知识量一旦上去，马上就撞上前面讲的两个限制：要么塞不进上下文窗口，要么塞进去了效果也不好（Lost in the Middle，模型对中间的信息注意力衰减）。

这就是 RAG 登场的背景。

## RAG 解决的问题

RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路很简单：**你不需要把所有知识都塞进上下文，只需要把跟当前问题相关的那部分找出来塞进去。**

```
用户提问 → 检索相关知识片段 → 拼入上下文 → LLM 基于这些片段生成回答
```

从"全量吞咽"变成"按需投喂"。

具体来说，RAG 解决了这几个问题：

**1. 绕过上下文窗口的长度限制**

知识库可能有几百万字，但每次只取跟问题相关的几段（通常几千 token），上下文窗口完全装得下。

**2. 不需要改模型参数**

知识更新了？改知识库就行，不需要重新训练模型。微调一次要几个月，知识库更新一条文档可能几分钟。

**3. 知识可追溯**

模型的回答基于哪些文档片段，可以一起返回给用户。出了问题能追溯到知识源头，而不是面对一个黑箱模型猜"它为什么这么说"。

**4. 减少幻觉**

模型在有明确参考资料的情况下回答，比纯靠参数"凭记忆"回答，幻觉率通常会降低。当然不是完全消除——模型仍然可能对检索到的内容做"创造性解读"。

### RAG 的基本流程

```
入库阶段（离线）：
┌──────┐    ┌──────┐    ┌───────────┐    ┌──────────┐
│ 文档  │ →  │ 分块  │ →  │ Embedding  │ →  │ 向量数据库 │
└──────┘    └──────┘    └───────────┘    └──────────┘

检索阶段（在线）：
┌──────┐    ┌───────────┐    ┌────────┐    ┌──────┐    ┌─────┐
│ 提问  │ →  │ Embedding  │ →  │ 向量检索 │ →  │ 拼接  │ →  │ LLM │
└──────┘    └───────────┘    └────────┘    └──────┘    └─────┘
                                  ↑
                             Top-K 最相似
```

入库时把文档切成小块，每块用 Embedding 模型转成向量，存入向量数据库。检索时把用户的问题也转成向量，在向量库里找最相似的 K 个片段，拼进 Prompt 让 LLM 回答。

关于 RAG 的具体实现，我在 LangBot RAG 迁移那篇文章里结合实际项目写过一遍：分块策略怎么选，Embedding 模型怎么选，向量数据库怎么选，查询改写怎么做。这里先不展开。

## RAG 也有硬伤

RAG 解决了"怎么把领域知识按需送进大模型"的问题，但它自身有一条致命的依赖链：**垃圾进，垃圾出（Garbage In, Garbage Out）**。

整个 RAG 的效果高度依赖**检索阶段的准确率**。如果这一步没召回正确的知识片段，后面的 LLM 再强也没用——给它错误的参考资料，它只会一本正经地胡说八道。

这里有一个能力断层：RAG 的检索通常依赖关键词匹配（如 BM25）或 Embedding 模型（如 BGE、text-embedding 系列）。这些检索模型的语义理解能力跟大模型本身比，差距很大。这种"小模型前置辅助大模型"的模式，天然就存在漏召和误召的风险。

举个例子：用户问"服务器连不上怎么办"，知识库里有"SSH 连接超时排查指南"和"网络不通排查手册"。Embedding 模型可能因为"连不上"这个表述的歧义性，召回了不太相关的文档，导致 LLM 给出偏离预期的回答。

这也是为什么在实际项目里，RAG 的时间经常花在检索环节，不花在 LLM 上。分块粒度、Embedding 选型、混合检索策略、查询改写、重排序，这些地方都在做同一件事：尽量先把送进去的那批材料弄对。

## 从 RAG 往后看

RAG 解决了单轮知识检索的问题，但真实场景往往比"一问一答"复杂得多。当需求从"回答一个问题"变成"完成一个任务"时，就需要更多的架构支撑。

简单列几个方向，后面会展开讨论：

- **Workflow（工作流编排）**：当一个任务需要多步执行，且步骤间的逻辑是确定的（比如"先查订单→再查物流→生成回复"），用工作流编排比让 LLM 自由发挥更可靠、更可控。
- **ReAct（推理+行动）**：让 LLM 在 Thought → Action → Observation 的循环中自主决策——该调什么工具、怎么用结果。这是 Agent 最基础的运行模式。
- **Agent（智能体）**：把 LLM 从"回答者"升级为"执行者"，给它工具（搜索、写代码、调 API），让它自主规划和执行任务。
- **MCP（Model Context Protocol）**：Anthropic 提出的协议标准，定义了模型如何发现和调用外部工具。可以理解为 Agent 工具调用的"USB 接口"——不管什么工具，只要实现了 MCP 协议，模型就能用。
- **Skills（可复用能力）**：把 Agent 反复做的事情封装成可复用的"技能"，避免每次都从零开始写 Prompt。

这几条路其实是在补同一件事：别让大模型只停在“会聊天”这一步。RAG 解决知识怎么喂进去，Workflow 解决流程怎么钉住，Agent 再往前一步，开始让模型自己决定下一步干什么。

### 一个值得思考的方向

回头看这些架构，会发现它们都像外挂。模型记不住太多东西，我们就做 RAG；模型推理容易飘，我们就把流程写死；模型碰不到外部系统，我们就给它包一层工具。它们看起来花样很多，其实都很朴素，就是在替基座模型补眼下还补不上的缺口。

但如果基座模型的能力持续增长呢？

上下文窗口从 4K → 128K → 1M，推理能力越来越强，工具调用越来越原生……**那些我们在外部显式构建的约束和机制，是不是会逐渐被模型内化？**

这个趋势其实已经在发生。早期需要精心设计的 Chain-of-Thought 提示词，现在模型原生就会推理；早期需要外部编排的工具调用，现在很多模型内置了 function calling。显式的外部架构正在被隐式的模型能力吸收。

当然，这不等于这些架构很快就没用了。账单、延迟、可控性都还在。就算模型真能稳稳吃下 1M token，你也未必愿意每次查询都这么烧钱。更现实的判断是：基座越强，外面那层脚手架可以一点点变薄，但短期内还拆不掉。

下一篇，从 RAG 这个起点出发，看看当任务变得更复杂时，Agent 架构是怎么一步步演化出来的。
