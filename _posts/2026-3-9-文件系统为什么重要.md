最近连着看了《Linux 内核设计与实现》和《操作系统导论》（OSTEP）。在阅读过程中，我发现自己一直存在一个盲区：每当书里花极大的篇幅去讲文件系统（甚至单拿出一个大章来写 ext2、ext3、ext4 的迭代），我总是习惯性地快速翻过。

我潜意识里觉得文件系统“好像没那么重要”，觉得它无非就是怎么在硬盘上存取数据而已。我不理解 ext 系列为什么要这么迭代，更不清楚文件系统到底解决了计算机世界里的哪些核心矛盾。

为了搞清楚这个问题，我查阅了一些资料和源码设计，才猛然发现：**文件系统被严重低估了，它不仅是存储工具，更是整个操作系统的设计哲学基石。**我之所以觉得它不重要，纯粹是因为现代 OS 把它包装得太好，以至于我没有感受过它底层的阵痛。

这篇笔记，权当是我对文件系统认知的一次“重塑”。

---

## 1. 认知的起点：文件系统是操作系统的“根隐喻”

为什么要用这么大篇幅讲文件系统？因为“**一切皆文件**”是 Linux 的终极抽象。

我在梳理资料时，总结了操作系统底层面临的一些核心矛盾，以及文件系统抽象是如何优雅地解决这些瓶颈的：

| 矛盾点 | VFS/文件系统的解决方案 | 为什么这很重要？ |
| :--- | :--- | :--- |
| **硬件设备千差万别**（磁盘、键盘、网卡、显卡等操作方式完全不同） | 统一抽象为“文件” | 用户和程序只需要掌握最基础的 API（`open`/`read`/`write`/`close`），屏蔽了硬件复杂性。 |
| **底层文件系统实现差异巨大**（ext4、XFS、NTFS 等机制各异） | VFS（虚拟文件系统层） | 向上提供统一视图，应用侧开发完全不需要关心自己落盘的是什么格式的文件系统。 |
| **内核需要管理海量资源，但必须保持接口极简** | 文件描述符（fd）作为万能句柄 | 仅仅通过一个整数，就能把进程、内存、网络连接、硬件设备全部统一聚合管理。 |

更为致命的是：进程会死、内存会丢、CPU 会停，只有文件系统中的数据能跨越重启而存在。**文件系统的一致性，就是操作系统的一致性。**

## 2. 矛盾驱动的演进：Ext2 → Ext3 → Ext4 到底在折腾什么？

以前看这些 ext 版本迭代，只觉得是加了些新特性。现在从工程和系统矛盾的视角来看，每一次迭代都是为了解决生死攸关的难题。

### **Ext2（1993）：没有日志的灾难**

*   **核心矛盾**：**性能 vs 崩溃一致性**

Ext2 是一套很经典的结构，但它存在一个致命缺陷：**没有日志**。向文件追加一个数据块，往往需要 3 次独立的写操作（写 inode 更新元数据、写数据位图标记块分配、写实际数据块）。

如果在这 3 次写入之间系统意外断电崩溃，灾难就来了：inode 可能指向了一个未分配的块，或者空间被分配了但毫无记录。系统重启后，唯一的自救手段就是靠 `fsck`（文件系统检查器）去全盘扫描修复。在 TB 级存储时代，重演一次崩溃可能需要好几个小时的扫描！

### **Ext3（2001）：用空间换时间的日志革命**

*   **核心矛盾**：**崩溃恢复速度 vs 写入性能**

为了解决长达数小时的灾难恢复，Ext3 引入了最重要的机制——**日志（Journaling）**。
这其实也是数据库经典的 Write-Ahead Logging (WAL) 思想：**先写日志，再写实际数据**。

`事务开始 → 收集要修改的数据 → 写入日志（提交）→ 写入实际位置 → 擦除日志`

这样一来，崩溃后系统再去重启，只需要扫描一小段日志区域（秒级）就能重放恢复。
当然，新机制也带来了新矛盾：**写放大**。数据要先写日志再写磁盘，性能必然受损。为此，我在查阅 Ext3 设计时发现，它妥协出了三种日志模式来权衡一致性与性能：

| 日志模式 | 记录内容 | 一致性保证 | 性能表现 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **Writeback** | 仅记录元数据 | 弱（可能读到旧数据或乱码） | 最高 | 临时文件、缓存等对丢失不敏感的场景 |
| **Ordered**（默认） | 仅记录元数据，但强制数据先落盘 | 中（保证元数据更新前，数据已写完） | 中等 | 绝大多数的通用场景 |
| **Journal** | 元数据 + 实际数据 | 最强（所有操作均可重做 Redo） | 最低 | 对数据一致性要求极高的关键业务 |

这点在 MySQL 等数据库的刷盘策略（如 `innodb_flush_log_at_trx_commit`）上也能看到极其相似的权衡艺术。

### **Ext4（2008）：告别旧时代的架构重构**

*   **核心矛盾**：**传统索引结构 vs 大容量/大文件时代的爆发**

Ext4 不再是修修补补，而是为了应对海量数据做出的架构级重构：

1.  **废弃块映射，引入 Extents（连续范围描述）**：
    在 Ext2/3 里，每个 4KB 的块都需要一个指针记录。存一个 1GB 的文件，大概需要 25 万个指针！这就导致大文件的元数据极其臃肿。Ext4 引入了 Extents 的概念，直接用 `[起始块号, 长度]` 来描述连续的一段空间。25 万个指针压缩成了一个描述符，树的高度降低了，查询快得飞起。
2.  **延迟分配（Delayed Allocation）**：
    以前的数据来一点分配一点，极易导致大量磁盘碎片。Ext4 则是把数据全缓存在内存里，不到最后刷盘的时候绝不分配物理块，这样系统就能一次性给文件寻找并分配最大最连续的物理空间。

## 3. 文件系统到底在对抗哪些底层矛盾？

顺着演进历史梳理下来，我终于理解了那些被用来讲文件系统的冗长章节，其实都在试图解决计算机世界里最苛刻的 IO 状态管理问题：

1.  **速度 vs 安全（易失与持久的矛盾）**：
    内存很快但断电即丢，磁盘很慢但永久保存。操作系统为了调和两者，引入了 Page Cache、延迟写入、日志、COW（写时复制），让我们既能享受到内存般的速度，又有了磁盘的安全感。
2.  **随机访问 vs 顺序写入的偏好**：
    机械磁盘因为物理构造，顺序写比随机写快几十倍。但用户的读写往往是随机的。文件系统只能想尽办法把随机转成顺序，比如通过日志结构、B+ 树索引、分配连续的 extents。
3.  **原子性 vs 性能的撕裂**：
    文件操作要么全成功，要么全失败。现实中磁盘是分块写入的，很容易被中断。日志和 COW 机制就是为了在保证性能的前提下兜底原子性。
4.  **小文件 vs 大文件的兼顾**：
    小文件需要层级低、查找快；大文件需要高效的连续存储。文件系统必须通过动态的索引结构（从小文件的直接指针，到大文件的 extents）来平衡二者。
5.  **并发访问的一致性**：
    多个进程同时读写同一个文件，如何保证读不到脏数据或者半写数据？需要通过 inode 级别的锁、目录项缓存（dentry cache）、原子操作机制来约束并发。

## 4. 延伸思考：数据库的 WAL 和 ext4 的日志，到底有什么不一样？

在上面 Ext3 的部分，我提到了日志机制和 WAL 思想的相似性。当时顺嘴一提，但后来越想越觉得这个问题值得深挖：数据库的 WAL（比如 InnoDB 的 redo log、PostgreSQL 的 WAL）和 ext4 的 journal，都是”先写日志再写数据”，看上去原理一模一样，那它们到底有什么本质区别？如果文件系统已经有了日志保护，数据库为什么还要自己再搞一套？更进一步，为什么很多数据库（InnoDB、PostgreSQL、RocksDB）倾向于用 `O_DIRECT` 绕过操作系统的 Page Cache，直接管理磁盘 IO？

### 同源但不同层

两者确实都源自 WAL 思想，但它们保护的对象和粒度完全不同：

| 维度 | ext4 Journal | 数据库 WAL（以 InnoDB 为例） |
| :--- | :--- | :--- |
| **保护对象** | 文件系统元数据的一致性（inode、位图、目录项） | 事务级别的数据一致性（行记录、索引页） |
| **操作粒度** | 物理块级别（固定大小的磁盘块） | 逻辑/页级别（数据页的修改记录，带有事务语义） |
| **日志结构** | 固定大小的环形缓冲区，写满即覆盖 | 可增长的日志文件（或分段日志），支持归档 |
| **恢复目标** | 重启后文件系统结构不损坏（不保证文件内容正确） | 重启后所有已提交事务的数据不丢失、未提交事务回滚 |
| **额外用途** | 仅用于崩溃恢复 | 崩溃恢复 + 主从复制 + 增量备份 + 时间点恢复（PITR） |

最关键的区别在第四行：**ext4 的 journal 不关心文件里写的是什么内容**。它只保证文件系统的元数据结构在崩溃后能自洽——inode 不会指向未分配的块，位图不会出现不一致。但文件里的业务数据写到一半断电了？对不起，那不是文件系统该管的事。

在默认的 Ordered 模式下，ext4 只保证”元数据更新前数据已落盘”，但不保证数据本身的原子性。如果 InnoDB 正在写一个 16KB 的数据页，而 ext4 的块大小是 4KB，那么一次页写入被拆成 4 次块写入。中间断电，可能出现一个数据页里有一半是新数据、一半是旧数据的”撕裂页”（torn page）。这就是为什么 InnoDB 要搞自己的 **doublewrite buffer**——先把数据页完整地写到一个连续的备份区域，再写到真实位置。万一真实位置写了一半断了，可以从 doublewrite 区域恢复出完整的页。

换句话说，文件系统的日志保护了”容器”不坏，但不保护”容器里的东西”完好。数据库必须自己保护自己的数据。

### 日志套日志——双重保险还是双重浪费？

这就引出一个有趣的问题：如果数据库自己已经有了 WAL 保证数据一致性，底下的 ext4 再写一层 journal，岂不是重复劳动？

确实如此。在生产环境中，DBA 经常会把 ext4 的日志模式调成 `data=writeback`（最弱模式，只记录元数据），因为数据层面的一致性已经由数据库的 WAL 兜底了。文件系统的 journal 此时只需要保证元数据不乱即可，没必要再操心数据块的写入顺序。PostgreSQL 的官方文档也建议在 ext4 上使用 `data=writeback` 或者关闭 `barrier`（需谨慎），以减少不必要的写放大。

这实际上是一个**分层职责冲突**的典型案例：文件系统和数据库都在试图解决”崩溃后数据不丢”这同一个问题，但各自的语义粒度不同。当它们叠加在一起时，如果不做协调，就会出现”我保护了一遍、你又保护了一遍”的冗余开销。

### 为什么数据库倾向于用 O_DIRECT？

理解了上面的分层冲突，O_DIRECT 的动机就水到渠成了。

正常情况下，应用程序写文件时，数据会先进入操作系统的 **Page Cache**（页缓存），由内核决定什么时候真正刷到磁盘。这层缓存对大多数应用来说是好事——它让重复读写变快，让内核来优化 IO 调度。

但对数据库来说，Page Cache 带来了几个严重的问题：

1. **双重缓存（Double Buffering）**：数据库自己有 Buffer Pool（比如 InnoDB 的 Buffer Pool、PostgreSQL 的 Shared Buffers），操作系统又有 Page Cache。同一份数据页在内存里存了两份——一份在数据库的 Buffer Pool 里，一份在 OS 的 Page Cache 里。白白浪费了一倍内存。对于一个分配了 128GB Buffer Pool 的数据库实例来说，OS 可能又额外缓存了几十 GB 的同样数据。

2. **刷盘时机不可控**：数据库需要精确控制”什么时候数据真正落盘”。比如 InnoDB 在事务提交时，必须保证 redo log 已经 `fsync` 到磁盘（`innodb_flush_log_at_trx_commit = 1`），否则就无法保证持久性（Durability）。如果中间隔着 Page Cache，数据库调用 `write()` 后数据可能还躺在 Page Cache 里，内核说不定几秒后才真正刷盘。虽然可以用 `fsync()` 强制刷盘，但 Page Cache 的存在增加了不确定性和延迟。

3. **淘汰策略冲突**：操作系统的 Page Cache 用自己的 LRU 变体来决定淘汰哪些页。但数据库对自己的数据访问模式了如指掌——哪些是热点索引页（永远不该淘汰）、哪些是一次性的全表扫描数据（扫完就扔）。OS 的通用 LRU 无法区分这些场景，可能把数据库急需的热点页给淘汰了，反而留下了冷数据。

使用 `O_DIRECT` 打开文件后，读写操作直接绕过 Page Cache，数据在用户空间的 Buffer Pool 和磁盘之间直接搬运。数据库获得了对内存和 IO 的**完全控制权**：

- 内存只有一份缓存（Buffer Pool），没有浪费
- 刷盘时机由数据库自己决定，`fsync` 的语义更清晰
- 淘汰策略由数据库自己的算法管理（InnoDB 用改良 LRU，将年轻页和老页分区管理）

### 代价与权衡

当然，O_DIRECT 也不是银弹。绕过 Page Cache 意味着：

- **必须自己管理对齐**：O_DIRECT 要求读写的内存地址和偏移量都按磁盘扇区对齐（通常 512 字节或 4KB），否则直接报错。这增加了实现复杂度。
- **小文件和元数据操作变慢**：对于配置文件、binlog 的顺序写入等场景，Page Cache 反而更高效。所以数据库通常只对数据文件和 redo log 使用 O_DIRECT，其他文件仍然走 Page Cache。InnoDB 的 `innodb_flush_method = O_DIRECT` 就是只对数据文件和 redo log 生效。
- **不是所有文件系统都能完美支持**：在某些老旧内核或网络文件系统上，O_DIRECT 的行为可能不符合预期。

这又回到了本文反复出现的主题：**没有完美的方案，只有不同场景下的权衡**。文件系统提供了通用的缓存和日志保护，对绝大多数应用足够好；但数据库作为对 IO 要求最苛刻的软件之一，选择绕过通用层，自己掌控一切。

### 一个更深的问题

把视角再往上拉一层，这个现象其实在[《架构整洁之道》]({% post_url 2025-1-25-架构整洁之道 %})中也有对应的讨论：**当上层系统对底层的”通用服务”有了超出其设计范围的要求时，上层不得不自己重新实现一套**。操作系统的 Page Cache 和文件系统日志是面向所有应用的”通用服务”，数据库的需求已经超出了这层通用服务的设计假设，所以它选择了 O_DIRECT + 自己的 WAL + 自己的 Buffer Pool。

这和 miniob/OceanBase 的存储引擎设计也是一脉相承的。在做 miniob 的存储层时，我也注意到 OceanBase 对 IO 路径的管理非常精细——它不依赖文件系统的缓存策略，而是自己管理内存池和刷盘节奏。不是因为文件系统做得不好，而是因为数据库比文件系统更了解自己的 IO 模式。

这大概也是”没有银弹”的又一个注脚：通用方案追求的是 80% 场景下的 80 分，而极端场景下的 95 分，只能靠”自己动手”来拿。

## 5. 后知后觉的感悟：它掌管着计算机的”时间”

回顾自己之前为何会觉得它不重要，主要还是因为现代操作系统把底层的设计和灾难屏蔽得太好。我缺乏因为断电导致文件损坏、或者面对百万级碎小文件 IO 暴跌的痛苦体验。但我忘了，它的任何一个细微的设计，都对应着真实的生产灾难：

| 文件系统设计 | 解决的潜在灾难 | 真实的生产痛点场景 |
| :--- | :--- | :--- |
| **引入日志（Ext3）** | 意外断电后，文件系统完全损坏状态 | 数据中心突然停电/宕机，数百台服务器无需再等待十几小时的 `fsck` 扫描，即可快速恢复上线。 |
| **引入 Extents（Ext4）** | 大文件元数据爆炸，导致内存耗尽 | 视频编辑软件处理 100GB 的 4K 视频，或者大数据平台写入巨型日志文件。 |
| **延迟块分配** | 文件碎片化严重，导致磁盘寻道性能暴跌 | 数据库运行一年后，由于频繁的随机小写入，导致物理存储极其碎片化，查询速度锐减 90%。 |
| **元数据校验和** | 磁盘出现的静默数据损坏 | 企业级 PB 存储中，坏道造成的位翻转未被察觉，最终导致业务数据错乱。 |

当我重读那些介绍文件系统机制的章节时，换了一种全新的视角：**这不再是刻板的数据结构背诵，而是前人在充满混沌和不可靠硬件的世界中，艰难建立软硬件秩序的过程。**

进程管理，处理的是“现在”（正在运行的程序）；
内存管理，关注的是“近期”（活跃的工作集）；
唯独文件系统，掌管着计算机的“永恒”（跨越断电与重启的数据）。

原来，它的每一次迭代和进化，都不是为了炫技式地追求“极客范”，而是要在每一次宕机、断电、极限 IO 压力等恶劣环境下，保证系统能够“不死”。
