我以前一直有个很模糊的困惑。

为什么说kafka比别的mq性能高，高在哪里？，为什么说es全文索引比别的数据库性能高，高在哪里？为什么redis就比mysql性能高（以前觉得都是数据库，还不知道什么是关系型和kv型），高在哪里？

刚学编程的时候，我其实不知道他们的区别。看别人的项目里用了多么高级的技术栈，又是mysql又是redis又是minio，好像什么场景都用这几个，好像用起来方便顺手。后来我学着做项目的时候，不管数据要不要存，我都放到redis里，因为redis是有持久化的内存级别缓存，性能好像很高，我就用了，好像这是工程范式。

后来我发现，如果我要自己存取数据，似乎可以不用别人弄好的数据库，因为我没有那么多需求。于是我尝试自己存取数据。那时候 IO 就是编程语言里的 `read()`、`write()`。有一些数据，我要写到磁盘上，或者从磁盘上读出来。程序重启之后数据还在，也蛮快，我觉得就满足我的要求了。

我当时摆弄玩具项目，感受不出来我写的持久化和redis的持久化有什么区别，至少在我看来就是放进去取出来，我知道他性能肯定比我的高，但是，为什么？

后来这个理解慢慢不够用了。做块存储管理后端的时候，我看过 IOPS、延迟、吞吐这些指标，但不太理解怎么跟软件联系起来；到 OB 做存储相关测试，又会碰共享存储、冷热缓存、造数这些东西。再往后读到 Kafka 为什么快、数据库为什么总想绕开文件系统，我才慢慢意识到，前面遇到的其实不是几类分散的问题，它们底下连着同一条线。

这篇文章我不打算把文件系统、块存储、对象存储一口气全塞进去。我先收窄一点，只讲一条更贴近程序员直觉的线：我一路碰到的几个系统调用，以及我是怎么慢慢把这些调用背后的路径和代价串起来的。

文件系统、块存储、对象存储这些抽象，我留到下一篇单独讲：[《文件系统为什么重要》]({% post_url 2026-3-26-文件系统为什么重要 %})。

## 最开始，IO 就是 `read()` 和 `write()`

大学的时候，我自己拿 Netty 写过一个很简单的文件传输系统。那时候还不知道什么是异步运行时，也说不清系统调用到底发生了什么。对 IO 的理解很朴素：内存快，磁盘安全；数据在内存里改完了，最后总得落到磁盘上。

我当时用 `read`、`write` 存了一些元数据，把它们塞进 `HashMap`，再序列化到磁盘里。程序重启之后看到数据还在，感觉很牛逼，有点应用程序的意思了。

现在回头看，那套做法很符合直觉但是很原始。为了保证持久化，我每改一次数据就立刻做一次序列化，再直接落盘。那时候我还意识不到这有多低效，只觉得能存住就行。

但这种朴素经验也有价值。我的场景，io根本不是瓶颈。

---

## 后来我才知道，`read`、`write` 只是入口

再往后学，才发现 `read`、`write` 只是最外层的两个门把手。你真想把 IO 看清楚，至少还得继续碰到几类调用。

### `lseek`：原来“读文件”不一定是顺着读

一开始我对文件的理解很线性：打开，读，读完，关掉。后来知道了 `lseek`，才意识到文件不是只能从头往后读。你可以跳到中间、跳到末尾，也可以先写一块，再回头补另一块。

不过我也有疑惑，那为什么不直接把文件全部读出来就完了。后来想起来有的游戏就几十个g，或者电影就超过内存大小了，为什么我们还能流畅的看电影？应该是有点东西在里面的。

这个认识对理解数据库、索引、B+ 树都很重要。因为很多系统并不是顺着把文件读完，它们一直在“跳”。

你把文件看成一个可以随机定位的大数组，很多事就通了：
- 为什么随机 IO 会慢；
- 为什么 page cache（虽然当时根本不懂什么是page cache） 会有命中和 miss；
- 为什么顺序读写和随机读写差别会那么大。

### `fsync` / `fdatasync`：`write()` 完了，不等于真的落盘了

这个点最开始很反直觉。

程序里 `write()` 返回成功，你很容易觉得“好了，写进去了”。后来才知道，这很多时候只代表数据先进了内核缓冲。至于什么时候真正落到磁盘，操作系统自己决定。

所以才有 `fsync()` 和 `fdatasync()` 这种调用。你可以把它们理解成一句很硬的话：别再拖了，现在就刷。

我后来再往回看，很多现象都卡在这里：
- 为什么一条条同步写很慢；
- 为什么某些系统宁可晚一点刷盘，也要先把吞吐做上去；
- 为什么数据库、消息队列、文件系统都在用不同方式处理“什么时候算真正提交成功”这件事。

我后来再看数据库事务、WAL、broker 刷盘策略，脑子里一直挂着这个问题：你到底是在 `write`，还是在 `fsync`？这两个调用挨得很近，代价却不是一个量级。

### `mmap`：文件和内存之间那条线，会突然变软

后来学操作系统课，我知道操作系统会保护内核，所以你的程序不能随便访问磁盘，所以数据是从磁盘到内核内存，再到用户内存的。当时也简单思考过，这样也对，但是要多拷贝和刷回一次，不能有更高效的方法吗？

然后会碰到 `mmap`。这个调用第一次让我意识到，文件和内存之间的边界也没有想象中那么硬。是不是，只是少了两次拷贝？

你可以把文件映射进进程地址空间，然后像改数组一样去改它。写起来很优雅，但背后还是 page cache、缺页异常、脏页回写这些老问题。

然后我知道了确实可以少拷贝，但是性能上的提升有多高呢？我当时没有做实验，没有一个具体的数值。

---

## 再往下，开始碰到“少拷贝一点”的问题

如果说 `read`、`write`、`lseek`、`fsync` 这些调用解决的是“能不能读写”和“什么时候算写完”，那 `sendfile` 和 `io_uring` 这类东西，解决的就是另一个问题：

**这条路径里有没有多余的路？**

### `sendfile`：少搬一次，就是赚到

我后来碰到 `sendfile`，是在看网络传输和对象存储上传的时候。

这个调用最吸引人的地方很直接：它把数据从磁盘直接送到 socket，中间不再经过用户态那两次来回搬运。

以前那条经典路径是这样的：

1. 磁盘 -> 内核缓冲区
2. 内核缓冲区 -> 用户态缓冲区
3. 用户态缓冲区 -> 内核 socket buffer
4. socket buffer -> 网卡

写成图会更直观点：

```text
read + write:
disk -> kernel buffer -> user buffer -> socket buffer -> NIC

sendfile:
disk -> kernel buffer -> socket buffer -> NIC
```

`sendfile` 至少把中间两步里一大块拷贝省掉了。这样 CPU 不用老忙着搬数据，吞吐就更容易上去。

但是我没想到使用场景，从哪里来？到哪里去？我要处理这个文件，然后把它发给别人吗？那我不还得从用户态拷贝到内核态吗？

### `io_uring`：我先把它理解成“少折返”

再很久很久以后，碰到 `io_uring`，感觉也差不多，这时候我已经知道，批处理要比每次提交都要好。

我第一次看 `io_uring`，提交请求，拿结果，用户态和内核态之间少折返几次，感觉也就那样啊。请求可以一批批塞进 ring，结果也能一批批拿回来。

我写造数工具的时候，后来观测到，io是瓶颈的时候，就开始寻求解决方法，寻求到了io_uring。数据量一大，你就不再只关心“能不能写”，而是开始关心：
- 每次系统调用的固定开销有多大；
- 能不能批量提交；
- 能不能减少上下文切换；
- 数据路径是不是足够短。

这些问题在小程序里不显眼。数据一大，差别就全出来了。

## 系统调用为什么会一步步变复杂

我后来回头看，会觉得自己接触这些调用的顺序，其实挺自然的。

- 一开始只需要 `read`、`write`
- 文件要随机读写时，碰到 `lseek`
- 真正在乎“落没落盘”时，碰到 `fsync`
- 想少拷贝一点时，碰到 `sendfile`
- 想把系统调用本身的固定成本压低时，碰到 `io_uring`

我后来觉得，倒不是自己忽然更爱底层了。是手里的问题变了。

程序小的时候，调用看起来只是接口。

等数据量上来，路径就露出来了。再往后，连业务假设也跟着露出来。这个过程我自己也不是一下就想清楚的。

这也是我后来越来越在意的一点：系统调用不是孤零零的 API。你一旦开始碰吞吐、延迟、拷贝次数、刷盘时机，讨论就会从“这个调用怎么用”慢慢滑到“它为什么会长成这样”。

## Kafka 为什么快

其实很早追求装逼的时候，看过公众号文章讲《Kafka 为什么这么快》，当时完全看不懂，后来再看，已经不太发懵了。

我后来看了数据密集型应用，发现其实只要关注三件事：数据怎么写，数据怎么读，数据怎么同步。用这个思路，能够理解关系型数据库和mq的区别。

kafka的场景很明确：

- 写多读少；
- 追加写为主；
- 消费时大多顺序读；
- 对吞吐特别敏感；
- 可以接受由 broker 统一批量刷盘，而不是每条消息都立刻 `fsync()`。

我最早对上号的是 page cache。就是从这里开始，我才觉得这事不再只是“Kafka 做了很多优化”。

数据库系统里，很多事务提交之后要立刻 `fsync()`，因为你得马上告诉用户“这笔数据已经稳了”。Kafka 沿着另一条路走。消息先顺序追加到日志文件里，先进 page cache，再按策略批量刷盘。

再往下看，是它的访问模式很规矩。这个规矩很值钱。

Kafka 的 broker 主要就是顺序写日志，消费者主要就是顺序读日志。这跟 B+ 树那套面向随机读写、频繁改页、频繁命中不同数据页的模式差别很大。顺序写对磁盘和 page cache 都友好，顺序读也更容易把预读和缓存效果吃满。而 关系型数据库很多时候可能会page miss，去磁盘上拉页面出来。

还有一层是数据拷贝路径。

消费者拉消息时，Kafka 可以用 `sendfile` 这种零拷贝方式，把数据直接从 page cache 送到 socket。中间少掉两次用户态和内核态之间的搬运，CPU 就不会白白耗在拷贝上。

这些东西拆开看都不玄。

真正难的地方，是你得先把业务场景收住，然后再让软硬件路径跟它一层层对齐。

写到这里，脑子里很自然会冒出另一个问题：不用这些技术，能不能直接拿 MySQL 模拟一个 MQ？

当然能。

对外接口完全可以做得很像：一张表存消息，生产者插入，消费者轮询，消费完改状态。功能上没问题，很多小系统一开始也就是这么干的。

但性能差很多。差在哪？不在“SQL 比日志落后”，而在底层路径根本不是为这个场景准备的。

MySQL 要处理的是更通用的事务语义：

- 它要支持随机读写；
- 要支持索引查找；
- 要维护 B+ 树；
- 要处理页分裂、脏页回写、锁、MVCC；
- 提交时通常还要更严格地保证持久化。

这些能力都很值钱，但它们不是白来的。你拿一个为“通用 OLTP”设计的系统去硬扛“高吞吐顺序日志”场景，当然也能跑，只是不会跑得像 Kafka 那样顺。

我现在回头看，学 IO 这件事最该记住的一点，大概就在这里：**接口一样，不代表路径一样；功能一样，不代表代价一样。**

你当然可以拿通用系统模拟专用系统。功能往往能做出来，很多时候也确实够用。可一旦性能差距被拉开，大家比的就是底下那整套针对场景做过的软硬件优化。Kafka 值钱的地方，我现在更愿意这样理解：它没有更万能，它只是把“高吞吐顺序日志”这件事做得更专，更狠。
