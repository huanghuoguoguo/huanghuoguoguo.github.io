2025.12.1 开始读这本书。之前在工作中遇到有关性能指标收集的问题，尽管我秉持着能收集则收集、应收尽收的原则，但其中仍然会有一些指标是从一级指标计算得来的。所以我对**哪些指标反映哪些问题、代表什么含义**很感兴趣。就像人体体检报告，肌酐、血糖偏高偏低有哪些意义——我也希望能看懂系统的"体检报告"。

看完之后觉得前四章对我用处最大。后面十几章讲的是具体的观测工具（perf、Ftrace、BPF），很多 shell 命令我没怎么用过，但前四章建立的思维框架已经足够受益了。

这本书最核心的贡献不是教你敲哪个命令，而是回答了一个更根本的问题：**性能到底是什么？怎么定义它？怎么度量它？**

## "性能好"到底是什么意思？

日常开发中我们经常说"这个系统性能好"、"这个接口太慢了"。但"好"和"慢"都是主观判断——好是多好？慢是多慢？慢在哪里？

Brendan Gregg 在书里做的第一件事，就是把"性能"这个模糊的概念拆成可度量的指标体系。性能不是一个单一的数字，而是多个维度的组合：

**延迟（Latency）**：一个操作花了多长时间。这是用户最直接感知到的指标——页面加载 200ms 和 2s，体验天差地别。但只看平均延迟是不够的，你需要看分布：P50 是 10ms 不代表所有请求都快，P99 可能是 500ms，意味着每 100 个用户里有 1 个在骂你。

**吞吐量（Throughput）**：单位时间内完成了多少工作。QPS、TPS、MB/s，都是吞吐量的不同表达。吞吐量和延迟之间往往存在 trade-off——为了提高吞吐量做 batch 处理，单个请求的延迟可能会上升。

**利用率（Utilization）**：资源被使用了多少比例。CPU 利用率 80% 意味着还有 20% 的余量。但利用率高不一定是问题——100% CPU 利用率如果全在做有用功，那是好事；如果大部分在做 GC 或者锁等待，那就是瓶颈。

**饱和度（Saturation）**：资源的排队程度。利用率看的是"用了多少"，饱和度看的是"来不来得及"。一个 CPU 利用率 100% 但运行队列长度为 1，和利用率 100% 但运行队列长度为 50，完全不是一个严重程度。

**错误（Errors）**：操作失败了多少。超时、重试、异常，这些都算。错误率往往是最应该先看的——如果 10% 的请求都在报错，优化延迟毫无意义。

这五个维度，书里给了一个结构化的检查方法：**USE 法（Utilization, Saturation, Errors）**。对每一种资源（CPU、内存、磁盘、网络），都检查这三个维度。这比漫无目的地敲命令有效率得多。

## 产品承诺了什么：SLA / SLO / SLI

性能指标不是孤立存在的，它背后对应的是产品对用户的**承诺**。书里虽然没有直接用 SRE 的术语体系，但讨论的本质是一样的：

**SLI（Service Level Indicator）**：你选哪些指标来度量性能？比如"接口的 P99 延迟"、"请求成功率"。

**SLO（Service Level Objective）**：这些指标要达到什么水平？比如"P99 延迟 < 200ms"、"成功率 > 99.9%"。

**SLA（Service Level Agreement）**：如果没达到 SLO，后果是什么？比如"可用性低于 99.9% 时赔偿用户"。

为什么要提这个？因为**性能优化不是无止境的，它有一个"够了"的标准。** 把 P99 从 200ms 优化到 180ms，如果 SLO 是 200ms，那这 20ms 的优化可能不值得你花一周时间。但如果 P99 已经到了 500ms 而 SLO 是 200ms，那就是着火了，必须立刻定位原因。

性能指标只有跟承诺关联起来，才知道什么时候该动手、什么时候该停手。

## 核心方法论：先看方向，再挖细节

这是前四章最值钱的部分——不是教你用什么工具，而是教你**按什么顺序去看问题**。

### USE 法：对每种资源检查三个维度

```
对于每种资源（CPU、内存、磁盘、网络）：
  1. Utilization：利用率多高？
  2. Saturation：有没有排队？
  3. Errors：有没有报错？
```

这个方法的价值在于**穷举**。不是你怀疑哪里有问题就去看哪里，而是把所有资源过一遍，避免遗漏。书里反复强调一句话我印象很深：*"Performance issues can originate from anywhere, including areas of the system that you know nothing about and didn't suspect."* 后来我自己优化 db_gen 的时候，五次瓶颈都不在我预想的地方，深有体会。

### 观测工具的层次

书里把观测工具分成了四层，从粗到细：

```
counters（计数器）         → 看方向：哪里红了？
  ↓
profiling（剖析）          → 看分布：CPU 时间花在哪个函数？
  ↓
tracing（跟踪）            → 看细节：每一次调用花了多少时间？
  ↓
BPF（可编程跟踪）           → 写脚本：自定义你想观测的任何东西
```

对应的数据来源：`/proc`、`/sys`（counters），perf events（profiling），ftrace / tracepoint（tracing），BPF maps（BPF）。

现场套路是：**counters 先看方向，perf 再拍火焰图，bpftrace 最后写脚本定位根因。** 不要一上来就上重武器——`top` + `vmstat` + `iostat` 这些轻量工具往往就能定位到 80% 的问题。

### 60 秒 Checklist

书里给了一个"60 秒快速检查"的命令列表，新手拿来直接用：

```bash
uptime          # 看负载趋势
dmesg -T        # 看内核消息（OOM、硬件错误）
vmstat 1        # CPU、内存、I/O 概览
mpstat -P ALL 1 # 每个 CPU 核的利用率
pidstat 1       # 每个进程的 CPU 消耗
iostat -xz 1    # 磁盘 I/O 详情
free -m         # 内存使用
sar -n DEV 1    # 网络接口吞吐量
sar -n TCP 1    # TCP 连接统计
top             # 综合概览
```

这十个命令不需要记住每个参数的含义，先跑一遍看数字有没有异常的，有异常再深入。这就是 counters 层的作用——快速定方向。

## 统计方法：别被平均值骗了

书里花了不少篇幅讲统计方法，我觉得最重要的一点是：**不要只看平均值。**

平均延迟 10ms，听起来不错。但如果延迟分布是这样的：

```
90% 的请求：5ms
10% 的请求：55ms

平均值 = 0.9 × 5 + 0.1 × 55 = 10ms
```

平均值确实是 10ms，但每 10 个用户里有 1 个在等 55ms。如果只看平均值，你会觉得没问题；看 P90 才能发现问题。

更极端的情况是**多重模态分布**——延迟有两个峰，比如大部分请求 5ms，少部分请求 500ms（可能命中了不同的代码路径或缓存状态）。这时候平均值和中位数都没有意义，你需要看直方图才能发现有两个完全不同的群体。

书里推荐的指标组合是：

| 指标 | 反映什么 |
|------|---------|
| P50（中位数） | 典型用户的体验 |
| P95 | 大部分用户的体验上限 |
| P99 | 长尾用户的体验 |
| Max | 最差情况 |
| 标准差 / 变异系数 | 稳定性 |

P99 特别重要，因为在分布式系统中，一个用户请求可能扇出到几十个后端服务。如果每个服务的 P99 是 100ms，那整个请求的延迟大概率会被某个后端的长尾拖住。**微服务越多，P99 越重要。**

## 在 AI 时代，可观测性比经验更重要

这是我读完这本书之后，结合最近做 LLM Agent 的经验产生的一个想法。

传统的性能分析高度依赖**经验**。一个资深工程师看到 CPU 利用率 90%、sys 态占比高、`futex` 调用频繁，能凭经验直接判断"这是锁竞争"。经验越丰富，定位越快。

但在 AI 做性能诊断的场景下，情况反过来了。LLM 不缺"经验"——它在训练数据中见过大量的性能问题和解决方案。**它缺的是信息。**

回忆一下 ReAct 架构的工作方式：

```
Thought:  根据当前信息做判断
Action:   调用工具获取新信息
Observation: 工具返回的数据
→ 重复，直到得出结论
```

整个循环的质量取决于 Observation 的质量——**你能给 Agent 看到什么，决定了它能判断什么。** 如果系统没有打正确的日志、没有暴露关键指标、perf 数据缺失，Agent 就像一个蒙着眼睛的医生，再有经验也诊断不了。

我自己做性能诊断 Agent 的时候就深有体会。Agent 的第一个动作总是调 `get_system_metrics()`，如果 `top` 的输出不够结构化（比如没有分 usr/sys/iowait），它很难做出准确的下一步决策。后来我加了工具输出预处理，只保留 top 5 进程和关键指标，效果反而更好——不是信息越多越好，而是**有用的信息越多越好**。

这跟书里强调的观测方法论是一脉相承的：

| 传统模式 | AI Agent 模式 |
|---------|-------------|
| 经验驱动：知道该看什么 | 数据驱动：能看到什么就分析什么 |
| 脑内推理：凭经验关联现象和原因 | 上下文推理：在 Observation 中找模式 |
| 瓶颈：人的经验天花板 | 瓶颈：可观测性的完备程度 |

所以这本书的方法论在 AI 时代不但没有过时，反而变得更重要了：

- **USE 法的穷举思路**告诉你该暴露哪些指标——Agent 需要看到每种资源的 Utilization、Saturation、Errors，你的监控系统就得把这些数据准备好。
- **工具层次（counters → profiling → tracing）**告诉你该怎么组织 Agent 的工具集——先给轻量的概览工具，Agent 发现异常再调用重量级的剖析工具，而不是一上来就跑 `perf record`。
- **60 秒 checklist** 直接可以编码成 Agent 的初始诊断流程。

换句话说：**这本书定义的不只是人类性能工程师的工作方法，它同时也在定义 AI 性能诊断 Agent 的工具集和观测接口。** 如果你想让 Agent 做好性能诊断，先确保你的系统按照这本书的方法论做好了可观测性。
