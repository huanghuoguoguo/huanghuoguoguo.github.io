标题党了。我的意思是，盲目使用 Harness，是错的。

上个月给小伙伴分享 Harness Engineering 的时候，其实我已经用了 SP 和 OpenSpec 一阵了。tlive 用了 OpenSpec，langbot 试了 Superpowers + OpenSpec。

当时在腾讯会议里讲了那一套：上下文窗口是核心，Harness 补的是模型外面那层壳，Skills 是方法论，Specs 是契约。讲的时候觉得挺对的。但最近越来越觉得，那篇分享里有些话说得太满了。

先说 SP 那次。做 langbot 的沙箱 sandbox 功能。agent 在那跑，我在旁边看着。聊的方向还行，但生成的 coding 步骤太多了，一层套一层，我根本看不过来。

其实我不关心具体实现。我关心的是模块边界清不清晰、职责对不对、有没有依赖倒置、架构有没有坏味道。具体代码写得好不好，单测能抓一部分，AI review 能抓一部分。所以那次我没看实现细节，直接让它跑。

跑完发现执行得挺好。后来才发现错了。因为没看具体实现，漂移发生在你看不到的地方，你不知道，等你知道了已经是两周以后。

搭配中转站 + Opus，一个任务跑一个半小时，出来一堆你懒得看的步骤。太沉。

---

OpenSpec 轻很多，但也暴露了另一个问题。

specs/ 目录下那些 proposal、design、tasks，全是英文的。不是自然语言那种英文，是代码变更文档，具体到哪个文件、哪一行、改什么。AI 读起来很舒服，人读起来很折磨。

我越来越不想点开看。每次都是 AI 生成 spec 之后直接让它执行。实在太多，太细，太不像给人看的东西。

那问题来了。Harness 到底是给谁用的？给人，还是给 AI？

如果是给人的，写出来的文档人根本不想看，那它的价值在哪？如果是给 AI 的，好模型真的需要这些吗？

---

我以前一直默认一个前提：**Harness 不管怎样，总能提高模型的能力。**

最近不这么想了。

Claude Code 之父 Boris 在红杉的闭门访谈里，把 Harness 定义为 `Agent - Model`，模型外面那层壳。权限控制、安全机制、human-in-the-loop、Skills、Specs，全在壳里。他的暴论是：这层壳会越来越不重要。模型每强一代，就吃掉上一代的产品设计。Sonnet 3.5 出来之前，Tab 补全是最先进的 coding 方式；Opus 4 出来之后，agent 直接写全部代码成了新的起点。今天 CC 里的权限控制、安全机制，等模型对齐做好了，模型直接做对，Harness 存在的理由就消失了。

不是某个产品变差了，是模型变强了，壳就悄悄变薄了。

他自己的做法更极端。一部手机，远程遥控几十个对话，白天数百个 agent 并行，晚上几千个 agent 做深度重构。loop 模式 24/7 干活。Zero-Human-in-the-Loop。

这些听着很震撼。但震撼之后我想的是另一个问题。

---

**Harness 到底在哪个维度上提高了模型？**

对弱模型，有用。提前搭好框、定好步骤、写好约束，能兜住模型的底。这是 Harness 真正立功的地方。

对强模型，我现在觉得不太需要。它知道怎么做，你给它塞一堆 spec 让它按这个来，它也能做，但做出来的东西不会比不塞 spec 更好，有时候还更差。spec 框住了它本来能做得更好的方向。然后你自己还不看这些 spec，连判断都省了。

Boris 说 Harness 会被模型当早餐吃掉。我信。但我的体感比这个更具体。**有些 Harness，连当早餐都不配，因为它从一开始就不是给你用的。**

它是一堆 AI 写给 AI 看的英文文档，人从来没读过。模型变强之后，这些文档连模型都不需要了。那它们是什么？磁盘垃圾。

---

我现在换了一种做法。

让 GPT-5.5 写 plan 文档。需求分析、为什么要做、怎么做、预计影响哪些文件。和 OpenSpec 做的事一样，产出一份结构化的执行计划。但要求不一样：文档里要有我聊的过程，有具体的决策点，有明确要做和明确不能做的边界，有我的判断在里面。

写完之后我自己过一遍。能读懂，能记住，能随时翻开看。然后让 GPT-5.5 再出一份执行文档。那份不用给人看，AI 自己读就行。扔给 GLM 执行，执行完 GPT-5.5 review，吐下一步。

CC 开 agent team + orchestrator + worktree，跑起来很舒服。快，质量也不差。没跑过严格的 benchmark，目测和纯 SP 或者纯 GPT-5.5 单干差不多。

---

Boris 在访谈最后说：未来写出最好财务软件的人，绝不是优秀的软件工程师，而是极其优秀的会计师。Domain 决定价值，Coding 只是手段。

这话对。但他漏了半句：一个懂 coding 的会计师，比一个纯会计师更快到那个未来。Domain 决定价值，但 coding 是让你能亲自下场验证 domain 直觉的前提。不然连 GPT-5.5 写的 plan 对不对你都判断不了。

---

Boris 用手机遥控几千个 agent，那是他的做法。我用 GPT-5.5 + GLM + CC agent team，这是我的做法。没有什么 Harness 框架是银弹，连 Claude Code 的亲爹说的话也不能照搬。

试一圈，留下顺手的，扔掉不顺手的。61 个 commit 都能扔，一个 Harness 框架有什么扔不得的。
