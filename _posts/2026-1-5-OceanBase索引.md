我最早接触分布式数据库的时候，脑子里一直有个很具体的问题：以前在 MySQL 上做分库分表，很多路由逻辑都得业务自己背着走。那换成 OceanBase 这种原生分布式数据库以后，这些事到底挪到哪里去了？

比如，数据怎么切成多个分片，索引怎么命中，查不到本地数据的时候请求怎么继续往下走。再往前一步，主键和索引该怎么设计，才不会把后面的路由和扩容一起拖垮。

这篇主要就是顺着这些问题，把分片、主键和索引设计里几件我当时最容易卡住的事情串一下。

---

## 一、MySQL 分库分表 vs 原生分布式分片

在分布式数据库架构中，**分片（Sharding）** 是将海量数据分散到多台物理节点上以实现横向扩展（Scale-Out）的核心技术。

### 1. MySQL 传统分库分表机制
在 MySQL 体系下，开发者往往需要借助中间件（如 ShardingSphere、MyCat 等）对数据关键列（通常为主键或业务 ID）进行 Hash 或范围取模，从而将数据打散到不同的数据库实例和表中。这种做法的痛点在于：分片逻辑强侵入业务层，跨节点事务处理复杂，且动态扩缩容的运维成本极高。

### 2. OceanBase 的原生分片机制
作为原生分布式数据库，OceanBase 提供了更为高级、透明的底层分片自动化管理：
- **数据分片单位（Tablet）**：OceanBase 将大表聚簇打散为多个较小的数据分片（Tablet），分布式架构下的多节点共同担负这些分片的存储与计算，不仅增强了并行处理能力，还天然实现了负载均衡。
- **丰富的关联分片策略**：支持基于 Hash 分片（均匀打散）和 Range 范围分片（适合时间序/递增型数据的扫描），业务无需显式感知。
- **局部索引与路由流转**：OceanBase 会在每个分片所在节点独立维护局部索引（Local Index）。当发起查询时，计算引擎通过预编译的执行计划，根据分片键精准定位到目标主节点。若查询不得不跨越多个分片，计算节点（SQL 层）会自动发起分布式查询计划，协调并行调度并在最终汇聚计算结果。

### 3. PolarDB 的分片机制设计
拿 PolarDB-X 这一类产品来说，思路也差不多：分片规则可以通过 DDL 显式声明，数据分布和路由对业务相对透明。你还是要关心分区键选得对不对，但不用像传统分库分表那样，把大量路由细节硬塞回应用层。

我现在更愿意把这类原生分布式数据库理解成：它把“怎么切数据、怎么找分片、怎么把分布式执行串起来”这些事情尽量收回数据库内核和 SQL 层去做，业务代码不用再自己维护一套分库分表中间件逻辑。

## 二、分布式场景下的主键选择痛点

主键（Primary Key）到了分布式场景里，至少得先满足一件事：它不能把全局唯一性搞乱。单机时代很好用的 **自增主键（Auto Increment）**，到了这里就要多想一步。

### 分片下的自增冲突
自增 ID 最大的问题在于，它往往无法在数据插入前就在应用层确切感知其值，必须依赖单台数据库节点执行完 `last_insert_id()` 以后方可确定。这意味着：**如果各个数据分片独立维护自增逻辑，必定会出现重复的自增 ID 存在于不同分片中**。

拿电商订单表 `orders` 举个例子。假设我们用 `O_CUSTKEY` 做分片键，用 `O_ORDERKEY` 做主键：

```sql
CREATE TABLE `orders` (
  `O_ORDERKEY` int NOT NULL auto_increment,
  `O_CUSTKEY` int NOT NULL,
  `O_ORDERSTATUS` char(1) NOT NULL,
  `O_TOTALPRICE` decimal(15,2) NOT NULL,
  `O_ORDERDATE` date NOT NULL,
  ...
  `O_COMMENT` varchar(79) NOT NULL,
  PRIMARY KEY (`O_ORDERKEY`),
  KEY (`O_CUSTKEY`)
) ENGINE=InnoDB;
```

如果 `O_ORDERKEY` 还是单机式自增，那“订单编号 = 1” 这种值，就可能在分片 A 生成一次，在分片 B 又生成一次。

<!-- ![OceanBase 自增主键冲突分析](/images/OceanBase-suoyin-1.jpeg) -->

按我的理解，更稳的做法是不要把“全局唯一主键”这件事继续押给单机自增。常见办法还是在业务侧或独立发号服务里生成全局唯一 ID，比如雪花算法，或者带业务编码的有序 ID。

## 三、分布式索引：怎么把查询路由得更准

我后来越来越相信一个很朴素的判断：**高频点查最好能直接落到一个分片上。**

### 1. 跨分片查询的代价
假设我们的订单表以买家 ID (`O_CUSTKEY`) 为分区键，但运营系统发起了一个按照订单号 (`O_ORDERKEY`) 检索详情的诉求：

```sql
SELECT * FROM orders WHERE o_orderkey = 1;
```
由于条件不涉及分片键，分布式计算引擎不得不向底部所有的 1000 个分片全量下发 RPC 查询请求，再等待汇总。这是典型的并发灾难。

为了绕开这种全分片扫描（Full Partition Scan），常见做法就是拿一点额外存储去换查询效率：

### 2. 传统方案：异构索引表映射
通过监听主表的变更，异步建立一张极小的**索引映射表**：

```sql
CREATE TABLE idx_orderkey_custkey (
  o_orderkey INT,
  o_custkey INT,
  PRIMARY KEY (o_orderkey)
);
```

查询时，将其拆分为两步回表操作：
1. `SELECT o_custkey FROM idx_orderkey_custkey WHERE o_orderkey = 1;`
2. 拿到分片信息后，带着分片键去主表查询。

这种设计虽然避免了全分片扫，但这增加了事务的复杂性与多余的网络交互开销。

### 3. 最优方案：基于编码规则的主键内嵌分片键
分布式架构最优雅的设计，是将**路由信息直接编码到你将要高频查询的列中**。

一种常见思路，是把订单主键 `O_ORDERKEY` 设计成带路由信息的编码。
比如可以把它组织成：`业务原始订单号 + 分片键信息`。

此时查询：
```sql
SELECT * FROM Orders WHERE o_orderkey = '1000-1';
```
这样查 `o_orderkey = '1000-1'` 时，路由层就能从编码里直接拿到分片信息，请求更容易直达目标分片，不用先去扫一轮映射表。

这一设计思想在业界大厂有着广泛运用。以**淘宝订单业务**的设计为例：

## 四、全局字典表与分布式唯一索引

### 1. 全局读多写少小表最佳实践（Broadcast Table）
分布式体系中往往存在一些脱离于分片逻辑的基准数据表：例如省市字典、国家区域（tpch 库中的 `nation`）。由于应用查询频繁执行 `JOIN` 会跨分片传递大量数据，最优解是：**采用冗余同步机制，将这类小表完整复制广播到所有的集群节点（分片）计算层内存中。**

<!-- ![全局表冗余复制机制](/images/OceanBase-suoyin-3.jpeg) -->

### 2. 分布式唯一索引与局部陷阱
这一块我最早也容易想当然，以为写了 `UNIQUE KEY(xxx)` 就天然全局唯一了。真到分布式表上，就要先看索引和分区键是什么关系。

如果唯一键本身覆盖了分区键，那局部唯一索引就有机会成立；因为同一组键值只会落到一个分区里。可如果唯一键不包含分区键信息，只靠局部索引通常没法兜住全局唯一性。这时候要么依赖数据库提供的全局索引能力，要么把全局唯一放回发号器或者业务侧设计里。

## 五、我现在会先记住的几点

写到这里，我现在更愿意先记住这几件事：

1. **分布式表不要想当然沿用单机自增主键**：全局唯一 ID 怎么来，最好在设计一开始就想清楚。
2. **`UNIQUE` 先看它和分区键的关系**：局部唯一不等于天然全局唯一，这里很容易踩坑。
3. **高频点查最好带上路由信息**：不管是直接用分片键，还是把路由信息编码进查询列里，目标都是少扫分片。
4. **小而高频的维表适合冗余分发**：这样做往往比跨分片 `JOIN` 更省事。
