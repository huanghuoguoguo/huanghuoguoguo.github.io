上一篇[服务器性能对比]({% post_url 2026-3-23-服务器性能对比背后的并发模型之战 %})里聊到 Actix 的 Actor 模型，当时说"我对 Actor 模型还没完全搞明白"。后来又翻了翻的资料，发现并发编程里最难的部分其实不是怎么起一个线程或者怎么调度——那些事情运行时帮你干了。真正让人掉头发的是：线程（或者协程、Actor、随便叫什么）之间怎么安全地交换信息。

这件事有好几种截然不同的思路。学 Java 的时候满脑子都是锁。学 Go 遇到了 channel。碰 Actix 碰到了 Actor。它们解决的是同一个问题，但哲学完全不同。

## 共享内存 + 锁：先从最直觉的说起

这是大多数人学并发时接触的第一个模型。数据放在一块共享的内存里，谁要读写就先拿锁。

```java
synchronized (account) {
    if (account.balance >= amount) {
        account.balance -= amount;
    }
}
```

直觉，好理解。数据就在那，锁是保安，一次只放一个人进去。Java 围绕这个思路搭了一整套工具：`synchronized`、`ReentrantLock`、`ReadWriteLock`、`StampedLock`、`Condition`、`CountDownLatch`、`Semaphore`……[《Java 并发编程的艺术》]({% post_url 2025-2-13-java并发编程的艺术 %})那本书基本就在讲这些。

在曙光做性能指标采集的时候，场景是典型的读多写少——几十个线程并发读指标数据，偶尔有一个写线程更新。用了 `StampedLock` 的乐观读和 `CopyOnWriteArrayList`，性能比 `synchronized` 好不少。当时觉得：锁选对了，问题就解决了。

后来才发现，锁本身就是问题。

### 锁的几个坑

死锁不用多说。线程 A 持有锁 1 等锁 2，线程 B 持有锁 2 等锁 1，两个人互相等到天荒地老。教科书上的经典，但真实项目里一不小心就踩到。尤其是锁的粒度没设计好，几个模块之间的锁顺序不一致的时候。

死锁有四个必要条件：互斥、持有并等待、不可剥夺、循环等待。四个同时满足才会死锁，破坏任何一个就能解。操作系统课上讲过好几遍，面试也老问。上次被问到的时候只说出来三个，"不可剥夺"这条死活想不起来。回来一翻书，就在那明晃晃印着呢。说来也巧，面试的时候脑子一紧，想不起来的往往是最简单的那条。

性能墙更隐蔽。上一篇文章里提过，锁竞争的开销不只是等待时间。现代 CPU 上，一次锁操作会触发缓存行失效，跨核同步一次要 100-300ns。100 个线程抢一把锁，CPU 大部分时间不在算东西，在等缓存同步。在[线程池那篇]({% post_url 2026-3-10-线程池没有银弹 %})里聊过，线程本身就有开销，锁竞争再叠上去，并发度一高性能就塌。

心智负担可能是最大的问题。共享内存模型要求程序员时刻想着"这块数据谁还会碰"。漏加一把锁，可能跑一万次才复现一次 bug。加了太多锁，性能掉下去，还可能死锁。锁的粒度、锁的顺序、锁和业务逻辑的交织——这些东西让并发代码的复杂度远超串行代码。

### 锁的变体

不是所有锁都一样粗暴。几种常见的改进：

| 锁类型 | 思路 | 适用场景 |
|-------|------|---------|
| Mutex（互斥锁） | 同一时间只有一个线程能进 | 通用，但并发度低 |
| RWLock（读写锁） | 读和读不互斥，读和写互斥 | 读多写少 |
| StampedLock | 乐观读，不加锁先读，读完验证有没有被改 | 读远多于写 |
| SpinLock（自旋锁） | 不挂起线程，忙等 | 临界区极短（几十 ns） |
| 分段锁 | 把数据分成多段，每段一把锁 | `ConcurrentHashMap` |

这些都是在"共享内存"这个大框架下做优化。锁选得好可以缓解问题，但改变不了根本矛盾：**多个线程能看到同一块数据，就必须有协调机制**。

Go 有句名言跳出了这个框架。

## CSP / Channel：别通过共享内存来通信

> Do not communicate by sharing memory; instead, share memory by communicating.

这句话来自 Go 的官方文档，思路来自 Tony Hoare 1978 年的 CSP（Communicating Sequential Processes）论文。

核心想法很简单：别让多个线程去碰同一块数据了。你有数据想给我？往 channel 里丢。我需要数据？从 channel 里取。数据的所有权随着消息传递而转移。

```go
ch := make(chan Order, 100)  // 带缓冲的 channel

// 生产者
go func() {
    for _, order := range orders {
        ch <- order  // 发到 channel
    }
    close(ch)
}()

// 消费者
for order := range ch {
    process(order)  // 从 channel 取
}
```

这段代码里没有锁，没有 `synchronized`，没有 `Mutex`。生产者和消费者通过 channel 传递 Order，谁持有 Order 谁就有权操作它。数据在任意时刻只被一个 goroutine 拥有。

### Go 的 channel 里面到底是什么

说 channel 没有锁，其实不完全对。翻 Go 的 `runtime/chan.go`，channel 的核心结构 `hchan` 长这样：

```go
type hchan struct {
    buf      unsafe.Pointer // 环形缓冲区
    sendx    uint           // 发送位置
    recvx    uint           // 接收位置
    sendq    waitq          // 等待发送的 goroutine 队列
    recvq    waitq          // 等待接收的 goroutine 队列
    lock     mutex          // 这里有把锁
}
```

环形缓冲区、读写指针、等待队列、一把锁。跟 Java 的 `ArrayBlockingQueue` 一比，结构几乎一样——也是 array + ReentrantLock + 两个 Condition。

我当时就在想：那 Go 的 channel 到底比 Java 的 BlockingQueue 多了什么？功能上差不多，为什么体验差这么远？

想了想，大概有三个差距。

第一个是调度集成。goroutine 在 channel 上阻塞的时候，Go 的运行时不是让 OS 线程傻等，而是把这个 goroutine 挂起，让同一个 OS 线程去跑别的 goroutine。这一步很轻，因为 goroutine 是用户态调度的。Java 的 BlockingQueue 阻塞的是 OS 线程，调度开销大得多。Loom 出来之后这个差距在缩小，但 Go 的调度器和 channel 是一起设计的，配合更紧。

第二个是 `select`。这个可能是最大的差距。Go 的 `select` 可以同时等多个 channel，谁先来处理谁。Java 的 BlockingQueue 没法做到这一点。要实现类似效果，得自己搞轮询或者用 `CompletableFuture` 组合，写起来完全不是一个画风。

第三个是语法和语义。`ch <- data` 和 `<-ch` 是语言级操作符，`close(ch)` 能广播"没有更多数据了"给所有消费者。Java 这边 `queue.put()` 和 `queue.take()` 是库方法，关闭语义只能靠放毒丸（poison pill）模拟。写并发代码的时候，语法越简洁越不容易出错，这不是小事。

那用 Java 第三方库能不能搞一个 channel 出来？功能上大概能搞到八成。比如 LMAX 的 Disruptor 是一个高性能环形缓冲区，吞吐量比 BlockingQueue 高一个数量级。但 Disruptor 的 API 更像在配置一个框架，和 channel 那种"往管道里丢一下"的简洁感完全不同。

说到底，Go 赢在"原生"。语言、运行时、调度器围绕 goroutine + channel 一起设计的，每个环节配合得上。Java 用库可以模拟大部分功能，但 `select`、close 语义、调度集成这些东西，库层面补不齐。

### 和 Java 的 Object Monitor 比一比

从锁那节一路讲到 channel，容易搞混一个事情。Java 的 `synchronized` + `wait()`/`notify()` 是 Object Monitor 模式，Go 的 channel 是 CSP 模式。都能协调线程，但出发点不一样。

Object Monitor 的想法：数据是共享的，大家都看得到。但得有个门卫。这个门卫就是 monitor 对象。你要操作数据，先拿到 monitor 的锁。操作完了释放锁。条件不满足（比如队列空了），就 `wait()` 挂起，等别人 `notify()` 叫你。

CSP 的想法：别共享了。你有数据给我？丢到管道里。我要数据？从管道里取。数据跟着消息走，发出去之后你就别碰了。

两个模式都有队列，但队列里装的东西不一样：

```
Object Monitor:
  线程 A ──→ ┌──────────┐ ←── 线程 B
             │  Monitor  │
             │ (锁+等待队列) │
             │  共享数据   │
             └──────────┘
  等待队列里放的是线程。数据不动，线程轮流进。

CSP / Channel:
  goroutine A ──→ [ channel 缓冲区 ] ──→ goroutine B
  缓冲区里放的是数据。数据从左流到右，A 发完不管了，B 拿到就是自己的。
```

我之前一直觉得 channel 不就是个花哨的 BlockingQueue 吗？后来想了想，有个微妙的区别。Java 的 BlockingQueue 你 `take()` 出一个对象，原来 `put()` 的线程照样能访问同一个对象——因为 Java 是引用传递，两边拿到的是同一个引用。channel 传递的是所有权，至少在 Go 的惯例里是这样：发了就别碰了。语言层面不强制，但社区约定如此。

Rust 的 channel 就真的强制了。所有权转移之后编译器不让你再用那个变量，直接编译报错。

### Channel 不止是队列

乍一看 channel 就是个线程安全的队列。但它比队列多了几个关键能力。

前面提过 `select` 是 Go channel 最大的杀手锏，看看具体怎么用：

```go
select {
case order := <-orderCh:
    process(order)
case <-timeout:
    log.Println("timeout")
case <-ctx.Done():
    return
}
```

这个模式在网络编程里特别常见。服务器同时等新连接、等已有连接的数据、等超时、等关闭信号——`select` 把这些事情统一了。写过 Java 的 `Selector` 就知道，NIO 那套代码有多痛苦。Go 的 `select` 就非常简洁。

有界 channel 天然带背压。`make(chan Order, 100)` 创建一个容量 100 的 channel，满了之后生产者会阻塞。不需要额外写限流逻辑，channel 本身就是个速率调节器。

还有关闭语义。`close(ch)` 之后，所有从这个 channel 读的 goroutine 会收到零值 + false。一个信号同时通知所有消费者"没有更多数据了"。

### Rust 也有 channel

Go 不是唯一有 channel 的语言。Rust 的标准库有 `std::sync::mpsc`（多生产者单消费者），Tokio 有异步版本的 `mpsc` 和 `broadcast`，`crossbeam` 库有性能更好的实现。

```rust
let (tx, rx) = tokio::sync::mpsc::channel(100);

tokio::spawn(async move {
    tx.send(order).await.unwrap();
});

while let Some(order) = rx.recv().await {
    process(order);
}
```

写 slap 造数工具的时候，行不定长的路径用了环形缓冲区 + 多生产者单消费者模型。后来想想，这不就是一个自己手搓的 channel 吗？多个 Rust 线程往缓冲区里丢数据，一个消费者线程负责写磁盘。当时对 Rust 的 channel 不熟，觉得自己控制内存布局能快一点，就直接让ai实现了。现在回头看，用现成的 `crossbeam` channel 大概会更好。不过手搓一遍之后对环形缓冲区的理解确实深了，也算没白费。

### Channel 的局限

Channel 解决了"别碰同一块数据"的问题，但不是万能的。

复杂的协调模式用 channel 写起来会很绕。比如多个 worker 需要汇总结果给一个 aggregator，aggregator 算完再通知所有 worker——这种扇入扇出的模式，channel 倒是能搞，但管道接管道，代码容易变成一碗意大利面。

另外，channel 是匿名的。你往 channel 里丢消息，不知道谁会收到。发送方和接收方通过 channel 解耦，但也因此失去了"给特定的某个人发消息"的能力。

Actor 模型在这个地方走了不同的路。

## Actor：每个人管好自己

Actor 模型是 Carl Hewitt 1973 年提出的，比 CSP 还早五年。但让它真正活起来的是 Erlang。

核心思想：把并发系统想象成一群人。每个人（Actor）有自己的脑子（状态）和邮箱。你想让某个人做事，给他发一封信。他按顺序拆信处理，一次只拆一封。

```erlang
%% Erlang —— Actor 是语言级原语
loop(Balance) ->
    receive
        {deposit, Amount} ->
            loop(Balance + Amount);
        {withdraw, Amount} when Amount =< Balance ->
            loop(Balance - Amount);
        {get_balance, From} ->
            From ! {balance, Balance},
            loop(Balance)
    end.
```

每个 Actor：
- 有自己的状态，别人摸不到
- 有一个邮箱，消息按到达顺序排队
- 一次只处理一条消息，内部天然串行
- 可以创建新的 Actor，可以给别的 Actor 发消息

不需要锁。不需要共享内存。因为根本没有共享。

### 和 Channel 到底差在哪

我一开始也觉得 Actor 就是 goroutine + channel 换了个马甲。后来想了想，差别还是不小的。

Channel 以管道为中心，Actor 以实体为中心。Channel 是匿名的管道——你往管道里丢消息，不关心谁收。Actor 有明确的身份（地址），你给某个 Actor 发消息，就像给一个人写信。

```
Channel 模式:   goroutine → channel → goroutine
                         （关注管道）

Actor 模式:     Actor → message → Actor
                      （关注实体）
```

生命周期管理也不同。Actor 有出生和死亡。Erlang 的 supervisor 树可以监控 Actor，一个 Actor 挂了，supervisor 自动重启它。这套机制叫 let it crash——不做防御性编程，出了错就让它挂掉，supervisor 会把它拉起来。Channel 没有这层管理，goroutine panic 了就是 panic 了。

状态封装更彻底。Actor 的状态只有自己能碰。Channel 模式下，goroutine 可以同时持有 channel 和共享变量——Go 不阻止你这么干，只是建议你别这么干。Actor 模型从设计上消灭了这个可能性。

| | 共享内存 + 锁 | CSP / Channel | Actor |
|---|---|---|---|
| 通信方式 | 直接读写共享数据 | 通过匿名管道传递 | 给有名字的 Actor 发消息 |
| 状态管理 | 共享，靠锁保护 | 建议不共享，但不强制 | 强制私有，不可能共享 |
| 协调机制 | 锁、条件变量、屏障 | select 多路复用 | 消息匹配 + supervisor |
| 错误处理 | try-catch，防御性编程 | panic + recover | let it crash + 监督树 |
| 典型实现 | Java, C++, Python | Go, Rust (channel) | Erlang, Akka, Actix |
| 适合场景 | 简单的局部共享 | 流水线、扇入扇出 | 有状态的服务、分布式系统 |

### Erlang 的 Actor 和 Actix 的 Actor

不同语言里 Actor 的纯度差别很大。

我没用过 Erlang，只是后来补资料的时候，慢慢觉得它大概是最“纯”的 Actor 实现。Erlang 的 Actor 叫 process，是语言级原语。每个 process 有独立的堆，GC 是 per-process 的，一个 process 挂了不影响别的。调度器也是自己的，不依赖 OS 线程。整门语言从上到下都是为 Actor 设计的。

Actix 的 Actor 是 Rust 库级别的抽象，这个我稍微熟一点。底下还是 Tokio 的异步运行时。Actor 的邮箱其实就是 channel，调度靠 Tokio 的 task。它没有 Erlang 那种 process 级别的隔离，但继承了 Actor 的核心好处：状态封装、消息驱动、无锁。

JVM 上还有 Akka，据说功能比较完整，有 supervisor 树、有持久化。但我没用过，不太好评价。

```
Erlang:  语言 = Actor。没得选，也不用选。
Akka:    语言 ≠ Actor，但框架完整。没用过，不好说。
Actix:   语言 ≠ Actor，偏轻量。主要用来做 Web 框架的内部模型。
```

## 还有几条路

写到这里我才发现，并发模型这件事不是只有“锁 vs channel vs actor”这么简单。后来顺手翻资料的时候，还看到几条别的路。有些我只碰过边，有些只看过一点实现，先记一下我现在能理解到的程度。

### 无锁数据结构（Lock-free）

不是没有并发控制，是用 CAS（Compare-And-Swap）原子操作替代锁。CPU 提供的原子指令保证"比较并交换"是不可被打断的。

```rust
use std::sync::atomic::{AtomicU64, Ordering};

let counter = AtomicU64::new(0);
counter.fetch_add(1, Ordering::Relaxed);  // 原子加 1，无锁
```

Java 的 `AtomicInteger`、`ConcurrentLinkedQueue`、`LongAdder` 都是这个路子。`ConcurrentHashMap` 的分段锁也部分用了 CAS。

写 slap 的时候，环形缓冲区的读写指针就是用原子操作同步的。多个生产者线程用 CAS 抢写位置，不需要 Mutex。当时觉得挺爽，少了一层锁，好像就高级了。后来才发现，无锁编程只是把麻烦换了个地方。内存序（Ordering）、ABA 问题、cache line bouncing，这些词看着就够让人头疼了，更别说真的调 bug。

无锁适合数据结构层面的优化，不太适合当作通用的并发模型来用。

### 分片（Sharding）

思路很直觉：既然大家抢一把锁太慢，那就把数据切成多份，每份一把锁，各管各的。

`ConcurrentHashMap` 就是这个路子。Java 7 用的是分段锁（Segment），每段一个 ReentrantLock，16 个段就是 16 把锁，并发度直接上去了。Java 8 改成了 CAS + synchronized 锁单个桶，粒度更细。但核心思想没变：把一把大锁拆成很多小锁。

Redis 的哈希槽也是一样的道理。16384 个槽分散在不同节点上，每个节点只管自己那部分数据。Kafka 的 partition 也是，每个 partition 独立写入，消费者各自消费各自的 partition，天然并行。OB 里很多事情其实也是这个思路，先把问题切开，再让每一块尽量少互相打架。

分片的好处是简单直觉，不需要换编程模型，锁还是那个锁，只是竞争的范围小了。坏处是分片策略要设计好。数据倾斜（某个分片特别热）会让优化效果打折扣。还有跨分片操作——两个 key 落在不同分片上要做原子操作，要么用分布式锁，要么用两阶段提交，复杂度一下就上去了。

不过大部分场景下，分片是性价比最高的并发优化手段。不用换语言，不用学新模型，把数据切一切就行。

### STM（软件事务内存）

把数据库事务的思路搬到内存操作里。读写共享变量不加锁，而是开一个事务，事务内部的修改在提交时检查有没有冲突，有冲突就重试。

```clojure
;; Clojure 的 STM
(def balance (ref 1000))

(dosync
  (alter balance - 100))  ;; 事务内修改，提交时检查冲突
```

Clojure 和 Haskell 有 STM 支持。我没实际用过这两门语言，只是看资料的时候觉得思路挺有意思。写起来像串行代码，不用操心锁的顺序。刚看到的时候我还下意识想，这不就是 CAS 换个皮吗？后来想想也不是一回事。CAS 更像一把底层原子工具，STM 想给的是一整套“事务式修改共享内存”的写法。只是这套东西离我平时写的 Java/Go/Rust 还是有点远，所以目前也只是停留在“知道有这么条路”。

### Dataflow / Reactive Streams

不显式管理线程，而是描述数据怎么流转。RxJava、Reactor 都是这个路子。

```java
Flux.range(1, 100)
    .filter(n -> n % 2 == 0)
    .map(n -> n * n)
    .subscribe(System.out::println);
```

之前在[并发模型与语言选型]({% post_url 2026-3-9-并发模型与语言选型 %})里提过 WebFlux 的背压机制。Reactive Streams 的核心价值就是背压，下游告诉上游“慢点”。这个思路我能理解，但说实话我对 Reactive 这套东西一直不算喜欢。只是用 WebFlux 的时候接触过一点，代码可读性差，调试也痛苦，异常栈几乎不可读。Java 社区推了好几年 Reactive，Loom 虚拟线程出来之后，很多人觉得没那么需要了。我现在也更倾向这个判断。能用同步写法解决的事，何必把自己先绕晕。

## 怎么选

写到最后我反而觉得，选并发模型这件事没必要一上来就搞得很大。大部分业务代码直接上锁就够了。Mutex 或 RWLock，一个简单的计数器用 `AtomicInteger` 就行，犯不着上 Actor。简单场景下锁的代码量最少，也最直觉。

channel 最自然的场景是流水线——一个阶段的输出喂给下一个阶段。Go 社区那些 pipeline 例子基本都是这个模式。

Actor 适合有状态的实体。游戏服务器里的每个玩家、即时通讯里的每个会话、IoT 里的每个设备，每个实体有自己的生命周期和行为，天然就是一个 Actor。分布式场景据说也是 Actor 的地盘，给本地 Actor 发消息和给远端 Actor 发消息可以写成一样的代码。但这块我没真做过，所以也只敢先记一个印象，不敢下太死的结论。

无锁数据结构留给确定瓶颈在锁竞争的时候再上。对内存模型没有足够理解的话，无锁代码比有锁的还难调。

## 还没想清楚的

写到这里有个问题一直在想：Actor 和微服务的边界在哪？

一个微服务有自己的状态、接受请求、返回响应——这不就是一个大号 Actor 吗？看到过一个说法，大意是"微服务是穷人的 Actor 系统"。如果语言本身支持 Actor，很多需要拆成微服务的东西可以在一个进程内用 Actor 解决，省掉网络开销和运维复杂度。

但微服务有 Actor 做不到的东西：独立部署、独立扩缩容、技术栈异构。这些是组织层面的需求，不是并发模型能覆盖的。

还有一个更实际的问题：Go 的 goroutine + channel 组合已经足够好用，Actor 框架在 Go 生态里几乎没有存在感。是因为 CSP 足够了？还是因为 Go 社区偏好简单？我倾向于后者。Actor 的完整形态（supervisor 树、错误传播、持久化）确实比 channel 复杂，Go 的文化不太接受这种"重型"抽象。

Rust 倒是两条路都走得通。Tokio channel 给你 CSP 的简洁，Actix 给你 Actor 的结构。到底选哪个，目前我的判断是：做 Web 服务大部分时候 channel 就够了，做有状态的长连接服务（游戏、聊天）Actor 可能更自然。但这个判断还没被项目验证过，先记着。

---

## 回过头来想想

写到最后发现两个比喻一直在脑子里转，觉得还挺帮助我自己理解的，记一下。

Channel 像工厂车间的传送带。也可以说是进程内的迷你 Kafka。生产者往上丢，消费者从另一头取，两边互相不认识。管道是主角，人是配角。谁挂了换一个接着干就行。天然适合流水线。

Actor 像微缩版的 Redis。你想想 Redis 怎么做到高并发还不出数据错乱的？请求从网络进来排成队列，一个单线程按顺序拿出来处理。因为是单线程，所以不需要锁。Actor 在我脑子里就有点这个意思，每个 Actor 有自己的队列（信箱）和私有状态，自己一个人按顺序拆信。所以处理有状态的实体（游戏房间、玩家、会话）时，Actor 会显得很顺。

这两个比喻帮我理清了一个困惑：Channel 解耦的是过程，Actor 封装的是实体。

顺着这个比喻再往下想，我脑子里就会开始把 Actor 和微服务拿来对着看。传统 Web 开发，尤其 Spring 那套，更像无状态服务。请求进来，几十个节点都冲向数据库抢同一行数据，靠行级锁或者 Redisson 分布式锁来保证不乱。Actor 的思路反过来，不去数据库抢，而是把状态拉到 Actor 自己的内存里，请求排队发给这个 Actor，单线程处理。这样一想，微服务和 Actor 确实有点像。一个微服务有自己的状态、接受请求、返回响应，这不就是一个大号 Actor 吗？所以才会有人说“微服务是穷人的 Actor 系统”。

但我越往下想，越觉得这两个东西还是不能简单画等号。微服务有 Actor 做不到的东西，比如独立部署、独立扩缩容、技术栈异构。这些更多是组织和交付层面的需求，不是并发模型本身能覆盖的。也就是说，Actor 解决的是进程内甚至分布式系统里的状态和消息问题，微服务解决的则不只是状态，还有团队协作、部署边界、运维拆分这些更外层的事。

这时候另一个问题又会冒出来：如果 Actor 真这么自然，为什么 Go 生态里几乎没什么存在感？Go 的 goroutine + channel 组合已经足够好用，大家写流水线、写 worker pool、写网络服务都很顺手。我现在更倾向于一个很朴素的解释，不一定是 CSP 在理论上更强，而是 Go 社区整体就偏爱简单。Actor 的完整形态，像 supervisor 树、错误传播、持久化这些东西，确实更重。Go 这门语言从气质上就不太喜欢这种“大而全”的抽象。

Rust 看起来倒是两条路都走得通。Tokio channel 给你 CSP 的简洁，Actix 给你 Actor 的结构。到底选哪个，我现在的感觉还是：做 Web 服务，大部分时候 channel 就够了；做有状态的长连接服务，比如游戏、聊天，Actor 可能更顺一点。但这个判断我还没拿项目真的验证过，所以只能先记在这里。

再顺着想下去，就会碰到两个我目前还没想透的问题。第一个是，如果状态全在 Actor 所在机器的内存里，那机器挂了怎么办？传统 Web 服务至少还能把状态甩给数据库，Actor 这条路如果真把状态放进内存里，恢复就成了绕不过去的事。看资料说 Akka 和 Erlang 有一套 Event Sourcing 的方案，不存状态本身，而是存操作日志，挂了之后重放日志恢复。和 MySQL 的 redo log、Redis 的 AOF 其实是一个思路。但它到底怎么把这件事做顺，我还没认真看过，先留个坑。

第二个问题是分布式场景下 Actor 怎么定位。给“张三”这个 Actor 发消息，框架怎么知道它在哪台机器上？据说 Akka Cluster 有一套 Cluster Sharding 的机制来做路由，和微服务里的注册中心思路有点像。但这块我确实不了解，也不想在没真碰过的时候硬装懂，只能先把问题记下来，等以后真的写到这种系统再回来补。
