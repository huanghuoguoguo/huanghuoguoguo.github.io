## 为什么看这本书

在前有《人月神话》和《测试驱动开发》的基础上，我对开源项目或公司内部项目中如何持续迭代、分支管理很感兴趣。

一个经常困扰我的问题：在持续学习中，学到了一种更好的方法，意识到之前的实现可以改进，但代码已经写出去了，修改还会影响别处。这时候怎么评估？怎么避免步子迈得太大？这个问题在 Dooba 的迭代中反复出现——重构比 commit 还勤快，结果步子太大，交付日期只能往后挪。

这本书给出的答案是：不是不能改，而是要有一条自动化的流水线来保证每次改动都是安全的。只要测试全绿、流水线通过，任何改动都可以放心合入。

---

## 这本书讲什么

《持续交付》的核心主张：**软件应该随时处于可发布状态**。

不是说每次提交都要发布，而是说每次提交都*能*发布——构建、测试、部署的整条链路是自动化的、可重复的。当发布变成日常而非战役，团队就从"攒一堆功能赌一把上线"变成了"小步快跑持续验证"。

其本质是将软件交付从手工艺术变为可靠工程：通过工具链与流程标准化，让部署成为日常，最终达成业务与技术的同步响应能力。

全书围绕三大支柱：

| 支柱 | 含义 | 反面 |
|------|------|------|
| 一切自动化 | 构建、测试、部署、环境配置全部脚本化 | 手工操作，"在我机器上能跑" |
| 质量内建 | 自动化测试金字塔、代码审查、持续集成 | 事后补测试、上线前突击测 |
| 快速反馈 | 失败即停、可视化流水线、监控驱动 | 问题攒到上线才发现 |

如何让一切可复现？将一切代码化——schema、脚本、机器配置、部署步骤——全部纳入版本控制。

## 部署流水线

部署流水线是全书的核心概念：代码提交后自动经历一系列阶段，每个阶段都对构建产物增加一层信心。

```
提交 → 编译 + 单元测试 → 集成测试 → 验收测试 → 性能测试 → 预发布 → 生产
```

越靠前的阶段越快（秒-分钟级），越靠后的阶段越慢但越接近真实环境。每个阶段都是一道关卡：通过就继续，失败就停下来修。

回头看 Dooba 的开发过程，问题就出在没有这条流水线。每次改动都是"改完跑跑看能不能用"，没有自动化测试兜底，于是每次重构都心惊胆战——要么不敢改，要么改完一堆回归问题。如果当时有一条哪怕最简单的流水线——编译通过 + 核心用例测试绿——改动的信心会完全不同。

在曙光实习时，内部项目的构建系统相对完善，build system 的熟练度涨了不少。但即便如此，"一个人扛架构、需求、迭代、维护"的模式下，流水线的价值反而被低估了——因为只有一个人的时候，总以为自己记得所有改动的影响范围。实际上不可能。

## 配置管理：一切版本化

书中反复强调：不只是代码要放进 Git。数据库的 schema 迁移、环境变量的配置、服务器的初始化脚本、依赖的版本锁定，都应该是版本化的、可追溯的、可回滚的。

"在我机器上能运行"之所以成为顽疾，本质就是环境没有代码化。开发环境、测试环境、生产环境之间的差异，是大量线上事故的根源。书中提出的解决方案是**基础设施即代码**（Infrastructure as Code）：服务器配置用脚本管理，环境差异通过配置文件区分，每个环境都可以从零快速重建。

这和数据库领域的经典思路相通：数据库通常是先写 WAL 再写数据页——先记录操作日志，再执行更新操作，系统崩溃了可以通过重放日志恢复。基础设施即代码也一样：先以代码记录环境应该是什么样，再用脚本自动搭建出来，搭错了可以推倒重来。

Dooba 在插件管理上做得还不错——任务通过文件系统扫描自动注册，插件用 git clone 远程拉取再动态装载，核心代码不用动。但存储层一开始就耦合了具体实现，切换存储方案时改动范围远超预期。随着对架构设计边界隔离思路的理解加深，我现在意识到，如果一开始将存储层也设计为可插拔、可通过配置切换的模块，这个问题根本不存在。

## 持续集成

持续集成的核心不是 Jenkins 或 GitHub Actions，而是一种纪律：**所有人每天都把代码合入主干，每次合入都触发自动化构建和测试。**

几个关键实践：

- **保持构建快速**：提交阶段（编译 + 单元测试）应该在几分钟内完成。构建太慢，开发者就会攒一堆改动再提交，失去"小步快跑"的意义
- **失败即停**：构建红了，所有人停下来先修，不修好不提交新代码
- **主干开发 vs 分支开发**：书中倾向于主干开发（trunk-based development），配合特性开关控制未完成的功能。长期分支会导致合并地狱

这让我想到之前的一个困惑：开源项目中怎么管理分支？大多数成熟的开源项目也倾向于短生命周期的特性分支 + 快速合入主干，而不是长期维护多个开发分支。xxl-job 的代码仓库就是这种模式。

对于 Dooba 这种一个人开发的项目，分支管理倒不是核心问题，但"持续集成"的纪律仍然适用——每次改动都应该跑测试，而不是攒一堆改动后"重构一把"。之前的问题就是重构太猛，每次都是大范围改动，测试跟不上，交付一推再推。

## 测试策略

书中提出的**测试金字塔**：

```
          /  \
         / E2E \        ← 少量，慢，验证完整流程
        /-------\
       / 集成测试 \      ← 适量，验证组件间交互
      /-----------\
     /   单元测试   \    ← 大量，快，验证单个模块
    /_______________\
```

越往下的测试越多、越快、越稳定；越往上的测试越少、越慢、越脆弱。金字塔倒过来（大量 E2E、很少单元测试），构建就会又慢又不稳定。

如果借鉴架构设计中的"谦卑对象"（Humble Object）模式，测试策略还可以进一步落地：把系统边缘跟外部环境交互的部分（比如网络、数据库、文件系统）隔离为薄封装层，核心计算和业务逻辑用纯内存实现单元测试。这在我们以前的分布式调度框架测试中就有过实践——先把调度策略和任务状态机写成内存版的单元测试，接着用 InProcessTransport 模拟多节点，最后才在真实集群中测试网络超时和分区容错。每一层只聚焦验证自己引入的问题。

TDD 和持续交付的关系也在这里接上了：TDD 保证每次提交都有测试覆盖，持续交付保证每次提交都经过流水线验证。前者是纪律，后者是基础设施，缺一不可。有了 TDD 作地基，只要所有测试用例都是绿的，代码就永远处于可交付的健康状态。

## 发布策略

几种降低发布风险的策略：

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| 蓝绿部署 | 两套环境，新版本部署到备用环境，验证后切流量 | 可承受双倍资源成本 |
| 金丝雀发布 | 少量流量导到新版本，观察指标无异常后逐步扩大 | 用户量大，需要渐进验证 |
| 特性开关 | 新功能用开关控制，发布后可动态开关 | 功能未完成但代码已合入主干 |
| 暗发布 | 新功能已部署但用户不可见，只在内部验证 | 需要在生产环境验证性能 |

特性开关对我来说是一个很有启发的概念。之前 Dooba 的做法是在分支上开发新功能，完成后合入主干——但如果开发周期长，分支和主干的差异越来越大，合并时问题成堆。用特性开关的话，代码始终在主干上，只是新功能被开关挡住，用户看不到。主干始终是最新的、可发布的。

这些策略在分布式系统中更为关键。Redis 用哨兵实现故障转移，Kafka 用 ISR 保证副本一致性——这些本质上都是"发布/切换"问题的不同层面：怎么在不停服的情况下，安全地从旧状态过渡到新状态。

## 回到最初的问题

开头提到的问题："学到了更好的方法，但代码已经写出去了，怎么办？"

这本书给出的框架：

1. **先有流水线**：自动化构建 + 测试，确保每次改动都有反馈
2. **小步重构**：每次只改一小部分，确保测试全绿后再改下一部分
3. **保持可发布**：重构过程中，主干始终是可发布的状态。用特性开关隔离未完成的改动
4. **先补测试再改代码**：没有测试覆盖的代码不要动，先补测试再重构

Dooba 的教训就是反面案例：没有流水线、没有测试覆盖、大范围重构、交付中断。若重新开发，应先交付最小可用功能，建立基本的测试和构建流程，然后在持续交付的保护下逐步演进。

## 与其他几本书的关系

把前几本书串起来看，它们各自解决了不同层面的问题：

| 书 | 解决的问题 |
|----|-----------|
| 《人月神话》 | 团队协作：加人不等于加速，沟通成本指数级上升 |
| 《测试驱动开发》 | 代码质量：先写测试再写实现，保证每次提交都有覆盖 |
| 《持续交付》 | 交付流程：自动化流水线，让软件随时可发布 |
| 《架构整洁之道》 | 系统结构：边界划分、依赖方向，让系统可演进 |

持续交付解决了开发过程中因分支混乱导致的延期问题，使版本发布节奏可控；架构方法则提供了可遵循的设计原则，让项目初期就能建立清晰、可演进的技术框架。两者结合，回答了我一直在问的那个问题：**如果再重来一次，在开始之初应该怎么做？**

答案是：先跑通最小用例，建立流水线，然后在持续交付的保护下，一边交付价值一边演进架构。书的立意比我最初的疑问大多了——我只是想知道"怎么安全地改代码"，它告诉我的是"怎么让整个交付过程变成可靠工程"。
